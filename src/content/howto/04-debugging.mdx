## Step 1: The Debugging Mindset

Debugging is not guessing. It's a **scientific process**:

1. **Observe** — What exactly is happening vs. what should happen?
2. **Hypothesize** — What could cause this difference?
3. **Test** — Check your hypothesis with the smallest possible experiment
4. **Fix** — Make one change at a time
5. **Verify** — Confirm the fix works and doesn't break anything else

The biggest mistake beginners make: changing multiple things at once. If you change three things and the bug disappears, you don't know which change fixed it — and the other two might introduce new bugs.

## Step 2: Console Methods You Should Know

`console.log()` is just the beginning:

```javascript
// Basic logging
console.log("value:", value);

// Structured objects (expandable in DevTools)
console.table(users);

// Grouping related logs
console.group("API Request");
console.log("URL:", url);
console.log("Method:", method);
console.log("Body:", body);
console.groupEnd();

// Timing operations
console.time("fetch-users");
const users = await fetchUsers();
console.timeEnd("fetch-users"); // → fetch-users: 142ms

// Conditional logging
console.assert(users.length > 0, "Users array is empty!");

// Warning and error (styled differently in console)
console.warn("Deprecated: use fetchUsersV2 instead");
console.error("Failed to load user:", error);
```

**Pro tip:** Use template literals for quick context:

```javascript
console.log(`[UserList] users count: ${users.length}, loading: ${isLoading}`);
```

## Step 3: Browser DevTools Breakpoints

Breakpoints let you pause code execution and inspect every variable:

**Setting breakpoints in Chrome DevTools:**

1. Open DevTools (F12 or Cmd+Option+I)
2. Go to the **Sources** tab
3. Find your file in the left panel
4. Click the line number to set a breakpoint
5. Trigger the code (click a button, reload the page)
6. Execution pauses — inspect variables in the **Scope** panel

**Breakpoint controls:**

| Button      | Action                                    |
| ----------- | ----------------------------------------- |
| ▶ Resume    | Continue to next breakpoint               |
| ⤵ Step Over | Execute current line, move to next        |
| ⤓ Step Into | Jump into the function being called       |
| ⤴ Step Out  | Finish current function, return to caller |

**Conditional breakpoints:** Right-click a line number → "Add conditional breakpoint" → enter a condition like `userId === "abc123"`. The code only pauses when the condition is true.

<Checkpoint id="breakpoints" label="Breakpoints" status="active">
  Set a breakpoint in your code and step through it line by line.
</Checkpoint>

## Step 4: VS Code Debugger

VS Code has a built-in debugger that's often easier than browser DevTools:

**For Node.js:**

1. Open the file you want to debug
2. Click the gutter (left of line numbers) to set breakpoints
3. Press F5 or go to Run → Start Debugging
4. Select "Node.js" as the environment

**For Next.js / React:**

Create `.vscode/launch.json`:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Next.js: debug",
      "type": "node",
      "request": "launch",
      "runtimeExecutable": "npm",
      "runtimeArgs": ["run", "dev"],
      "port": 9230
    }
  ]
}
```

**Watch expressions:** In the Debug sidebar, add expressions to monitor (e.g., `users.length`, `isLoading`, `error?.message`). They update live as you step through code.

## Step 5: Network Debugging

Many bugs come from API calls. Use the **Network** tab in DevTools:

1. Open DevTools → Network tab
2. Trigger the action (page load, button click)
3. Click on a request to inspect it

**What to check:**

- **Status code** — 200 is success, 4xx is client error, 5xx is server error
- **Request headers** — Is the auth token being sent?
- **Request body** — Is the data formatted correctly?
- **Response body** — Is the server returning what you expect?
- **Timing** — Is the request slow? Failing silently?

```javascript
// Add error handling to see failed requests
try {
  const response = await fetch("/api/users");
  if (!response.ok) {
    console.error(`API error: ${response.status} ${response.statusText}`);
    const body = await response.text();
    console.error("Response body:", body);
  }
  const data = await response.json();
} catch (error) {
  console.error("Network error:", error);
}
```

## Step 6: Rubber Duck Debugging

When you're truly stuck, explain the problem out loud — to a rubber duck, a pet, or an imaginary colleague.

**Why it works:**

1. **Forces precision** — You can't explain vaguely. You have to state exactly what the code does.
2. **Reveals assumptions** — "Well, this variable should be..." — wait, _is_ it?
3. **Triggers connections** — Speaking activates different brain pathways than reading.

**The process:**

1. State the problem: "This function should return a sorted array, but it returns undefined."
2. Walk through the code line by line: "First, we filter the array... then we sort... oh. `.sort()` returns the array but I'm not returning it from the function."
3. Bug found.

This is not a joke technique. Senior engineers do this regularly.

<Checkpoint id="rubber-duck" label="Rubber Duck Method" status="active">
  Try explaining your next bug out loud before reaching for the debugger.
</Checkpoint>

## Step 7: The Debugging Checklist

When you encounter a bug, run through this checklist:

1. **Read the error message** — All of it, including the stack trace
2. **Reproduce it** — Can you trigger the bug reliably?
3. **Isolate it** — What's the smallest code change that causes/fixes it?
4. **Check your assumptions** — Log the actual values, not what you think they are
5. **Check recent changes** — `git diff` to see what you changed since it last worked
6. **Search for the error** — Paste the exact error message (without file paths) into a search engine
7. **Take a break** — If you've been stuck for 30+ minutes, walk away. The solution often appears when you return.
