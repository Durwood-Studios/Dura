## Step 1: Understand Why Git Exists

Imagine writing a 20-page essay and only having Ctrl+Z to undo mistakes. That's coding without git.

Git gives you:

- **Unlimited undo** — Go back to any previous version of your code
- **Parallel work** — Work on a new feature without breaking what already works
- **Collaboration** — Multiple people can work on the same project without conflicts
- **History** — See who changed what, when, and why

Every professional software project uses git. Learning it now pays dividends forever.

## Step 2: Initialize a Repository

```bash
mkdir my-project
cd my-project
git init
```

This creates a hidden `.git` folder that tracks all changes. You only run `git init` once per project.

**Check the status:**

```bash
git status
```

You'll see "No commits yet" — that's normal for a fresh repo.

## Step 3: The Core Loop — Add, Commit, Push

This is the workflow you'll use hundreds of times:

```bash
# 1. See what changed
git status

# 2. Stage the files you want to commit
git add src/components/Button.tsx

# 3. Commit with a descriptive message
git commit -m "feat(ui): add primary button component"

# 4. Push to the remote repository
git push origin main
```

**Key concepts:**

- **Working directory** — Your actual files
- **Staging area** — Files you've selected for the next commit
- **Repository** — The committed history

Think of it like packing a box (staging) and shipping it (committing).

## Step 4: Write Good Commit Messages

Bad commit messages:

```text
"fix stuff"
"update"
"asdfasdf"
"changes"
```

Good commit messages follow **Conventional Commits**:

```text
feat(auth): add login form with email validation
fix(api): handle null response from user endpoint
docs(readme): add setup instructions for new developers
refactor(db): extract query builder into separate module
```

Format: `type(scope): description`

| Type       | When to use                                       |
| ---------- | ------------------------------------------------- |
| `feat`     | New feature                                       |
| `fix`      | Bug fix                                           |
| `docs`     | Documentation only                                |
| `refactor` | Code change that doesn't add features or fix bugs |
| `test`     | Adding or updating tests                          |
| `chore`    | Tooling, dependencies, config                     |

## Step 5: Branching

Branches let you work on features in isolation:

```bash
# Create and switch to a new branch
git checkout -b feature/user-profile

# Do your work...
git add .
git commit -m "feat(profile): add user avatar upload"

# Switch back to main
git checkout main

# Merge your feature
git merge feature/user-profile
```

**Branch naming conventions:**

- `feature/description` — New features
- `fix/description` — Bug fixes
- `chore/description` — Maintenance tasks

## Step 6: Handling Merge Conflicts

Conflicts happen when two branches change the same line. Git marks them like this:

```text
<<<<<<< HEAD
const color = "blue";
=======
const color = "red";
>>>>>>> feature/theme
```

**To resolve:**

1. Open the file in VS Code (it highlights conflicts)
2. Choose which version to keep (or combine them)
3. Remove the `<<<<<<<`, `=======`, and `>>>>>>>` markers
4. Stage and commit:

```bash
git add src/theme.ts
git commit -m "fix: resolve color conflict, keep theme value"
```

**Pro tip:** VS Code shows "Accept Current", "Accept Incoming", and "Accept Both" buttons above each conflict.

## Step 7: Connect to GitHub

```bash
# Add a remote repository
git remote add origin git@github.com:your-username/my-project.git

# Push your main branch
git push -u origin main
```

After the first push with `-u`, you can just use `git push` for subsequent pushes.

**Creating a Pull Request (PR):**

1. Push your feature branch: `git push -u origin feature/user-profile`
2. Go to your repo on GitHub
3. Click "Compare & pull request"
4. Add a description of your changes
5. Request a review from teammates

## Step 8: Essential Git Commands Cheat Sheet

| Command                     | What it does                         |
| --------------------------- | ------------------------------------ |
| `git status`                | Show changed files                   |
| `git log --oneline`         | Show commit history (compact)        |
| `git diff`                  | Show unstaged changes                |
| `git stash`                 | Temporarily save uncommitted changes |
| `git stash pop`             | Restore stashed changes              |
| `git reset HEAD~1`          | Undo the last commit (keep changes)  |
| `git blame file.ts`         | See who changed each line            |
| `git log --oneline --graph` | Visual branch history                |
