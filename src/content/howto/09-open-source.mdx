## Step 1: Why Contribute to Open Source?

Open source isn't just altruism — it's one of the most effective career accelerators in tech:

- **Real-world experience** — Work on codebases larger than anything you'd build alone
- **Portfolio proof** — Merged PRs demonstrate ability better than personal projects
- **Network** — Maintainers and contributors become professional connections
- **Learning** — Code review from experienced developers is free mentorship
- **Job opportunities** — Many companies actively recruit from their contributor community

You don't need to be an expert to contribute. Every project needs documentation fixes, test coverage, accessibility improvements, and bug reports.

## Step 2: Find Your First Project

**Look for projects that are:**

- Actively maintained (commits in the last month)
- Welcoming to newcomers (look for CONTRIBUTING.md)
- Using technology you know or want to learn
- Something you actually use — motivation matters

**Where to find beginner-friendly issues:**

- GitHub labels: `good first issue`, `help wanted`, `beginner-friendly`
- [goodfirstissue.dev](https://goodfirstissue.dev) — Curated list of beginner issues
- [up-for-grabs.net](https://up-for-grabs.net) — Projects that tag easy issues
- Projects you use daily — you understand the context already

**Evaluate before committing:**

| Signal        | Good sign                         | Red flag                       |
| ------------- | --------------------------------- | ------------------------------ |
| Response time | PRs reviewed within a week        | PRs sitting for months         |
| Tone          | Welcoming, constructive feedback  | Dismissive or hostile comments |
| Documentation | Clear CONTRIBUTING.md             | No contribution guide          |
| Issue labels  | Organized, labeled for difficulty | Hundreds of unlabeled issues   |

<Checkpoint id="find-project" label="Find a Project" status="active">
  Browse GitHub and bookmark 2-3 projects you'd like to contribute to.
</Checkpoint>

## Step 3: Set Up the Project Locally

Every contribution starts with getting the project running on your machine:

```bash
# 1. Fork the repository (click "Fork" on GitHub)

# 2. Clone your fork
git clone git@github.com:YOUR-USERNAME/project-name.git
cd project-name

# 3. Add the original repo as "upstream"
git remote add upstream git@github.com:ORIGINAL-OWNER/project-name.git

# 4. Install dependencies
npm install  # or yarn, pnpm — check the README

# 5. Run the project
npm run dev  # or whatever the README says

# 6. Run tests to make sure everything works
npm test
```

**Read these files first:**

- `README.md` — Project overview and setup instructions
- `CONTRIBUTING.md` — Contribution guidelines and workflow
- `.github/PULL_REQUEST_TEMPLATE.md` — What to include in your PR
- `CODE_OF_CONDUCT.md` — Community behavior standards

## Step 4: Make Your Contribution

**The contribution workflow:**

```bash
# 1. Create a branch for your change
git checkout -b fix/typo-in-readme

# 2. Make your changes
# (edit files, run tests, verify it works)

# 3. Commit with a clear message
git commit -m "fix(docs): correct typo in installation instructions"

# 4. Push to your fork
git push origin fix/typo-in-readme

# 5. Open a Pull Request on GitHub
```

**Good first contributions:**

- Fix a typo in documentation
- Add a missing test case
- Improve an error message
- Fix a bug tagged as `good first issue`
- Add TypeScript types to an untyped function
- Improve accessibility (add alt text, aria labels)

**Contribution quality checklist:**

- [ ] Follows the project's code style
- [ ] Tests pass (existing and any new ones you added)
- [ ] Lint checks pass
- [ ] Commit message follows the project's convention
- [ ] PR description explains what and why

<Checkpoint id="make-contribution" label="Make a Contribution" status="active">
  Start small. A merged typo fix is infinitely more valuable than an unfinished feature PR.
</Checkpoint>

## Step 5: Write a Great Pull Request

Your PR description should make the reviewer's job easy:

```markdown
## What

Fix the installation command in README.md — the npm package name
was incorrect (missing the @scope prefix).

## Why

New users following the getting started guide get a "package not found"
error because `npm install project` should be `npm install @org/project`.

## How

Updated the installation command in README.md line 42.

## Testing

- Verified the correct package name on npmjs.com
- Followed the updated instructions from a clean directory
```

**PR etiquette:**

- Keep PRs focused — one logical change per PR
- Respond to review comments promptly
- Don't take feedback personally — it's about the code, not you
- If a maintainer requests changes, make them and push to the same branch
- Say thank you when your PR gets merged

<Checkpoint id="great-pr" label="Writing PRs" status="active">
  A well-written PR description significantly increases your chances of getting merged.
</Checkpoint>

## Step 6: Handle Code Review

Code review is where you learn the most. Here's how to handle it:

**When you receive feedback:**

```markdown
// Reviewer says: "Could you extract this into a helper function?"

// Good response:
"Great idea! I've extracted it into `formatUserName()` in utils.ts.
Let me know if the naming works for you."

// Bad response:
"But it works fine as is."
```

**Common review requests and what they mean:**

| Request                     | What they want                               |
| --------------------------- | -------------------------------------------- |
| "Could you add a test?"     | Verify your change works and doesn't regress |
| "Nit: naming"               | Rename for clarity (not a blocker)           |
| "What about edge case X?"   | Handle null, empty, or unusual inputs        |
| "Could you simplify this?"  | The logic is correct but hard to follow      |
| "Let's discuss in an issue" | The change is bigger than expected           |

**If you disagree with feedback:**

- Explain your reasoning respectfully
- Ask for their perspective: "What would you suggest instead?"
- If the maintainer insists, defer to their judgment — it's their project

<Checkpoint id="code-review" label="Code Review" status="active">
  Code review is mentorship, not criticism. Embrace it.
</Checkpoint>

## Step 7: Build a Contribution Habit

**Weekly cadence:**

- Monday: Check your bookmarked projects for new `good first issue` labels
- During the week: Work on one small contribution
- Weekend: Review open PRs in projects you've contributed to

**Level up over time:**

1. **Month 1-2:** Documentation fixes, typo corrections
2. **Month 3-4:** Bug fixes tagged as beginner-friendly
3. **Month 5-6:** Small feature additions
4. **Month 7+:** Regular contributor — help triage issues, review others' PRs

**Track your contributions:**

- Pin your best PRs on your GitHub profile
- Mention specific contributions in your resume
- Write a blog post about what you learned from a contribution

<Checkpoint id="habit" label="Contribution Habit" status="active">
  Consistency compounds. One small contribution per week transforms your career.
</Checkpoint>
