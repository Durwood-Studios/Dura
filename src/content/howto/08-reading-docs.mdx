## Step 1: Why Documentation Matters

The fastest developers aren't the ones who memorize everything — they're the ones who can **find answers quickly**. Documentation is your superpower.

**Types of documentation you'll encounter:**

| Type              | Example                         | When to use                |
| ----------------- | ------------------------------- | -------------------------- |
| **Tutorials**     | "Getting Started with React"    | Learning something new     |
| **How-to guides** | "How to add authentication"     | Solving a specific problem |
| **API Reference** | Function signatures, parameters | Looking up exact syntax    |
| **Conceptual**    | "How React rendering works"     | Understanding the "why"    |

Most developers waste time reading tutorials when they need a reference, or reading references when they need a conceptual explanation. Match the doc type to your need.

## Step 2: The Search-First Strategy

Never read documentation linearly like a book. Use this approach:

1. **Search for your specific question** — Use the docs search bar or Ctrl+F
2. **Scan headings** — Look for the section most relevant to your problem
3. **Read the code examples first** — They communicate faster than paragraphs
4. **Read the surrounding text** if the code example doesn't fully answer your question

**Where to search:**

- **Official docs** first (react.dev, nextjs.org/docs, developer.mozilla.org)
- **GitHub README** for library-specific usage
- **TypeScript types** — hover over functions in VS Code to see their signatures
- **Source code** — when docs are outdated, the code is always right

<Checkpoint id="search-first" label="Search-First Strategy" status="active">
  Train yourself to search before scrolling.
</Checkpoint>

## Step 3: Reading Code Examples

Code examples are the most valuable part of any documentation. Here's how to extract maximum value:

**Step-by-step approach:**

1. **Read the entire example** before trying to understand each line
2. **Identify the imports** — What packages/modules are being used?
3. **Find the core pattern** — What's the essential 3-5 lines?
4. **Note the configuration** — What options are being set and why?
5. **Look for comments** — Docs authors put important caveats in comments

**Example — reading Next.js docs:**

```typescript
// The docs show this Server Action pattern:
"use server";

export async function createPost(formData: FormData) {
  const title = formData.get("title") as string;
  // ... save to database
  revalidatePath("/posts");
}
```

**What to extract:**

- `"use server"` directive is required at the top
- Function receives `FormData` (not a plain object)
- `revalidatePath` refreshes the page data after mutation
- This runs on the server even though it's called from a form

## Step 4: MDN — Your JavaScript Bible

[MDN Web Docs](https://developer.mozilla.org) is the definitive reference for JavaScript, HTML, CSS, and Web APIs.

**How to use MDN effectively:**

Every MDN page for a method follows this structure:

1. **Syntax** — The exact signature with parameter types
2. **Parameters** — What each argument does
3. **Return value** — What the method gives back
4. **Description** — Detailed explanation
5. **Examples** — Working code snippets
6. **Browser compatibility** — Which browsers support it

**Example: Looking up `Array.prototype.reduce`**

```javascript
// Syntax section tells you:
array.reduce(callbackFn, initialValue);

// Parameters section explains:
// callbackFn(accumulator, currentValue, currentIndex, array)
// initialValue — optional, but recommended

// Example section shows:
const sum = [1, 2, 3].reduce((acc, val) => acc + val, 0);
// → 6
```

**Pro tip:** Search "mdn array reduce" instead of navigating the site manually.

</Checkpoint>

## Step 5: Reading TypeScript Types as Documentation

TypeScript types are self-documenting. Learn to read them:

```typescript
// This tells you everything about the function:
function createElement(
  type: string | ComponentType,
  props?: Record<string, unknown> | null,
  ...children: ReactNode[]
): ReactElement;
```

**What you learn from the types alone:**

- First argument: a string (HTML tag) or a component
- Second argument: props object (optional, can be null)
- Remaining arguments: children (any number of React nodes)
- Returns: a ReactElement

**In VS Code:** Hover over any function to see its type signature. Press F12 to jump to the type definition. Press Ctrl+Space for autocompletion with type information.

```typescript
// Hover over fetch() to see:
function fetch(input: RequestInfo | URL, init?: RequestInit): Promise<Response>;

// Now you know: it takes a URL and optional config, returns a Promise
```

## Step 6: Build Your Reference Library

Create a personal system for documentation you use often:

**Bookmarks to organize:**

- **Daily tools** — VS Code shortcuts, git commands, npm scripts
- **Framework docs** — React, Next.js, Tailwind (the pages you visit most)
- **Language reference** — MDN for JS, TypeScript handbook
- **Package docs** — Libraries you use in your current project

**Create snippets for patterns you look up repeatedly:**

```json
// VS Code snippet: .vscode/snippets.code-snippets
{
  "React Component": {
    "prefix": "rfc",
    "body": [
      "interface ${1:Component}Props {",
      "  $2",
      "}",
      "",
      "export function ${1:Component}({ $3 }: ${1:Component}Props) {",
      "  return <div>$0</div>;",
      "}"
    ]
  }
}
```

**When docs are wrong or outdated:**

1. Check the GitHub issues for the project
2. Look at the source code
3. Check the changelog for recent changes
4. Search for recent blog posts or conference talks

  Build a personal reference system. Future-you will thank present-you.
</Checkpoint>
