Browser DevTools are the most powerful debugging tool you already have installed. Most developers use maybe 10% of what they offer. This guide covers the panels and techniques that senior engineers use daily to debug, profile, and understand web applications.

Open DevTools with `Cmd+Option+I` (Mac) or `Ctrl+Shift+I` (Windows/Linux). Right-click any element and select "Inspect" to jump directly to it.

## Step 1: Elements Panel — Inspect and Edit the DOM

The Elements panel shows the live DOM tree and its styles. This is where you debug layout issues, test CSS changes, and understand why something looks wrong.

**Edit anything live:**

- Double-click any text in the DOM to edit it
- Double-click an attribute to change it
- Right-click an element and choose "Edit as HTML" for bigger changes
- Changes are temporary — refreshing the page resets everything

**Debug CSS:**

- Select an element and look at the Styles pane on the right
- Styles are listed in specificity order — the winning rule is on top
- Strikethrough styles are overridden by a more specific rule
- Click any value to edit it live. Use arrow keys to increment numbers.
- Toggle any property on/off with the checkbox next to it

**Find layout problems:**

- Hover over elements to see their box model (margin, border, padding, content) highlighted on the page
- Click the box model diagram in the Computed tab to see exact values
- For flexbox containers, look for the flex badge on the element — click it to see the flex overlay
- For grid containers, click the grid badge to see grid lines and track sizes

**Useful tricks:**

- `Cmd+Shift+C` activates element picker — click anything on the page to inspect it
- In the Console, `$0` refers to the currently selected element in the Elements panel
- Right-click an element and "Scroll into view" to find it on the page
- "Force state" lets you toggle `:hover`, `:focus`, `:active` without actually interacting

## Step 2: Console Panel — Debug with Code

The Console is a JavaScript REPL connected to your page. It is far more powerful than `console.log`.

**Beyond console.log:**

```javascript
// Table view for arrays and objects
console.table([
  { name: "Alice", age: 30 },
  { name: "Bob", age: 25 },
]);

// Group related logs
console.group("User data");
console.log("Name:", user.name);
console.log("Role:", user.role);
console.groupEnd();

// Measure execution time
console.time("fetch");
await fetch("/api/users");
console.timeEnd("fetch"); // fetch: 142ms

// Conditional logging
console.assert(user.role === "admin", "User is not an admin!", user);

// Count how many times something happens
console.count("render"); // render: 1, render: 2, render: 3...
```

**Query the page like a pro:**

```javascript
// Same as document.querySelector
$("nav a.active");

// Same as document.querySelectorAll (returns a real array)
$$("img:not([alt])"); // Find all images missing alt text

// Monitor function calls
monitor(fetchUserData); // Logs every time it's called with arguments

// Monitor events on an element
monitorEvents($0, "click"); // Logs every click on the selected element
```

**Debug React/Next.js apps:**

Install the React Developer Tools extension. It adds Components and Profiler tabs that show your component tree, props, state, and hooks. You can edit state directly to test different scenarios.

## Step 3: Network Panel — Understand Every Request

The Network panel shows every HTTP request your page makes. This is where you debug API calls, find slow requests, and understand data flow.

**Read the waterfall:**

Each row is a request. The waterfall chart on the right shows timing:

- **DNS Lookup** — resolving the domain name
- **Connection** — TCP handshake
- **SSL** — TLS negotiation
- **Waiting (TTFB)** — time to first byte from the server
- **Content Download** — receiving the response body

If TTFB is high, the problem is server-side. If Content Download is high, the response is too large.

**Filter and search:**

- Filter by type: JS, CSS, Img, Font, Fetch/XHR, Doc
- The search bar (`Cmd+F`) searches inside request/response bodies
- Check "Preserve log" to keep requests across page navigations
- Check "Disable cache" to test without browser caching

**Inspect a request:**

Click any request to see:

- **Headers** — request and response headers, status code
- **Payload** — the request body (for POST/PUT requests)
- **Preview** — a formatted view of the response
- **Response** — the raw response text
- **Timing** — detailed breakdown of the request lifecycle

**Simulate conditions:**

- Throttle the network to "Slow 3G" or "Fast 3G" to test on slow connections
- Block specific requests by right-clicking and choosing "Block request URL"
- Right-click a request and "Copy as cURL" to replay it in the terminal

## Step 4: Performance Panel — Find What Makes Your Page Slow

The Performance panel records a timeline of everything the browser does — JavaScript execution, layout, paint, compositing.

**Record a performance trace:**

1. Click the record button (or `Cmd+Shift+E` to record a page reload)
2. Interact with the page — scroll, click, navigate
3. Stop recording

**Read the results:**

- **Summary pie chart** — Shows time split between Scripting, Rendering, Painting, and Idle
- **Main thread flame chart** — Each bar is a function call. Width is duration. Tall stacks are deep call chains.
- **Long tasks** — Red-flagged tasks that block the main thread for over 50ms

**Common problems and what to look for:**

- **Layout thrashing** — Purple bars labeled "Layout" happening repeatedly. Caused by reading and writing DOM properties in a loop.
- **Forced reflows** — Yellow warning triangles on layout events. Something triggered layout recalculation unnecessarily.
- **Expensive re-renders** — Look for React component render functions that take too long. The React Profiler is better for this.

**Quick performance check without the Performance panel:**

Open the Console and run:

```javascript
// Check Core Web Vitals
new PerformanceObserver((list) => {
  list.getEntries().forEach(console.log);
}).observe({ type: "largest-contentful-paint", buffered: true });

// Check how long scripts take
performance
  .getEntriesByType("resource")
  .filter((r) => r.initiatorType === "script")
  .sort((a, b) => b.duration - a.duration)
  .slice(0, 5)
  .forEach((r) => console.log(r.name, Math.round(r.duration) + "ms"));
```

## Step 5: Application Panel — Inspect Storage and State

The Application panel (called "Storage" in some browsers) shows everything your app stores on the client.

**Local Storage and Session Storage:**

- View all key-value pairs
- Click any value to edit it
- Right-click to delete entries
- Useful for testing how your app handles missing or corrupted stored data

**IndexedDB:**

- Browse databases, object stores, and indexes
- View individual records
- Useful for debugging offline-first apps that cache data locally

**Cookies:**

- View all cookies for the current domain
- See attributes: path, expiry, HttpOnly, Secure, SameSite
- Edit or delete cookies to test authentication and session handling

**Cache Storage:**

- Inspect Service Worker caches
- See what assets are cached and their sizes
- Delete specific cached responses to test cache invalidation

**Service Workers:**

- See registered service workers
- Force update a service worker
- Simulate offline mode with the "Offline" checkbox (also available in Network panel)
- Check "Update on reload" during development

**A debugging workflow using multiple panels:**

1. Something is broken on the page. Start in **Elements** — is the DOM what you expect?
2. Check the **Console** — are there any errors?
3. Check the **Network** panel — did the API call succeed? What data did it return?
4. Check **Application** — is stored state correct?
5. If the issue is performance, record a **Performance** trace.

DevTools is a diagnostic tool. The more comfortable you are with it, the less time you spend guessing and the more time you spend fixing. Keep it open while you develop. Not just when something breaks.
