## The concurrency problem

At any given moment a production database might be handling thousands of concurrent transactions. Two users update the same account balance. A report reads millions of rows while new orders are being inserted. Without coordination, these concurrent operations corrupt data in subtle, hard-to-detect ways. Concurrency control is the set of mechanisms databases use to make concurrent transactions behave as if they ran one at a time.

The naive solution — one global lock that serializes all transactions — is correct but catastrophically slow. Modern databases use far more sophisticated approaches that allow high concurrency while maintaining correctness guarantees.

## MVCC: the key insight

**Multi-version concurrency control** (MVCC) is the dominant concurrency control strategy in systems like PostgreSQL, Oracle, and MySQL InnoDB. The core insight is simple: instead of locking rows so readers and writers block each other, the database keeps multiple **versions** of each row simultaneously.

When a transaction updates a row, it does not overwrite the old version. Instead, it creates a new row version stamped with the current transaction ID, and marks the old version as superseded. When a transaction reads a row, it sees the most recent version that was committed _before the reading transaction started_ — not necessarily the latest version.

This design achieves **readers never block writers, and writers never block readers**. A long-running report can scan a table for minutes while ongoing transactions insert and update rows freely. The report sees a consistent snapshot of the database as of its start time; newer row versions are simply invisible to it.

**The tradeoff:** Old row versions accumulate until they are no longer needed by any active transaction. PostgreSQL's VACUUM process periodically reclaims this "dead tuple" space. If VACUUM cannot keep up with your write rate, the table bloats. Very long-running transactions are particularly harmful — they prevent VACUUM from cleaning up any version newer than the transaction's start time.

## Transaction IDs and visibility rules

PostgreSQL assigns every transaction a monotonically increasing **transaction ID** (XID). Each row version is stamped with the XID of the transaction that created it (`xmin`) and, if deleted or superseded, the XID that removed it (`xmax`).

Visibility rule: a row version is visible to transaction T if:

- `xmin` was committed before T started, AND
- `xmax` is either not set, or committed after T started

This simple rule, applied to every row version, produces snapshot isolation: each transaction sees a consistent point-in-time view with no lock contention between readers and writers.

## Pessimistic vs. optimistic locking

**Pessimistic locking** assumes conflicts are frequent. Transactions acquire locks on resources before accessing them, blocking other transactions that want the same resources. This is safe but reduces concurrency — a slow transaction holding many locks starves other transactions.

PostgreSQL's `SELECT ... FOR UPDATE` is pessimistic locking: it acquires a row-level lock immediately, preventing other transactions from updating those rows until the current transaction commits or rolls back.

**Optimistic locking** assumes conflicts are rare. Transactions proceed without acquiring locks, reading and modifying data freely. At commit time, the transaction verifies that none of the data it read has changed. If a conflict is detected, the transaction is aborted and retried. This works well when conflicts genuinely are rare (many readers, few writers on the same rows). When conflicts are frequent, the retry overhead makes it worse than pessimistic locking.

A common application-level optimistic locking pattern: add a `version` integer column. When reading a row, remember its version. Before updating, check that the version matches. If another transaction has incremented it, abort.

## Isolation levels

SQL defines four isolation levels that trade correctness guarantees for concurrency:

**Read Uncommitted:** Transactions see uncommitted changes from other transactions ("dirty reads"). Rarely used; usually forbidden.

**Read Committed:** Transactions see only committed data, but may see different data if they read the same row twice within a transaction (non-repeatable read). PostgreSQL's default.

**Repeatable Read:** The same query returns the same data no matter how many times you run it within a transaction (no non-repeatable reads). New rows from concurrent inserts may still appear (phantom reads). PostgreSQL implements this with MVCC snapshot isolation.

**Serializable:** Full isolation — concurrent transactions produce exactly the results that some serial execution would produce. PostgreSQL implements this with Serializable Snapshot Isolation (SSI), detecting dangerous read-write cycles at commit time. Performance penalty is usually acceptable for the safety it provides.

The right isolation level depends on your application. Most OLTP applications work correctly at Read Committed. Financial applications handling account transfers often need Repeatable Read or Serializable.

## What you learned

- MVCC stores multiple versions of each row, letting readers see a consistent snapshot without blocking writers
- Each row version is stamped with the creating transaction's ID (xmin) and the superseding transaction's ID (xmax)
- Visibility rules determine which version a transaction sees based on its start-time snapshot
- Pessimistic locking acquires locks before access; optimistic locking detects conflicts at commit time
- Isolation levels trade correctness guarantees for concurrency — Read Committed is the default; Serializable is the safest
- Long-running transactions prevent MVCC cleanup (VACUUM), causing table bloat
