## When one machine is not enough

Replication copies data to multiple machines, which helps with fault tolerance and read scalability. But what happens when a single table grows so large that one machine cannot store it? Or when write throughput exceeds what a single leader can handle? This is the problem **partitioning** (also called **sharding**) solves.

Partitioning splits a dataset into smaller subsets called **partitions** or **shards**, each stored on a different node. Where replication gives you copies of everything, partitioning gives you fractions of everything. A table with one billion rows might be split across 10 nodes, each storing 100 million rows. Writes and reads are distributed across nodes, scaling both storage and throughput horizontally.

The challenge is deciding which rows go to which partition. This is the **partition key** decision, and it is one of the most consequential architectural choices you'll make when designing a distributed data system. A bad partition key can turn a well-resourced distributed system into a bottleneck worse than a single machine.

## Range partitioning

**Range partitioning** assigns rows to partitions based on a range of values of the partition key. The key space is divided into contiguous ranges, and each range maps to a specific shard.

For example, a table of sales records partitioned by `sale_date`:

- Shard 1: 2020-01-01 to 2020-12-31
- Shard 2: 2021-01-01 to 2021-12-31
- Shard 3: 2022-01-01 to 2022-12-31

**Advantage:** Range queries are efficient. `WHERE sale_date BETWEEN '2021-01-01' AND '2021-06-30'` touches only Shard 2. The database can prune all other shards without scanning them.

**Disadvantage: hotspots.** If all new writes have today's date, 100% of writes go to the current shard while all other shards sit idle. This is the definition of a **hotspot** — one partition receiving disproportionate load while others are underutilized. A hotspot eliminates the scaling benefit of partitioning. Cassandra's time-series data frequently suffers from this: the "current time" partition is always hot.

The fix is to add entropy to the partition key. Instead of partitioning on `user_id` alone, some systems hash `user_id` first, then range-partition on the hash. Or they prefix the partition key with a random value (sharding key = `random_prefix + timestamp`), distributing writes across many shards while keeping logical ordering within each prefix group.

## Hash partitioning

**Hash partitioning** computes a hash of the partition key and assigns the row to `hash(key) % N` where N is the number of shards. This distributes writes and reads evenly across all shards — no hotspots for typical random-key workloads.

**Advantage:** Even distribution of load. A user table partitioned by `hash(user_id) % 8` will distribute 8 million users approximately uniformly across 8 shards.

**Disadvantage:** Range queries fan out across all shards. `WHERE user_id BETWEEN 1000 AND 2000` cannot be answered by a single shard — the user IDs in that range are scattered across all shards by the hash function. The query must be sent to all shards and the results merged. This is called a **scatter-gather** query and is expensive.

**Consistent hashing** is a refinement used by systems like DynamoDB and Cassandra. Instead of `hash(key) % N`, consistent hashing maps keys and nodes onto a ring. Adding or removing a node only requires redistributing the keys adjacent to that node on the ring, rather than rehashing all keys. This makes scaling the cluster (adding/removing shards) far less disruptive.

## Composite partitioning

Many production systems use **composite partitioning** — partitioning on multiple keys in a hierarchy. Cassandra uses this model explicitly: the first part of the key is the **partition key** (hash-partitioned across nodes), and additional columns are the **clustering key** (sorted within each partition). This gives you even distribution across nodes while preserving sort order within each node — the best of both worlds for specific query patterns.

## Cross-partition queries and the scatter-gather problem

The fundamental challenge of sharding is that queries not aligned with the partition key must touch all shards. In a system with 100 shards, a scatter-gather query sends the query to all 100, waits for all 100 to respond, and merges the results. Latency is determined by the slowest shard. One slow node (a "stragglers" problem) delays every cross-partition query.

This is why **partition key selection drives schema design** in distributed databases. Your most common queries must align with the partition key. If they don't, you're paying for distributed infrastructure while getting single-machine performance.

## What you learned

- Partitioning splits a dataset across multiple nodes, scaling both storage capacity and throughput
- Range partitioning assigns contiguous key ranges to shards — efficient for range queries but prone to hotspots with sequential keys
- Hash partitioning uses a hash function to distribute writes evenly but makes range queries expensive (scatter-gather)
- Hotspots occur when one partition receives disproportionate load — they destroy the benefit of sharding
- Composite partitioning (partition key + clustering key) combines even distribution with sorted order within each shard
- Partition key selection is a schema-level architectural decision that must match your most common query patterns
