## The cost of looking things up

Imagine a table with ten million rows and no index. To find the row where `user_id = 42`, the database must read every single row from disk and check the condition — a **sequential scan** that touches potentially gigabytes of data. An index is a separate data structure that trades some write overhead and storage space for dramatically faster lookups. Understanding how different index types work internally tells you which index to create for each situation.

## B+ trees: the universal index structure

The **B+ tree** is the data structure underlying nearly every index in every major relational database. It is a variant of the B-tree with one critical difference: all actual data values live in the **leaf nodes**, while internal nodes store only keys used for navigation.

This design has a profound consequence for range scans. In a standard B-tree, values can be at any level of the tree, so a range scan must traverse up and down the tree to collect all matching values. In a B+ tree, all leaf nodes are linked in a doubly-linked list. Once the search finds the starting leaf for a range query, it can follow the leaf links horizontally to collect all matching rows without ever going back up the tree. This makes range scans — `WHERE price BETWEEN 100 AND 200`, `WHERE created_at > '2024-01-01'` — extremely efficient.

**B+ tree insertion** maintains sorted order. Inserting a key finds the correct leaf, inserts it in sorted position, and splits the leaf if it overflows. The split promotes the middle key to the parent, which may itself split, cascading toward the root. In practice, splits are infrequent and the tree stays balanced automatically.

**Lookup cost** is O(log_f n), where f is the branching factor (number of children per node, typically in the hundreds) and n is the number of indexed rows. For ten million rows with a branching factor of 200, the tree is only about three levels deep — meaning at most three disk reads to find any row.

## Hash indexes: O(1) lookups with a catch

A **hash index** computes a hash of the indexed column value and uses it as a bucket address in a hash table. Lookups are O(1) — no tree traversal, just hash the value and go to the bucket. This sounds strictly better than a B+ tree's O(log n), but hash indexes have a significant limitation: they only support equality comparisons.

`WHERE user_id = 42` — hash index works perfectly.
`WHERE user_id > 42` — hash index is useless. A hash table has no concept of ordering.

In PostgreSQL, hash indexes were historically considered unreliable (they were not WAL-logged until version 10) and are rarely used in practice. For equality-heavy workloads on high-cardinality columns, a B+ tree index on a single column already performs extremely well, so the benefits of a hash index rarely justify the tradeoff.

## Composite indexes: column order is everything

A **composite index** covers multiple columns. The index `(last_name, first_name)` can answer queries that filter on `last_name` alone, or on both `last_name` and `first_name`. But it cannot efficiently answer queries that filter on `first_name` alone.

This is because a composite index sorts rows first by the first column, then by the second within each first-column group, and so on. Think of a phone directory sorted by last name then first name. You can flip to the "S" section and find all Smiths quickly. Within the Smiths, you can find "Smith, Alice" quickly. But if you want all people named "Alice" regardless of last name, you're stuck scanning the whole directory.

The **selectivity rule** says: place the most selective column (the one that eliminates the most rows) first — unless your query patterns dictate otherwise. If every query filters on `status` and sometimes also on `created_at`, the index `(status, created_at)` serves both use cases; `(created_at, status)` serves only queries that filter on `created_at`.

## Covering indexes: eliminating the table fetch

An **index scan** in most databases involves two steps: find the matching row's location in the index, then fetch the full row from the table (a "heap fetch"). Each heap fetch is potentially a random disk read. For a query returning many rows, these random reads can be slower than a sequential scan of the entire table.

A **covering index** includes all the columns the query needs — not just the filter columns but also the select-list columns. When every needed column is in the index, the database never touches the table at all. PostgreSQL calls this an "Index Only Scan."

```sql
-- A query that needs: filter on user_id, return title and created_at
SELECT title, created_at FROM posts WHERE user_id = 42;

-- A covering index includes all three columns
CREATE INDEX idx_posts_user_covering ON posts (user_id) INCLUDE (title, created_at);
-- PostgreSQL will use an Index Only Scan — zero table heap reads
```

The `INCLUDE` clause (supported by PostgreSQL 11+) adds columns to the leaf level of the B+ tree without affecting the sort order. This keeps the index compact while eliminating heap fetches for common queries.

## When indexes hurt

Indexes are not free. Every `INSERT`, `UPDATE`, and `DELETE` must update all relevant indexes in addition to the table itself. A table with 10 indexes incurs 10x the index maintenance overhead on writes. Indexes also consume disk space and buffer cache, potentially evicting hot table data.

The right number of indexes depends on your read/write ratio. For a reporting database that ingests data once and queries it constantly, many indexes are appropriate. For a write-heavy event log, keep indexes minimal.

## What you learned

- B+ trees store all values in leaf nodes linked in a sorted list, making range scans highly efficient
- Hash indexes support only equality comparisons but are O(1) — useful for exact-match lookups at very high throughput
- Composite index column order determines which query shapes the index can serve — leading columns must appear in filters
- Covering indexes include all columns needed by a query, enabling Index Only Scans that never touch the main table
- Indexes trade write performance and storage for read speed; the right index count depends on your read/write ratio
