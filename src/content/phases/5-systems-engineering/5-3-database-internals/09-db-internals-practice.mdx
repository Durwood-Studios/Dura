## Connecting theory to practice

You have learned how storage engines work, how indexes are structured, how the optimizer chooses plans, how MVCC handles concurrency, and how replication and partitioning scale databases beyond a single machine. Now you put it together. Real database performance work is a cycle of measurement, explanation, and change.

The most valuable tool in that cycle is `EXPLAIN ANALYZE` (in PostgreSQL, MySQL's `EXPLAIN FORMAT=JSON`, or equivalent in any production database). It bridges theory and reality: the optimizer's cost model tells you what it _planned_ to do; the actual execution statistics tell you what actually happened and where time was spent.

## Reading an EXPLAIN plan

An EXPLAIN plan is a tree. Each node is an operator. Leaf nodes are data sources (table scans, index scans). Internal nodes are operations (joins, sorts, aggregates). The root node produces the final result. Operators execute bottom-up: leaves produce rows that are consumed by their parents.

For each node, you see:

- **Operation type** (Seq Scan, Index Scan, Hash Join, etc.)
- **Estimated cost** (startup cost to first row, total cost to last row)
- **Estimated rows** — how many rows the optimizer thinks this operator will produce
- **Actual time** (in EXPLAIN ANALYZE) — actual milliseconds to first and last row
- **Actual rows** — how many rows it actually produced

**The most important diagnostic:** compare estimated rows to actual rows. When these are far apart (orders of magnitude), the optimizer was working with wrong statistics and made a bad plan choice. Running `ANALYZE` on the table updates the statistics the optimizer uses and often fixes the plan.

**Identifying the slow node:** total cost flows through the tree. Look for operators with actual time that is disproportionately high compared to their cost estimate. A `Seq Scan` on a large table when an index exists means the index is missing or the query isn't using it. A `Sort` operator touching millions of rows suggests adding an index to avoid the sort. Nested loops over large sets suggest the planner should use a hash join instead.

## Schema design principles

Good schema design is not just normalization theory — it is the intersection of data integrity requirements and access patterns. A schema that is correct but ignores access patterns will perform poorly. A schema optimized purely for performance may become unmaintainable.

**Normalization** eliminates data redundancy by storing each fact once. A user's address stored in the `addresses` table and referenced by foreign key in the `users` table is normalized. This ensures consistency — update the address once and it's updated everywhere. The cost: reading user-with-address requires a join.

**Denormalization** intentionally duplicates data to avoid joins on the hot path. An e-commerce site might store the product name on each order line item even though it's already in the products table. This violates third normal form but means displaying order history never requires joining to products — faster for the read-heavy order history page.

**The right answer** depends on your read/write ratio and consistency requirements. Normalize first. Denormalize only where you have measured a join performance problem and can afford the consistency complexity.

**Indexing principles from this module:**

- Index columns that appear in WHERE clauses, ORDER BY clauses, and JOIN conditions
- Put the most selective column first in composite indexes — unless all queries filter on a specific leading column
- Use covering indexes (INCLUDE) to eliminate heap fetches for high-frequency queries
- Audit indexes after schema changes — indexes can become invalid or redundant

## Query log analysis

Production databases can log slow queries — any query taking longer than a configurable threshold (e.g., 100ms). Analyzing the slow query log is often the first step in performance work. Patterns to look for:

- **Same query structure, different parameters:** the query plan is fine but data distribution makes some parameter values slow (no statistics update can fix this — you need a better index or query structure)
- **N+1 queries:** one query per item in a list, instead of a single batch query — a classic application-layer bug
- **Missing index:** a frequently-slow query whose EXPLAIN shows a Seq Scan on a large table with high selectivity
- **Lock waits:** queries slow because they're waiting for locks held by other transactions — a concurrency design problem, not an index problem

## What you learned

- EXPLAIN ANALYZE bridges theory and reality: compare estimated rows to actual rows to diagnose stale statistics and bad plans
- Sequential scans are not always wrong — they are correct when a query reads most of the table
- Stale statistics (estimated rows far from actual rows) cause the optimizer to make bad plan choices — fix with ANALYZE
- Slow query log analysis reveals patterns: N+1 queries, missing indexes, lock waits, and parameter-sensitive plans
- Normalize schemas first for correctness; denormalize only after measuring a specific, proven performance problem
- The full internals stack — storage engines, indexes, query optimizer, MVCC, WAL, replication, and partitioning — works together: understanding each layer lets you diagnose problems at the right level
