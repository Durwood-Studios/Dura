## Why storage engines matter

When you write `INSERT INTO users (name) VALUES ('Alice')`, you probably assume that row lands somewhere on disk. But _how_ it lands — in what data structure, in what order, with what overhead — is entirely the storage engine's job. The choice of storage engine determines whether your database is fast at reads, fast at writes, or somewhere in between. It is one of the most consequential decisions in database design.

Two families of storage engines dominate the industry: **B-trees** (used by PostgreSQL, MySQL InnoDB, SQLite) and **LSM-trees** (used by RocksDB, Cassandra, LevelDB, and increasingly by newer systems like CockroachDB). They make radically different tradeoffs.

## B-trees: the read-optimized classic

A B-tree is a self-balancing tree where each node holds multiple keys and pointers to child nodes. The "B" does not stand for binary — a B-tree node can have hundreds of children. This wide branching factor means a B-tree with millions of rows is only 3–4 levels deep. Finding any record requires at most 3–4 disk seeks, which is why B-trees have dominated relational databases for fifty years.

B-tree nodes correspond to fixed-size disk pages (typically 4KB or 8KB). When you insert a row, the engine finds the correct leaf page and writes the new key in sorted position. If the page is full, it splits into two pages and updates the parent node to point to both. This split can cascade up the tree, but it is rare in practice.

**The write penalty:** Every update to a B-tree is an in-place modification of an existing page. If you update a single byte in a row, the engine still rewrites the entire 4KB page. More importantly, if one logical write requires modifying multiple pages (a split, or a cascade of splits), you need to ensure all those pages are written atomically. This is why B-tree databases require a **write-ahead log** (WAL) — if the system crashes mid-split, the WAL lets the database recover to a consistent state. That means every write actually involves at least two disk writes: one to the WAL, one to the data page.

**The read advantage:** Because B-tree pages are always in sorted order and stored in a stable location on disk, range scans are highly efficient. Reading all users with IDs between 1000 and 2000 means finding the leaf page for ID 1000 and then scanning forward — the pages are physically adjacent (or nearly so) on disk.

## LSM-trees: the write-optimized challenger

An LSM-tree — Log-Structured Merge-tree — takes a completely different approach. Writes never modify existing data. Instead, every write is appended to an in-memory buffer called a **memtable**, which is also mirrored in an append-only log on disk for crash safety. When the memtable fills up, it is flushed to disk as an immutable, sorted file called an **SSTable** (Sorted String Table).

This means every write is a sequential append — the fastest possible disk operation. There is no hunting for the right page to modify. You simply tack new data onto the end. This is why LSM-tree databases can sustain dramatically higher write throughput than B-tree databases under heavy write load.

**The read penalty:** Data for a single key might be spread across the memtable, multiple levels of SSTables, and the most recent SSTable that supersedes older ones. To read a key, the engine must check all of these in sequence, from newest to oldest, until it finds the key. In the worst case, this touches many files. Bloom filters (probabilistic data structures) help skip SSTables that definitely don't contain the key, but reads are still more expensive than in a B-tree.

**Compaction:** Left unchecked, SSTables accumulate indefinitely, making reads slower and consuming more disk. **Compaction** is the background process that merges older SSTables together, discarding deleted and overwritten data. Compaction keeps read performance bounded but introduces its own cost: the engine must read and rewrite data that might have already been written once. This is **write amplification** — the ratio of actual bytes written to disk versus logical bytes written by the application. A write amplification factor of 10 means a 1MB logical write causes 10MB of actual disk I/O.

## Choosing the right engine

The choice depends on your workload:

| Workload                                 | Choose                                    |
| ---------------------------------------- | ----------------------------------------- |
| Heavy reads, moderate writes             | B-tree (PostgreSQL, MySQL)                |
| Write-heavy, time-series, event logs     | LSM-tree (RocksDB, Cassandra)             |
| Mixed, needs SQL                         | B-tree (battle-tested, mature)            |
| Mobile / embedded with high write bursts | LSM-tree (SQLite in WAL mode is a hybrid) |

Modern databases blur these lines. PostgreSQL's heap storage is a B-tree variant, but it borrows LSM ideas (like the WAL). RocksDB allows tunable compaction strategies. Understanding the tradeoffs means you can make informed decisions when picking a database and tuning its configuration.

## What you learned

- B-trees organize data in sorted, fixed-size pages on disk; reads are fast because range scans are physically sequential
- B-tree writes modify pages in-place and require a WAL for crash safety, adding write overhead
- LSM-trees append all writes sequentially to memory then flush to immutable sorted files (SSTables), making writes extremely fast
- Compaction merges SSTables in the background, but introduces write amplification
- Write amplification is the ratio of actual disk I/O to logical application writes
- B-trees win for read-heavy workloads; LSM-trees win for write-heavy workloads like event streams and time-series data
