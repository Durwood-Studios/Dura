## Why this matters

The request-response model of HTTP works for most of the web — a client asks, a server answers. But a growing class of applications need something different: live stock prices, collaborative document editing, chat applications, game state updates, deployment logs streaming to your terminal. These need data to flow from server to client at any time, not just in response to a client request. Understanding sockets and the alternative patterns built on top of HTTP lets you choose the right tool for each problem rather than retrofitting the wrong one.

## What is a socket?

A **socket** is an endpoint for two-way network communication. In the OS model, a socket is a file descriptor — a reference that applications use to send and receive data. When you open a TCP connection, both sides get a socket. The server's socket is created by _binding_ to a port and _accepting_ incoming connections. The client's socket is created by _connecting_ to the server's address and port.

A TCP socket is identified by a 4-tuple: `(src IP, src port, dst IP, dst port)`. This uniquely identifies every connection in the network. A server listening on port 443 can handle thousands of simultaneous connections because each connection has a unique source IP/port combination on the client side.

UDP sockets work differently — they do not maintain connection state. You bind to a port and send or receive datagrams, each carrying the sender's address.

## WebSockets: full-duplex over HTTP

**WebSockets** upgrade an HTTP connection into a persistent, **bidirectional** channel. A WebSocket connection starts life as an HTTP/1.1 request with a special header:

```
GET /ws HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

The server responds with `101 Switching Protocols`, and from that point on the connection is no longer HTTP. Both sides can send messages at any time — there is no request-response pattern, no waiting for the other side to finish. A WebSocket message can be a text string or binary data, and messages are framed (each has a length prefix so the receiver knows where it ends).

WebSockets are ideal for: chat, collaborative editing, live dashboards, multiplayer games, financial data feeds. They require a persistent connection per client, which means connection limits and memory on the server side become operational concerns at scale.

## Long-polling: fake real-time over HTTP

Before WebSockets were widely supported, **long-polling** provided a workaround. The client sends a normal HTTP GET request. The server holds the response open — it does not reply until it has new data, or until a timeout expires. The client, upon receiving the response, immediately sends another request. This creates an illusion of a persistent connection.

```
Client: GET /events          Server holds request...
                             ...new event occurs...
                     200 OK  Server sends data, connection closes
Client: GET /events          Client immediately reconnects
                             (repeat)
```

Long-polling works everywhere HTTP works, which was its main advantage. Downsides: each message requires a full HTTP request/response overhead, headers are resent each time, and the server must manage many suspended request handlers. For most modern use cases, WebSockets or SSE are better.

## Server-Sent Events: one-directional HTTP streaming

**SSE** (Server-Sent Events) is a simpler alternative to WebSockets when you only need server-to-client data flow. The client makes one HTTP GET request, and the server holds the connection open and sends data in a specific text format:

```
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache

data: {"type":"price","symbol":"AAPL","value":182.50}

data: {"type":"price","symbol":"GOOG","value":140.20}

event: heartbeat
data: ping

```

Each event is separated by a blank line. The browser's `EventSource` API automatically handles reconnection if the connection drops. SSE is simpler than WebSockets, works over HTTP/2 multiplexing (you can have many SSE streams without many connections), and benefits from standard HTTP infrastructure (proxies, load balancers, auth middleware).

Use SSE when data flows only server-to-client. Use WebSockets when the client also needs to send data to the server continuously (chat, games, collaborative editing).

## Choosing between the options

| Pattern        | Direction       | Latency | Complexity | Best for                           |
| -------------- | --------------- | ------- | ---------- | ---------------------------------- |
| HTTP polling   | Pull            | High    | Low        | Infrequent updates, simple clients |
| Long-polling   | Pull-push       | Medium  | Medium     | Legacy clients, rare updates       |
| SSE            | Server → Client | Low     | Low        | Live feeds, logs, notifications    |
| WebSocket      | Bidirectional   | Low     | Medium     | Chat, games, collaboration         |
| Raw TCP socket | Bidirectional   | Lowest  | High       | Custom protocols, low-level work   |

constructor() {
this.state = "CONNECTING";
this.clientToServer = [];
this.serverToClient = [];
this.onClientMessage = null;
this.onServerMessage = null;
}

connect() {
if (this.state !== "CONNECTING") throw new Error("Already connected or closed");
this.state = "OPEN";
}

clientSend(message) {
if (this.state !== "OPEN") throw new Error("Cannot send — connection is " + this.state);
this.clientToServer.push(message);

serverSend(message) {

this.serverToClient.push(message);

const serverMsgs = this.serverToClient.splice(0);

}
for (const msg of serverMsgs) {
if (this.onServerMessage) this.onServerMessage(msg);
}
}

}

## What you learned

- A socket is the OS endpoint for network communication, identified by a 4-tuple of source/destination IP and port; a server on one port handles thousands of connections because each client has a unique source port
- WebSockets upgrade HTTP to a persistent bidirectional channel, ideal for chat, games, and collaborative tools; SSE provides simpler server-to-client streaming; long-polling is a legacy fallback
- Choosing between WebSocket, SSE, and polling depends on directionality, connection persistence requirements, and infrastructure constraints (SSE benefits from HTTP/2 multiplexing transparently)
