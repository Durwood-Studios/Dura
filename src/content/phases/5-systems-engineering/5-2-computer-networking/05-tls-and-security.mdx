## Why this matters

Every time you type a password, submit a form, or make an API call over the web, **TLS** is what makes it safe to do so. Without TLS, any network between you and the server — your ISP, the Wi-Fi router at the coffee shop, a malicious router in a data center — can read and modify every byte of your traffic. TLS prevents eavesdropping, prevents tampering, and proves you are talking to the server you think you are. As an engineer building anything that touches sensitive data, you must understand what TLS does, how it works, and what can go wrong.

## What TLS provides

**TLS** (Transport Layer Security, successor to SSL) sits between TCP and the application protocol (HTTP, SMTP, database protocols). It provides three properties:

**Confidentiality** — data is encrypted so third parties cannot read it, even if they capture the raw packets.

**Integrity** — data cannot be modified in transit without detection. A message authentication code (MAC) covers every record; tampering invalidates the MAC.

**Authentication** — the server proves its identity via a **certificate** signed by a trusted authority. The client knows it is talking to `api.example.com` and not an impersonator.

Note that authentication is one-directional by default — the client authenticates the server, not the other way around. Mutual TLS (mTLS) adds client certificates and is used for machine-to-machine communication in service meshes.

## Certificates and the trust chain

A **certificate** is a document that binds a public key to an identity (a domain name, an organization). It is signed by a **Certificate Authority** (CA) — an organization trusted by browsers and operating systems. When your browser connects to `example.com`, it:

1. Receives the server's certificate
2. Checks that the certificate was signed by a trusted CA
3. Checks that the certificate's domain matches the server's domain
4. Checks that the certificate has not expired and has not been revoked

Operating systems and browsers ship with a list of trusted root CAs (Mozilla, Apple, Google each maintain their own). A certificate chain typically has three levels: root CA → intermediate CA → end-entity certificate. This reduces the exposure of the root CA's private key (root CAs keep their keys offline and use intermediates for day-to-day signing).

## Public-key cryptography: the foundation

**Public-key** cryptography uses mathematically linked key pairs: a public key that anyone can know and a private key that only the owner holds. Data encrypted with the public key can only be decrypted with the private key. A signature created with the private key can be verified by anyone with the public key.

TLS uses public-key cryptography for two purposes:

1. **Authentication** — the server proves it owns the certificate's private key (without revealing it) by signing something with it
2. **Key exchange** — the two sides negotiate a shared symmetric key securely, using algorithms like ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)

After the key exchange, all data uses symmetric encryption (AES-GCM is common). Symmetric encryption is orders of magnitude faster than asymmetric, so the asymmetric phase is kept short.

## Cipher suites

A **cipher suite** is the specific combination of algorithms used for a TLS connection. A typical TLS 1.3 cipher suite name like `TLS_AES_256_GCM_SHA384` specifies:

- **AES_256_GCM** — the symmetric encryption algorithm (AES with 256-bit key in Galois/Counter Mode)
- **SHA384** — the hash algorithm for the MAC

TLS 1.3 simplified cipher suites dramatically by removing weak legacy options. TLS 1.2 had hundreds of suites including dangerously weak ones (RC4, DES, NULL encryption). TLS 1.3 has only five, all strong.

## The TLS 1.3 handshake

TLS 1.3 reduced the handshake from 2 RTTs (TLS 1.2) to 1 RTT:

```
Client                           Server
  |                                |
  | -- ClientHello (key_share) --> |   Client sends supported cipher suites
  |                                |   and its ECDHE key share
  |                                |
  | <-- ServerHello ------------   |   Server picks cipher suite, sends its
  | <-- Certificate     --------   |   key share, certificate, and proof of
  | <-- CertificateVerify ------   |   private key ownership
  | <-- Finished        --------   |   (all encrypted from this point)
  |                                |
  | -- Finished ----------------> |   Client verifies certificate, confirms
  |                                |   handshake is complete
  |                                |
  | ======= Application Data ====  |   1 RTT total before data flows
```

TLS 1.3 also supports **0-RTT resumption** for previously-connected clients — the client can send application data with the first message, using a pre-shared key from the previous session. This has a replay attack risk for non-idempotent requests and must be used carefully.

## What you learned

- TLS provides confidentiality, integrity, and authentication — without it, any network intermediary can read and modify all traffic
- Certificates bind a server's public key to a domain identity, verified through a chain of trust rooted in CAs that your OS and browser pre-trust
- TLS 1.3 completes the handshake in 1 RTT (versus TLS 1.2's 2 RTTs) and supports 0-RTT resumption for known servers; it uses ephemeral key exchange so past sessions cannot be decrypted even if the server's private key is later compromised
