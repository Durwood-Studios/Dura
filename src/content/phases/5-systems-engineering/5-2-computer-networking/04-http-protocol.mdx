## Why this matters

HTTP is the language of the web. Every API call, every page load, every webhook is an HTTP conversation. Senior engineers know HTTP well enough to read raw requests and responses, diagnose performance problems at the protocol level, and make deliberate choices between HTTP versions. If you have been writing `fetch()` calls without understanding what is on the wire, this lesson fills that gap. Understanding HTTP also makes you better at security, caching, debugging, and API design.

## The request-response cycle

**HTTP** (HyperText Transfer Protocol) is a stateless, text-based protocol built on top of TCP. Every interaction is a **request-response** pair: the client sends a request, the server sends a response, and the transaction is complete. HTTP itself has no memory of previous transactions — any state (sessions, authentication) is layered on top via **headers** and cookies.

A raw HTTP/1.1 request looks like this:

```
GET /api/users/42 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
User-Agent: Mozilla/5.0
```

And a response:

```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 87
Cache-Control: max-age=3600

{"id": 42, "name": "Alice", "email": "alice@example.com", "role": "admin"}
```

The structure is always: **method + path + version**, then **headers** (one per line), then a blank line, then an optional **body**.

## HTTP methods

Methods tell the server what operation to perform:

| Method  | Semantics                                  | Safe? | Idempotent? |
| ------- | ------------------------------------------ | ----- | ----------- |
| GET     | Retrieve a resource                        | Yes   | Yes         |
| POST    | Create or submit                           | No    | No          |
| PUT     | Replace a resource                         | No    | Yes         |
| PATCH   | Partial update                             | No    | No          |
| DELETE  | Remove a resource                          | No    | Yes         |
| HEAD    | GET without body (check existence/headers) | Yes   | Yes         |
| OPTIONS | What methods does this endpoint support?   | Yes   | Yes         |

**Safe** means the method has no side effects (read-only). **Idempotent** means calling it multiple times produces the same result as calling it once. GET is both; DELETE is idempotent (deleting something twice leaves it deleted) but not safe.

## Status codes you must know

Status codes communicate outcome at a glance. Memorize the categories:

- **2xx — Success**: `200 OK`, `201 Created`, `204 No Content`
- **3xx — Redirect**: `301 Moved Permanently`, `302 Found`, `304 Not Modified` (use cached version)
- **4xx — Client error**: `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`, `422 Unprocessable Entity`, `429 Too Many Requests`
- **5xx — Server error**: `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`, `504 Gateway Timeout`

Returning `200 OK` with `{ "error": "user not found" }` in the body is an antipattern. Use the appropriate status code — it enables caching infrastructure, monitoring, and client error handling to work correctly without parsing response bodies.

## HTTP headers carry metadata

**Headers** are key-value pairs that carry metadata about the request or response. They control caching, authentication, content negotiation, and more.

Important request headers:

- `Content-Type` — what format is the body (e.g., `application/json`)
- `Accept` — what formats the client can handle
- `Authorization` — credentials (Bearer token, Basic auth)
- `Cache-Control` — caching instructions sent by the client
- `If-None-Match` — conditional request (ETag-based)

Important response headers:

- `Content-Type` — format of the response body
- `Cache-Control` — how long clients and intermediaries may cache the response
- `ETag` — a hash of the resource for conditional requests
- `Location` — redirect target (used with 3xx)
- `Retry-After` — when to retry after a 429 or 503

## HTTP/2: multiplexing ends head-of-line blocking

**HTTP/1.1** allows only one outstanding request per TCP connection. Browsers open 6-8 parallel connections per domain to compensate, but this is wasteful and still limits concurrency. Each new request also bears the overhead of resending large headers (cookie headers alone can be hundreds of bytes, sent with every request).

**HTTP/2** introduced three major improvements:

**Multiplexing** — multiple requests and responses interleave over a single TCP connection. A slow response does not block a fast one. This is **multiplexing**: many streams share one connection without waiting for each other.

**Header compression (HPACK)** — headers are compressed using a shared table. Repeated headers (like `Authorization`) are sent as a single index reference instead of the full string.

**Server push** — the server can proactively send resources the client will need (e.g., CSS and JS for a page) before the client requests them. In practice this feature is rarely used correctly and may be deprecated in HTTP/3.

## HTTP/3: QUIC replaces TCP

HTTP/2's multiplexing has a subtle flaw: it still runs over TCP. If a single TCP packet is lost, all streams on that connection stall until the packet is retransmitted — this is **TCP head-of-line blocking** at the transport layer.

**HTTP/3** solves this by replacing TCP with **QUIC** (a UDP-based protocol). QUIC implements reliable, ordered delivery per stream independently, so a lost packet only stalls the stream it belongs to, not all streams. QUIC also combines the TLS handshake with the connection handshake, reducing connection setup from 2 RTTs (TCP + TLS) to 1 RTT or even 0 RTT for known servers.

i++;

const body = lines.slice(i + 1).join("\\n").trim() || null;

## What you learned

- HTTP's request-response model uses methods (GET, POST, PUT, DELETE) and status codes (2xx, 4xx, 5xx) to communicate operations and outcomes; using the wrong status code breaks caching and monitoring
- Headers carry essential metadata for authentication, content negotiation, and caching without touching the body
- HTTP/2 multiplexing solves application-layer head-of-line blocking; HTTP/3 on QUIC additionally eliminates transport-layer head-of-line blocking and reduces connection setup latency
