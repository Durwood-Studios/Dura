## Why this matters

Engineers who only understand TCP reach for it by reflex, even when it is the wrong tool. A real-time voice call does not need reliability — a retransmitted word that arrives 500ms late is useless. A multiplayer game cares more about position data arriving _now_ than arriving _eventually_. DNS lookups happen billions of times a day; adding a three-way handshake to each one would add intolerable latency. Understanding **UDP** — what it gives up, what it gains, and when that trade is correct — is the difference between a protocol-aware engineer and one who cargo-cults TCP for everything.

## UDP: deliberately minimal

**UDP** (User Datagram Protocol) is TCP's lean sibling. Its header is only 8 bytes versus TCP's minimum 20 bytes. It provides exactly three services:

1. **Port multiplexing** — like TCP, UDP uses source and destination ports so the OS knows which application gets which packet
2. **Checksum** — detects corruption (mandatory in IPv6, technically optional in IPv4 but always used in practice)
3. **Datagram boundaries** — each UDP send is a distinct **datagram** received as a unit; TCP has no message boundaries

That is it. No connection establishment. No acknowledgments. No retransmission. No ordering. No flow control. No congestion control. UDP gives you the raw ability to send labeled packets and receive them — or not.

## Connectionless means stateless

Because UDP is **connectionless**, there is no handshake and no state maintained between packets. Each datagram is independent. The OS does not track whether the remote host is alive, whether previous datagrams arrived, or how many bytes are in flight. This makes UDP:

- **Faster to start** — first packet can carry application data immediately, no RTT for setup
- **Lower overhead** — no ACKs, no window management, no state tables
- **Naturally multicast-friendly** — you can send one datagram to many receivers simultaneously (TCP connections are point-to-point)

The cost is that your application becomes responsible for any reliability it needs. If you need acknowledgment, you implement it. If you need ordering, you implement it. Many real-time protocols (RTP for audio/video, QUIC for HTTP/3) are built on UDP with custom reliability tailored to their use case.

## Packet loss and jitter in practice

Two failure modes dominate real-time UDP applications:

**Packet loss** — some percentage of datagrams never arrive. On a LAN this is typically under 0.1%. On mobile networks it can reach 5% or higher. Real-time applications must decide: retransmit (adding latency), use forward error correction (FEC — send redundant data proactively), or simply conceal the gap (interpolate position, repeat the last audio frame).

**Jitter** — even when packets arrive, they may arrive with variable spacing. A stream of packets sent every 20ms might arrive at 18ms, 25ms, 19ms, 32ms intervals. **Jitter** is the variation in inter-packet arrival times. Applications use a **jitter buffer** — a small queue that holds incoming packets briefly and releases them at a steady rate — to smooth out this variation. A larger jitter buffer reduces glitches but adds latency; too small and packets underrun the buffer.

## Use cases for UDP

**DNS** — a query and response fit in a single packet. The overhead of a TCP handshake would triple the latency of every DNS lookup. DNS uses UDP by default and falls back to TCP only for large responses (>512 bytes in the original spec, now up to 4096 with EDNS).

**Online gaming** — game state (positions, inputs) is time-sensitive. A 200ms-old position update is less useful than no update at all. Games send frequent position snapshots over UDP and accept that some will be lost. They implement their own acknowledgment for critical events (item pickups, kills) when needed.

**Live video streaming and VoIP** — a retransmitted video frame that arrives after the decoder has moved on is useless. Better to decode with one frame missing than to stall waiting. RTP (Real-time Transport Protocol) runs over UDP and lets the application decide how to handle loss.

**QUIC / HTTP/3** — Google's QUIC protocol (now IETF-standardized) implements its own reliability, ordering, and congestion control on top of UDP. This lets it avoid TCP's head-of-line blocking while still being reliable. HTTP/3 uses QUIC as its transport.

## What you learned

- UDP sacrifices reliability, ordering, and connection state for dramatically lower latency and overhead — making it ideal when timeliness matters more than completeness
- Applications built on UDP must implement any reliability they need themselves, tailoring it to their use case (e.g., FEC for video, custom ACK for game events)
- Jitter is variation in packet arrival timing; jitter buffers smooth it at the cost of a small fixed latency increase
