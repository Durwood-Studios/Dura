## Why this matters

Theoretical networking knowledge is valuable. The ability to observe _actual_ network traffic is indispensable. When a production incident involves intermittent connection drops, unexpected latency spikes, or a suspected middlebox interfering with traffic, tools like **tcpdump**, **Wireshark**, and **traceroute** are your eyes on the wire. Every backend engineer and every SRE needs to be comfortable with these tools — not expert-level, but fluent enough to capture the right traffic, filter the noise, and identify the problem. This lesson connects the theory from previous lessons to the tools you will use in production.

## tcpdump: the packet capture workhorse

**tcpdump** is a command-line tool that captures packets flowing through a network interface. It runs on Linux, macOS, and most Unix systems. Because it captures raw packets, it requires root or `CAP_NET_RAW` privileges.

```bash
# Capture all traffic on eth0
sudo tcpdump -i eth0

# Capture only TCP traffic on port 443
sudo tcpdump -i eth0 'tcp port 443'

# Capture traffic to/from a specific host, save to a file
sudo tcpdump -i eth0 -w capture.pcap host 93.184.216.34

# Read a saved capture file
sudo tcpdump -r capture.pcap

# Show packet contents in ASCII (-A) or hex (-x)
sudo tcpdump -i eth0 -A 'tcp port 80'

# Limit capture to first 100 packets
sudo tcpdump -i eth0 -c 100 'udp port 53'
```

tcpdump output shows one line per packet: timestamp, protocol, source address, destination address, flags, sequence numbers, and length. Reading TCP flags in tcpdump is essential:

- `[S]` — SYN (connection initiation)
- `[S.]` — SYN-ACK
- `[.]` — ACK (the `.` means pure acknowledgment)
- `[P.]` — PSH-ACK (data segment)
- `[F.]` — FIN-ACK (connection teardown)
- `[R]` — RST (abrupt connection reset)

A clean HTTP connection in tcpdump:

```
10:21:30 IP client.54321 > server.80: Flags [S], seq 0
10:21:30 IP server.80 > client.54321: Flags [S.], seq 0, ack 1
10:21:30 IP client.54321 > server.80: Flags [.], ack 1
10:21:30 IP client.54321 > server.80: Flags [P.], seq 1:78, ack 1  ← GET request
10:21:30 IP server.80 > client.54321: Flags [P.], seq 1:512, ack 78  ← response
10:21:30 IP client.54321 > server.80: Flags [F.], seq 78, ack 512
10:21:30 IP server.80 > client.54321: Flags [F.], seq 512, ack 79
```

## Wireshark: GUI packet analysis

**Wireshark** is the graphical counterpart to tcpdump. It reads the same `.pcap` files and adds powerful display filters, protocol dissection, stream reassembly, and statistical views. Where tcpdump is fast and scriptable, Wireshark is deep and visual.

Key Wireshark features for debugging:

- **Display filters** — `http.response.code == 500`, `tcp.analysis.retransmission`, `dns.qry.name == "example.com"` — these filter the view without stopping capture
- **Follow TCP stream** — right-click any packet in a connection and select "Follow TCP Stream" to see the complete HTTP request and response as text
- **Statistics → TCP Stream Graphs** — visualize the sawtooth congestion window pattern, identify retransmissions, measure throughput
- **Expert Information** — Wireshark automatically flags retransmissions, out-of-order segments, duplicate ACKs, and TCP zero-window events

**Capture filter** (applied before capture, uses BPF syntax, same as tcpdump): reduces what is recorded.
**Display filter** (applied after capture, more powerful syntax): hides what you are not interested in without discarding data.

## traceroute: mapping the path

**traceroute** (or `tracert` on Windows) exploits the **TTL** (Time to Live) field in IP packets to discover the sequence of routers between you and a destination. Each router decrements the TTL by 1 before forwarding. When TTL reaches 0, the router discards the packet and sends back an ICMP "Time Exceeded" message — including its own IP address. By sending packets with TTL=1, then TTL=2, and so on, traceroute reveals each hop.

```bash
# Basic traceroute
traceroute example.com

# Use TCP SYN probes instead of UDP (bypasses some firewalls)
sudo traceroute -T -p 80 example.com

# Show numeric IPs, no DNS reverse lookup (faster)
traceroute -n example.com
```

Sample traceroute output:

```
traceroute to example.com (93.184.216.34)
 1  192.168.1.1       1.2 ms   1.1 ms   1.0 ms   ← home router
 2  10.0.0.1          5.3 ms   5.1 ms   5.2 ms   ← ISP edge
 3  72.14.232.1      12.4 ms  12.1 ms  12.2 ms   ← backbone
 4  209.85.240.1     13.0 ms  13.1 ms  12.9 ms
 5  * * *                                         ← router drops probes (firewall)
 6  93.184.216.34    15.2 ms  15.1 ms  15.3 ms   ← destination
```

Reading traceroute:

- **Three latency columns** — traceroute sends three probes per TTL; the values are their round-trip times
- **`* * *`** — a router did not respond (firewall drops TTL-expired packets; the hop still forwarded traffic)
- **Latency spikes** — a sudden jump at one hop usually indicates that hop is the bottleneck, not necessarily the router itself (ICMP responses are low-priority)
- **High latency at all hops after a point** — confirms the bottleneck is at or near that hop

## Diagnosing common problems

**Packet loss** — if `ping` to a host shows intermittent losses, run a **mtr** (my traceroute — combines ping and traceroute in real time) to identify which hop is dropping packets.

**High latency to a specific service** — traceroute to that service. Compare the path to a geographically similar service. Sudden jumps indicate a slow router or suboptimal routing.

**Connection resets** — tcpdump shows `[R]` flags unexpectedly. This indicates a firewall, load balancer, or NAT gateway is resetting connections. Check timeout settings and keepalive configuration.

**TLS handshake failures** — Wireshark can show the TLS handshake in detail. Look for the server sending a `close_notify` alert or the client receiving an unexpected certificate.

## What you learned

- tcpdump captures raw packets with BPF filter expressions; reading TCP flags (S, S., ., P., F., R) lets you trace connections and spot resets or retransmissions at the protocol level
- Wireshark adds GUI analysis, display filters, TCP stream reconstruction, and expert annotations that automatically flag retransmissions and zero-window events
- traceroute reveals the path between two hosts by exploiting TTL; latency jumps between consecutive hops indicate slow links, while `* * *` hops are silent firewalls that still forward traffic
