## Why this matters

Every time you open a browser and load a page, dozens of protocols collaborate invisibly — HTTP asks for the resource, TCP breaks it into pieces and guarantees delivery, IP routes those pieces across the internet, and Ethernet or Wi-Fi physically transmits the bits. Understanding how these protocols are organized is not academic trivia. It is the mental map that lets you diagnose network problems, design distributed systems, and reason about where failures occur. Engineers who understand the layers find and fix production incidents faster because they know exactly which layer to inspect.

## The OSI model: seven layers of abstraction

The **OSI model** (Open Systems Interconnection) was designed in the 1970s as a universal framework for understanding networking. It splits communication into seven layers, each responsible for a specific job, each relying on the layer below and serving the layer above.

| Layer | Name         | What it does                  | Example protocols    |
| ----- | ------------ | ----------------------------- | -------------------- |
| 7     | Application  | User-facing protocols         | HTTP, DNS, SMTP, FTP |
| 6     | Presentation | Data encoding, encryption     | TLS, JPEG, ASCII     |
| 5     | Session      | Managing sessions, dialogs    | NetBIOS, RPC         |
| 4     | Transport    | End-to-end reliability, ports | TCP, UDP             |
| 3     | Network      | Routing between networks      | IP, ICMP             |
| 2     | Data Link    | Node-to-node delivery         | Ethernet, Wi-Fi      |
| 1     | Physical     | Bits over the wire            | Copper, fiber, radio |

The mnemonic "**P**lease **D**o **N**ot **T**hrow **S**ausage **P**izza **A**way" gives you layers 1–7 from bottom to top. The OSI model is conceptual — real protocols do not map cleanly to it — but it gives you a shared vocabulary for discussing where problems live.

## The TCP/IP model: what the internet actually uses

The **TCP/IP model** is a pragmatic four-layer model that describes how the real internet works. It collapses the OSI model's top three layers into one and combines the bottom two.

| TCP/IP Layer   | Corresponds to OSI | Example protocols |
| -------------- | ------------------ | ----------------- |
| Application    | Layers 5–7         | HTTP, DNS, TLS    |
| Transport      | Layer 4            | TCP, UDP          |
| Internet       | Layer 3            | IP, ICMP          |
| Network Access | Layers 1–2         | Ethernet, Wi-Fi   |

Engineers use TCP/IP terminology in practice and OSI terminology when discussing abstractions. You need both.

## Encapsulation: wrapping data as it descends

**Encapsulation** is the process of each layer adding its own header (and sometimes trailer) to the data as it passes downward through the **network stack**. The receiving end reverses the process — each layer strips its header and passes the remaining data upward. This is called de-encapsulation.

Think of it as a series of envelopes. The application writes a letter (HTTP request). The transport layer puts it in an inner envelope labeled with source and destination port numbers. The network layer puts that envelope inside an outer envelope labeled with IP addresses. The data link layer puts the whole thing in a box with the physical addresses (MAC addresses) of the next hop.

```
[ HTTP request ]                              ← Application data
[ TCP header | HTTP request ]                 ← Transport segment
[ IP header | TCP header | HTTP request ]     ← Network packet
[ Frame header | IP | TCP | HTTP | Frame CRC ]← Data link frame
```

Each header carries metadata needed by that layer's **protocol** — the agreed-upon rules that both sides must follow. A protocol without both sides agreeing on the format is just noise.

## Why layers matter in practice

Layering provides two critical properties: **interoperability** and **replaceability**. Any Wi-Fi hardware can carry IP packets because they agree on the network access layer contract. Any TCP implementation can carry HTTP because TCP provides the transport contract. You can swap IPv4 for IPv6 without rewriting your HTTP server. You can encrypt with TLS without changing TCP. The layers are interfaces, and interfaces enable substitution.

When something breaks, layering tells you where to look. A packet not arriving at all? Check the network layer (IP routing, firewall rules). Data arriving out of order? Transport layer (TCP handles this, but UDP does not). Application getting garbled data? Look at the presentation layer — possibly an encoding mismatch.

## What you learned

- The OSI model organizes networking into seven conceptual layers; the TCP/IP model collapses those into four practical layers that describe the real internet
- Encapsulation means each layer wraps the payload from above with its own header, forming progressively larger units: data → segment → packet → frame
- Layering provides interoperability and replaceability — you can swap implementations at one layer without changing others, and failures can be localized to a specific layer
