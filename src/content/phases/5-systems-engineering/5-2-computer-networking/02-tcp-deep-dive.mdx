## Why this matters

TCP is the backbone of the reliable internet. Every HTTP request, database query, SSH session, and API call depends on TCP to deliver data completely and in order, even though the underlying network makes no such guarantees. Understanding TCP deeply changes how you think about performance: why does a connection to a distant server feel slow even on fast hardware? Why does a dropped packet cause a cascade of slowdowns? Why does "retry with exponential backoff" appear in virtually every distributed systems design? The answers are in TCP's internals.

## What TCP guarantees

The **TCP** (Transmission Control Protocol) sits at Layer 4 and provides four guarantees over the unreliable IP layer below it:

1. **Reliable delivery** — every byte sent is acknowledged; lost segments are retransmitted
2. **Ordered delivery** — bytes arrive in the order they were sent, regardless of how packets were reordered in transit
3. **Error detection** — a checksum covers every segment; corrupted data is discarded and retransmitted
4. **Flow control** — the sender never overwhelms the receiver's buffer

These guarantees come at a cost: TCP requires setup (the handshake), teardown (FIN/ACK exchange), and ongoing overhead (acknowledgments, timers). That cost is worth paying for most applications. For others, it is not — which is why UDP exists.

## The three-way handshake

Before any data flows, TCP performs a **handshake** to synchronize both sides. This establishes the connection and exchanges initial **sequence numbers** — the counters that both sides use to track how many bytes have been sent and received.

```
Client                          Server
  |                               |
  |  -- SYN (seq=x) ----------->  |   Client picks random initial seq x
  |                               |
  |  <-- SYN-ACK (seq=y,ack=x+1) |   Server picks random initial seq y
  |                               |   Server acknowledges client's seq
  |  -- ACK (ack=y+1) ----------> |   Client acknowledges server's seq
  |                               |
  |  =====  DATA FLOWS  ========  |
```

The exchange takes **1.5 round trips** before data can flow. On a transcontinental connection with 100ms RTT (round-trip time), that is 150ms of mandatory delay before a single byte of application data moves. This is why connection reuse (HTTP keep-alive, HTTP/2 multiplexing) matters enormously for performance.

## Sequence numbers and acknowledgments

Every byte in a TCP stream is numbered. The **sequence number** in a segment's header indicates the position of the first byte in that segment's payload. The **acknowledgment number** tells the sender: "I've received everything up through byte N-1; send me byte N next."

This mechanism enables the receiver to detect gaps. If segments arrive out of order (which IP can do), TCP buffers them and reassembles them correctly before delivering to the application. If a segment is lost entirely, the absence of an acknowledgment triggers retransmission after a timeout.

## Flow control: the receive window

The receiver has a finite buffer. **Flow control** prevents the sender from overwhelming it. The receiver advertises a **receive window** (rwnd) in every ACK — the number of bytes it can currently accept. The sender must not have more than rwnd bytes of unacknowledged data in flight at any time.

```
Receiver buffer: [####....] → rwnd = 4 bytes available
Sender: "I can only have 4 bytes in flight until more ACKs arrive"
```

When an application reads data from the buffer slowly (a slow disk write, a blocking operation), the window shrinks. If it reaches zero, the sender stops entirely and polls until the window reopens. A receiver that cannot drain its buffer fast enough is a common performance bottleneck.

## Congestion control: protecting the network

Flow control protects the receiver. **Congestion control** protects the network itself. The network between sender and receiver may have routers with limited queue sizes. If the sender floods the network, routers drop packets — and dropped packets trigger retransmissions, making congestion worse.

TCP uses a **congestion window** (cwnd) to limit how much data is in flight network-wide. It works in phases:

**Slow start** — begin with a small cwnd (1-10 MSS) and double it every RTT. Despite the name, this phase grows fast (exponentially).

**Congestion avoidance** — once cwnd reaches a threshold (ssthresh), grow linearly (one MSS per RTT) to probe for more bandwidth without causing drops.

**Loss detection** — when a segment is lost (timeout or duplicate ACKs), cut cwnd significantly and return to congestion avoidance. This is the "sawtooth" pattern visible in TCP throughput graphs.

Modern variants like CUBIC and BBR use different algorithms to probe bandwidth more aggressively while still being fair to competing flows.

## What you learned

- TCP guarantees reliable, ordered, error-checked, flow-controlled delivery by adding mechanisms on top of unreliable IP
- The three-way handshake (SYN → SYN-ACK → ACK) synchronizes sequence numbers and costs at least one round-trip before data flows
- Flow control uses a receive window to prevent overwhelming the receiver's buffer; congestion control uses a congestion window to prevent overwhelming the network itself
