## Why this matters

No single server handles unlimited traffic. At some point — whether due to CPU, memory, connection count, or I/O limits — you need multiple servers. A **load balancer** is the traffic cop that distributes incoming connections across those servers, hides individual server failures from clients, and enables you to deploy and scale without downtime. Understanding load balancing algorithms and their tradeoffs is essential for designing systems that stay up under real-world conditions. When your on-call alert fires at 3am because one backend server is handling 90% of traffic, you need to know why and how to fix it.

## What a load balancer does

A **load balancer** sits between clients and backend servers. Clients connect to the load balancer's address — they never know how many backend servers exist or which one will handle their request. The load balancer:

1. Accepts incoming connections or requests
2. Selects a backend server using a distribution algorithm
3. Forwards the traffic to that server
4. Optionally maintains session state
5. Monitors backend health and removes failed servers

A **reverse proxy** does everything a load balancer does, plus more: SSL/TLS termination, caching, compression, request rewriting, and security filtering. Nginx and HAProxy are commonly deployed as both. In contrast, a _forward proxy_ sits in front of clients and mediates their access to the internet — the opposite direction.

## Layer 4 vs Layer 7

The most important distinction in load balancing is _which layer_ the balancer operates at:

**Layer 4 (L4)** — works at the transport layer. The balancer sees source/destination IP and port numbers but not the content of the request. It forwards TCP/UDP connections without inspecting application data. This is faster (less processing per connection) but dumb — it cannot make routing decisions based on URL paths, request headers, or response content.

**Layer 7 (L7)** — works at the application layer. The balancer terminates the TCP connection, reads the HTTP request, makes a routing decision based on the full request (URL, headers, cookies), then opens a new connection to the backend. This is slower (two TCP connections per request) but powerful — it can route `/api/*` to one server group and `/static/*` to another, add or strip headers, enforce rate limits, and rewrite URLs.

Modern systems typically use L7 load balancing for HTTP traffic. L4 is used for non-HTTP protocols (database connections, game servers) and for performance-critical paths.

## Distribution algorithms

**Round-robin** — requests are distributed sequentially across all servers in order. Request 1 goes to server A, request 2 to B, request 3 to C, request 4 back to A. Simple, fair, and stateless. Works well when requests have similar cost. Breaks down when requests have wildly different execution times — a slow request on server A does not slow down server B.

**Weighted round-robin** — like round-robin, but servers with more capacity receive more requests. Server A (8 cores) might receive 4 requests for every 1 that server B (2 cores) receives. Useful when backend servers have different capacities.

**Least connections** — the next request goes to the server with the fewest active connections. More sophisticated than round-robin; naturally handles requests with variable duration because heavily-loaded servers receive fewer new requests. Requires the load balancer to track connection counts.

**Least response time** — combines least connections with response time measurements. Prefers servers that are both lightly loaded and responding quickly. Catches a scenario where a server has few connections but is responding slowly (perhaps due to a slow database query or GC pause).

**IP hash** — the source IP address is hashed to always route a given client to the same server. This is one way to implement **sticky sessions** — ensuring a client's requests always reach the same backend. Useful when backend servers maintain in-memory session state. Breaks if the server goes down; consistent hashing reduces the disruption.

**Random** — select a backend at random. Surprisingly effective at scale due to the law of large numbers. The "power of two choices" variant picks two random backends and routes to the one with fewer connections — this outperforms pure round-robin in many simulations.

## Health checks

A **health check** is a periodic probe the load balancer sends to each backend to verify it is alive and able to serve traffic. Two types:

**Active health check** — the load balancer probes the backend on a schedule (e.g., every 5 seconds). A probe might be a TCP connect, an HTTP GET to `/health`, or a more complex check that verifies database connectivity. If N consecutive probes fail, the backend is marked down and removed from rotation.

**Passive health check** — the load balancer watches real traffic responses. If a backend returns 5xx errors or connection timeouts above a threshold, it is temporarily removed. Less overhead than active checks, but slower to detect failures that do not affect current requests.

When a backend recovers, most load balancers slowly reintroduce it (send a fraction of traffic) before restoring it to full rotation — this protects against flapping.

## Sticky sessions and their cost

**Sticky sessions** (also called session persistence) route all requests from a given client to the same backend. This allows stateful backends — servers that store session data in memory. The load balancer typically uses a cookie (`Set-Cookie: SERVERID=A`) to remember which server a client was assigned to.

Sticky sessions create problems at scale:

- If the assigned server fails, the session is lost regardless
- Traffic becomes uneven if some users are much more active than others
- Scaling becomes harder (adding a server does not help existing users)

The modern alternative is to store session state outside the backend (Redis, a database) so any server can handle any request. Stateless backends scale horizontally without sticky session complexity. When you see sticky sessions in a design, treat it as a flag that session state management needs rethinking.

## What you learned

- Layer 4 load balancers operate at the TCP/UDP level (fast but protocol-blind); Layer 7 balancers read HTTP requests and can route based on URL, headers, and cookies (slower but powerful)
- Round-robin is simple and stateless; least-connections is more adaptive for variable-duration requests; IP hash enables sticky sessions but at the cost of resilience and even distribution
- Health checks detect failed backends and remove them from rotation; sticky sessions are an antipattern to avoid by making backends stateless with external session storage
