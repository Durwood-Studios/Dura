# Threads, Concurrency, and Race Conditions

Processes give every program its own isolated execution environment — its own memory, its own file descriptors, its own identity. But sometimes isolation is the problem. What if you want multiple parts of the same program to run simultaneously and share data efficiently?

That's what **threads** are for. A thread is a unit of execution that lives inside a process. Every process starts with one thread (the main thread), but it can create additional threads that share the process's memory space and run concurrently. Threads within the same process can read and write the same data directly — there's no copying, no inter-process communication protocol, no message passing overhead.

This power comes with a sharp edge: if multiple threads modify the same data at the same time without coordination, the results are unpredictable. Understanding this problem — and how to solve it — is one of the most important skills in systems programming.

## Threads vs. Processes: What's Shared

When a process forks, the child gets its own **copy** of the parent's memory. Changes the child makes don't affect the parent's memory, and vice versa. Processes communicate through explicit channels: pipes, sockets, shared memory segments, or files.

Threads are fundamentally different. All threads within a process share:

- The **heap** — dynamically allocated memory
- **Global variables** — any module-level or static data
- **File descriptors** — open files, sockets, pipes
- **Code segments** — all threads run the same program code

Each thread has its own:

- **Stack** — local variables and function call frames are private per thread
- **CPU registers and program counter** — each thread has its own execution position
- **Thread ID (TID)** — a unique identifier within the process

Because heap memory is shared, threads can pass data to each other simply by writing to the same variable. This is called **shared memory** communication, and it is dramatically faster than inter-process communication. It is also dramatically more dangerous.

## Concurrency and the Illusion of Simultaneity

On a single-core machine, true parallelism is impossible — only one thread can execute at a time. But the OS scheduler rapidly switches between threads, giving each a brief slice of CPU time. This creates the illusion of simultaneous execution, called **concurrency**.

On a multi-core machine, threads can achieve true parallelism: multiple threads run on separate CPU cores simultaneously. This is where the speedup from threading actually comes from — but it's also where race conditions become most severe, because operations that might have been naturally serialized on a single core can now genuinely overlap.

## Race Conditions: When Timing Is Everything

A **race condition** occurs when the correctness of a program depends on the relative timing or ordering of events in multiple threads. The outcome is non-deterministic — it depends on how the OS happens to schedule the threads, which can vary between runs.

Consider a classic example: two threads both want to increment a shared counter. In high-level code, `counter++` looks like a single operation. But at the machine level, it's three separate operations:

1. **Read** the current value of `counter` from memory into a register
2. **Increment** the value in the register
3. **Write** the new value back to memory

If two threads execute these three steps without coordination, their operations can interleave in destructive ways. Thread A reads the value (say, 5). Thread B also reads the value (5). Thread A writes 6. Thread B also writes 6. The counter should be 7, but it's 6 — one increment was silently lost.

The portion of code that accesses shared data is called the **critical section**. The race condition occurs when multiple threads are in the critical section at the same time.

## Mutexes: Enforcing Exclusivity

A **mutex** (short for "mutual exclusion lock") is a synchronization primitive that ensures only one thread can be in the critical section at a time. Before entering the critical section, a thread acquires the mutex. If another thread already holds it, the acquiring thread blocks — it waits — until the mutex is released.

The basic protocol looks like this:

1. Thread calls `mutex.lock()` — if another thread holds it, block; otherwise, acquire it
2. Thread executes its critical section (reads/writes shared data)
3. Thread calls `mutex.unlock()` — releases the mutex, potentially unblocking a waiting thread

A mutex serializes access to shared data. Only one thread can hold a given mutex at a time, so the read-increment-write sequence for the counter happens atomically from the perspective of other threads — they can't interleave with it.

Mutexes are not free. Every lock/unlock pair has overhead, and excessive locking can eliminate the performance benefit of threading entirely. Good concurrent programming minimizes the time spent inside critical sections, holding locks only as long as strictly necessary.

## Deadlock: The Danger of Multiple Locks

When code uses multiple mutexes, a deadlock can occur: Thread A holds mutex 1 and waits for mutex 2. Thread B holds mutex 2 and waits for mutex 1. Both threads wait forever — the program is frozen, with no cycle to break the standoff.

Avoiding deadlock requires careful design: always acquire multiple locks in a consistent order across all threads, use timeout-based locking, or redesign to use fewer locks.
