# Synchronization: Mutexes, Semaphores, and Deadlock

Concurrency would be simple if every thread had its own private data and never communicated. But real systems share resources: a database connection pool, a buffer of network packets, a print queue, a shared cache. Controlling access to these shared resources correctly — without deadlock, without starvation, without data corruption — is one of the hardest problems in systems programming.

This lesson goes deeper than the mutex introduction from the threads lesson. You'll understand the full synchronization toolkit that OS designers provide and learn to reason precisely about correctness.

## Recap: What Mutexes Guarantee

A **mutex** (mutual exclusion lock) is the simplest synchronization primitive. It guarantees that at most one thread holds the lock at any given time. A thread that tries to acquire an already-held mutex blocks until the holder releases it.

Mutexes are appropriate when you need to protect a **critical section** — a block of code where shared data is read or modified. The invariant is simple: only the thread holding the mutex can be executing that critical section.

What mutexes don't tell you: anything about how many threads should be allowed through, or whether a resource is available. For those questions, you need semaphores.

## Semaphores: Counting Permits

A **semaphore** is a synchronization primitive with an integer counter. It supports two atomic operations:

- **acquire()** (also called `wait`, `P`, or `down`): Decrements the counter. If the counter would go below zero, the thread blocks until another thread increments it.
- **release()** (also called `signal`, `V`, or `up`): Increments the counter. If any threads are blocked waiting, one is woken up.

A **binary semaphore** (counter initialized to 1) behaves exactly like a mutex — only one thread can acquire it at a time.

A **counting semaphore** (counter initialized to N) allows up to N threads to proceed simultaneously. This makes semaphores ideal for managing pools of resources: a database connection pool with 10 connections, a thread pool with a fixed number of workers, or a bounded buffer.

The key insight: the semaphore counter represents **how many units of a resource are currently available**. Acquiring decrements (one fewer available), releasing increments (one more available).

## The Producer-Consumer Problem

The **producer-consumer** problem (also called the bounded buffer problem) is a classic synchronization scenario:

- **Producers** create data and add it to a shared buffer
- **Consumers** remove data from the buffer and process it
- The buffer has a maximum capacity — producers must wait when it's full
- Consumers must wait when it's empty

This pattern appears everywhere: web servers buffering incoming requests, audio systems buffering decoded samples, OS kernels buffering network packets, log pipelines buffering messages.

The challenge is coordinating producers and consumers so that:

1. No data is lost (producers don't overwrite unread items)
2. No phantom reads occur (consumers don't read from an empty buffer)
3. The buffer's internal data structures are protected from concurrent modification

A correct solution requires **three** synchronization objects:

- A semaphore `empty` initialized to buffer capacity — tracks available empty slots
- A semaphore `full` initialized to 0 — tracks available filled slots
- A mutex protecting the buffer's internal structure

**Producer:** acquire `empty` (wait for a free slot), acquire mutex, add item, release mutex, release `full` (signal that a new item is available).

**Consumer:** acquire `full` (wait for an item), acquire mutex, remove item, release mutex, release `empty` (signal that a slot is free).

This pattern is elegant: producers and consumers automatically block when they need to, and wake up precisely when their condition is satisfied.

## Deadlock: Formal Analysis

A deadlock requires four conditions to hold simultaneously (Coffman conditions):

1. **Mutual exclusion** — resources cannot be shared; only one thread can hold a resource at a time
2. **Hold and wait** — a thread holds at least one resource while waiting for another
3. **No preemption** — the OS cannot forcibly take a resource from a thread; it must be released voluntarily
4. **Circular wait** — there is a cycle of threads where each waits for a resource held by the next

Eliminating any one of these four conditions prevents deadlock entirely. In practice:

- Eliminating hold-and-wait: Require threads to acquire all locks atomically upfront. Impractical in general but works in specific scenarios.
- Eliminating circular wait: Enforce a global ordering on all locks. All threads must acquire locks in the same order. If Thread A always acquires lock 1 before lock 2, and Thread B does the same, no cycle can form.
- Eliminating no-preemption: Use try-lock with timeout. If a thread can't acquire a lock within a deadline, it releases all held locks and retries later. Risk: livelock (threads keep retrying without progress).

Real systems also use deadlock detection: periodically build a resource allocation graph and check for cycles. If a cycle exists, terminate one thread in the cycle to break it.

## Livelock: The Zombie of Concurrency Bugs

Related to deadlock but distinct: livelock occurs when threads are not blocked, but cannot make progress because they keep reacting to each other. Imagine two people in a hallway stepping aside to let each other pass — but always in the same direction. They're both "active" but neither gets through.

Livelock typically occurs with retry-based strategies where all threads retry at the same rate and keep colliding. The fix is randomized backoff — each thread waits a random amount of time before retrying, which staggers their attempts.
