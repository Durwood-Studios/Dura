# CPU Scheduling: Sharing the Processor

Your laptop might have dozens of processes running simultaneously — a browser, a code editor, a music player, background services, and more. Yet your machine likely has 4 to 16 CPU cores. How does it give all those processes the CPU time they need?

The answer is the **scheduler** — one of the most consequential subsystems in the operating system. The scheduler decides, at every moment, which process or thread gets to run on which CPU core. Get it right and the system feels fast and responsive. Get it wrong and your mouse stutters, your music skips, or your compilation crawls.

## The Core Problem

If there are 40 runnable processes and 4 CPU cores, only 4 processes can actually run at any given moment. The scheduler must continuously choose which 4 get CPU time and for how long. It does this many thousands of times per second, with microsecond-level precision.

The scheduler must balance multiple competing goals:

- **Throughput** — maximize the total amount of work completed per unit time
- **Fairness** — every process should eventually get CPU time; no process should starve indefinitely
- **Responsiveness** — interactive programs (like your GUI and mouse cursor) should feel snappy, with low latency
- **Efficiency** — the scheduling mechanism itself shouldn't consume so much CPU that it undermines the other goals

No algorithm perfectly satisfies all of these simultaneously. Different workloads benefit from different scheduling strategies.

## Preemption: Taking Back the CPU

Early operating systems were **cooperative** — a process ran until it voluntarily yielded the CPU. If a program entered an infinite loop or froze, the entire system froze with it.

Modern schedulers are **preemptive**. The kernel uses a hardware timer that fires at regular intervals (typically 1 millisecond to 10 milliseconds). When the timer fires, the CPU hands control back to the kernel, which decides whether to let the current process keep running or switch to another. Preemption means no single process can monopolize the CPU — the OS can always reclaim it.

The maximum time a process can run before being preempted is called its **time quantum** or **time slice**. Shorter time slices improve responsiveness but increase overhead from frequent context switches. Longer time slices improve throughput but make the system feel less responsive.

## Context Switching: Saving and Restoring State

When the scheduler switches from one process to another, it must perform a **context switch**. The CPU is a stateful machine — it has registers, a program counter, a stack pointer, and memory mappings. All of this state belongs to the currently running process and must be saved before another process can use the CPU.

A context switch involves:

1. Saving the current process's CPU registers, program counter, and stack pointer into its kernel-managed saved state block (the Process Control Block or PCB)
2. Switching the memory map (updating the page table base register so the MMU uses the new process's page tables, and flushing or tagging the TLB)
3. Restoring the new process's saved CPU state from its PCB
4. Jumping to the restored program counter — the new process resumes where it left off

Context switches are fast (microseconds) but not free. A system doing thousands of context switches per second has measurable overhead. This is why heavily multithreaded programs with lots of lock contention can be slower than simpler single-threaded implementations — the overhead of coordinating many threads eats into the gains.

## First-Come, First-Served (FCFS)

The simplest scheduling algorithm: run each process to completion in the order it arrived. No preemption, no priority. Simple to implement, terrible in practice. A single long-running process can delay all shorter ones — a phenomenon called the **convoy effect**. Completely inappropriate for interactive systems.

## Shortest Job First (SJF)

Run the process expected to finish soonest. Theoretically optimal for minimizing average waiting time, but useless in practice because you can't know in advance how long a process will run. Approximations exist (using past behavior to predict future runtime), and SJF concepts influence modern schedulers.

## Round Robin

**Round Robin** is the classic preemptive algorithm. Every runnable process gets added to a circular queue. The scheduler gives the front process a time quantum of CPU time. When the quantum expires, the process goes to the back of the queue, and the next process runs. Every process gets a fair turn, in rotation.

Round robin is excellent for interactive fairness but doesn't distinguish between a video game (which needs frequent, short CPU bursts to respond to input) and a background compression task (which needs long, uninterrupted CPU time). Modern schedulers extend round robin with **priority levels** — higher-priority processes get more CPU time or are given preference in scheduling.

## Priority Scheduling

Each process is assigned a **priority** (a number). The scheduler always runs the highest-priority runnable process. Within the same priority level, round robin applies.

Priorities can be **static** (fixed at process creation) or **dynamic** (adjusted by the kernel based on behavior). Linux's Completely Fair Scheduler (CFS) tracks how much CPU time each process has received relative to what it "should" have received, and always schedules the process furthest behind its fair share.

Priority scheduling risks **starvation** — low-priority processes may never run if high-priority processes continuously occupy the CPU. The solution is **aging**: the OS gradually increases the priority of processes that haven't run in a while, ensuring they eventually get scheduled.
