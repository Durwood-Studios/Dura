# Memory Management and Virtual Memory

Every program you write assumes it has access to a large, contiguous block of memory starting near address 0. In a typical 64-bit process, this address space spans terabytes — far more than any physical RAM could provide. Yet the program works. How?

The answer is **virtual memory** — one of the most elegant abstractions in operating systems design. Virtual memory creates the illusion that each process has its own private, enormous address space, even though the physical RAM might be small and shared among dozens of processes simultaneously.

## The Problem Virtual Memory Solves

Early computers had no virtual memory. Programs used physical memory addresses directly. This created severe problems: a bug in one program that wrote to the wrong address would corrupt another program's data. Programs had to be carefully arranged in memory to avoid collisions. The total size of all running programs had to fit in physical RAM. Relocating a program in memory required recompiling or relinking it.

Virtual memory solves all of these problems in one elegant system.

## Address Spaces

Every process gets its own **address space** — a private map of memory that the process sees as entirely its own. When a process reads from or writes to a memory address, it is using a _virtual address_. The CPU and operating system silently translate that virtual address to the corresponding _physical address_ in RAM before the actual memory access occurs.

Two different processes can use the exact same virtual address (say, `0x00400000`) without conflict because they map to entirely different physical memory locations. Process A's virtual address `0x00400000` might map to physical RAM byte `0x01234000`, while Process B's same virtual address maps to `0x05678000`. From each process's perspective, it owns that memory. In physical reality, they're in completely different places.

## Pages: The Unit of Memory Management

The OS doesn't track virtual-to-physical mappings one byte at a time — that would require enormous bookkeeping overhead. Instead, it works in chunks called **pages**.

A page is a fixed-size block of memory — on most modern systems, 4 kilobytes (4096 bytes). Both virtual and physical memory are divided into page-sized chunks. Physical pages are called **page frames**. The translation from virtual to physical always works at page granularity: a virtual page number maps to a physical page frame number.

For example, if your process accesses virtual address `0x00401234`, the CPU computes the virtual page number as `0x00401234 / 4096 = 0x401` (page 1025) and the offset within the page as `0x00401234 % 4096 = 0x234` (byte 564 within the page). It then looks up page 1025 in the process's page table to find which physical frame it maps to.

## The Page Table

For each process, the OS maintains a **page table** — a data structure that records which virtual pages are mapped to which physical frames, and with what permissions (readable, writable, executable).

The page table entry for each virtual page contains:

- The **physical frame number** — which physical page this maps to
- A **present bit** — whether the page is currently in RAM or on disk
- **Permission bits** — read, write, execute
- **Dirty bit** — whether the page has been modified since it was loaded
- **Accessed bit** — whether the page has been recently used (helps the OS decide what to evict)

## The MMU: Hardware Translation

Translating every memory access through software would be impossibly slow. The **Memory Management Unit (MMU)** is dedicated hardware inside the CPU that performs page table lookups automatically, on every memory access, in nanoseconds.

The MMU uses a hardware cache called the **Translation Lookaside Buffer (TLB)** to cache recent virtual-to-physical translations. A TLB hit (the translation is cached) is extremely fast — just a single cycle overhead. A TLB miss requires a full page table walk, which takes dozens of cycles. This is why programs with good memory locality (accessing nearby addresses) are much faster than programs that jump randomly through memory — they generate far more TLB hits.

## Page Faults: On-Demand Loading

What happens when a process accesses a virtual page that isn't currently in RAM — perhaps because it was never loaded, or because the OS swapped it out to disk to free memory for another process?

The MMU detects that the page isn't present (the present bit is 0) and raises a **page fault** — a hardware exception that hands control to the OS's page fault handler. The kernel then:

1. Determines whether the access is legitimate (is this a valid virtual address the process is allowed to use?)
2. If invalid: delivers a segmentation fault signal (SIGSEGV) — the program has accessed memory it shouldn't
3. If valid: finds a free physical frame (or evicts an existing page to disk to make room), loads the required page from disk into the frame, updates the page table, and resumes the process as if nothing happened

Page faults are transparent to the process. The application never knows its memory was loaded on demand. This mechanism enables several powerful features: programs larger than physical RAM can run (using disk as overflow), copy-on-write forking (parent and child share pages until one writes, triggering a fault and a copy), and memory-mapped files.

## Why This Matters for Performance

Virtual memory is not free. Every context switch between processes must either flush the TLB or manage process-specific TLB tags. Page faults are expensive — a page fault that requires reading from disk can take milliseconds, compared to nanoseconds for a cached memory access. Programs that cause many page faults (exhibiting "thrashing") perform catastrophically poorly.

Understanding virtual memory helps you understand why: working set size matters enormously for performance, why large random memory accesses are slow, why `mmap` can be faster than `read` for large files, and what "swap" means when your system's memory is full.
