# Cloud Fundamentals: IaaS, PaaS, and SaaS

Before cloud computing existed, running software in production meant buying physical servers, racking them in a data center, installing an operating system, and then finally deploying your application. Capacity planning was a guessing game — buy too little hardware and your service falls over under load; buy too much and you're paying for idle steel. Cloud computing changed this by turning infrastructure into a metered utility. You pay for what you use, scale up in minutes instead of months, and let someone else worry about the physical hardware.

But "the cloud" is not one thing. It is a spectrum of services, each offering a different trade-off between control and convenience.

## The Three Service Models

**Infrastructure as a Service (IaaS)** gives you raw compute, storage, and networking. You get virtual machines, raw disks, and network interfaces — and you are responsible for everything above that layer: the operating system, the runtime, the application, the security patches. AWS EC2, Google Compute Engine, and Azure VMs are classic examples. IaaS gives maximum control and maximum responsibility.

**Platform as a Service (PaaS)** handles the infrastructure and runtime for you. You bring your code; the platform handles deployment, scaling, patching, and often the database. Heroku, Google App Engine, and AWS Elastic Beanstalk are PaaS offerings. The trade-off: you get speed and simplicity, but you operate within the constraints the platform imposes. If your app needs a custom kernel module, PaaS probably cannot help you.

**Software as a Service (SaaS)** is the fully managed end of the spectrum — you consume an application entirely. Gmail, Salesforce, and GitHub are SaaS. As a developer, you may build on top of SaaS APIs (sending email via SendGrid, processing payments via Stripe), but you control nothing about how the underlying software runs.

The right model depends on your team's size, expertise, and the value of operational control. A two-person startup usually wants PaaS to move fast. A bank running compliance-sensitive workloads might prefer IaaS for full auditability.

## Regions and Availability Zones

Cloud providers divide the world into **regions** — geographically distinct locations like `us-east-1` (Northern Virginia), `eu-west-1` (Ireland), or `ap-southeast-1` (Singapore). Choosing a region affects latency to your users, data residency compliance (GDPR requires EU data to stay in the EU), and service availability.

Within each region, providers operate multiple **availability zones (AZs)**. An AZ is an independent data center — or small cluster of data centers — with its own power supply, cooling, and physical network connections. AZs within a region are connected by low-latency fiber but are far enough apart that a fire, flood, or power failure in one AZ cannot take down another.

The practical implication: if you run all your servers in a single AZ and that AZ loses power, your service goes offline. If you spread your servers across three AZs and use a load balancer to distribute traffic, losing one AZ costs you at most a third of capacity — and your load balancer routes around it automatically. Multi-AZ deployment is the baseline for any production system that needs to stay up.

## Why This Layering Matters

The IaaS/PaaS/SaaS distinction is not just marketing — it determines where your operational responsibility ends. Every layer you offload to a cloud provider is a layer you no longer need to patch, monitor, or wake up at 3 a.m. to fix. But it is also a layer you no longer control. Understanding this contract is essential before you decide where to run a system.

When you design a cloud architecture, the first questions you ask are: What region? What service model? How many AZs? The answers cascade into everything else — cost, latency, compliance, and how many on-call engineers you need.

constructor(azNames, capacityPerAZ) {
this.capacity = capacityPerAZ;
this.azNames = azNames;
this.azUsage = {};
this.roundRobinIndex = 0;
for (const az of azNames) {
this.azUsage[az] = [];
}
}

place(serviceName) {
const startIndex = this.roundRobinIndex;
for (let i = 0; i < this.azNames.length; i++) {
const idx = (startIndex + i) % this.azNames.length;
const az = this.azNames[idx];
if (this.azUsage[az].length < this.capacity) {
this.azUsage[az].push(serviceName);
this.roundRobinIndex = (idx + 1) % this.azNames.length;
return { success: true, az, service: serviceName };
}
}
return { success: false, reason: "No capacity" };
}

utilization() {
const result = {};
for (const az of this.azNames) {
result[az] = this.azUsage[az].length;
}
return result;
}
}
