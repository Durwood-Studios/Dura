# Compute: VMs, Containers, Serverless, and Edge

Every application needs somewhere to run. The cloud offers four fundamentally different answers to the question "where does my code execute?" — and each answer comes with a different set of trade-offs around startup time, cost, operational complexity, and control. Picking the wrong abstraction can mean paying ten times too much, or suffering latency your users will never forgive.

## Virtual Machines

A **virtual machine (VM)** is an emulated computer running inside a physical host. The cloud provider's hypervisor divides the host's CPU, memory, and storage among multiple VMs, each completely isolated from the others. From your VM's perspective, it behaves exactly like a dedicated physical server — it has its own operating system, its own kernel, its own filesystem.

VMs offer maximum isolation and control. You can install any software, configure any kernel parameter, and run any workload. The cost is overhead: a VM is heavy. It takes minutes to start, consumes gigabytes of RAM just for the OS, and requires you to manage patching, security hardening, and OS upgrades yourself. VMs are the right choice for stateful workloads, workloads requiring specific kernel features, or legacy applications that assume a traditional server environment.

## Containers

A **container** shares the host operating system's kernel while isolating the application's filesystem, network namespace, and process tree. This makes containers dramatically lighter than VMs — they start in seconds, use far less memory, and pack more workloads onto the same hardware.

The trade-off is weaker isolation: all containers on a host share one kernel. A kernel vulnerability could, in principle, allow a container to escape its isolation. For most workloads this is acceptable; for highly sensitive multi-tenant environments, VMs offer stronger guarantees.

Containers have become the dominant unit of deployment in modern cloud infrastructure. Tools like Docker package an application and all its dependencies into a portable image, and platforms like Kubernetes orchestrate thousands of containers across clusters of VMs. The portability of containers — the same image runs the same way on your laptop, in CI, and in production — has transformed how software is shipped.

## Serverless

**Serverless** functions push abstraction further: you provide a function handler, and the platform manages everything else. No VMs, no containers you manage, no servers to patch. You pay only for actual execution time, typically measured in milliseconds. AWS Lambda, Cloudflare Workers, and Vercel Functions all follow this model.

The cost of this convenience is **cold starts**. When no instance of your function is currently running, the platform must provision one — downloading your code, starting a runtime, and initializing your application. This process can take anywhere from a few hundred milliseconds to several seconds, depending on the runtime and the size of your bundle. A cold start on a user-facing request introduces a latency spike that can be noticeable and frustrating.

Strategies to mitigate cold starts include keeping your function bundle small (faster to download and initialize), using "provisioned concurrency" (pre-warming instances at a cost), and choosing runtimes with fast startup characteristics (JavaScript/Python start faster than Java/Scala).

## Edge Functions

**Edge functions** are a variant of serverless that run not in a central cloud region, but on servers distributed close to users — at the "edge" of the network. A user in Tokyo hits an edge node in Tokyo rather than routing to a data center in Virginia. This slashes geographic latency for latency-sensitive work like authentication, routing, and personalization.

Edge functions typically run in stripped-down environments. Most edge platforms (Cloudflare Workers, Vercel Edge, Deno Deploy) do not support the full Node.js API — there is no filesystem access and limited networking. They are designed for lightweight request transformation, not heavy computation.

## Choosing the Right Model

| Model      | Startup                              | Cost Model                    | Control       | Best For                       |
| ---------- | ------------------------------------ | ----------------------------- | ------------- | ------------------------------ |
| VM         | Minutes                              | Per-hour (always-on)          | Full OS       | Stateful apps, legacy          |
| Container  | Seconds                              | Per-CPU/memory (when running) | Application   | Most web services              |
| Serverless | Milliseconds (warm) / Seconds (cold) | Per-invocation                | Function only | Event-driven, sporadic traffic |
| Edge       | Milliseconds (always warm)           | Per-request                   | Very limited  | Auth, routing, personalization |

The decision is rarely black-and-white. Real systems mix models: a container-based API service might call serverless functions for image processing, while edge functions handle authentication at the CDN layer.
