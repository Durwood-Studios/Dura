# Observability: Metrics, Logs, and Traces

A system you cannot observe is a system you cannot operate. When something goes wrong in production — and it will — you need to be able to answer three questions: What happened? Where did it happen? Why did it happen? Observability is the discipline of building systems that can answer these questions without requiring you to redeploy code or guess at the cause.

The field has converged on three complementary types of telemetry: **metrics**, **logs**, and **traces**. Each answers different questions. The most effective observability stacks use all three.

## Metrics

**Metrics** are numeric measurements collected over time. They are optimized for answering "what is the current state and how has it changed?" — questions like: What is the current request rate? What is the 99th percentile response time? How much memory is this service using? How many errors occurred in the last five minutes?

Metrics have three fundamental types:

- **Counter**: A value that only increases. Total requests served, total errors, total bytes transferred. Counters are reset to zero only when a process restarts. You usually query rate-of-change rather than absolute value: "requests per second" rather than "total requests."
- **Gauge**: A value that can go up or down. Current memory usage, current queue depth, current number of active connections. A gauge represents a snapshot of a value at a point in time.
- **Histogram**: A distribution of observations across configurable buckets. Response time histograms let you compute percentiles: "50% of requests finished in under 100ms, 99% finished in under 500ms." Histograms are the right tool for measuring latency, request sizes, and similar bounded distributions.

**Prometheus** is the dominant open-source metrics system. Applications expose a `/metrics` HTTP endpoint in Prometheus's text format, and Prometheus scrapes it on a configurable interval. Prometheus stores metrics in a time-series database and provides PromQL — a query language for computing rates, percentiles, aggregations, and alerts. Grafana is typically paired with Prometheus to visualize these metrics as dashboards.

## Logs

**Logs** are time-stamped, human-readable (or machine-parseable) records of discrete events. "User 12345 authenticated successfully." "Database query took 3.2s — exceeded threshold." "Failed to parse request body: unexpected EOF."

Logs are the best tool for debugging specific incidents — they carry context that metrics cannot. But logs have costs: they generate enormous volume, require storage and indexing infrastructure, and can expose sensitive data if not filtered carefully.

The evolution of logging has moved from unstructured text to structured JSON. A structured log event like `{"level":"error","user_id":12345,"duration_ms":3200,"msg":"query timeout"}` is machine-parseable — you can filter, aggregate, and alert on specific fields without fragile regex parsing.

## Distributed Traces

In a microservices system, a single user request may traverse a dozen services: API gateway → auth service → product service → database → cache → recommendation service → response. If the overall request is slow, which service is responsible?

**Distributed traces** answer this question by following a request across service boundaries. Each service in the call chain records a **span** — a timed unit of work — and spans are linked by a shared **trace ID** propagated in request headers. When a trace is complete, you can see the entire call tree: which services were called, in what order, how long each one took, and where time was actually spent.

Without traces, debugging latency in a microservices system is guesswork. With traces, you can pinpoint in seconds: "The recommendation service is adding 800ms to every checkout request because it's making N+1 database queries."

## OpenTelemetry

Historically, every observability vendor had its own SDK. Instrumenting your application with Datadog's library meant a complete rewrite to switch to New Relic. **OpenTelemetry** (OTel) solves this by providing a vendor-neutral API and SDK for emitting metrics, logs, and traces. You instrument your code once against the OTel API, and route the telemetry to any backend — Prometheus, Jaeger, Honeycomb, Datadog — by swapping the exporter, not the instrumentation.

OpenTelemetry has become the standard. All major cloud providers and observability vendors now support it as an input format.

## The Three Pillars Together

The three pillars are complementary, not redundant. A common workflow: an alert fires because Prometheus detects that the error rate exceeded 1% (metrics). An engineer opens Grafana, identifies the spike started at 14:32 (metrics). They look at log entries around 14:32 and find "connection pool exhausted" errors appearing (logs). They pull a trace from that time window and see that the database layer is timing out because the connection pool is empty (traces). Root cause identified in under five minutes — because all three pillars were in place.

  solution={`class MetricsCollector {
constructor() {
this.metrics = {};
}

defineCounter(name) {
this.metrics[name] = { type: "counter", value: 0 };
}

defineGauge(name, initialValue = 0) {
this.metrics[name] = { type: "gauge", value: initialValue };
}

defineHistogram(name, buckets) {
const bucketCounts = {};
for (const b of buckets) bucketCounts[b] = 0;
bucketCounts["+Inf"] = 0;
this.metrics[name] = { type: "histogram", buckets: buckets.slice().sort((a,b)=>a-b), bucketCounts, count: 0, sum: 0 };
}

increment(name, amount = 1) {
const m = this.metrics[name];
if (!m || m.type !== "counter") throw new Error(\`Counter '\${name}' not found\`);
m.value += amount;
}

setGauge(name, value) {
const m = this.metrics[name];
if (!m || m.type !== "gauge") throw new Error(\`Gauge '\${name}' not found\`);
m.value = value;
}

observe(name, value) {
const m = this.metrics[name];
if (!m || m.type !== "histogram") throw new Error(\`Histogram '\${name}' not found\`);
for (const bound of m.buckets) {
if (value <= bound) m.bucketCounts[bound]++;
}
m.bucketCounts["+Inf"]++;
m.count++;
m.sum += value;
}

query(name) {
const m = this.metrics[name];
if (!m) return null;
if (m.type === "counter" || m.type === "gauge") return m.value;
return {
buckets: m.bucketCounts,
count: m.count,
sum: m.sum,
avg: m.count > 0 ? (m.sum / m.count).toFixed(2) : 0
};
}
}

const metrics = new MetricsCollector();

metrics.defineCounter("http_requests_total");
metrics.defineGauge("active_connections", 0);
metrics.defineHistogram("request_duration_ms", [10, 50, 100, 500, 1000]);

metrics.increment("http_requests_total");
metrics.increment("http_requests_total", 5);
metrics.setGauge("active_connections", 42);
metrics.observe("request_duration_ms", 8);
metrics.observe("request_duration_ms", 45);
metrics.observe("request_duration_ms", 120);
metrics.observe("request_duration_ms", 480);
metrics.observe("request_duration_ms", 1200);

console.log("Requests:", metrics.query("http_requests_total"));
console.log("Connections:", metrics.query("active_connections"));
console.log("Duration histogram:", metrics.query("request_duration_ms"));
