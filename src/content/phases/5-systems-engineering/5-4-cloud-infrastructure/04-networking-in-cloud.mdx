# Networking in the Cloud: VPCs, Subnets, and Security Groups

When you deploy a workload to the cloud, it doesn't simply appear on the public internet. It lives inside a carefully constructed private network that you design and control. Understanding cloud networking is essential because the network is your first and most important line of defense — and because misconfigured networking is one of the most common sources of security breaches and outages in production systems.

## Virtual Private Clouds

A **VPC (Virtual Private Cloud)** is an isolated, private network that you own within the cloud provider's infrastructure. Think of it as your own section of a data center — a network space where you control the IP address range, routing rules, and connectivity to the outside world. AWS, GCP, and Azure all use this model.

When you create a VPC, you specify a CIDR block — a range of private IP addresses your resources will use. A common choice is `10.0.0.0/16`, which gives you 65,536 addresses. Resources inside your VPC can communicate freely using private IP addresses. Nothing from the public internet can reach your VPC unless you explicitly configure a path for it to do so.

VPCs solve an important problem: in the early days of cloud computing, all EC2 instances were placed on a shared, flat network. Any instance could potentially reach any other instance. VPCs enforce strict isolation between customers — your network is completely separate from every other customer's network, even on the same physical hardware.

## Subnets: Public and Private

A **subnet** is a subdivision of your VPC's IP address space, tied to a specific availability zone. A VPC spans all AZs in a region; a subnet lives in exactly one AZ.

The critical design choice is whether a subnet is **public** or **private**. A public subnet has a route to an **Internet Gateway**, meaning resources in it can receive inbound traffic from the internet (if security groups allow it) and can reach the internet directly. A private subnet has no direct internet route — resources in it can only communicate with other resources inside the VPC, or reach the internet indirectly.

A well-designed architecture uses both: your web servers or load balancers sit in public subnets (they need to accept incoming requests), while your application servers and databases sit in private subnets (they should never be directly reachable from the internet). Your database should be in a private subnet in every production system, without exception.

## Security Groups

A **security group** is a stateful firewall that controls what traffic is allowed to reach a resource. Every EC2 instance, RDS database, and Lambda function is associated with one or more security groups.

Security group rules are simple: an inbound rule says "allow traffic from source X on port Y using protocol Z." A common configuration for a web server: allow inbound TCP on port 443 from `0.0.0.0/0` (the entire internet) for HTTPS. A database security group might allow inbound TCP on port 5432 only from the security group of the application tier — meaning only the app servers can reach the database, and no other source can even attempt a connection.

Security groups are **stateful** — if you allow inbound traffic from a source, the response traffic is automatically allowed back, without needing a separate outbound rule. This stateful behavior is why security groups are simpler to reason about than traditional stateless packet-filter firewalls.

## NAT Gateways

Private subnet resources sometimes need to reach the internet — to download a software package, call an external API, or send metrics to a monitoring service — without being reachable from the internet themselves.

A **NAT gateway** (Network Address Translation) solves this. It sits in a public subnet and acts as an intermediary: private subnet resources route outbound internet traffic through the NAT gateway, which translates their private IPs to its own public IP. Responses come back to the NAT gateway and are forwarded to the originating private resource. The outside world only ever sees the NAT gateway's IP — the private resource remains invisible.

## CDN: Content Delivery Networks

A **CDN** (Content Delivery Network) is a globally distributed caching layer. CDNs like CloudFront (AWS), Fastly, or Cloudflare operate hundreds of **edge nodes** around the world. When you configure a CDN in front of your origin server, users fetch content from the nearest edge node rather than from your server in a single region.

A CDN caches static assets (images, CSS, JavaScript, video) at the edge. When a user in Tokyo requests an image served from a CDN, they hit a Tokyo edge node that already has the image cached — sub-10ms latency. Without a CDN, that request would travel all the way to a Virginia data center and back — potentially 150-200ms.

CDNs also provide DoS protection at the edge, TLS termination, and HTTP/2 and HTTP/3 acceleration. For any public-facing web application, a CDN in front of your origin is standard practice.

constructor(name) {

this.inboundRules = [];

addRule(protocol, port, source) {
this.inboundRules.push({ protocol, port, source });
}

allows(protocol, port, sourceIP) {
return this.inboundRules.some(rule => {
if (rule.protocol !== protocol) return false;
if (rule.port !== port) return false;
if (rule.source === "0.0.0.0/0") return true;
if (rule.source === "10.0.0.0/16" && sourceIP.startsWith("10.0.")) return true;
if (rule.source === sourceIP) return true;
return false;
});
}
}
