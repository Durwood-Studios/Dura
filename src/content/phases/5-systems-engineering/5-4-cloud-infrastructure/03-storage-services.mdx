# Storage Services: Object, Block, and File

Data outlives code. An application you wrote five years ago might be rewritten in a completely different language today, but the data it generated still needs to be accessible, durable, and retrievable. Cloud providers offer three fundamentally different storage abstractions — object, block, and file storage — each designed for a different category of use case.

Understanding the differences is not academic: choosing the wrong storage type can result in an architecture that doesn't scale, costs dramatically more than necessary, or simply doesn't work for your access patterns.

## Object Storage

**Object storage** stores data as discrete objects in a flat namespace. Each object has a unique key (a string identifier), a blob of data (bytes of any size), and metadata (key-value pairs describing the object). There are no directories, no hierarchies, no random-access reads to byte offsets. You interact with objects through simple operations: `put`, `get`, `delete`, and `list`.

**Amazon S3** (Simple Storage Service) is the canonical example of object storage. An S3 object lives in a bucket, is identified by its key, and can be retrieved via HTTP. A key like `images/user-avatars/12345.png` looks like a path, but it isn't — S3 has no filesystem. The slash is just part of the key string; S3 lets you filter by prefix to simulate directory-like behavior.

Object storage is designed for three things: massive scale, high durability, and cheap cost. S3 is designed to survive the loss of an entire availability zone without data loss. Under the hood, your objects are replicated across at least three AZs within a region. AWS quotes 99.999999999% (eleven nines) durability — meaning if you store ten million objects, you might expect to lose one over 10,000 years.

Object storage is the right choice for: static assets (images, videos, CSS, JavaScript bundles), backups and archives, data lake raw files, log aggregation, and any file that is written once and read many times. It is the wrong choice for anything that needs random byte-level writes, low-latency random reads, or filesystem semantics (like a database's data files).

## Block Storage

**Block storage** exposes raw storage as a device — similar to a hard drive connected to your VM. You format it with a filesystem, and your OS can read and write arbitrary byte ranges at any offset. It behaves exactly like local disk but lives on the network.

**Amazon EBS** (Elastic Block Store) is the AWS block storage product. An EBS volume attaches to exactly one EC2 instance at a time (with one exception: `io2` volumes support multi-attach in limited scenarios). It is persistent — data survives a VM reboot or termination as long as you don't delete the volume.

Block storage is the right choice for databases, operating system boot volumes, any workload that needs low-latency random reads/writes, and applications that require filesystem semantics. Most managed databases (RDS, self-hosted PostgreSQL) run on block storage under the hood.

The trade-off: block storage is more expensive than object storage per gigabyte, and it doesn't scale horizontally — you can't easily stripe across hundreds of volumes without specialized software.

## File Storage

Cloud **file storage** (AWS EFS, Azure Files, Google Filestore) provides a network filesystem that multiple instances can mount simultaneously. If your architecture has ten application servers that all need to read and write to the same shared directory, file storage solves that problem cleanly. Neither object storage nor block storage supports concurrent multi-instance access with POSIX semantics.

File storage is typically used for: shared configuration, content management systems where multiple web servers need the same media files, or any workflow using tools that assume a POSIX filesystem.

## Choosing the Right Storage

| Type        | Access Pattern                 | Cost     | Durability     | Use Case                    |
| ----------- | ------------------------------ | -------- | -------------- | --------------------------- |
| Object (S3) | Key-value get/put              | Very low | Extremely high | Assets, backups, data lakes |
| Block (EBS) | Random byte-level              | Medium   | High           | Databases, boot volumes     |
| File (EFS)  | POSIX filesystem, multi-client | Higher   | High           | Shared app storage          |

The most common mistake: using a database to store binary blobs (images, videos, documents). Databases are optimized for structured queries, not large binary transfers. The correct pattern is to store the blob in S3 and store the S3 key in the database.

constructor(bucketName) {
this.bucketName = bucketName;
this.objects = {};
}

put(key, data, metadata = {}) {
this.objects[key] = {
data,
metadata,
size: data.length,
createdAt: new Date().toISOString(),
};
return { success: true, key, size: data.length };
}

get(key) {
const obj = this.objects[key];
if (!obj) return null;
return { key, ...obj };
