## Why this matters

Every app you build eventually needs to run somewhere — but "somewhere" is not a single answer. Choosing the wrong deployment model wastes money, burns engineering time, and creates scaling headaches before you even have users. Understanding the spectrum from fully managed to fully raw lets you match infrastructure to the actual needs of your project rather than defaulting to whatever you've heard of.

## The infrastructure spectrum

Deployment options sit on a dial between "someone else manages everything" and "you manage everything." Moving left saves engineering time; moving right gives control.

```
SaaS → PaaS → CaaS → IaaS → Bare Metal
       ↑ most teams live here ↑
```

**SaaS** (Software as a Service) means you use someone else's product directly — think Stripe, Sendgrid, Supabase. There's nothing to deploy; you call an API.

**PaaS** (Platform as a Service) is where most applications start. You give the platform your code, and it handles the OS, runtime, dependencies, networking, scaling, and TLS certificates. Examples: Vercel, Railway, Render, Heroku. Deployment is as simple as `git push`. You pay for this simplicity with less control over the underlying machine.

**CaaS** (Container as a Service) takes a Docker image from you and runs it, managing the infrastructure around it. Examples: Fly.io, Google Cloud Run, AWS ECS. You get more control — any runtime, any dependencies, custom ports — while still avoiding VM management.

**IaaS** (Infrastructure as a Service) gives you a virtual machine. You decide the OS, install your runtime, configure your firewall, set up TLS, and handle everything above the hypervisor. Examples: AWS EC2, GCP Compute Engine, DigitalOcean Droplets. This is the right tool when you have specific hardware requirements, legacy software, or need to squeeze maximum performance from known workloads.

**Bare metal** is a physical server you own or rent. No virtualization overhead, maximum performance — but you are the data center now.

## Serverless and edge

Two deployment models sit outside the traditional spectrum because they trade the concept of "a running server" for something different.

**Serverless** (AWS Lambda, Vercel Functions, Netlify Functions) runs your code in response to events — an HTTP request, a queue message, a scheduled trigger. There is no persistent process. Each invocation spins up in isolation and disappears when it finishes. The benefits: you pay only for what you use, and scale is theoretically infinite. The catch: **cold starts**. When no recent invocation has occurred, the platform must initialize your runtime before running your code. This adds 100–500ms of latency on the first request after a quiet period. Keeping functions small and avoiding heavy initialization mitigates this.

**Edge** (Cloudflare Workers, Vercel Edge Functions, Deno Deploy) runs JavaScript at CDN points of presence around the world — 200+ locations. Instead of your server in `us-east-1` being 200ms from Tokyo, your code runs in Tokyo. The tradeoff: edge runtimes are deliberately constrained. No Node.js APIs, limited memory, short execution time. They excel at response transformation, auth checks, A/B testing, and request routing — not database-heavy business logic.

## Choosing the right model

| Situation                             | Recommendation                        |
| ------------------------------------- | ------------------------------------- |
| New app, unknown scale                | PaaS (Vercel, Railway, Render)        |
| Need WebSockets or persistent process | CaaS (Fly.io, Cloud Run)              |
| Sporadic traffic, event-driven        | Serverless (Lambda, Vercel Functions) |
| Global, latency-sensitive routing     | Edge (Cloudflare Workers)             |
| Specific OS/hardware needs            | IaaS (EC2, Compute Engine)            |
| Maximum performance, known load       | Bare metal                            |

Start with PaaS. It handles 90% of use cases, costs you almost no ops time, and you can migrate to CaaS or IaaS later when you have a concrete reason. Premature infrastructure optimization is as real a problem as premature code optimization.

## Cost models matter

PaaS typically charges a flat monthly rate plus bandwidth/compute overages. Serverless charges per invocation and per GB-second (memory × duration). IaaS charges by instance-hour regardless of load. For low-traffic apps, serverless is often the cheapest. For high-throughput apps with predictable load, flat-rate PaaS or reserved IaaS instances win. The sandbox exercise below lets you estimate and compare costs across models.

{`// Deployment cost estimator
// Calculate estimated monthly cost across PaaS, Serverless, and IaaS models

function estimateMonthlyCost(config) {
// config: { type, requestsPerMonth, avgResponseTimeMs, memorySizeMB, instances }
const { type, requestsPerMonth, avgResponseTimeMs, memorySizeMB, instances = 1 } = config;

switch (type) {
case 'paas': {
// Flat rate per instance + bandwidth overage
const basePerInstance = 7; // $7/month per dyno/service
const includedRequests = 1_000_000;
const overageRequests = Math.max(0, requestsPerMonth - includedRequests);
const overageCost = (overageRequests / 1_000_000) _ 0.50;
const estimatedCost = (basePerInstance _ instances) + overageCost;
return {
estimatedCost: parseFloat(estimatedCost.toFixed(2)),
breakdown: {
baseInstances: basePerInstance \* instances,
requestOverage: parseFloat(overageCost.toFixed(2)),
},
};
}

    case 'serverless': {
      // Per-invocation + per-GB-second
      const invocationCost = (requestsPerMonth / 1_000_000) * 0.20;
      const gbSeconds = (memorySizeMB / 1024) * (avgResponseTimeMs / 1000) * requestsPerMonth;
      const computeCost = gbSeconds * 0.0000166667;
      const freeTierInvocations = 1_000_000;
      const freeTierGbSeconds = 400_000;
      const billableInvocations = Math.max(0, requestsPerMonth - freeTierInvocations);
      const billableGbSeconds = Math.max(0, gbSeconds - freeTierGbSeconds);
      const estimatedCost = (billableInvocations / 1_000_000) * 0.20 + billableGbSeconds * 0.0000166667;
      return {
        estimatedCost: parseFloat(estimatedCost.toFixed(2)),
        breakdown: {
          invocations: parseFloat(((billableInvocations / 1_000_000) * 0.20).toFixed(4)),
          computeGbSeconds: parseFloat((billableGbSeconds * 0.0000166667).toFixed(4)),
          totalGbSeconds: parseFloat(gbSeconds.toFixed(0)),
        },
      };
    }

    case 'iaas': {
      // Per instance-hour (t3.micro = ~$0.0104/hr)
      const hourlyRatePerInstance = 0.0104;
      const hoursPerMonth = 730;
      const estimatedCost = hourlyRatePerInstance * hoursPerMonth * instances;
      return {
        estimatedCost: parseFloat(estimatedCost.toFixed(2)),
        breakdown: {
          instanceHours: hoursPerMonth * instances,
          hourlyRate: hourlyRatePerInstance,
          instances,
        },
      };
    }

    default:
      throw new Error(\`Unknown deployment type: \${type}\`);

}
}

// Test scenarios
const lowTraffic = { requestsPerMonth: 100_000, avgResponseTimeMs: 200, memorySizeMB: 512 };
const highTraffic = { requestsPerMonth: 10_000_000, avgResponseTimeMs: 150, memorySizeMB: 256 };

console.log("=== Low traffic (100K req/month) ===");
console.log("PaaS: ", estimateMonthlyCost({ ...lowTraffic, type: 'paas', instances: 1 }));
console.log("Serverless:", estimateMonthlyCost({ ...lowTraffic, type: 'serverless' }));
console.log("IaaS: ", estimateMonthlyCost({ ...lowTraffic, type: 'iaas', instances: 1 }));

console.log("\\n=== High traffic (10M req/month) ===");
console.log("PaaS: ", estimateMonthlyCost({ ...highTraffic, type: 'paas', instances: 3 }));
console.log("Serverless:", estimateMonthlyCost({ ...highTraffic, type: 'serverless' }));
console.log("IaaS: ", estimateMonthlyCost({ ...highTraffic, type: 'iaas', instances: 2 }));`}

</SandboxExercise>

## What you learned

- The infrastructure spectrum runs from SaaS (fully managed) through PaaS, CaaS, IaaS, to bare metal (fully raw)
- PaaS handles OS, runtime, scaling, and TLS — you just push code
- Serverless has no persistent process; you pay per invocation but must manage cold starts
- Edge runs JavaScript at CDN locations globally for low-latency routing and transformation
- Start with PaaS and migrate when you have a concrete reason to need more control
