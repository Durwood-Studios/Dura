## Why this matters

Your application will fail in production. Servers crash, third-party APIs go down, a bad deploy introduces a bug that affects 1% of requests. The question is not whether these things happen — it is how quickly you find out and how much information you have when you do. Good observability turns a catastrophe into an investigation with a clear resolution path.

## The three pillars of observability

**Logs** are the record of what happened. Each event — a request received, a database query executed, an error thrown — produces a log entry. Logs are your primary debugging tool.

**Metrics** are aggregations over time: how many requests per second, what percentage are errors, what is the 95th-percentile latency. Metrics tell you that something is wrong; logs tell you why.

**Traces** connect a single user request across multiple services. When a request flows through an API gateway, a backend service, and a database, a trace shows the entire journey with timing for each segment.

Most small teams start with logs and basic metrics. Add distributed tracing when you have multiple services and need to understand cross-service latency.

## Structured logging

Unstructured logs are `console.log("User 123 logged in at " + new Date())`. They are human-readable but machine-hostile — you cannot filter them, aggregate them, or query them efficiently.

Structured logs are JSON:

```json
{
  "timestamp": "2025-04-09T14:23:01.442Z",
  "level": "info",
  "service": "api",
  "requestId": "req_abc123",
  "userId": "usr_456",
  "message": "User logged in",
  "durationMs": 42
}
```

Every field is queryable. In your log aggregator, you can instantly answer: "Show me all errors for user `usr_456` in the last hour." With unstructured logs, you'd need regex and luck.

**Log levels** signal the severity of an event:

| Level   | Use when                               | Production behavior         |
| ------- | -------------------------------------- | --------------------------- |
| `debug` | Internal details useful in development | Usually suppressed          |
| `info`  | Normal, notable events                 | Stored, sampled             |
| `warn`  | Something unexpected but recoverable   | Stored, alerted if frequent |
| `error` | Something failed and needs attention   | Stored, alerted immediately |

In production, set your log level to `info` or `warn`. Logging every debug message generates enormous volumes and buries the signal in noise.

## Log aggregators

A log aggregator collects logs from all your service instances, indexes them, and provides search and alerting:

- **Datadog**: full observability platform — logs, metrics, traces, APM
- **Grafana + Loki**: open-source; Loki stores logs, Grafana visualizes them
- **AWS CloudWatch**: built-in for AWS-hosted services
- **Axiom**: developer-friendly, generous free tier, built for modern cloud apps

Most platforms accept logs via HTTP or as output to stdout (12-factor apps log to stdout; the platform captures and forwards them).

## Health check endpoints

A health check is an HTTP endpoint your infrastructure calls to verify the service is running correctly. There are two types:

**Liveness** (`/health`): Is the process alive? This is the simplest check. If it fails, restart the process.

```javascript
// Express — liveness check
app.get("/health", (req, res) => {
  res.json({ status: "ok", uptime: process.uptime() });
});
```

**Readiness** (`/ready`): Is the process ready to handle traffic? This checks dependencies — database connectivity, external service availability. Traffic should not route to a process that cannot reach its database.

```javascript
// Express — readiness check
app.get("/ready", async (req, res) => {
  try {
    await pool.query("SELECT 1"); // verify DB connection
    res.json({ status: "ready", db: "connected" });
  } catch (err) {
    res.status(503).json({ status: "not ready", db: "disconnected" });
  }
});
```

Kubernetes uses both: liveness to decide whether to restart a pod, readiness to decide whether to route traffic to it. Railway and Fly.io use similar concepts in their health check configuration.

## Uptime monitoring

Uptime monitors call your health endpoint on a schedule (every 30 seconds to 5 minutes) from outside your infrastructure. If the endpoint fails to respond, you get an alert.

Popular options:

- **Better Uptime**: modern UI, on-call scheduling, status pages
- **Uptime Robot**: generous free tier, 5-minute check intervals
- **Cronitor**: good for both uptime and cron job monitoring

Set up an uptime monitor before your first real user. Knowing your service went down 3 hours before you noticed is bad. Getting paged within 2 minutes is normal operations.

## Error tracking with Sentry

Log aggregators are great for searching known events. Sentry is for capturing the unknown — exceptions that escape your try/catch blocks, unhandled promise rejections, client-side JavaScript errors.

Sentry captures the full stack trace, the request context, the user who experienced the error, and the breadcrumb trail of events leading up to it. It groups duplicate errors so you see "this error happened 847 times in the last hour" rather than 847 separate alerts.

```javascript
import * as Sentry from "@sentry/node";

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV,
  tracesSampleRate: 0.1, // trace 10% of requests
});

// Automatically captures unhandled exceptions
// Manually capture with context:
try {
  await processOrder(order);
} catch (err) {
  Sentry.captureException(err, { extra: { orderId: order.id } });
  throw err;
}
```

function log(level, message, meta = {}) {

      timestamp: new Date().toISOString(),
      level,
      service,
      message,
      ...meta,
    };

info: (msg, meta) => log('info', msg, meta),

## What you learned

- Observability rests on three pillars: logs (what happened), metrics (trends over time), traces (cross-service request flow)
- Structured logging uses JSON with consistent fields, making logs searchable and filterable by machines
- Log levels (debug, info, warn, error) let you control signal-to-noise ratio — suppress debug in production
- Liveness checks confirm a process is alive; readiness checks confirm it can serve traffic
- Sentry captures and groups unhandled exceptions with full stack traces and user context
