## Why this matters

Manual deployments are how bugs reach production at 5pm on a Friday. CI/CD — Continuous Integration and Continuous Deployment — is the practice of automating the path from code change to running software. When you get it right, every merged pull request either deploys successfully or gets caught by automated tests before it can do damage. It turns deployment from a risky event into a routine non-event.

## Continuous Integration vs Continuous Deployment

These terms are related but distinct, and the "CD" abbreviation covers two different things:

**Continuous Integration (CI)** is the practice of merging code frequently — ideally multiple times per day — and running an automated suite of tests on every merge. The goal is to catch integration problems early, when they are cheap to fix. The longer branches diverge, the more painful the eventual merge.

**Continuous Delivery** means every build that passes CI is in a deployable state. A human still decides when to deploy, but the act of deploying is a single button press, not a ceremony.

**Continuous Deployment** goes one step further: every passing build deploys automatically to production without human intervention. This requires high confidence in your test suite. Most teams reach Continuous Delivery; the subset with robust testing reach Continuous Deployment.

## The pipeline stages

A CI/CD pipeline is a sequence of automated steps that your code passes through. A typical Node.js pipeline looks like this:

```yaml
trigger (push to any branch)
└── install dependencies
└── lint (ESLint)
└── typecheck (tsc --noEmit)
└── test (jest / vitest)
└── build (next build / tsc)
└── deploy (if branch = main)
```

Each stage gates the next. If lint fails, tests never run — you get fast feedback and don't waste compute.

## GitHub Actions anatomy

GitHub Actions is the most common CI/CD platform for open-source and small-to-medium teams. A **workflow** is a YAML file in `.github/workflows/`. It contains one or more **jobs**, each of which contains a sequence of **steps**.

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, dev]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm # cache node_modules between runs

      - run: npm ci # clean install — never npm install in CI
      - run: npm run lint
      - run: npm run typecheck
      - run: npm run test

  deploy:
    needs: test # only runs if test job passes
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      # trigger Railway or Vercel deploy hook
      - run: curl -X POST "${{ secrets.DEPLOY_HOOK_URL }}"
```

Key concepts from the above:

- **`on`**: what triggers the workflow. `push` and `pull_request` are most common.
- **`jobs`**: multiple jobs run in **parallel by default**. Use `needs:` to express dependencies between them.
- **`steps`**: run sequentially within a job. Each step is either a reusable `uses: action@version` or a shell `run:` command.
- **`secrets`**: sensitive values stored in the GitHub repo settings, referenced as `${{ secrets.MY_SECRET }}`. Never hardcode API keys in workflow files.
- **`cache:`** in `setup-node` stores `node_modules` between runs, cutting install time from 60s to under 5s on cache hits.

## Matrix builds

When you need to verify your code works across multiple environments, matrix builds run the same job in parallel for each combination:

```yaml
strategy:
  matrix:
    node-version: [18, 20, 22]
    os: [ubuntu-latest, windows-latest]
```

This creates six parallel jobs. Useful for libraries that need to support multiple Node versions.

## Self-hosted runners

GitHub provides hosted runners (Ubuntu, Windows, macOS) for free up to a limit. For large builds, GPU workloads, or access to private network resources, you can run your own runner on a machine you control. It registers with GitHub and receives jobs just like hosted runners.

## What you learned

- CI runs tests on every push; CD makes every passing build deployable or deployed automatically
- A pipeline gates each stage — lint must pass before tests run, tests before build, build before deploy
- GitHub Actions workflows live in `.github/workflows/` and consist of jobs containing steps
- Jobs run in parallel by default; `needs:` creates sequential dependencies between them
- Secrets are stored in GitHub settings and referenced as `${{ secrets.KEY }}` — never hardcoded
