## Why this matters

Almost every serious application stores data in a relational database. Social networks, banking systems, e-commerce platforms, healthcare records — they all depend on relational databases because the guarantees they provide (integrity, consistency, powerful queries) are not optional for those use cases. Understanding _why_ relational databases exist — not just _how_ to use them — makes you a dramatically better engineer.

## What is a relational database?

A relational database organizes data into **tables** — think of a table as a spreadsheet with a fixed, defined structure. Each table has **columns** (fields) that define what kind of data the table holds, and **rows** (records) that are individual entries.

```sql
-- A users table
CREATE TABLE users (
  id        BIGSERIAL PRIMARY KEY,
  email     TEXT      NOT NULL UNIQUE,
  username  TEXT      NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

Every row in `users` has an `id`, `email`, `username`, and `created_at`. The column definitions are enforced — you cannot store a number in a `TEXT` column, and you cannot leave `email` empty.

The **primary key** is the column (or combination of columns) that uniquely identifies each row. No two rows in `users` can have the same `id`. This is what lets you reliably refer to a specific record.

## Relationships

The "relational" in relational database refers to relationships _between_ tables, not rows within a table. You express these relationships using **foreign keys** — a column in one table that references the primary key of another.

**One-to-many** — the most common relationship. One user can have many posts.

```sql
CREATE TABLE posts (
  id         BIGSERIAL PRIMARY KEY,
  user_id    BIGINT    NOT NULL REFERENCES users(id),
  title      TEXT      NOT NULL,
  body       TEXT      NOT NULL,
  published_at TIMESTAMPTZ
);
```

`user_id` is a foreign key: it must match an `id` in the `users` table. The database _enforces_ this. You cannot insert a post with a `user_id` that doesn't exist, and (by default) you cannot delete a user who has posts.

**Many-to-many** — one post can have many tags; one tag can belong to many posts. You model this with a **join table** (also called a junction table or association table):

```sql
CREATE TABLE tags (
  id   BIGSERIAL PRIMARY KEY,
  name TEXT NOT NULL UNIQUE
);

CREATE TABLE post_tags (
  post_id BIGINT NOT NULL REFERENCES posts(id),
  tag_id  BIGINT NOT NULL REFERENCES tags(id),
  PRIMARY KEY (post_id, tag_id)
);
```

`post_tags` has no data of its own — just two foreign keys. The composite primary key ensures the same post/tag pair can't appear twice.

## ACID: the guarantees that matter

ACID is an acronym for the four properties that relational databases guarantee for transactions:

- **Atomicity** — a transaction either completes entirely or not at all. Transfer $100 between accounts: both the debit and the credit happen, or neither does.
- **Consistency** — every transaction takes the database from one valid state to another. Constraints (NOT NULL, UNIQUE, REFERENCES) are always satisfied.
- **Isolation** — concurrent transactions don't see each other's in-progress changes. Two users checking out the last item in stock at the same moment don't both succeed.
- **Durability** — once a transaction commits, it's permanent. Even if the server crashes immediately after, the data is safe on disk.

These four properties are why you trust your bank's balance. They're also why relational databases are the right choice for anything where data correctness is non-negotiable.

## Normalization: eliminating redundancy

**Normalization** is the process of organizing your tables to eliminate redundant data. The basic idea: store each fact in exactly one place. If a user's email appears in five different tables, updating it requires five updates — and if you miss one, your data is inconsistent.

The three most important normal forms are:

1. **1NF** — each column holds a single atomic value. No comma-separated lists in a column.
2. **2NF** — every non-key column depends on the _whole_ primary key (matters for composite keys).
3. **3NF** — no column depends on another non-key column (no transitive dependencies).

In practice, most well-designed schemas are in 3NF without the engineers thinking about it explicitly. The rule of thumb: if you're duplicating data, ask whether a relationship and a foreign key would serve better.

## PostgreSQL vs MySQL vs NoSQL

**PostgreSQL** is the most feature-rich open-source relational database. It supports JSON columns, arrays, window functions, full-text search, and advanced indexing types. It has a permissive license and a strong community. It's the right default choice for new projects.

**MySQL** (and MariaDB) are older and more widely deployed but have historically lagged PostgreSQL in features and standards compliance. Many legacy applications run on MySQL.

**NoSQL databases** (MongoDB, DynamoDB, Cassandra) offer a fundamentally different trade-off: flexible schemas, horizontal scaling across many machines, and often very high write throughput — but they sacrifice consistency guarantees and relational query power. They're the right choice for specific workloads: event logs, document storage with highly variable shapes, or systems that genuinely need to scale to millions of writes per second.

The hype around NoSQL databases in the 2010s led many teams to adopt them for use cases where a relational database was the better tool. For most applications — user data, orders, content, settings — PostgreSQL handles the load comfortably and gives you data integrity that NoSQL databases require you to enforce in application code.

tables[name] = { schema, rows: [], nextId: 1 };

    const newRow = { id: table.nextId++, ...row };

    // Validate required fields
    for (const [field, rules] of Object.entries(table.schema)) {

        throw new Error(\`Field "\${field}" is required in "\${tableName}"\`);
      }
    }

    table.rows.push(newRow);
    return newRow;

}

function findAll(tableName, where = {}) {
const table = tables[tableName];
if (!table) throw new Error(\`Table "\${tableName}" does not exist\`);

    return table.rows.filter(row =>
      Object.entries(where).every(([field, value]) => row[field] === value)

// on: { t1Field: t2Field }

const t1Rows = findAll(table1Name);
const t2Rows = findAll(table2Name);

    const result = [];

        }

      }
    }
    return result;

## What you learned

- Relational databases organize data into tables with defined schemas — rows are records, columns are fields
- Primary keys uniquely identify rows; foreign keys express relationships between tables
- One-to-many relationships use a foreign key column; many-to-many relationships use a join table
- ACID guarantees (Atomicity, Consistency, Isolation, Durability) are why relational databases are trusted for financial and user data
- Normalization eliminates data redundancy by storing each fact in exactly one place
- PostgreSQL is the most feature-rich open-source relational database and the right default for new projects
