## Why this matters

Everything in this module comes together here: normalization, constraints, indexes, joins, and advanced queries — applied to a real, cohesive schema. Designing and querying a multi-table system from scratch is the skill that separates engineers who know SQL syntax from engineers who can ship a production-ready data layer.

## The blog platform schema

The domain: a blogging platform where users write posts, posts have tags, and readers leave comments.

### The tables

**users** — authors and commenters

```sql
CREATE TABLE users (
  id         BIGSERIAL   PRIMARY KEY,
  email      TEXT        NOT NULL UNIQUE,
  username   TEXT        NOT NULL UNIQUE,
  avatar_url TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_users_email ON users (email);
```

**posts** — articles written by users

```sql
CREATE TABLE posts (
  id           BIGSERIAL   PRIMARY KEY,
  user_id      BIGINT      NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title        TEXT        NOT NULL,
  slug         TEXT        NOT NULL UNIQUE,
  body         TEXT        NOT NULL,
  status       TEXT        NOT NULL DEFAULT 'draft'
                CHECK (status IN ('draft', 'published', 'archived')),
  published_at TIMESTAMPTZ,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_posts_user_id    ON posts (user_id);
CREATE INDEX idx_posts_published  ON posts (published_at DESC)
  WHERE status = 'published';
```

**comments** — reader comments on posts

```sql
CREATE TABLE comments (
  id         BIGSERIAL   PRIMARY KEY,
  post_id    BIGINT      NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  user_id    BIGINT      NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  body       TEXT        NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_comments_post_id ON comments (post_id);
```

**tags** — labels that categorize posts

```sql
CREATE TABLE tags (
  id   BIGSERIAL PRIMARY KEY,
  name TEXT      NOT NULL UNIQUE,
  slug TEXT      NOT NULL UNIQUE
);
```

**post_tags** — the join table for the many-to-many relationship between posts and tags

```sql
CREATE TABLE post_tags (
  post_id BIGINT NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  tag_id  BIGINT NOT NULL REFERENCES tags(id)  ON DELETE CASCADE,
  PRIMARY KEY (post_id, tag_id)
);
```

### Migration file

A production deployment would ship all of the above as a single migration:

```sql
-- 001_create_blog_schema.sql
-- UP: Run all CREATE TABLE statements above (in dependency order: users → posts → tags → comments → post_tags)
-- DOWN: Drop in reverse order
DROP TABLE IF EXISTS post_tags;
DROP TABLE IF EXISTS comments;
DROP TABLE IF EXISTS tags;
DROP TABLE IF EXISTS posts;
DROP TABLE IF EXISTS users;
```

## The five production queries

**Query 1: All published posts with author name**

```sql
SELECT
  posts.id,
  posts.title,
  posts.slug,
  posts.published_at,
  users.username AS author
FROM   posts
INNER  JOIN users ON users.id = posts.user_id
WHERE  posts.status = 'published'
ORDER  BY posts.published_at DESC;
```

**Query 2: Post count per author (including authors with zero posts)**

```sql
SELECT
  users.username,
  COUNT(posts.id) AS post_count
FROM   users
LEFT   JOIN posts ON posts.user_id = users.id AND posts.status = 'published'
GROUP  BY users.id, users.username
ORDER  BY post_count DESC;
```

**Query 3: Top 5 most-commented posts**

```sql
SELECT
  posts.title,
  users.username AS author,
  COUNT(comments.id) AS comment_count
FROM   posts
INNER  JOIN users    ON users.id = posts.user_id
LEFT   JOIN comments ON comments.post_id = posts.id
WHERE  posts.status = 'published'
GROUP  BY posts.id, posts.title, users.username
ORDER  BY comment_count DESC
LIMIT  5;
```

**Query 4: Posts with a specific tag**

```sql
SELECT
  posts.title,
  posts.slug,
  users.username AS author
FROM   posts
INNER  JOIN users     ON users.id = posts.user_id
INNER  JOIN post_tags ON post_tags.post_id = posts.id
INNER  JOIN tags      ON tags.id = post_tags.tag_id
WHERE  tags.slug = 'postgresql'
AND    posts.status = 'published'
ORDER  BY posts.published_at DESC;
```

**Query 5: Comments with post title and author**

```sql
SELECT
  comments.body,
  comments.created_at,
  commenter.username  AS commenter,
  posts.title         AS post_title,
  post_author.username AS post_author
FROM   comments
INNER  JOIN users post_author ON post_author.id = posts.user_id
INNER  JOIN posts             ON posts.id = comments.post_id
INNER  JOIN users commenter   ON commenter.id = comments.user_id
ORDER  BY comments.created_at DESC;
```

This query joins `users` twice with different aliases — once for the commenter, once for the post's author. This is the self-join pattern applied in a practical context.

## What you learned

- A complete blog schema uses five tables: users, posts, comments, tags, and post_tags (join table)
- Constraints (CHECK, UNIQUE, REFERENCES, NOT NULL) enforce data integrity at the database level
- The post_tags composite primary key prevents duplicate tag associations automatically
- Joining users twice with different aliases retrieves both the post author and commenter in one query
- Partial indexes on status = 'published' make published post queries fast without indexing draft rows
- Seeding with realistic data enables meaningful testing of complex queries across all five tables
