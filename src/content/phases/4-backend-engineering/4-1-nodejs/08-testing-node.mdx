## Why this matters

Code without tests is code you are afraid to change. Every significant refactor in an untested codebase risks introducing regressions that only appear in production. Tests give you the freedom to improve code confidently — if the tests pass, the behavior is preserved. In Node.js backends, where a bug can affect thousands of concurrent users simultaneously, this safety net is not optional. It is the difference between deploying on a Friday afternoon and deploying on a Monday morning with your fingers crossed.

## Test anatomy: describe, it, expect

Every test framework structures tests the same way. Here is a Vitest example:

```javascript
// src/utils/math.test.js
import { describe, it, expect } from "vitest";
import { add, divide } from "./math.js";

describe("add", () => {
  it("adds two positive numbers", () => {
    expect(add(2, 3)).toBe(5);
  });

  it("handles negative numbers", () => {
    expect(add(-1, 1)).toBe(0);
  });
});

describe("divide", () => {
  it("divides correctly", () => {
    expect(divide(10, 2)).toBe(5);
  });

  it("throws on division by zero", () => {
    expect(() => divide(10, 0)).toThrow("Cannot divide by zero");
  });
});
```

- `describe` groups related tests with a label
- `it` (or `test`) defines a single test case
- `expect` creates an assertion — `toBe` for primitives, `toEqual` for objects

## Vitest vs Jest

Both work. Vitest is the better choice for new projects:

| Feature    | Vitest              | Jest                     |
| ---------- | ------------------- | ------------------------ |
| Native ESM | Yes                 | Requires config          |
| Speed      | Faster (Vite-based) | Slower (Babel transform) |
| API        | Identical to Jest   | The original             |
| TypeScript | Zero config         | Requires setup           |
| Watch mode | Excellent           | Good                     |

Since Vitest's API is identical to Jest's, skills transfer completely. If a company uses Jest, your Vitest knowledge applies directly.

## The AAA pattern: Arrange, Act, Assert

Every well-written test follows three phases:

```javascript
it("creates a user and returns the id", async () => {
  // Arrange — set up inputs and state
  const db = createTestDatabase();
  const userService = new UserService(db);
  const userData = { name: "Alice", email: "alice@example.com" };

  // Act — call the function under test
  const user = await userService.createUser(userData);

  // Assert — verify the outcome
  expect(user.id).toBeDefined();
  expect(user.name).toBe("Alice");
  expect(user.email).toBe("alice@example.com");
});
```

Keep each phase distinct. When a test fails, the three-phase structure tells you immediately whether the setup was wrong, the function had a bug, or the assertion was incorrect.

## Mocking dependencies

Unit tests test one unit in isolation. When your function depends on a database, file system, or external API, you mock those dependencies — replace them with controlled fakes:

```javascript
// src/userService.js
export async function sendWelcomeEmail(user, emailClient) {
  await emailClient.send({
    to: user.email,
    subject: "Welcome!",
    body: `Hi ${user.name}, welcome to the platform.`,
  });
}
```

```javascript
// src/userService.test.js
import { describe, it, expect, vi } from "vitest";
import { sendWelcomeEmail } from "./userService.js";

describe("sendWelcomeEmail", () => {
  it("sends an email with the correct subject", async () => {
    // Create a mock email client
    const mockEmailClient = {
      send: vi.fn().mockResolvedValue({ messageId: "123" }),
    };

    const user = { name: "Alice", email: "alice@example.com" };
    await sendWelcomeEmail(user, mockEmailClient);

    // Assert that send was called once with correct arguments
    expect(mockEmailClient.send).toHaveBeenCalledOnce();
    expect(mockEmailClient.send).toHaveBeenCalledWith({
      to: "alice@example.com",
      subject: "Welcome!",
      body: "Hi Alice, welcome to the platform.",
    });
  });
});
```

`vi.fn()` creates a mock function that records all calls. `mockResolvedValue` makes it return a resolved promise. `toHaveBeenCalledWith` asserts it was called with specific arguments.

## Mocking modules

When the dependency is imported directly (not injected), use `vi.mock()`:

```javascript
// src/mailer.js
export async function sendEmail(to, subject, body) {
  // Real implementation calls SendGrid API
}

// src/notifications.js
import { sendEmail } from "./mailer.js";

export async function notifyUser(userId, message) {
  const user = await db.findUser(userId);
  await sendEmail(user.email, "Notification", message);
}
```

```javascript
// src/notifications.test.js
import { vi, it, expect, beforeEach } from "vitest";

vi.mock("./mailer.js", () => ({
  sendEmail: vi.fn().mockResolvedValue(undefined),
}));

import { sendEmail } from "./mailer.js"; // now the mock
import { notifyUser } from "./notifications.js";

beforeEach(() => {
  vi.clearAllMocks(); // Reset call counts between tests
});

it("sends an email when notifying a user", async () => {
  await notifyUser(1, "Your order shipped");
  expect(sendEmail).toHaveBeenCalledWith(
    expect.any(String), // email address
    "Notification",
    "Your order shipped"
  );
});
```

## Testing async code

Vitest handles async tests automatically — just use `async/await`:

```javascript
it("resolves with user data", async () => {
  const user = await fetchUser(1);
  expect(user.name).toBe("Alice");
});

it("rejects for unknown users", async () => {
  await expect(fetchUser(999)).rejects.toThrow("User not found");
});
```

## Code coverage

Coverage measures which lines of code your tests execute. Run it with:

```bash
npx vitest run --coverage
```

Coverage output shows:

- **Statements** — individual executable statements covered
- **Branches** — if/else paths taken
- **Functions** — functions called at least once
- **Lines** — lines executed

100% coverage is not the goal. Chasing 100% leads to tests that exercise code without testing meaningful behavior. A test that calls `getUser()` to hit a line is worthless if it doesn't assert what `getUser()` returned. Aim for coverage of business logic, not boilerplate.

What coverage tells you: which code paths no test exercises. That is where you have the most risk of undetected regressions.

<SandboxExercise language="javascript">
{`// Implement a simple test runner.
// createTestSuite(name) returns { test(name, fn), run() }
// run() executes all test functions and returns:
//   { passed: number, failed: number, results: [{name, status, error?}] }
// Implement expect(value) with .toBe(), .toEqual(), .toThrow()

function createTestSuite(suiteName) {
const tests = [];

function test(name, fn) {
tests.push({ name, fn });
}

async function run() {
const results = [];
let passed = 0;
let failed = 0;

    for (const { name, fn } of tests) {
      try {
        await fn();
        results.push({ name, status: "passed" });
        passed++;
      } catch (error) {
        results.push({ name, status: "failed", error: error.message });
        failed++;
      }
    }

    console.log(\`\\n[\${suiteName}] Results: \${passed} passed, \${failed} failed\`);
    results.forEach((r) => {
      const icon = r.status === "passed" ? "✓" : "✗";
      const detail = r.error ? \` — \${r.error}\` : "";
      console.log(\`  \${icon} \${r.name}\${detail}\`);
    });

    return { passed, failed, results };

}

return { test, run };
}

function expect(value) {
return {
toBe(expected) {
if (value !== expected) {
throw new Error(\`Expected \${JSON.stringify(expected)}, got \${JSON.stringify(value)}\`);
}
},
toEqual(expected) {
if (JSON.stringify(value) !== JSON.stringify(expected)) {
throw new Error(
\`Expected \${JSON.stringify(expected)}, got \${JSON.stringify(value)}\`
);
}
},
toThrow(message) {
if (typeof value !== "function") throw new Error("toThrow requires a function");
try {
value();
throw new Error("Expected function to throw, but it did not");
} catch (error) {
if (error.message === "Expected function to throw, but it did not") throw error;
if (message && !error.message.includes(message)) {
throw new Error(\`Expected error containing "\${message}", got "\${error.message}"\`);
}
}
},
};
}

// Use the test runner
const suite = createTestSuite("Math utilities");

suite.test("adds two numbers", () => {
const add = (a, b) => a + b;
expect(add(2, 3)).toBe(5);
expect(add(-1, 1)).toBe(0);
});

suite.test("toEqual compares objects", () => {
expect({ name: "Alice" }).toEqual({ name: "Alice" });
});

suite.test("detects division by zero", () => {
const divide = (a, b) => {
if (b === 0) throw new Error("Cannot divide by zero");
return a / b;
};
expect(() => divide(10, 0)).toThrow("Cannot divide by zero");
});

suite.test("intentionally failing test", () => {
expect(1 + 1).toBe(3); // This will fail
});

await suite.run();`}

## What you learned

- Tests use `describe` to group related cases, `it` to define individual cases, and `expect` with matchers to assert outcomes
- The AAA pattern (Arrange, Act, Assert) structures tests for clarity and debuggability
- `vi.fn()` creates mock functions that record calls; `vi.mock()` replaces entire modules with controlled fakes
