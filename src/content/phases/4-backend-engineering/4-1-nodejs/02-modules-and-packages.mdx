## Why this matters

Every non-trivial Node.js project is a composition of modules — your own code split into files, plus third-party packages pulled from npm. Understanding how modules work (and why there are two completely different module systems) prevents the cryptic errors you get when you mix CommonJS and ESM incorrectly. Understanding semantic versioning prevents the silent breaking changes that appear weeks after a `npm install` when a dependency updated its minor version and changed a behavior you depended on.

## CommonJS: the original module system

CommonJS (CJS) was the module system Node.js shipped with in 2009. It uses `require()` to load modules and `module.exports` to expose values.

```javascript
// math.js — CommonJS module
function add(a, b) {
  return a + b;
}

function multiply(a, b) {
  return a * b;
}

module.exports = { add, multiply };
```

```javascript
// app.js — importing from math.js
const { add, multiply } = require("./math");

console.log(add(2, 3)); // 5
console.log(multiply(4, 5)); // 20
```

Key characteristics of CommonJS:

- **Synchronous** — `require()` blocks execution until the module loads. This is fine on disk but terrible for network loading (which is why CJS never worked in browsers natively).
- **Dynamic** — you can call `require()` conditionally inside an if-statement or a function. The module path can be a variable.
- **Cached** — the first `require()` of a module executes it and caches the result. Every subsequent `require()` of the same path returns the cached exports object.

## ESM: the standard module system

ECMAScript Modules (ESM) was standardized in ES2015 and added to Node.js in version 12. It uses `import` and `export` statements.

```javascript
// math.mjs — ESM module
export function add(a, b) {
  return a + b;
}

export function multiply(a, b) {
  return a * b;
}

export default { add, multiply };
```

```javascript
// app.mjs — importing ESM
import { add, multiply } from "./math.mjs";
import mathLib from "./math.mjs"; // default import

console.log(add(2, 3)); // 5
```

Key characteristics of ESM:

- **Static** — `import` statements must be at the top level. The module graph is analyzed before any code runs, enabling tree-shaking.
- **Asynchronous** — supports top-level `await` (you can `await` at the module's top level, not just inside functions).
- **Not cached the same way** — modules are linked, not copied.

To use ESM in Node.js: name files `.mjs`, or set `"type": "module"` in your `package.json` (which makes all `.js` files in that project ESM by default).

## package.json: the project manifest

Every Node.js project has a `package.json` in its root. It describes the project and controls how npm behaves.

```json
{
  "name": "my-api",
  "version": "1.0.0",
  "description": "A simple REST API",
  "main": "src/index.js",
  "type": "module",
  "scripts": {
    "start": "node src/index.js",
    "dev": "node --watch src/index.js",
    "test": "vitest run",
    "lint": "eslint src/"
  },
  "dependencies": {
    "express": "^4.18.2",
    "zod": "^3.22.0"
  },
  "devDependencies": {
    "vitest": "^1.0.0",
    "eslint": "^8.0.0"
  }
}
```

- `main` — the entry point when someone requires/imports your package
- `exports` — the modern replacement for `main`, with more control over what gets exported
- `dependencies` — packages needed in production
- `devDependencies` — packages only needed during development (testing, linting, building)
- `scripts` — shortcuts you run with `npm run <name>`

## Semantic versioning

npm uses **semver** (semantic versioning) with the format `MAJOR.MINOR.PATCH`:

- **PATCH** (`1.2.3` → `1.2.4`) — bug fixes, backward compatible
- **MINOR** (`1.2.3` → `1.3.0`) — new features, backward compatible
- **MAJOR** (`1.2.3` → `2.0.0`) — breaking changes

Version prefixes in `package.json` control what updates are allowed:

| Range    | Meaning               | Example             |
| -------- | --------------------- | ------------------- |
| `1.2.3`  | Exact version only    | Only 1.2.3          |
| `~1.2.3` | Patch updates only    | 1.2.3, 1.2.4, 1.2.9 |
| `^1.2.3` | Minor updates allowed | 1.2.3, 1.3.0, 1.9.9 |
| `*`      | Any version           | Latest always       |

`^` (caret) is the npm default. It allows minor and patch updates but not major. This is usually safe — minor versions are supposed to be backward compatible — but "supposed to" and "actually are" sometimes differ.

## package-lock.json and reproducible installs

When you run `npm install`, npm writes a `package-lock.json` that records the exact version of every installed package (including transitive dependencies). The next time anyone runs `npm install` — on another machine, in CI, six months later — they get identical versions.

**Never manually edit** `package-lock.json`. **Do commit** it to version control. Without it, `npm install` might install different patch versions on different machines, causing "works on my machine" bugs.

## npm vs yarn vs pnpm

All three are package managers for the npm registry. They are interchangeable for most tasks:

- **npm** — ships with Node.js, safest default
- **yarn** — faster historically, introduced workspaces and the lockfile format npm later copied
- **pnpm** — fastest, uses content-addressable storage to share packages across projects (saves gigabytes of disk space)

For new projects, pnpm is excellent. For teams with existing projects, npm is the path of least friction.

{`// Implement a minimal module system.
// register(name, factory) — registers a module. factory receives require and returns exports.
// require(name) — returns the module's exports, executing the factory on first call.
// Handle circular dependencies by returning the partially-built exports object.

function createModuleSystem() {
const registry = {}; // factories registered with register()
const cache = {}; // exports cached after first require()

function require(name) {
if (cache[name]) return cache[name];
if (!registry[name]) throw new Error(\`Module not found: \${name}\`);

    // Initialize cache entry early to handle circular deps
    cache[name] = {};

    const exports = registry[name](require);

    // Copy all exported keys into the cached object
    Object.assign(cache[name], exports);
    return cache[name];

}

function register(name, factory) {
registry[name] = factory;
}

return { require, register };
}

const { require, register } = createModuleSystem();

// Register a math module
register("math", (require) => {
return {
add: (a, b) => a + b,
multiply: (a, b) => a \* b,
};
});

// Register an app module that depends on math
register("app", (require) => {
const math = require("math");
return {
run: () => {
console.log("2 + 3 =", math.add(2, 3));
console.log("4 \* 5 =", math.multiply(4, 5));
},
};
});

const app = require("app");
app.run();

// Verify caching — same object reference
const math1 = require("math");
const math2 = require("math");
console.log("Cached (same reference):", math1 === math2); // true`}

## What you learned

- CommonJS uses `require()` and `module.exports` — synchronous and dynamic, the original Node module format
- ESM uses `import`/`export` — static, tree-shakeable, and supports top-level `await`
- Semantic versioning (MAJOR.MINOR.PATCH) defines compatibility guarantees, and the `^` prefix allows minor updates
