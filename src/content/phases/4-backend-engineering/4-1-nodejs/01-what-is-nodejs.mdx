## Why this matters

Before Node.js, running JavaScript outside the browser required a different language for server-side work — typically Python, Ruby, Java, or PHP. Node.js changed that in 2009 by extracting Chrome's V8 JavaScript engine and pairing it with a powerful asynchronous I/O library. Today, Node.js powers everything from small CLI tools to the backends of Netflix, LinkedIn, and NASA. Understanding what makes Node tick — and what its actual limitations are — lets you choose it confidently when it's the right tool, and reach for something else when it isn't.

## What Node.js actually is

Node.js is not a framework, and it is not a new programming language. It is a **runtime environment** that lets you execute JavaScript outside the browser. Three components work together to make that happen.

**V8** is the JavaScript engine originally built for Google Chrome. It compiles JavaScript to native machine code, which is why Node.js is fast. V8 handles all your JavaScript: parsing, optimizing, garbage collection.

**libuv** is a C library that provides Node's asynchronous I/O capabilities. When you read a file, make a network request, or set a timer in Node.js, libuv is doing the actual system-level work. It maintains a thread pool (default size: 4 threads) for operations the operating system cannot handle asynchronously — like certain file system operations — and uses the OS's native async APIs (epoll on Linux, kqueue on macOS, IOCP on Windows) for everything else.

**The Node.js standard library** is the set of built-in modules that come with every Node installation: `fs` for files, `http` for networking, `path` for file paths, `crypto` for hashing, `os` for system info, and more.

## The event loop

This is the concept that confuses new Node.js developers the most, but it's actually elegant once it clicks.

JavaScript in Node.js runs on a **single thread**. There is one call stack, and only one piece of JavaScript executes at a time. If Node is single-threaded, how does it handle thousands of simultaneous requests without blocking?

The answer is the **event loop** — a mechanism that lets Node.js perform non-blocking I/O by offloading operations to the system kernel or libuv's thread pool, then executing callbacks when those operations complete.

```
  ┌─────────────────────────────────────────────┐
  │              Your JavaScript                 │
  │          (runs on call stack)               │
  └─────────────────────────────────────────────┘
                       │
                       ▼
  ┌─────────────────────────────────────────────┐
  │              Event Loop                      │
  │  Checks: is the call stack empty?           │
  │  If yes: pick the next callback from queue  │
  └─────────────────────────────────────────────┘
                       │
          ┌────────────┴─────────────┐
          ▼                          ▼
  ┌──────────────┐          ┌──────────────────┐
  │  libuv       │          │  OS kernel       │
  │  thread pool │          │  (sockets, etc.) │
  │  (file I/O)  │          │                  │
  └──────────────┘          └──────────────────┘
```

When your code calls `fs.readFile()`, Node registers a callback and immediately returns to the event loop. libuv performs the actual disk read on a background thread. When the read finishes, libuv places your callback into the event queue. The next time the call stack is empty, the event loop picks up that callback and runs it.

This is why a Node.js web server can handle 10,000 simultaneous connections on a single thread — it is never actually waiting. It is always doing something or moving to the next callback.

## Why single-threaded non-blocking beats multi-threaded blocking for I/O

Traditional web servers (Apache with PHP, for example) use the **thread-per-request** model: each incoming request gets its own OS thread. The thread reads from a database, waits for the response, then sends the HTTP reply. While waiting, the thread just sits there consuming memory (typically 1-8 MB per thread). With 10,000 concurrent users, you need 10,000 threads — which exhausts server memory.

Node's model is the opposite. One thread handles all requests. When any request hits an I/O operation, Node hands it off and immediately picks up the next request. The single thread is never idle because I/O is never blocking it.

The trade-off: Node.js is not good at **CPU-intensive work**. If a request requires heavy computation — image processing, video encoding, cryptographic cracking, running a machine learning model — that computation blocks the single thread and starves every other request. For CPU-bound work, use worker threads, child processes, or a different language entirely.

## Where Node.js shines

- **REST APIs and GraphQL servers** — pure I/O, tons of network calls, perfect fit
- **Real-time applications** — chat apps, collaborative editing, live dashboards (WebSockets)
- **CLI tools** — npm itself is a Node.js CLI tool
- **Serverless functions** — fast cold starts, small memory footprint
- **Build tooling** — Webpack, Vite, ESLint, Prettier, TypeScript compiler

## Version management with nvm

Node.js releases two major versions per year. The even-numbered versions become **LTS** (Long-Term Support) with 30 months of security patches. Odd-numbered versions are **Current** with new features but shorter support windows. For production, always use an LTS version.

```bash
# Install nvm (Node Version Manager)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.0/install.sh | bash

# List available LTS versions
nvm ls-remote --lts

# Install the latest LTS
nvm install --lts

# Switch between versions per project
nvm use 20

# Set a default version
nvm alias default 20
```

Use nvm (or fnm, the faster Rust-based alternative) instead of installing Node directly. Project teams often need different Node versions, and nvm makes switching trivial.

{`// Simulate the event loop's task ordering.
// Implement an EventLoop class with:
//   addTask(fn, label)      — adds a macrotask (runs after microtasks)
//   addMicrotask(fn, label) — adds a microtask (runs before next macrotask)
//   run()                   — processes all tasks in correct order

class EventLoop {
constructor() {
this.macrotasks = [];
this.microtasks = [];
this.log = [];
}

addTask(fn, label) {
this.macrotasks.push({ fn, label });
}

addMicrotask(fn, label) {
this.microtasks.push({ fn, label });
}

run() {
// Process all microtasks first, then one macrotask,
// then all microtasks again, then next macrotask, etc.
while (this.macrotasks.length > 0 || this.microtasks.length > 0) {
// Drain microtask queue
while (this.microtasks.length > 0) {
const { fn, label } = this.microtasks.shift();
this.log.push(label);
fn();
}
// Run one macrotask
if (this.macrotasks.length > 0) {
const { fn, label } = this.macrotasks.shift();
this.log.push(label);
fn();
}
}
return this.log;
}
}

const loop = new EventLoop();

// Synchronous code runs first (before the loop)
console.log("1. synchronous");

loop.addTask(() => {}, "3. macrotask A");
loop.addMicrotask(() => {}, "2. microtask A");
loop.addMicrotask(() => {}, "2b. microtask B");
loop.addTask(() => {}, "4. macrotask B");

const order = loop.run();
console.log("Execution order:", order);
// Expected: microtask A, microtask B, macrotask A, macrotask B`}

## What you learned

- Node.js is a runtime built from Chrome's V8 engine plus the libuv async I/O library, not a new language or framework
- The event loop allows a single thread to handle thousands of concurrent I/O operations without blocking
- Node.js excels at I/O-bound work (APIs, real-time apps, CLI tools) but is unsuitable for CPU-intensive computation without worker threads
