## Why this matters

Understanding the difference between an image and a container — and knowing the handful of commands that control them — is the foundation of everything else in Docker. These are the commands you'll type dozens of times per day as a backend developer.

## Docker architecture

Docker has three main pieces:

**Docker daemon (`dockerd`)** — a background service that manages images, containers, networks, and volumes. It does the actual work. On Linux it runs natively; on macOS/Windows it runs inside a small Linux VM.

**Docker CLI (`docker`)** — the command you type in your terminal. It sends instructions to the daemon via a REST API.

**Registry** — a storage and distribution system for images. Docker Hub (`hub.docker.com`) is the default public registry. Companies also run private registries: GitHub Container Registry (`ghcr.io`), AWS ECR, Google Artifact Registry.

```
You type: docker run nginx
    │
    ▼
Docker CLI ──────── REST API ──────── Docker Daemon
                                           │
                               ┌───────────┴────────────┐
                               │                        │
                         Pull image from           Create container
                         Docker Hub (if needed)    from image
```

## Images vs containers

This distinction trips up every new Docker user. Here's the clearest way to think about it:

**Image** = the recipe (read-only, layered, stored on disk)
**Container** = what's running (a process using that recipe, with a writable layer on top)

```
Docker Image (read-only layers)
┌──────────────────────────┐
│  Layer 3: COPY app/ .   │ ← your source code
│  Layer 2: RUN npm install│ ← installed dependencies
│  Layer 1: FROM node:20  │ ← base Node.js OS
└──────────────────────────┘

Container = Image + thin writable layer on top
┌──────────────────────────┐
│  Writable layer          │ ← container can write here
├──────────────────────────┤
│  Image layers (shared)   │ ← read-only, shared between containers
└──────────────────────────┘
```

You can run ten containers from the same image simultaneously. They all share the read-only image layers (efficient disk use) but each gets its own writable layer for runtime data.

## Essential commands

### Pulling and running images

```bash
# Pull an image from Docker Hub (does not start a container)
docker pull nginx:1.27

# Run a container from an image (pulls automatically if not local)
docker run nginx:1.27

# Common run flags
docker run \
  -d \                        # detached: run in background
  -p 8080:80 \               # port mapping: HOST:CONTAINER
  -e NODE_ENV=production \    # set environment variable
  --name my-nginx \           # give the container a name
  --rm \                      # auto-remove when stopped
  nginx:1.27
```

### Listing and inspecting

```bash
# List running containers
docker ps

# List all containers (including stopped)
docker ps -a

# List local images
docker images

# Detailed info about a container (returns JSON)
docker inspect my-nginx

# Live resource usage (CPU, memory, network)
docker stats
```

### Managing containers

```bash
# Stop a running container (sends SIGTERM, then SIGKILL after 10s)
docker stop my-nginx

# Start a stopped container
docker start my-nginx

# Remove a stopped container
docker rm my-nginx

# Force-remove a running container
docker rm -f my-nginx

# Remove an image
docker rmi nginx:1.27
```

### Getting inside a container

```bash
# Run a command in a running container
docker exec -it my-nginx bash

# View container logs (stdout/stderr)
docker logs my-nginx

# Follow logs in real time
docker logs -f my-nginx

# Last 50 lines with timestamps
docker logs --tail 50 --timestamps my-nginx
```

The `-it` flags on `docker exec` mean: `-i` (keep stdin open) and `-t` (allocate a pseudo-TTY). Together they give you an interactive terminal session inside the container.

## Port mapping in depth

Containers have their own network namespace — ports inside are not automatically exposed to your host machine. The `-p` flag creates a mapping:

```bash
# Your machine's port 3000 → container's port 3000
docker run -p 3000:3000 node-app

# Different ports — host:container
docker run -p 8080:80 nginx         # visit localhost:8080 to reach nginx's port 80
docker run -p 5433:5432 postgres    # avoid conflicting with a local postgres on 5432

# Bind to a specific host interface (localhost only, not LAN-accessible)
docker run -p 127.0.0.1:5432:5432 postgres
```

## Registries

```bash
# Docker Hub (default) — no prefix needed
docker pull node:20-alpine

# GitHub Container Registry
docker pull ghcr.io/owner/repo:tag

# AWS ECR
docker pull 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest

# Log in before pushing to a registry
docker login
docker login ghcr.io -u USERNAME --password-stdin
```

Image naming follows the pattern: `[registry/][namespace/]name[:tag]`

- `nginx` → `docker.io/library/nginx:latest`
- `node:20-alpine` → `docker.io/library/node:20-alpine`
- `myuser/myapp:1.0` → `docker.io/myuser/myapp:1.0`

Always pin to a specific tag in production. The `latest` tag changes when the image maintainer pushes a new version — using it means your deployments are not reproducible.

<SandboxExercise language="javascript">
{`// Simulate a Docker registry and container runner
function createDockerSimulator() {
  // "Registry" — images available to pull
  const remoteRegistry = {
    'nginx:1.27': {
      startupOutput: ['nginx: starting...', 'nginx: ready on port 80'],
    },
    'node:20-alpine': {
      startupOutput: ['Node.js 20 runtime ready'],
    },
    'postgres:16': {
      startupOutput: [
        'PostgreSQL 16 starting...',
        'database system is ready to accept connections',
      ],
    },
  };

const localImages = {}; // pulled images
const containers = {}; // running containers
let nextId = 1;

return {
pull(imageName) {
if (!remoteRegistry[imageName]) {
return { ok: false, error: \`Unable to find image '\${imageName}'\` };
}
localImages[imageName] = remoteRegistry[imageName];
return { ok: true, message: \`Pulled \${imageName}\` };
},

    run(imageName, options = {}) {
      if (!localImages[imageName]) {
        // Auto-pull like real Docker does
        const pullResult = this.pull(imageName);
        if (!pullResult.ok) return pullResult;
      }

      const id = String(nextId++).padStart(12, '0');
      const name = options.name || \`container_\${id}\`;
      containers[id] = {
        id,
        name,
        image: imageName,
        status: 'running',
        ports: options.ports || [],
        env: options.env || {},
        logs: [...localImages[imageName].startupOutput],
        startedAt: new Date().toISOString(),
      };
      return { ok: true, containerId: id, name };
    },

    stop(containerId) {
      const c = containers[containerId];
      if (!c) return { ok: false, error: 'No such container' };
      c.status = 'exited';
      c.logs.push('Container stopped.');
      return { ok: true };
    },

    ps(all = false) {
      return Object.values(containers)
        .filter(c => all || c.status === 'running')
        .map(c => ({ id: c.id, name: c.name, image: c.image, status: c.status }));
    },

    logs(containerId) {
      const c = containers[containerId];
      if (!c) return { ok: false, error: 'No such container' };
      return { ok: true, output: c.logs };
    },

    images() {
      return Object.keys(localImages);
    },

};
}

const docker = createDockerSimulator();

// Pull an image explicitly
console.log('1. Pull nginx:');
console.log(docker.pull('nginx:1.27'));

// Run postgres (auto-pulls since we didn't pull first)
console.log('\\n2. Run postgres (auto-pull):');
const pgResult = docker.run('postgres:16', {
name: 'my-db',
ports: ['5432:5432'],
env: { POSTGRES_PASSWORD: 'secret' },
});
console.log(pgResult);

// Run nginx with a custom name
const ngResult = docker.run('nginx:1.27', {
name: 'web',
ports: ['8080:80'],
});

// See running containers
console.log('\\n3. docker ps:');
console.log(docker.ps());

// Check logs
console.log('\\n4. docker logs my-db:');
const logsResult = docker.logs(pgResult.containerId);
logsResult.output.forEach(line => console.log(' ', line));

// Stop and re-list
docker.stop(ngResult.containerId);
console.log('\\n5. docker ps -a (after stopping nginx):');
console.log(docker.ps(true));

// List local images
console.log('\\n6. docker images:');
console.log(docker.images());`}

## What you learned

- The Docker daemon manages resources; the CLI sends it commands; registries store images
- Images are read-only layered blueprints; containers are running instances with a writable layer on top
- `docker run -d -p HOST:CONTAINER --name` are the core flags you'll use daily
- `docker exec -it` opens a terminal inside a running container
- `docker logs` shows stdout/stderr; `docker ps` lists containers; `docker images` lists local images
- Always pin image tags in production — never use `latest`
