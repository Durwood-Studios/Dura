## Why this matters

An API with no documentation is useless to everyone except its author — and even then, only for a few weeks until memory fades. Documentation is not a nice-to-have you write after the API is done. It is a communication contract between you and every developer who will ever call your endpoints. The OpenAPI specification gives you a machine-readable, standardized way to document every endpoint, parameter, schema, and response. From that spec, you can automatically generate interactive documentation, client SDKs, mock servers, and test suites — all staying in sync with your actual API.

## What OpenAPI is

**OpenAPI 3.x** (formerly called Swagger) is a YAML or JSON document that describes a REST API in complete detail. It specifies:

- Every available endpoint (paths + methods)
- All parameters (path, query, header, cookie)
- Request body shapes with JSON Schema
- All possible response codes and their body shapes
- Authentication schemes
- Reusable component schemas

A minimal OpenAPI document looks like this:

```yaml
openapi: "3.0.3"
info:
  title: Bookshelf API
  version: "1.0.0"
  description: Manage your personal book collection

paths:
  /books:
    get:
      operationId: listBooks
      summary: List all books
      parameters:
        - name: author
          in: query
          schema:
            type: string
          description: Filter by author name
      responses:
        "200":
          description: Array of books
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: "#/components/schemas/Book"
    post:
      operationId: createBook
      summary: Create a new book
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: "#/components/schemas/CreateBookInput"
      responses:
        "201":
          description: Book created
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/Book"
        "400":
          description: Validation error
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/Error"

components:
  schemas:
    Book:
      type: object
      required: [id, title, author]
      properties:
        id:
          type: integer
        title:
          type: string
          example: "Clean Code"
        author:
          type: string
          example: "Robert C. Martin"
        isbn:
          type: string
          pattern: "^\\d{13}$"

    CreateBookInput:
      type: object
      required: [title, author]
      properties:
        title:
          type: string
          minLength: 1
          maxLength: 500
        author:
          type: string
          minLength: 1

    Error:
      type: object
      required: [error]
      properties:
        error:
          type: string
        details:
          type: array
          items:
            type: object

  securitySchemes:
    bearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT
```

## Swagger UI with Express

`swagger-jsdoc` generates the OpenAPI spec from JSDoc comments in your route files. `swagger-ui-express` serves the interactive documentation at a URL you choose:

```javascript
import swaggerJsdoc from "swagger-jsdoc";
import swaggerUi from "swagger-ui-express";

const swaggerDefinition = {
  openapi: "3.0.3",
  info: { title: "My API", version: "1.0.0" },
  components: {
    securitySchemes: {
      bearerAuth: { type: "http", scheme: "bearer", bearerFormat: "JWT" },
    },
  },
};

const spec = swaggerJsdoc({
  swaggerDefinition,
  apis: ["./src/routes/*.js"], // Files containing JSDoc comments
});

// Serve the interactive UI at /api-docs
app.use("/api-docs", swaggerUi.serve, swaggerUi.setup(spec));

// Serve the raw JSON spec for tooling that needs it
app.get("/api-docs.json", (req, res) => res.json(spec));
```

Then annotate your routes with JSDoc:

```javascript
/**
 * @openapi
 * /books/{id}:
 *   get:
 *     operationId: getBook
 *     summary: Get a book by ID
 *     parameters:
 *       - name: id
 *         in: path
 *         required: true
 *         schema:
 *           type: integer
 *     responses:
 *       "200":
 *         description: The book
 *         content:
 *           application/json:
 *             schema:
 *               $ref: "#/components/schemas/Book"
 *       "404":
 *         description: Book not found
 */
router.get(
  "/:id",
  catchAsync(async (req, res) => {
    const book = await Book.findById(req.params.id);
    if (!book) throw notFound("Book");
    res.json(book);
  })
);
```

## Contract-first vs code-first

**Code-first**: You write the route handlers, annotate them with JSDoc or decorators, and a tool generates the spec. This is the most common approach for Express APIs. The spec stays close to the code, but can drift if developers forget to update annotations.

**Contract-first**: You write the OpenAPI spec first, then generate server stubs and client SDKs from it. Forces you to design the API before implementation. Makes the spec the authoritative source of truth. Used in teams where frontend and backend development happens in parallel — the frontend can start building against the spec before the API exists.

For solo projects or small teams, code-first is pragmatic. For APIs consumed by external teams or published publicly, contract-first produces better-designed, more consistent APIs.

## API versioning

APIs change. When they change in breaking ways (removing a field, changing a response shape, making a required parameter required that was optional), clients break. Versioning lets you evolve the API while keeping old versions working:

**URL versioning** (most common): `/v1/books`, `/v2/books`. Versions are visible, cacheable, and easy to route to different handlers. This is what most public APIs use.

**Header versioning**: `Accept: application/vnd.myapi.v2+json`. Cleaner URLs but harder to test in a browser. Preferred by API purists.

**Query parameter versioning**: `/books?version=2`. Easy to implement but pollutes query strings.

For new APIs, start with `/v1`. When you need to make a breaking change, introduce `/v2` and maintain `/v1` for a deprecation period (typically 6-12 months). Announce deprecations clearly in headers (`Deprecation: true`, `Sunset: 2025-01-01`).

## Redoc and Scalar

Swagger UI is the default choice, but alternatives offer cleaner designs:

**Redoc** generates a three-panel documentation page — endpoints on the left, details in the middle, code samples on the right. Common on public API portals.

**Scalar** (newer, open-source) produces beautiful interactive docs with support for multiple languages, dark mode, and a cleaner UI than Swagger. Gaining popularity quickly.

Both accept the same OpenAPI JSON/YAML spec. Swap them by changing which UI library you serve — your spec does not change.

<SandboxExercise language="javascript">
{`// Build an API spec builder that generates OpenAPI-like structures.
// createSpec(title, version) returns a builder with addEndpoint() and build().

function createSpec(title, version) {
const paths = {};
const schemas = {};

return {
addEndpoint(method, path, { summary, operationId, params = [], body = null, responses = {} }) {
if (!paths[path]) paths[path] = {};

      const operation = { operationId, summary, parameters: [], responses: {} };

      // Add parameters
      for (const param of params) {
        operation.parameters.push({
          name: param.name,
          in: param.in || "query",
          required: param.required || false,
          schema: { type: param.type || "string" },
          description: param.description || ""
        });
      }

      // Add request body
      if (body) {
        operation.requestBody = {
          required: true,
          content: {
            "application/json": {
              schema: body
            }
          }
        };
      }

      // Add responses
      for (const [code, desc] of Object.entries(responses)) {
        operation.responses[code] = typeof desc === "string"
          ? { description: desc }
          : desc;
      }

      paths[path][method.toLowerCase()] = operation;
      return this; // fluent interface
    },

    addSchema(name, schema) {
      schemas[name] = schema;
      return this;
    },

    build() {
      return {
        openapi: "3.0.3",
        info: { title, version },
        paths,
        components: { schemas }
      };
    }

};
}

// Build a small spec for a Books API
const spec = createSpec("Books API", "1.0.0")
.addSchema("Book", {
type: "object",
required: ["id", "title", "author"],
properties: {
id: { type: "integer" },
title: { type: "string" },
author: { type: "string" }
}
})
.addEndpoint("GET", "/books", {
operationId: "listBooks",
summary: "List all books",
params: [
{ name: "author", in: "query", type: "string", description: "Filter by author" },
{ name: "page", in: "query", type: "integer", description: "Page number" }
],
responses: {
"200": "Array of books",
"500": "Internal server error"
}
})
.addEndpoint("POST", "/books", {
operationId: "createBook",
summary: "Create a new book",
body: { type: "object", required: ["title", "author"], properties: { title: { type: "string" }, author: { type: "string" } } },
responses: {
"201": "Book created",
"400": "Validation error"
}
})
.addEndpoint("GET", "/books/{id}", {
operationId: "getBook",
summary: "Get a single book",
params: [{ name: "id", in: "path", type: "integer", required: true }],
responses: {
"200": "The book",
"404": "Book not found"
}
})
.build();

// Inspect the result
console.log("OpenAPI version:", spec.openapi);
console.log("API title:", spec.info.title);
console.log("Paths:", Object.keys(spec.paths));
console.log("\\nGET /books operation:");
console.log(JSON.stringify(spec.paths["/books"].get, null, 2));
console.log("\\nSchemas:", Object.keys(spec.components.schemas));`}

## What you learned

- OpenAPI 3.x is a machine-readable spec format that describes every endpoint, parameter, schema, and response.
- `swagger-jsdoc` generates specs from JSDoc comments; `swagger-ui-express` serves the interactive Swagger UI.
- Contract-first (spec before code) produces better-designed APIs; code-first (annotate existing code) is more pragmatic for small teams.
- URL versioning (`/v1/`, `/v2/`) is the standard approach for introducing breaking API changes.
- Redoc and Scalar are drop-in Swagger UI alternatives that accept the same OpenAPI JSON spec.
