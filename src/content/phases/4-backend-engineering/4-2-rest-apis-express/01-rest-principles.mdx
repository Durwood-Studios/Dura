## Why this matters

Every API you consume as a developer — GitHub, Stripe, Twitter, Google Maps — is almost certainly a REST API. Understanding the architectural principles behind REST lets you design APIs that other developers find intuitive, predict how unfamiliar APIs will behave before reading their docs, and debug integration problems faster. REST is not a protocol or a standard you implement. It is a set of constraints that, when followed, produce APIs with predictable, consistent behavior.

## REST is a style, not a standard

REST stands for **Representational State Transfer**. Roy Fielding introduced it in his 2000 doctoral dissertation as a description of the architectural constraints that made the web work at massive scale. REST is an **architectural style** — a collection of six constraints. Following them consistently is what makes an API "RESTful." Violating them produces an API that technically works but confuses users and creates integration headaches.

The six constraints are:

**Client-server separation.** The UI and the data storage are decoupled. The client only knows how to make requests; the server only knows how to respond. This lets them evolve independently — a mobile app, a web app, and a CLI can all consume the same API.

**Statelessness.** Every request from a client must contain all information needed to understand and process that request. The server does not store session state between requests. If a user is logged in, their identity must be proven on every single request — typically with a token in the Authorization header. This is the constraint that makes REST APIs easy to scale horizontally: any server can handle any request because no sticky sessions exist.

**Cacheability.** Responses must declare whether they can be cached. Cacheable responses reduce load on the server and improve client performance. HTTP already has mechanisms for this: `Cache-Control`, `ETag`, `Last-Modified` headers.

**Uniform interface.** This is the most important constraint. Resources are identified by URIs. Resources are manipulated through representations (JSON, XML). Messages are self-descriptive. This standardization is what makes REST APIs learnable — once you know the pattern, every REST API works the same way.

**Layered system.** The client does not know whether it is talking to the actual server or an intermediary (load balancer, CDN, API gateway). This enables scaling and security without changing client code.

**Code on demand (optional).** Servers can send executable code to clients (like JavaScript). This is the only optional constraint and is rarely used in practice.

## Resources, not actions

The single biggest mistake developers make when designing REST APIs is thinking in terms of actions instead of resources.

**Wrong — verb-based:**

```
GET /getUser?id=42
POST /createOrder
POST /deletePost?id=7
POST /searchProducts
```

**Right — resource-based:**

```
GET  /users/42
POST /orders
DELETE /posts/7
GET  /products?q=widget
```

A **resource** is a noun — a thing your API exposes. Users are a resource. Orders are a resource. Blog posts are a resource. HTTP verbs tell you what to do with that resource.

URLs should be **lowercase kebab-case**, plural for collections:

```
/users           → the users collection
/users/42        → a specific user
/users/42/posts  → posts belonging to user 42
/blog-posts      → use kebab-case for multi-word resources
```

## HTTP verbs as semantic actions

HTTP defines verbs whose meanings REST repurposes consistently:

| Verb   | Meaning        | Safe? | Idempotent? |
| ------ | -------------- | ----- | ----------- |
| GET    | Read           | Yes   | Yes         |
| POST   | Create         | No    | No          |
| PUT    | Replace        | No    | Yes         |
| PATCH  | Partial update | No    | No          |
| DELETE | Remove         | No    | Yes         |

**Safe** means the operation does not modify data. **Idempotent** means calling it multiple times has the same effect as calling it once. DELETE is idempotent because deleting an already-deleted resource still results in it being gone. POST is neither — each POST typically creates a new resource.

## Status codes communicate outcomes

HTTP status codes are not suggestions. They carry semantic meaning that REST clients rely on:

```
2xx — Success
  200 OK            Standard success (GET, PUT, PATCH)
  201 Created       Resource created (POST)
  204 No Content    Success with no response body (DELETE)

3xx — Redirect
  301 Moved Permanently  Resource has a new URL
  304 Not Modified       Cached version is still valid

4xx — Client error (the request was wrong)
  400 Bad Request        Malformed request or validation error
  401 Unauthorized       Not authenticated
  403 Forbidden          Authenticated but not allowed
  404 Not Found          Resource does not exist
  409 Conflict           Duplicate or state conflict
  422 Unprocessable Entity  Validation failed (semantically wrong)

5xx — Server error (our fault)
  500 Internal Server Error  Unhandled exception
  503 Service Unavailable    Server overloaded or down
```

Returning 200 for a delete that failed, or 500 for missing input, breaks every client that follows HTTP semantics. Use the right code.

## The Richardson Maturity Model

Leonard Richardson described four levels of REST maturity that help evaluate how "RESTful" an API really is:

**Level 0 — The swamp of POX.** One URL, one method. Everything is a POST. Most SOAP services and many legacy APIs live here.

**Level 1 — Resources.** Different URLs for different resources, but all actions are POSTs.

**Level 2 — HTTP verbs.** Different URLs and the right HTTP verbs. This is what most developers mean when they say "REST API." It is the practical target for new APIs.

**Level 3 — Hypermedia (HATEOAS).** Responses include links to related actions. A GET /orders/42 response includes a link to cancel the order. Clients discover the API from within responses. Rare in practice, but elegant in theory.

Most production APIs operate at Level 2. Aim for Level 2 — it is achievable, learnable, and well-supported by every HTTP client library.

{`// Validate whether a request follows REST conventions.
// Returns { valid: boolean, issues: string[] }

function validateRestRequest(method, path, body) {
const issues = [];

// Check: path should use nouns, not verbs
const verbPatterns = [
/\\/get[A-Z]/i, /\\/create[A-Z]/i, /\\/delete[A-Z]/i,
/\\/update[A-Z]/i, /\\/fetch[A-Z]/i, /\\/list[A-Z]/i
];
if (verbPatterns.some(p => p.test(path))) {
issues.push("Path contains a verb — use a noun resource instead (e.g. /users not /getUsers)");
}

// Check: path segments should be lowercase-kebab-case
const segments = path.split("/").filter(Boolean);
for (const seg of segments) {
// Skip :param placeholders and numeric IDs
if (seg.startsWith(":") || /^\\d+$/.test(seg)) continue;
if (/[A-Z_]/.test(seg)) {
issues.push(\`Path segment "\${seg}" should be lowercase-kebab-case\`);
}
}

// Check: GET and DELETE should not have a body
if ((method === "GET" || method === "DELETE") && body !== null && body !== undefined) {
issues.push(\`\${method} requests should not have a request body\`);
}

// Check: POST and PUT should have a body
if ((method === "POST" || method === "PUT") && (body === null || body === undefined)) {
issues.push(\`\${method} requests must include a request body\`);
}

return { valid: issues.length === 0, issues };
}

// Test cases
const tests = [
{ method: "GET", path: "/users/42", body: null, label: "Valid GET" },
{ method: "GET", path: "/getUsers", body: null, label: "Verb in path" },
{ method: "DELETE", path: "/posts/7", body: { force: true}, label: "DELETE with body" },
{ method: "POST", path: "/orders", body: { item: "book"},label: "Valid POST" },
{ method: "POST", path: "/Orders", body: { item: "x" }, label: "Uppercase path" },
{ method: "PUT", path: "/users/1", body: null, label: "PUT without body" },
];

for (const { method, path, body, label } of tests) {
const result = validateRestRequest(method, path, body);
console.log(\`[\${label}] valid=\${result.valid}\`);
if (result.issues.length > 0) {
result.issues.forEach(i => console.log(" issue:", i));
}
}`}

## What you learned

- REST is an architectural style defined by six constraints, not a protocol or standard.
- Resources are nouns in URLs; HTTP verbs (GET, POST, PUT, PATCH, DELETE) express what to do with them.
- Statelessness means every request is self-contained — the server stores no session state between calls.
- HTTP status codes carry semantic meaning: 2xx success, 4xx client error, 5xx server error.
- The Richardson Maturity Model describes four levels of REST compliance; Level 2 (resources + verbs) is the practical target.
