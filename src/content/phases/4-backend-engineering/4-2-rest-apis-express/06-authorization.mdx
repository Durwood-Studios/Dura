## Why this matters

Authentication tells you who someone is. Authorization tells you what they can do. Without authorization, every authenticated user could read every other user's private data, delete records that do not belong to them, or call admin endpoints that should be locked down. Authorization bugs are security vulnerabilities — and unlike authentication, where failure is obvious (wrong password), authorization failures are often silent: a user gets data they should not have, and nobody notices until there is a breach. Building authorization correctly from the start is far easier than auditing and patching it later.

## 401 vs 403: the important distinction

These two HTTP status codes are related but carry different meanings:

**401 Unauthorized** — The request did not include valid authentication credentials. The server does not know who you are. Fix: log in and include a valid token.

**403 Forbidden** — The request included valid credentials, but the authenticated user does not have permission to perform this action. The server knows exactly who you are — you are just not allowed. Fix: get the right permissions (or stop trying to access things you should not).

Getting these backwards confuses API consumers. A 403 from a login endpoint makes no sense. A 401 from an admin dashboard implies the user just needs to log in, when in fact they will never have access.

## Role-Based Access Control (RBAC)

The most common authorization model is **RBAC**: assign users to roles, and define what each role can do.

A simple three-role system covers most small-to-medium applications:

| Role   | Can do                                               |
| ------ | ---------------------------------------------------- |
| admin  | Everything — create, read, update, delete any record |
| editor | Create and modify their own resources; read all      |
| viewer | Read only                                            |

Roles are typically stored with the user record in the database. For convenience, they are often included in the JWT payload so you do not need a database lookup on every request:

```javascript
// At login — include role in the token payload
const token = jwt.sign(
  { userId: user.id, email: user.email, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: "15m" }
);

// In the requireAuth middleware — role is already decoded
req.user = { userId: payload.userId, email: payload.email, role: payload.role };
```

## Middleware guards

Authorization logic lives in middleware — reusable guards that run before route handlers:

```javascript
// requireAuth — checks JWT is valid (from lesson 05)
function requireAuth(req, res, next) {
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith("Bearer ")) {
    return res.status(401).json({ error: "Authentication required" });
  }
  try {
    req.user = jwt.verify(authHeader.slice(7), process.env.JWT_SECRET);
    next();
  } catch {
    return res.status(401).json({ error: "Invalid or expired token" });
  }
}

// requireRole — checks the user's role is in the allowed list
function requireRole(...roles) {
  return (req, res, next) => {
    if (!req.user) {
      return res.status(401).json({ error: "Authentication required" });
    }
    if (!roles.includes(req.user.role)) {
      return res.status(403).json({ error: "Insufficient permissions" });
    }
    next();
  };
}

// requireOwner — checks that the user owns the resource
function requireOwner(getResourceOwner) {
  return async (req, res, next) => {
    if (!req.user) {
      return res.status(401).json({ error: "Authentication required" });
    }
    // Admins bypass ownership checks
    if (req.user.role === "admin") {
      return next();
    }
    try {
      const ownerId = await getResourceOwner(req.params.id);
      if (ownerId !== req.user.userId) {
        return res.status(403).json({ error: "You do not own this resource" });
      }
      next();
    } catch (err) {
      next(err);
    }
  };
}
```

## Composing guards on routes

Guards compose naturally in Express because middleware runs in sequence:

```javascript
import { getPost, deletePost } from "./models/posts.js";

// Public — no auth needed
router.get("/posts", listPosts);
router.get("/posts/:id", getPost);

// Requires authentication (any role)
router.post("/posts", requireAuth, createPost);

// Requires admin role
router.delete("/posts/:id", requireAuth, requireRole("admin"), async (req, res) => {
  await deletePost(req.params.id);
  res.status(204).send();
});

// Requires ownership (or admin)
router.put(
  "/posts/:id",
  requireAuth,
  requireOwner((id) => getPost(id).then((p) => p.userId)),
  async (req, res) => {
    const updated = await updatePost(req.params.id, req.body);
    res.json(updated);
  }
);
```

The `requireOwner` pattern is important: look up the resource to find its owner, compare to `req.user.userId`. Admins skip this check. Non-owners get 403.

## Where to store roles: JWT vs database

There is a real tradeoff between putting roles in the JWT payload versus looking them up on every request:

**Roles in JWT payload:**

- No database lookup per request — faster
- Roles are stale until the token expires: if you revoke admin access, the user keeps admin until their token expires
- Acceptable for most applications with short token expiry (15 minutes)

**Roles from database on every request:**

- Always current — role changes take effect immediately
- Costs one database query per request
- Required for high-security systems where immediate revocation is necessary

For most applications: put roles in the JWT, use short expiry (15 minutes), and let refresh tokens issue new tokens with updated roles. This gives you near-real-time role propagation at low cost.

## Attribute-Based Access Control (ABAC)

RBAC covers most use cases, but sometimes you need finer control. ABAC (Attribute-Based Access Control) bases decisions on any attribute of the user, resource, or environment:

- "A user can edit a document if they are in the same organization as the document owner"
- "Requests from IP ranges in the allowlist can access admin endpoints"
- "Users can only see records from their own team"

ABAC is more flexible but more complex to implement and audit. For most applications, start with RBAC. Add ABAC selectively where RBAC is not granular enough.

{`// Build an authorization system with roles and resource ownership.
// createAuthz(userRoles) — userRoles is Map<userId, role>
// Returns { can(userId, action, resourceOwnerId?) }
// Roles: admin > editor > viewer
// Actions: "read", "write", "delete"

function createAuthz(userRoles) {
// Permission table: which roles can perform which actions
const permissions = {
admin: {
read: () => true,
write: () => true,
delete: () => true
},
editor: {
read: () => true,
// Editors can write/delete only their own resources
write: (userId, ownerId) => ownerId === undefined || userId === ownerId,
delete: (userId, ownerId) => ownerId === undefined || userId === ownerId
},
viewer: {
read: () => true,
write: () => false,
delete: () => false
}
};

return {
can(userId, action, resourceOwnerId) {
const role = userRoles.get(userId);
if (!role) return false; // Unknown user — no access
const rolePerms = permissions[role];
if (!rolePerms) return false;
const check = rolePerms[action];
if (!check) return false;
return check(userId, resourceOwnerId);
}
};
}

// Setup: assign roles to users
const userRoles = new Map([
["alice", "admin"],
["bob", "editor"],
["carol", "editor"],
["dave", "viewer"]
]);

const authz = createAuthz(userRoles);

// Test scenarios
const tests = [
{ user: "alice", action: "delete", owner: "carol", desc: "Admin deletes carol's resource" },
{ user: "bob", action: "write", owner: "bob", desc: "Editor writes their own resource" },
{ user: "bob", action: "write", owner: "carol", desc: "Editor writes carol's resource" },
{ user: "carol", action: "delete", owner: "carol", desc: "Editor deletes their own resource" },
{ user: "dave", action: "read", owner: undefined, desc: "Viewer reads public list" },
{ user: "dave", action: "write", owner: "dave", desc: "Viewer writes their own resource" },
{ user: "dave", action: "delete", owner: "dave", desc: "Viewer deletes their own resource" },
{ user: "eve", action: "read", owner: undefined, desc: "Unknown user reads" },
];

for (const { user, action, owner, desc } of tests) {
const allowed = authz.can(user, action, owner);
console.log(\`[\${allowed ? "ALLOW" : "DENY "}] \${desc} (\${user} → \${action})\`);
}`}

## What you learned

- Authorization determines what an authenticated user can do; it always follows authentication.
- 401 means not authenticated; 403 means authenticated but not permitted — never confuse them.
- RBAC assigns users to roles (admin, editor, viewer) and defines what each role may do.
- Middleware guards (`requireAuth`, `requireRole`, `requireOwner`) compose cleanly on Express routes.
- Storing roles in JWT payloads is fast but stale until token renewal; short expiry minimizes the staleness window.
