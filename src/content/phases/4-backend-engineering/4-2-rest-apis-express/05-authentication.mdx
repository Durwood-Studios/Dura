## Why this matters

Authentication is the front door of every protected API. Get it wrong and attackers can impersonate users, steal data, or lock legitimate users out of their own accounts. The two most common mistakes are storing passwords in plain text (a catastrophic breach waiting to happen) and using insecure session management (predictable tokens, no expiry, improper storage). The patterns you will learn here — bcrypt for password storage, JWTs for stateless session tokens — are the same ones used by production systems handling millions of users.

## Authentication vs authorization

These words are often confused because they work together. They are different things:

**Authentication** proves identity: "Who are you?" A user provides credentials (username + password, OAuth token, API key) and the server verifies them. The result is a confirmed identity.

**Authorization** determines what an identity can do: "What are you allowed to do?" Covered in the next lesson.

You must authenticate before you can authorize. Authentication comes first.

## The register/login flow

```
REGISTER:
  1. Client sends { email, password }
  2. Server validates input
  3. Server hashes the password with bcrypt
  4. Server stores { email, hashedPassword } in the database
  5. Server returns 201 (or immediately logs the user in with a token)

LOGIN:
  1. Client sends { email, password }
  2. Server finds the user by email
  3. Server compares the plaintext password against the stored hash with bcrypt
  4. If they match, server creates a JWT signed with a secret
  5. Server returns the JWT in the response
  6. Client stores the JWT and sends it in Authorization header on future requests

PROTECTED REQUEST:
  1. Client sends Authorization: Bearer <token>
  2. Server extracts and verifies the JWT
  3. If valid and not expired, request proceeds
  4. If invalid or expired, server returns 401 Unauthorized
```

## Passwords: bcrypt

Never store passwords in plain text. Never store them with a reversible cipher. Only store a one-way hash — specifically, a **bcrypt** hash.

bcrypt works by:

1. Generating a random **salt** (unique per-hash, prevents rainbow table attacks)
2. Combining the salt with the password
3. Applying an expensive hashing function a configurable number of times (the **work factor** or rounds)
4. Producing a hash that includes the salt and the work factor

Because the salt is embedded in the hash, the same password produces a different hash every time — `bcrypt.hash('password', 10)` returns something different each call. Comparison works because `bcrypt.compare()` extracts the salt from the stored hash and re-derives the key.

```javascript
import bcrypt from "bcrypt";

// Hashing — happens at registration
const ROUNDS = 12; // Higher = slower = more secure. 10-12 is typical.
const hashedPassword = await bcrypt.hash(plainTextPassword, ROUNDS);
// Store hashedPassword in the database. Never store plainTextPassword.

// Comparing — happens at login
const isMatch = await bcrypt.compare(loginAttempt, hashedPassword);
// isMatch is true only if loginAttempt is the same password that was hashed
```

The work factor is crucial. 12 rounds means bcrypt runs the algorithm 2^12 = 4,096 times. An attacker cracking hashes offline can only try a few hundred per second instead of billions. As CPUs get faster, increase the rounds.

## JWTs: JSON Web Tokens

A JWT is a compact, self-contained token that proves who a user is. It has three parts separated by dots, each base64url-encoded:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOiI0MiIsImVtYWlsIjoiYWxpY2VAZXhhbXBsZS5jb20iLCJleHAiOjE3MDAwMDAwMDB9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Header.Payload.Signature
```

**Header** — algorithm and token type: `{"alg": "HS256", "typ": "JWT"}`

**Payload** — claims (data about the user): `{"userId": "42", "email": "alice@example.com", "exp": 1700000000}`

**Signature** — HMAC of `header.payload` using your server's secret key. Tamper with any part of the token and the signature check fails.

```javascript
import jwt from "jsonwebtoken";

const JWT_SECRET = process.env.JWT_SECRET; // Must be long, random, secret

// Issue a token at login
const token = jwt.sign(
  { userId: user.id, email: user.email }, // payload (claims)
  JWT_SECRET,
  { expiresIn: "15m" } // short expiry — use refresh tokens for longer sessions
);

// Verify on every protected request
function requireAuth(req, res, next) {
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith("Bearer ")) {
    return res.status(401).json({ error: "No token provided" });
  }
  const token = authHeader.slice(7);
  try {
    const payload = jwt.verify(token, JWT_SECRET);
    req.user = payload; // Attach decoded user to the request
    next();
  } catch (err) {
    // jwt.verify throws on invalid signature, expiry, etc.
    return res.status(401).json({ error: "Invalid or expired token" });
  }
}

// Protect a route
app.get("/me", requireAuth, (req, res) => {
  res.json({ userId: req.user.userId, email: req.user.email });
});
```

## Token storage: cookies vs localStorage

Where clients store JWTs is a security decision with real tradeoffs:

**httpOnly Cookie** — the browser stores the token but JavaScript cannot access it (`document.cookie` returns nothing). This prevents XSS attacks from stealing the token. The browser automatically sends the cookie with every request to your domain. Requires CSRF protection. Best choice for web apps.

**localStorage / sessionStorage** — JavaScript can read the token, which means any XSS vulnerability can steal it. Easier to implement. Does not require CSRF protection. Acceptable for SPAs that already have strong XSS mitigations, or for mobile/API clients.

For most web applications, prefer httpOnly cookies. The XSS protection is worth the extra CSRF setup.

## Refresh tokens

Short-lived access tokens (15 minutes) mean users get logged out frequently — bad UX. Refresh tokens solve this:

- **Access token**: short-lived (15 min), sent in every request header
- **Refresh token**: long-lived (30 days), stored securely, used only to get new access tokens
- When the access token expires, the client sends the refresh token to `POST /auth/refresh`
- The server verifies the refresh token, issues a new access token

This pattern lets you revoke access by invalidating refresh tokens (store them in a database and delete them on logout) while keeping access tokens stateless.

const padded = str.replace(/-/g, '+').replace(/\_/g, '/') + '=='.slice(0, (4 - str.length % 4) % 4);

// Mock HMAC: XOR each char code of payload with the secret's char codes
function mockSign(data, secret) {
let result = 0;
for (let i = 0; i < data.length; i++) {
result ^= data.charCodeAt(i) \* (secret.charCodeAt(i % secret.length) + 1);
}
return Math.abs(result).toString(36);
}

function createJWT(payload, secret) {
const header = base64urlEncode({ alg: "mock", typ: "JWT" });
const body = base64urlEncode({
...payload,
iat: Math.floor(Date.now() / 1000),
exp: payload.exp || Math.floor(Date.now() / 1000) + 900 // 15 min default
});
const signature = mockSign(header + "." + body, secret);
return \`\${header}.\${body}.\${signature}\`;
}

function verifyJWT(token, secret) {
const parts = token.split(".");
if (parts.length !== 3) throw new Error("Malformed token");
const [header, body, sig] = parts;
const expectedSig = mockSign(header + "." + body, secret);
if (sig !== expectedSig) throw new Error("Invalid signature — token was tampered with");
const payload = base64urlDecode(body);
if (!payload) throw new Error("Malformed payload");
return payload;
}

const parts = token.split(".");
if (parts.length !== 3) return true;
const payload = base64urlDecode(parts[1]);
if (!payload || !payload.exp) return true;
return payload.exp < Math.floor(Date.now() / 1000);
}

## What you learned

- Authentication proves identity; authorization determines permissions — authentication comes first.
- bcrypt hashes passwords with a unique random salt, making stored hashes irreversible and rainbow-table-resistant.
- JWTs consist of a header, payload, and signature — tampering with any part invalidates the signature check.
- Use httpOnly cookies for web apps to protect tokens from XSS attacks; localStorage is acceptable for API-only clients.
- Refresh tokens keep users logged in without compromising security — short-lived access tokens plus revocable refresh tokens.
