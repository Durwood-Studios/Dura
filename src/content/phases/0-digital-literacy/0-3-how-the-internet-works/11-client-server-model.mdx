## Why this matters

Almost every networked system you'll ever build follows the same pattern: one side asks, the other side answers. Knowing this pattern by name lets you reason about web apps, mobile apps, APIs, databases, and microservices using a single mental model.

## The two roles

A **client** asks for things. A **server** waits for requests and answers them. That's the entire model.

- Your browser is a client.
- The DURA web server is a server.
- Your phone's weather app is a client.
- The weather service is a server.

A given machine can play either role depending on context. When your laptop fetches a webpage, it's a client. When you run a tiny dev server on the same laptop, it's also a server.

## The cycle

Every interaction is a loop:

1. Client decides it needs something.
2. Client sends a request.
3. Server processes it.
4. Server sends a response.
5. Client does something with the response.

That's a single round trip. A modern web page loads dozens or hundreds of these in parallel — one for the HTML, one for each image, one for each script, one for each font.

```text
Client                Server
  |                     |
  |  ─── request ───>   |
  |                     |  (does work)
  |  <── response ───   |
  |                     |
```

## Why this model wins

Servers can handle many clients at once. They can be upgraded centrally without anyone reinstalling the client. They can hold the data that all clients share. Clients stay simple — they just need to know how to ask.

The downside: if the server goes down, every client loses access. The whole pattern depends on at least one party being reachable.

<SandboxExercise
  language="javascript"
  instructions="Build a tiny client/server pair. The server has a price list; the client requests prices for given items."

  prices: { apple: 1, bread: 3, milk: 4 },

prices: { apple: 1, bread: 3, milk: 4 },
handleRequest(item) {
return this.prices[item] ?? "not found";
}
};

function client(item) {
return server.handleRequest(item);
}

console.log(client("bread"));
console.log(client("steak"));`}

## What you learned

- A client asks; a server answers. Every networked interaction follows this pattern.
- A round trip is request → process → response. Real pages run many in parallel.
- The same machine can be a client in one conversation and a server in another.
