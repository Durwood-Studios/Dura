## Why this matters

Git lives on your computer. If your hard drive dies, your local commits go with it. **GitHub** (and similar services like GitLab and Bitbucket) is where developers put their Git repositories on the internet, so they're safe, shareable, and collaborate-able. The combination of Git + GitHub is the bedrock of modern software development.

## Sign up for GitHub

Go to `github.com` and create an account. Use your real name; this is professional infrastructure, not a social network. Pick a username you wouldn't mind a recruiter seeing.

GitHub is free for almost everything. The only paid features matter to large organizations. As an individual learner, you'll never hit a paywall.

## Create a repository

Click the green "New" button in the upper-left of your dashboard. Give it a name (`my-first-repo`), check "Initialize with a README," and click create.

Congratulations: you have a remote repository.

## Connect your local repo to it

If you already have local commits from the previous lesson, you can push them up:

```text
$ git remote add origin https://github.com/yourname/my-first-repo.git
$ git branch -M main
$ git push -u origin main
```

`git remote add` tells your local repo where the remote one lives. `git push` uploads your commits.

## The push/pull cycle

This becomes your daily rhythm:

1. **Pull** — `git pull` downloads any changes others made.
2. **Work** — edit, add, commit locally.
3. **Push** — `git push` uploads your local commits.

If two people change the same file at the same time, Git tries to merge their changes automatically. When it can't, you get a **merge conflict** — Git asks you to manually pick which version to keep. We'll cover those properly in Phase 8.

## The README

Every repository should have a README — a Markdown file that explains what the project is, how to install it, and how to use it. It's the first thing visitors see on GitHub.

A good README doesn't have to be long. It just has to answer "what is this and why should I care?"

## What you learned

- GitHub is where Git repositories live on the internet — it's not a separate tool, just a hosting service.
- The daily cycle is pull → work → push.
- Merge conflicts happen when two people change the same lines; you resolve them by hand.
