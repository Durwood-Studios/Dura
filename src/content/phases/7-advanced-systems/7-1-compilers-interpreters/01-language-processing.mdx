## Why this matters

Every time you run `node script.js` or `gcc main.c`, an invisible pipeline fires. Your text file — meaningless to the CPU — gets dismantled, analyzed, and reconstructed into something the machine understands. That pipeline is the same whether you write C, Python, Rust, or JavaScript. Understanding it changes how you read error messages, how you structure code, and how you reason about performance.

Compilers are not academic curiosities. ESLint is a partial compiler. Babel is a compiler. TypeScript is a compiler. Prettier is a compiler. If you use any modern JavaScript toolchain, you already depend on compiler technology daily.

## The language processing pipeline

Every language processor follows roughly the same sequence of stages:

```javascript
// The pipeline at a glance
const pipeline = [
  "Source Code", // what you write
  "Tokens", // words and symbols, tagged by type
  "AST", // a tree that captures the structure
  "Intermediate Rep", // a simplified, optimized form
  "Output", // machine code, bytecode, or result
];

// Each stage transforms one representation into the next.
// Errors at each stage have different characteristics:
// - Token stage:  "Unexpected character @"
// - Parse stage:  "Unexpected token }"
// - Semantic:     "Cannot read property of undefined"
// - Runtime:      "Stack overflow"
```

The first half — source to AST — is called the **front end**. The second half — AST to output — is the **back end**. This separation is why a single language can target multiple platforms, and why a single backend (like LLVM) can serve multiple languages.

## Compiled vs interpreted

A **compiler** translates the entire program before any of it runs. The output is a file (binary, bytecode, or another language) you can execute later. C, Rust, and Go are compiled.

An **interpreter** executes the program as it reads it, statement by statement. There is no separate output file. Python's default implementation and early JavaScript engines worked this way.

```javascript
// Simulating the difference in concept

function compileAndRun(source) {
  const compiled = compileToInstructions(source); // full pass first
  return execute(compiled); // then run
}

function interpret(source) {
  const lines = source.split("\n");
  for (const line of lines) {
    executeImmediately(line); // one at a time, no full pass
  }
}
```

In practice, the line is blurry. Most modern "interpreted" languages use a hybrid approach.

## JIT compilation: the best of both worlds

Modern JavaScript engines (V8, SpiderMonkey) do not simply interpret. They use **Just-In-Time (JIT) compilation**: start interpreting quickly, then watch which functions run often ("hot" code), and compile those to optimized machine code while the program is still running.

```javascript
// Conceptual model of V8's pipeline
function v8Pipeline(source) {
  const ast = parse(source);
  const bytecode = generateBytecode(ast); // Ignition interpreter

  // Run bytecode. Profile which functions are "hot."
  const profile = runAndProfile(bytecode);

  // Compile hot functions to optimized machine code
  for (const fn of profile.hotFunctions) {
    const optimized = turbofanCompile(fn); // TurboFan compiler
    replaceInRuntime(fn, optimized);
  }
}
```

This is why a JavaScript loop that runs millions of times can approach C-like speed — V8 compiles it to native machine code after observing it.

## The spectrum of language processing

Languages sit on a spectrum, not in binary categories:

```javascript
const spectrum = {
  "Fully compiled (AOT)": ["C", "Rust", "Go", "Haskell"],
  "Compiled to bytecode": ["Java (JVM)", "C# (CLR)", "Erlang (BEAM)"],
  "JIT compiled": ["JavaScript (V8)", "Java (HotSpot)", "C# (RyuJIT)"],
  "Interpreted + bytecode": ["Python (CPython)", "Ruby (YARV)"],
  Transpiled: ["TypeScript → JS", "Babel → JS", "Sass → CSS"],
};

// Transpilation is compilation where the output is another
// high-level language rather than machine code or bytecode.
```

TypeScript is a transpiler — it compiles TypeScript into JavaScript, then V8 JIT-compiles that JavaScript. Multiple stages, multiple compilers, one pipeline.

## Why every engineer should understand this

Knowing the pipeline helps you:

1. **Read error messages** — a syntax error (parse stage) vs a type error (semantic stage) vs a runtime crash have different root causes and different fixes.
2. **Write faster code** — understanding what JIT compilers optimize helps you avoid patterns that force deoptimization.
3. **Use tools better** — ESLint rules, Prettier formatting, Babel transforms, and TypeScript type checking all operate on ASTs. They are compiler stages you configure daily.

```javascript
// This is what ESLint actually does internally (simplified)
function lintFile(source) {
  const tokens = tokenize(source); // lexing
  const ast = parse(tokens); // parsing
  const violations = [];

  walkTree(ast, (node) => {
    // AST traversal
    for (const rule of enabledRules) {
      const result = rule.check(node);
      if (result) violations.push(result);
    }
  });

  return violations;
}
```

## What you learned

- Every language processor follows the same pipeline: source code → tokens → AST → intermediate representation → output.
- Compiled languages translate everything before execution; interpreted languages execute as they read; JIT compilation is a hybrid that optimizes hot code at runtime.
- Tools you use daily — ESLint, Prettier, TypeScript, Babel — are all compiler technology operating on ASTs.
