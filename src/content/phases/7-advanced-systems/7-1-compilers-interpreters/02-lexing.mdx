## Why this matters

Before a compiler can understand your code, it has to read it — character by character. The string `let x = 42 + y;` is just 16 characters to the machine. It has no idea that `let` is a keyword, `x` is a name, or `42` is a number. The **lexer** (also called a scanner or tokenizer) is the stage that transforms this flat character stream into a sequence of labeled chunks called **tokens**.

Every language tool you use starts here. When your editor highlights keywords in blue and strings in green, that is a lexer at work. When ESLint reports "unexpected token," a lexer found something it could not classify. Building one yourself removes the mystery.

## What a token is

A token is a chunk of source text plus a label describing what kind of chunk it is.

```javascript
// The source string "let x = 42 + y;"
// becomes these tokens:

const tokens = [
  { type: "Keyword", value: "let" },
  { type: "Identifier", value: "x" },
  { type: "Equals", value: "=" },
  { type: "Number", value: "42" },
  { type: "Plus", value: "+" },
  { type: "Identifier", value: "y" },
  { type: "Semicolon", value: ";" },
];
```

Whitespace and comments are consumed but (usually) not emitted as tokens — they separate tokens but carry no semantic meaning. Some tools (like Prettier) do preserve whitespace tokens because formatting is their job.

## Token types

Most languages need a small set of token types:

```javascript
const TOKEN_TYPES = {
  // Single-character
  PLUS: "+",
  MINUS: "-",
  STAR: "*",
  SLASH: "/",
  LPAREN: "(",
  RPAREN: ")",
  EQUALS: "=",
  SEMICOLON: ";",

  // Literals
  NUMBER: "NUMBER", // 42, 3.14
  STRING: "STRING", // "hello"

  // Words
  IDENTIFIER: "IDENTIFIER", // x, foo, myVar
  KEYWORD: "KEYWORD", // let, if, return

  // Special
  EOF: "EOF", // end of input
};
```

The lexer's job is to decide, for each chunk of characters, which type it belongs to.

## Pattern matching with regular expressions

Each token type corresponds to a pattern. Regular expressions describe these patterns concisely:

```javascript
const patterns = [
  { type: "Number", regex: /^\d+(\.\d+)?/ },
  { type: "String", regex: /^"[^"]*"/ },
  { type: "Keyword", regex: /^(let|if|else|return|while|function)\b/ },
  { type: "Identifier", regex: /^[a-zA-Z_]\w*/ },
  { type: "Plus", regex: /^\+/ },
  { type: "Minus", regex: /^-/ },
  { type: "Star", regex: /^\*/ },
  { type: "Slash", regex: /^\// },
  { type: "Equals", regex: /^=/ },
  { type: "LParen", regex: /^\(/ },
  { type: "RParen", regex: /^\)/ },
  { type: "Semicolon", regex: /^;/ },
];

// The ^ anchor ensures we match at the current position.
// Order matters: "Keyword" must come before "Identifier"
// so that "let" is classified as a keyword, not an identifier.
```

## Building a lexer

A lexer walks through the source string, matching patterns at each position and advancing past whatever it matches:

```javascript
function tokenize(source) {
  const tokens = [];
  let pos = 0;

  while (pos < source.length) {
    // Skip whitespace
    const wsMatch = source.slice(pos).match(/^\s+/);
    if (wsMatch) {
      pos += wsMatch[0].length;
      continue;
    }

    let matched = false;

    for (const { type, regex } of patterns) {
      const match = source.slice(pos).match(regex);
      if (match) {
        tokens.push({ type, value: match[0] });
        pos += match[0].length;
        matched = true;
        break;
      }
    }

    if (!matched) {
      throw new Error(`Unexpected character '${source[pos]}' at position ${pos}`);
    }
  }

  tokens.push({ type: "EOF", value: "" });
  return tokens;
}
```

The `throw` at the end is how lexers produce those "unexpected character" errors you have seen in every programming language.

## Seeing it work

```javascript
const source = "let x = 42 + y;";
const result = tokenize(source);

for (const token of result) {
  console.log(`${token.type.padEnd(12)} ${token.value}`);
}

// Output:
// Keyword      let
// Identifier   x
// Equals       =
// Number       42
// Plus         +
// Identifier   y
// Semicolon    ;
// EOF
```

Every subsequent stage — parsing, analysis, code generation — works with these tokens, never with raw characters again. The lexer draws the boundary between messy text and structured data.

## Edge cases that break lexers

Real lexers handle complications that our simple version skips:

```javascript
// Multi-character operators: ==, !=, <=, >=, &&, ||
// The lexer must look ahead to distinguish = from ==

// String escapes: "she said \"hello\""
// The regex /^"[^"]*"/ would fail here

// Nested comments: /* outer /* inner */ still in comment */
// Requires a counter, not a simple regex

// Unicode identifiers: const café = true;
// Pattern must handle characters beyond ASCII

// Numbers: 0xFF, 1_000_000, 3.14e-10
// Each format needs its own pattern branch
```

Production lexers (like the one in V8) are hand-written state machines for speed. Our regex-based approach is correct but slower. The concepts are identical.

## What you learned

- A **lexer** (scanner/tokenizer) transforms raw characters into labeled tokens — the first stage of every language processor.
- Each token type is defined by a pattern (often a regular expression), and order of pattern matching matters for correct classification.
- Building a lexer is straightforward: walk the string, match patterns at each position, advance past the match, and throw on unrecognized characters.
