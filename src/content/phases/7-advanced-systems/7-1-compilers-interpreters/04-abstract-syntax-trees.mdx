## Why this matters

An **Abstract Syntax Tree** (AST) is the representation that every serious code tool operates on. ESLint traverses an AST to find violations. Prettier traverses an AST to reformat your code. Babel traverses an AST to transform modern syntax into older JavaScript. TypeScript builds an AST to type-check your program.

If you understand ASTs, you understand how all of these tools work — and you can write your own custom lint rules, code transforms, and analysis scripts. The AST is where raw text becomes something a program can reason about.

## What makes it "abstract"

A **concrete** syntax tree mirrors the exact grammar, including every parenthesis, semicolon, and keyword. An **abstract** syntax tree strips away the syntactic noise and keeps only the structural meaning:

```javascript
// Source: (2 + 3) * 4
//
// Concrete syntax tree (includes parentheses as nodes):
// Expr → '(' Expr '+' Expr ')' '*' Expr
//
// Abstract syntax tree (parentheses are implicit in structure):
// BinaryExpr(*)
//   left:  BinaryExpr(+)
//            left:  NumberLiteral(2)
//            right: NumberLiteral(3)
//   right: NumberLiteral(4)

const ast = {
  type: "BinaryExpr",
  op: "*",
  left: {
    type: "BinaryExpr",
    op: "+",
    left: { type: "NumberLiteral", value: 2 },
    right: { type: "NumberLiteral", value: 3 },
  },
  right: { type: "NumberLiteral", value: 4 },
};
```

The parentheses disappear because the tree structure itself encodes the grouping. The `+` is a child of the `*`, so it evaluates first. No parentheses needed.

## Common AST node types

A real language has many node types. Here are the most common:

```javascript
// Literals — leaf nodes that hold values
const numberNode = { type: "NumberLiteral", value: 42 };
const stringNode = { type: "StringLiteral", value: "hello" };
const boolNode = { type: "BooleanLiteral", value: true };

// Identifiers — references to names
const idNode = { type: "Identifier", name: "x" };

// Expressions — compute values
const binaryNode = {
  type: "BinaryExpr",
  op: "+",
  left: { type: "Identifier", name: "a" },
  right: { type: "NumberLiteral", value: 1 },
};

const unaryNode = {
  type: "UnaryExpr",
  op: "-",
  operand: { type: "NumberLiteral", value: 5 },
};

const callNode = {
  type: "CallExpr",
  callee: { type: "Identifier", name: "print" },
  args: [{ type: "StringLiteral", value: "hello" }],
};

// Statements — perform actions
const varDecl = {
  type: "VarDeclaration",
  name: "x",
  initializer: { type: "NumberLiteral", value: 10 },
};

const ifStmt = {
  type: "IfStatement",
  condition: { type: "BinaryExpr", op: ">", left: idNode, right: numberNode },
  then: { type: "Block", statements: [] },
  else: null,
};
```

Every AST is a tree of these nodes. The root is usually a `Program` node containing a list of top-level statements.

## Walking the tree

To do anything useful with an AST, you need to visit every node. A **tree walker** recursively traverses the structure:

```javascript
function walk(node, visitor) {
  // Call the visitor for this node type
  if (visitor[node.type]) {
    visitor[node.type](node);
  }

  // Recurse into child nodes
  for (const key of Object.keys(node)) {
    const child = node[key];
    if (child && typeof child === "object" && child.type) {
      walk(child, visitor);
    }
    if (Array.isArray(child)) {
      for (const item of child) {
        if (item && typeof item === "object" && item.type) {
          walk(item, visitor);
        }
      }
    }
  }
}

// Count how many number literals appear in an AST
let count = 0;
walk(ast, {
  NumberLiteral(node) {
    count++;
    console.log("Found number:", node.value);
  },
});
console.log("Total numbers:", count);
```

## The visitor pattern

The tree walker above implements the **visitor pattern** — a design pattern where you separate the traversal logic from the operation logic. You write a visitor object with methods named after node types, and the walker calls the right method for each node:

```javascript
// A visitor that finds all identifier names
const identifierCollector = {
  Identifier(node) {
    this.names.push(node.name);
  },
  names: [],
};

// A visitor that checks for division by zero
const divisionChecker = {
  BinaryExpr(node) {
    if (node.op === "/" && node.right.type === "NumberLiteral" && node.right.value === 0) {
      console.warn("Warning: division by zero detected");
    }
  },
};
```

This is exactly how ESLint works. Each lint rule is a visitor that reacts to specific node types. The ESLint engine walks the AST once and notifies every active rule at each node.

## Real-world ASTs

You can see real JavaScript ASTs at astexplorer.net. Paste any code and it shows the tree. Here is what `let x = 1 + 2;` looks like in the standard ESTree format:

```javascript
// Simplified ESTree output for: let x = 1 + 2;
const estree = {
  type: "Program",
  body: [
    {
      type: "VariableDeclaration",
      kind: "let",
      declarations: [
        {
          type: "VariableDeclarator",
          id: { type: "Identifier", name: "x" },
          init: {
            type: "BinaryExpression",
            operator: "+",
            left: { type: "Literal", value: 1 },
            right: { type: "Literal", value: 2 },
          },
        },
      ],
    },
  ],
};

// Every JavaScript tool agrees on this shape (ESTree standard).
// That is why ESLint plugins, Babel plugins, and Prettier
// plugins can all operate on the same AST.
```

## What you learned

- An **AST** strips syntactic noise from parsed code and represents structure as a tree of typed nodes — the universal data structure for code analysis tools.
- **Tree walking** with the **visitor pattern** is how linters, formatters, transpilers, and type checkers all operate: traverse the tree, react to specific node types.
- Understanding ASTs lets you write custom lint rules, code transforms, and analysis tools using the same techniques that power ESLint, Babel, and TypeScript.
