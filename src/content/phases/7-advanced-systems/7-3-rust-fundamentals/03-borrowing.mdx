## Why this matters

Ownership without borrowing would be unusable. Every function call would consume its arguments, and you would spend most of your time passing values back and forth. Borrowing lets you lend data to a function without giving up ownership — like letting someone read your book without giving it away. The borrowing rules are what make Rust's concurrency story so powerful: they prevent data races at compile time, something no other mainstream language achieves without a runtime cost.

## Immutable references: &T

An **immutable reference** lets you read data without taking ownership. You create one with the `&` operator. The original owner keeps ownership, and the reference cannot modify the data.

```rust
fn print_length(s: &String) {
    println!("Length: {}", s.len());
} // s goes out of scope, but it was just a reference — nothing is dropped

fn main() {
    let greeting = String::from("hello");
    print_length(&greeting);   // Borrow greeting — do not move it
    println!("{}", greeting);   // Still valid — we only lent it out
}
```

You can have **multiple immutable references** at the same time. This is safe because nobody is modifying the data — it is like multiple people reading the same book simultaneously.

```rust
let s = String::from("shared");
let r1 = &s;
let r2 = &s;
let r3 = &s;
println!("{} {} {}", r1, r2, r3); // All valid — read-only access
```

## Mutable references: &mut T

A **mutable reference** lets you read and modify borrowed data. You create one with `&mut`. But there is a critical restriction: you can have **exactly one** mutable reference at a time, and no immutable references can coexist with it.

```rust
fn add_exclamation(s: &mut String) {
    s.push_str("!");
}

fn main() {
    let mut greeting = String::from("hello");
    add_exclamation(&mut greeting);
    println!("{}", greeting); // "hello!"
}
```

## The borrowing rule

This is the rule the borrow checker enforces:

> At any given time, you can have **either** one mutable reference **or** any number of immutable references — but **never both**.

```rust
let mut data = String::from("hello");

let r1 = &data;      // Immutable borrow — OK
let r2 = &data;      // Another immutable borrow — OK
// let r3 = &mut data; // ERROR: cannot borrow as mutable while immutable borrows exist

println!("{} {}", r1, r2);
// r1 and r2 are no longer used after this point

let r3 = &mut data;  // OK now — no active immutable borrows
r3.push_str(" world");
```

Rust's borrow checker is smart about **non-lexical lifetimes** — a reference's lifetime ends at its last use, not at the end of the scope. This is why `r3` works in the example above: `r1` and `r2` are no longer active.

## Why this prevents data races

A **data race** occurs when two or more threads access the same data concurrently, at least one is writing, and there is no synchronization. Data races cause corrupted memory, crashes, and security vulnerabilities that are notoriously hard to reproduce and debug.

Rust's borrowing rules make data races **structurally impossible**:

- If multiple threads hold references to the same data, they must all be immutable — no writes possible.
- If one thread holds a mutable reference, no other thread can access the data at all.

This guarantee is enforced at compile time. There is no mutex you might forget to lock, no race condition hiding in a code path that only triggers under load. The compiler simply will not let you write a data race.

```rust
// This will NOT compile — Rust prevents the data race
// let mut data = vec![1, 2, 3];
// let r = &data;
// data.push(4);        // ERROR: cannot mutate while an immutable borrow exists
// println!("{:?}", r); // r would see inconsistent data if push reallocated
```

## Borrowing and function signatures

Function signatures in Rust communicate ownership intent:

```rust
fn takes_ownership(s: String) { }     // Caller loses ownership
fn borrows(s: &String) { }            // Caller keeps ownership, read-only
fn borrows_mutably(s: &mut String) { } // Caller keeps ownership, allows mutation
```

This is part of Rust's API design philosophy. By reading a function signature, you know exactly whether calling it will consume your value, read it, or modify it. No surprises.

## What you learned

- **Immutable references** (`&T`) let you read data without taking ownership; you can have many at once.
- **Mutable references** (`&mut T`) let you modify borrowed data, but only one can exist at a time, with no concurrent immutable references.
- The borrow checker enforces these rules at compile time, making **data races structurally impossible** with zero runtime cost.
