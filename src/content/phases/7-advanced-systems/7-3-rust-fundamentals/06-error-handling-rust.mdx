## Why this matters

Error handling is where most programs break down. In JavaScript and Python, any function can throw an exception at any time, and nothing in the function signature tells you about it. You either wrap everything in try-catch or hope for the best. In Java, checked exceptions tried to solve this but are widely considered a failed experiment — developers just catch and ignore them. Rust takes a different approach: errors are values, returned from functions using `Result<T, E>`, and the type system ensures you handle them. This makes error handling explicit, composable, and impossible to ignore.

## Result as the standard error type

Every operation that can fail in Rust returns a `Result<T, E>`, where `T` is the success type and `E` is the error type:

```rust
use std::fs;

fn read_config() -> Result<String, std::io::Error> {
    fs::read_to_string("config.toml")
}

fn main() {
    match read_config() {
        Ok(contents) => println!("Config: {}", contents),
        Err(e) => println!("Failed to read config: {}", e),
    }
}
```

Because `Result` is a regular enum, not a special language feature, you must handle both cases. The compiler will not let you use the `Ok` value without first checking whether the operation succeeded.

## The ? operator

Writing `match` for every fallible operation would be verbose. The **`?` operator** is syntactic sugar for early return on error:

```rust
fn read_username() -> Result<String, std::io::Error> {
    let contents = fs::read_to_string("user.txt")?; // Returns Err early if this fails
    Ok(contents.trim().to_string())
}
```

The `?` does three things:

1. If the Result is `Ok`, unwrap the value and continue.
2. If the Result is `Err`, return that error from the current function immediately.
3. Convert the error type if needed (using the `From` trait).

This is dramatically cleaner than nested try-catch blocks. Error propagation becomes a single character.

```rust
// Chain multiple fallible operations elegantly
fn get_user_greeting() -> Result<String, std::io::Error> {
    let name = fs::read_to_string("name.txt")?.trim().to_string();
    let greeting = fs::read_to_string("greeting.txt")?.trim().to_string();
    Ok(format!("{}, {}!", greeting, name))
}
```

## Custom error types

For libraries and larger applications, you define custom error types that aggregate all possible errors:

```rust
#[derive(Debug)]
enum AppError {
    Io(std::io::Error),
    Parse(std::num::ParseIntError),
    NotFound(String),
}

impl std::fmt::Display for AppError {
    fn fmt(&self, f: &mut std::fmt::Formatter) -> std::fmt::Result {
        match self {
            AppError::Io(e) => write!(f, "IO error: {}", e),
            AppError::Parse(e) => write!(f, "Parse error: {}", e),
            AppError::NotFound(name) => write!(f, "Not found: {}", name),
        }
    }
}

// Implement From to enable ? to convert automatically
impl From<std::io::Error> for AppError {
    fn from(e: std::io::Error) -> Self {
        AppError::Io(e)
    }
}
```

## Panic vs recoverable errors

Rust distinguishes between two kinds of failure:

**Recoverable errors** use `Result`. File not found, invalid input, network timeout — these are expected and the caller should decide how to handle them.

**Unrecoverable errors** use `panic!`. Index out of bounds, violated invariants, bugs that should never happen in correct code — these crash the program immediately.

```rust
// Recoverable — the caller handles it
fn parse_port(s: &str) -> Result<u16, std::num::ParseIntError> {
    s.parse::<u16>()
}

// Unrecoverable — this is a bug, not a user error
fn get_element(v: &[i32], index: usize) -> i32 {
    if index >= v.len() {
        panic!("Index {} out of bounds for length {}", index, v.len());
    }
    v[index]
}
```

The guideline: use `Result` for anything a reasonable caller might want to recover from. Use `panic!` only for programming errors — situations that indicate a bug in the code, not a problem with the input.

## Why this is better than exceptions

In exception-based languages, every function can throw, and there is nothing in the type system that tells you what errors to expect. Rust's approach has concrete advantages:

1. **Errors are visible in the signature.** `fn read_file() -> Result<String, io::Error>` tells you exactly what can go wrong.
2. **Errors cannot be silently ignored.** The compiler warns if you discard a Result.
3. **Error propagation is explicit.** The `?` operator marks every point where an error can occur.
4. **No hidden control flow.** Exceptions unwind the stack invisibly; Result values flow through normal return paths.

## What you learned

- **Result\<T, E\>** is Rust's standard error type — errors are values you return, not exceptions you throw, making error handling explicit and type-checked.
- The **? operator** unwraps Ok values or returns Err early, enabling clean error propagation chains without nested match blocks.
- **panic!** is for unrecoverable bugs; **Result** is for expected failures — this distinction makes programs more robust and their error paths more predictable than exception-based languages.
