## Why this matters

Reading Rust syntax and understanding individual concepts is different from seeing them work together in a real program. In this lesson, you will walk through a complete Rust CLI tool — a line counter that reads a file, filters lines by a search term, and prints statistics. Every concept from this module appears: ownership, borrowing, lifetimes, structs, enums, error handling with Result and `?`, traits, and generics. You will then build the same logic in JavaScript, mapping each Rust concept to its JS analogy.

## The project: a line counter CLI

The tool reads a text file, counts total lines, counts lines matching a search pattern, and prints a summary. Here is the complete Rust implementation:

```rust
use std::env;
use std::fs;
use std::process;

// Struct to hold our results — ownership of the data is clear
struct LineStats {
    total: usize,
    matched: usize,
    pattern: String,     // LineStats owns this String
    filename: String,    // LineStats owns this String
}

// impl block adds methods — &self borrows immutably
impl LineStats {
    fn match_percentage(&self) -> f64 {
        if self.total == 0 {
            return 0.0;
        }
        (self.matched as f64 / self.total as f64) * 100.0
    }
}

// Display trait for human-readable output
impl std::fmt::Display for LineStats {
    fn fmt(&self, f: &mut std::fmt::Formatter) -> std::fmt::Result {
        write!(
            f,
            "{}: {} total lines, {} matching '{}' ({:.1}%)",
            self.filename,
            self.total,
            self.matched,
            self.pattern,
            self.match_percentage()
        )
    }
}

// Returns Result — errors propagate with ?
fn count_lines(filename: &str, pattern: &str) -> Result<LineStats, std::io::Error> {
    let contents = fs::read_to_string(filename)?; // ? returns Err early if file read fails

    let total = contents.lines().count(); // Iterator trait — .lines() yields &str
    let matched = contents
        .lines()                          // Iterator
        .filter(|line| line.contains(pattern)) // Closure borrows pattern
        .count();

    Ok(LineStats {
        total,
        matched,
        pattern: pattern.to_string(),    // Create owned String from borrowed &str
        filename: filename.to_string(),
    })
}

fn main() {
    let args: Vec<String> = env::args().collect();

    if args.len() != 3 {
        eprintln!("Usage: linecounter <filename> <pattern>");
        process::exit(1);
    }

    // args[1] and args[2] are borrowed as &str via .as_str() or auto-deref
    match count_lines(&args[1], &args[2]) {
        Ok(stats) => println!("{}", stats),  // Display trait
        Err(e) => {
            eprintln!("Error: {}", e);
            process::exit(1);
        }
    }
}
```

## Concept walkthrough

**Ownership**: `LineStats` owns its `pattern` and `filename` strings. When a `LineStats` value is dropped, those strings are freed automatically.

**Borrowing**: `count_lines` takes `&str` parameters — it borrows the strings rather than taking ownership. The caller keeps its data.

**Lifetimes**: The `contents.lines()` iterator yields `&str` references into `contents`. These references are valid as long as `contents` exists. The compiler ensures we do not return dangling references.

**Error handling**: `fs::read_to_string` returns `Result<String, io::Error>`. The `?` operator unwraps `Ok` or returns `Err` early. No try-catch, no unchecked exceptions.

**Traits**: We implement `Display` for `LineStats` so `println!("{}", stats)` works. We use the `Iterator` trait's `.lines()`, `.filter()`, and `.count()` methods.

**Pattern matching**: The `match` in `main` handles both `Ok` and `Err` — the compiler ensures we do not forget a case.

## What you learned

- A real Rust program combines **ownership** (structs own their data), **borrowing** (functions take `&str` to avoid moves), **error handling** (`Result` + `?`), **traits** (`Display`, `Iterator`), and **pattern matching** (`match`) into a cohesive, safe design.
- The `?` operator and `Result` chaining make error propagation clean and explicit — every failure point is visible in the code.
- Rust's CLI tools are fast, safe, and expressive — the same concepts that prevent memory bugs also produce clear, maintainable code.
