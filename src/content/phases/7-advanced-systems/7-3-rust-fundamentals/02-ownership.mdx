## Why this matters

Ownership is the concept that makes Rust unique. Every language has to decide how memory is managed — C says "you figure it out," Java says "the garbage collector handles it," and Rust says "the compiler enforces strict rules about who owns what." If you understand ownership, you understand why Rust programs cannot have use-after-free bugs, double-free bugs, or dangling pointers. You also understand why code that looks correct to a C or JavaScript developer sometimes will not compile in Rust.

## The three rules of ownership

Every value in Rust has exactly one **owner** — the variable that holds it. Rust's ownership system follows three rules:

1. **Each value has exactly one owner.** No shared ownership by default.
2. **When the owner goes out of scope, the value is dropped (freed).** No manual `free()` needed.
3. **When you assign a value to another variable, the original owner is invalidated.** This is called a **move**.

```rust
fn main() {
    let s1 = String::from("hello"); // s1 owns the String
    let s2 = s1;                     // Ownership MOVES to s2
    // println!("{}", s1);           // ERROR: s1 is no longer valid
    println!("{}", s2);              // Works — s2 is the owner now
}
```

This is fundamentally different from JavaScript, where `let s2 = s1` creates a second reference to the same object and both variables remain valid. In Rust, after a move, the original variable is dead. The compiler enforces this.

## Stack vs heap

Understanding ownership requires understanding where data lives. Rust uses two regions of memory:

**Stack**: Fixed-size data stored in a last-in-first-out structure. Allocation and deallocation are instant — just move the stack pointer. Integers, booleans, floats, and fixed-size types live here.

**Heap**: Dynamically-sized data. Allocating requires finding a free block, and freeing requires returning it. Strings, vectors, and any data whose size is not known at compile time live on the heap.

When you assign a stack value (like an integer), Rust **copies** it — both variables get their own independent copy. When you assign a heap value (like a String), Rust **moves** it — only one variable can own the heap data at a time.

```rust
fn main() {
    // Stack values: Copy (both x and y are valid)
    let x = 42;
    let y = x;
    println!("{} {}", x, y); // Works — integers implement Copy

    // Heap values: Move (only s2 is valid)
    let s1 = String::from("data");
    let s2 = s1;
    // println!("{}", s1); // ERROR — s1 was moved
    println!("{}", s2);    // Works
}
```

## The Copy trait

Types that are small and live entirely on the stack implement the **Copy trait**. When a Copy type is assigned to another variable, Rust copies the bits rather than moving ownership. All primitive types — integers, floats, booleans, characters — implement Copy. Strings, vectors, and other heap-allocated types do not.

This is not arbitrary. Copying a 64-bit integer is trivially cheap. Copying a 10-megabyte string would be expensive and surprising if it happened implicitly. By making heap types move-only by default, Rust forces you to be explicit about when you want an expensive copy — you must call `.clone()`.

```rust
let s1 = String::from("expensive to copy");
let s2 = s1.clone(); // Explicit deep copy — both s1 and s2 are valid
println!("{} {}", s1, s2); // Works because we cloned
```

## Why ownership prevents memory bugs

**Use-after-free**: Impossible. After a value is moved, the original variable cannot be used. The compiler rejects the code.

**Double-free**: Impossible. Only one owner exists at any time. When that owner goes out of scope, the value is freed exactly once.

**Memory leaks**: Largely prevented. When an owner goes out of scope, its destructor runs automatically. You cannot forget to free memory because you never explicitly free it — the compiler inserts the free for you.

This is not a runtime check. There is no overhead. The compiler statically proves your code is correct, and the resulting binary is as fast as hand-written C.

## Ownership and functions

Passing a value to a function moves it, just like assignment. The function becomes the new owner. After the call, the original variable is invalid — unless the function returns the value back.

```rust
fn take_ownership(s: String) {
    println!("{}", s);
} // s is dropped here — memory freed

fn main() {
    let greeting = String::from("hello");
    take_ownership(greeting);
    // println!("{}", greeting); // ERROR — greeting was moved into the function
}
```

This can feel restrictive. You will learn in the next lesson how **borrowing** lets you pass data to functions without giving up ownership.

## What you learned

- Rust's ownership model enforces three rules: each value has one owner, values are dropped when the owner goes out of scope, and assignment **moves** ownership (invalidating the original variable).
- Primitive types implement the **Copy trait** and are cheaply duplicated on the stack; heap types require explicit `.clone()` for deep copies.
- Ownership statically prevents use-after-free, double-free, and memory leaks at compile time with zero runtime cost.
