## Why this matters

Every year, Microsoft reports that roughly 70% of its security vulnerabilities stem from memory safety bugs — use-after-free, buffer overflows, null pointer dereferences. Google reports similar numbers for Chromium. These are not obscure edge cases; they are the dominant source of critical exploits in production software written in C and C++. Rust was designed specifically to eliminate this entire class of bugs at compile time, without paying the runtime cost of garbage collection. Understanding why Rust exists and what tradeoffs it makes will help you reason about language design, memory models, and the performance-vs-safety spectrum that every systems programmer must navigate.

## The problem Rust solves

Languages like C and C++ give you direct control over memory. You allocate it, you use it, you free it. This is fast — no garbage collector pausing your program — but it is dangerous. Forget to free memory and you leak it. Free it twice and you corrupt the heap. Use a pointer after freeing the memory it pointed to and you get **undefined behavior** — the program might crash, silently corrupt data, or open a security hole.

```rust
// In C, this compiles and runs — but it is a use-after-free bug
// char *name = malloc(64);
// free(name);
// printf("%s", name);  // Undefined behavior — name points to freed memory

// Rust prevents this at compile time:
fn main() {
    let name = String::from("DURA");
    drop(name); // Explicitly free the memory
    // println!("{}", name); // ERROR: value used after move
}
```

Languages like Java, Python, and JavaScript solve this with a **garbage collector** (GC). The runtime tracks which objects are still reachable and frees the rest automatically. This is safe, but it costs you: GC pauses are unpredictable, memory usage is higher, and you cannot control exactly when memory is freed. For a web app, this is fine. For an operating system kernel, a game engine, or a real-time audio processor, GC pauses are unacceptable.

## Rust's answer: ownership

Rust takes a third path. Instead of manual memory management (C) or garbage collection (Java), Rust uses a system of **ownership** with rules checked at compile time. The compiler tracks who owns each piece of data, when it is borrowed, and when it should be freed. If your code violates the rules, it does not compile. There is no runtime overhead — the checks happen entirely before the program runs.

This means Rust gives you the performance of C with the safety of a garbage-collected language. The Rust community calls this **zero-cost abstractions**: the safety mechanisms add zero runtime cost. The compiler does the work so the CPU does not have to.

## Where Rust is used

Rust is not a niche language. It is used in production at scale:

- **Systems programming**: The Linux kernel accepts Rust code alongside C. Android uses Rust for security-critical components. Windows is rewriting core libraries in Rust.
- **CLI tools**: ripgrep (faster than grep), fd, bat, exa — many beloved developer tools are written in Rust.
- **WebAssembly**: Rust compiles to WebAssembly with first-class support, powering high-performance browser applications like Figma's rendering engine.
- **Embedded**: Rust runs on microcontrollers with no operating system, no allocator, and no standard library.
- **Infrastructure**: Cloudflare, Discord, and Dropbox use Rust in performance-critical backend services.

## The tradeoff

Rust is not free of cost — the cost is simply shifted. Instead of paying at runtime (GC pauses, crashes), you pay at compile time. The Rust compiler is famously strict. Code that would compile instantly in C or JavaScript may require you to restructure your logic to satisfy the borrow checker. The learning curve is steep. Compile times are longer. Simple tasks sometimes require more lines of code.

But the payoff is real: if your Rust code compiles, entire categories of bugs — data races, use-after-free, null pointer dereferences, buffer overflows — are impossible. Not unlikely. Impossible.

// Build a MemoryChecker that tracks allocated and freed "addresses."
// If code tries to use freed memory, throw an error — just like Rust's compiler would.

constructor() {
this.allocated = new Set();

allocate(address) {
this.allocated.add(address);
console.log("Allocated:", address);

}

free(address) {
if (!this.allocated.has(address)) {
throw new Error("Cannot free " + address + " — not allocated");
}
if (this.freed.has(address)) {
throw new Error("Double free detected on " + address);
}
this.freed.add(address);
this.allocated.delete(address);
console.log("Freed:", address);
}

use(address) {
// TODO: throw an error if the address has been freed (use-after-free)
// TODO: throw an error if the address was never allocated

}
}

// Test it
const mem = new MemoryChecker();
const ptr = mem.allocate("0xA1");
mem.use(ptr); // Should work
mem.free(ptr);

mem.use(ptr); // Should throw: use-after-free

console.log("Caught:", e.message);
}

## What you learned

- Rust eliminates memory safety bugs (use-after-free, double-free, data races) at **compile time** through its ownership system, with no garbage collector overhead.
- **Zero-cost abstractions** mean safety checks happen during compilation, not at runtime — giving you C-level performance with high-level safety guarantees.
- Rust is used in production for systems programming, CLI tools, WebAssembly, embedded devices, and performance-critical infrastructure.
