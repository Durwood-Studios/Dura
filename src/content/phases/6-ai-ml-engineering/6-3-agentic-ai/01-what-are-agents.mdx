## Why this matters

A chatbot answers questions. An agent **does things**. The difference is not intelligence — it is architecture. When you give an LLM the ability to call tools, read results, and decide what to do next, you get a system that can book flights, query databases, write and run code, or orchestrate entire workflows. Understanding the agent pattern is the first step to building systems that act on the world instead of just describing it.

## The agent pattern

An agent is three things bolted together:

1. **An LLM** that reasons about what to do next
2. **Tools** it can call (search, database, calculator, API, file system)
3. **A loop** that keeps going until the task is complete

That is the entire idea. Everything else — frameworks, memory systems, multi-agent orchestration — is optimization on top of this core loop.

```javascript
// The simplest agent loop you can write
async function agentLoop(task, tools, llm) {
  const messages = [{ role: "user", content: task }];

  while (true) {
    const response = await llm.chat(messages, { tools });

    // If the model is done, return its answer
    if (response.stopReason === "end") {
      return response.content;
    }

    // Otherwise it wants to call a tool
    for (const toolCall of response.toolCalls) {
      const result = await tools[toolCall.name](toolCall.arguments);
      messages.push({ role: "tool", content: result, toolCallId: toolCall.id });
    }

    // Push the assistant response so the model sees its own reasoning
    messages.push({ role: "assistant", content: response.content });
  }
}
```

That `while (true)` is the heartbeat. The model reasons, acts, observes, and repeats. This is the **ReAct** pattern — Reason + Act — formalized by Yao et al. in 2022 and now the default behavior in every major LLM API.

## Agent vs chatbot

A chatbot is a single turn: user speaks, model responds. Even multi-turn chat is just a growing list of messages with no side effects. A chatbot cannot check the weather — it can only tell you what it knows about weather.

An agent closes the loop between language and action:

```javascript
// Chatbot: can only describe
// "The weather in Tokyo is typically warm in April..."

// Agent: can actually check
const weatherTool = {
  name: "get_weather",
  description: "Get current weather for a city",
  parameters: {
    type: "object",
    properties: {
      city: { type: "string", description: "City name" },
    },
    required: ["city"],
  },
};

// The model decides to call get_weather("Tokyo"),
// reads the result, and gives a grounded answer.
```

The key insight: the model does not "know" how to call APIs. You give it a description of what tools exist, and it outputs structured data saying which tool to call with which arguments. Your code executes the actual call. The model never touches the network directly.

## The ReAct cycle

Every iteration of the agent loop follows three steps:

1. **Thought** — the model reasons about the current state and what to do next
2. **Action** — the model emits a tool call
3. **Observation** — your code runs the tool and feeds the result back

```javascript
// A single ReAct iteration looks like this in the message history:
const messages = [
  { role: "user", content: "What is the population of France?" },
  {
    role: "assistant",
    content: "I need to look this up.",
    toolCalls: [{ name: "search", arguments: { query: "population of France 2026" } }],
  },
  {
    role: "tool",
    toolCallId: "call_1",
    content: "France population 2026: approximately 68.4 million",
  },
  {
    role: "assistant",
    content: "The population of France is approximately 68.4 million as of 2026.",
  },
];
```

The model sees the full history each time, so it can course-correct. If a tool call fails, it can try a different approach. If it gets partial data, it can make another call. This self-correcting behavior is what makes agents genuinely useful — they recover from errors the way a human would.

## When to use an agent

Not every problem needs an agent. Use the simplest approach that works:

- **Single LLM call** — the task is self-contained and needs no external data
- **RAG pipeline** — the task needs external data but follows a fixed retrieval pattern
- **Agent** — the task requires multiple steps, dynamic tool selection, or conditional logic that cannot be predetermined

Agents shine when you cannot write the workflow as a flowchart ahead of time. If the next step depends on what the model finds, you need the loop.

## What you learned

- An **agent** is an LLM + tools + a loop — it reasons, acts, observes, and repeats until the task is done.
- The **ReAct** pattern (Reason + Act) gives the model a structured cycle of thought, action, and observation.
- Agents are the right choice when the workflow cannot be predetermined — when the next step depends on what the model discovers.
