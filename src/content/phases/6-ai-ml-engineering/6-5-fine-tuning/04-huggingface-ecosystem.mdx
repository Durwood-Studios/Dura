## Why this matters

HuggingFace is to modern ML what npm is to JavaScript — the central registry where models, datasets, and tools live. When you fine-tune an open-source model, you will almost certainly use HuggingFace libraries. Understanding this ecosystem means you can navigate the thousands of available models, load them correctly, configure training, and share results. Every job posting for ML engineering assumes familiarity with this stack.

## The four core libraries

The HuggingFace fine-tuning stack has four main pieces that work together. Each handles a distinct responsibility.

**transformers** — The foundation. Loads pretrained models and tokenizers. Provides the `Trainer` class that handles the training loop. Supports thousands of model architectures.

**peft** — Parameter-Efficient Fine-Tuning. Implements LoRA, QLoRA, and other adapter methods. Wraps around a transformers model to inject trainable adapters while keeping the base frozen.

**trl** — Transformer Reinforcement Learning. Built on top of transformers and peft. Provides `SFTTrainer` (Supervised Fine-Tuning Trainer) which simplifies the common case of fine-tuning on chat data.

**datasets** — Data loading and processing. Handles JSONL, CSV, Parquet, and streaming from the Hub. Efficient memory-mapped storage so you can work with datasets larger than RAM.

## The Model Hub

The HuggingFace Hub hosts over 500,000 models. For fine-tuning, you typically start with a base model — a pretrained LLM that you then adapt to your task. Choosing the right base model is critical.

```javascript
// How to evaluate base models for fine-tuning
function evaluateBaseModel(model) {
  const criteria = {
    size: {
      value: model.params,
      assessment:
        model.params <= 3e9
          ? "Small — fast training, limited capacity"
          : model.params <= 13e9
            ? "Medium — good balance for most tasks"
            : "Large — high capacity but expensive to train",
    },
    license: {
      value: model.license,
      assessment: ["apache-2.0", "mit"].includes(model.license)
        ? "Permissive — commercial use OK"
        : model.license === "llama3"
          ? "Meta license — commercial with restrictions"
          : "Check terms carefully",
    },
    architecture: {
      value: model.arch,
      assessment:
        model.arch === "llama"
          ? "Well-supported, wide tooling compatibility"
          : model.arch === "mistral"
            ? "Efficient, good for fine-tuning"
            : "Verify peft/trl compatibility",
    },
    quantAvailable: {
      value: model.hasQuants,
      assessment: model.hasQuants
        ? "QLoRA-ready with GGUF/AWQ variants"
        : "May need to quantize yourself",
    },
  };

  console.log(`Evaluating: ${model.name}`);
  Object.entries(criteria).forEach(([key, data]) => {
    console.log(`  ${key}: ${data.assessment}`);
  });
  console.log("");
}

const models = [

    arch: "llama",
```

## The standard training configuration

While the actual training runs in Python, understanding the configuration translates directly. Every parameter maps to a concept you can reason about in any language.

```javascript
// Training configuration — these are the decisions you make before training starts
const trainingConfig = {
  // Model selection
  model: "meta-llama/Llama-3.1-8B",

  // LoRA configuration
  lora: {
    rank: 16, // Adapter capacity — higher = more expressive
    alpha: 32, // Scaling factor — typically 2x rank
    targetModules: [
      // Which layers get adapters
      "q_proj",
      "k_proj",
      "v_proj",
      "o_proj",
      "gate_proj",
      "up_proj",
      "down_proj",
    ],
    dropout: 0.05, // Regularization to prevent overfitting
  },

  // Training parameters
  training: {
    epochs: 3, // Full passes through the dataset
    batchSize: 4, // Examples processed simultaneously
    gradientAccumulationSteps: 4, // Simulate larger batch (effective batch = 16)
    learningRate: 2e-4, // Step size — too high = diverge, too low = slow
    weightDecay: 0.01, // Regularization on weights
    warmupRatio: 0.03, // Gradually increase LR at the start
    lrScheduler: "cosine", // Learning rate decreases following a cosine curve
    maxSeqLength: 2048, // Maximum tokens per example
  },

  // Quantization (for QLoRA)
  quantization: {
    loadIn4bit: true,
    bnb4bitComputeType: "float16",
    bnb4bitQuantType: "nf4", // NormalFloat4 — optimized for normally distributed weights
  },
};

// Calculate effective batch size
const effectiveBatch =
  trainingConfig.training.batchSize * trainingConfig.training.gradientAccumulationSteps;
console.log(`Effective batch size: ${effectiveBatch}`);

// Estimate training steps
const datasetSize = 500; // training examples
const stepsPerEpoch = Math.ceil(datasetSize / effectiveBatch);
const totalSteps = stepsPerEpoch * trainingConfig.training.epochs;
console.log(`Steps per epoch: ${stepsPerEpoch}`);
console.log(`Total training steps: ${totalSteps}`);
console.log(`Warmup steps: ${Math.ceil(totalSteps * trainingConfig.training.warmupRatio)}`);
```

## Loading and inspecting models

Before fine-tuning, you inspect the model to understand its architecture — how many layers, what the attention dimensions are, and which modules you can target with LoRA.

```javascript
// Simulating model inspection — what you see when you load a model
const modelArchitecture = {
  name: "Llama-3.1-8B",
  layers: 32,
  hiddenSize: 4096,
  intermediateSize: 14336,
  attentionHeads: 32,
  kvHeads: 8, // Grouped-query attention
  vocabSize: 128256,
  maxPosition: 131072,

  // Each layer contains these modules (LoRA targets)
  layerModules: {
    selfAttention: ["q_proj", "k_proj", "v_proj", "o_proj"],
    mlp: ["gate_proj", "up_proj", "down_proj"],
  },
};

// Calculate total parameters
const paramsPerAttentionLayer =
  modelArchitecture.hiddenSize * modelArchitecture.hiddenSize * 3 + // q, k, v (simplified)
  modelArchitecture.hiddenSize * modelArchitecture.hiddenSize; // o_proj

const paramsPerMLPLayer = modelArchitecture.hiddenSize * modelArchitecture.intermediateSize * 3; // gate, up, down

const totalParams = modelArchitecture.layers * (paramsPerAttentionLayer + paramsPerMLPLayer);

console.log(`Architecture: ${modelArchitecture.name}`);
console.log(`Layers: ${modelArchitecture.layers}`);
console.log(`Hidden dim: ${modelArchitecture.hiddenSize}`);
console.log(`Estimated parameters: ${(totalParams / 1e9).toFixed(1)}B`);
console.log(`\nLoRA-targetable modules per layer:`);
Object.entries(modelArchitecture.layerModules).forEach(([group, modules]) => {
  console.log(`  ${group}: ${modules.join(", ")}`);
});
```

## What you learned

- The HuggingFace stack has four core libraries: **transformers** (models), **peft** (LoRA adapters), **trl** (training recipes), and **datasets** (data loading).
- The **Model Hub** hosts hundreds of thousands of pretrained models — choosing the right base model means evaluating size, license, architecture, and quantization support.
- Training configuration involves **LoRA hyperparameters** (rank, alpha, target modules) and **training hyperparameters** (learning rate, batch size, epochs) that together determine quality and cost.
