## Why this matters

Fine-tuning a large language model costs real money — compute hours, data preparation time, ongoing maintenance of a custom model. Choosing the wrong approach wastes all of it. Most tasks that developers think require fine-tuning are actually solved faster and cheaper with prompt engineering or retrieval-augmented generation (RAG). Understanding when fine-tuning is genuinely the right tool saves you from the most expensive mistake in applied AI: over-engineering.

## The three-level decision tree

When you need an LLM to behave differently from its default, you have three levers. They form a natural progression from cheapest to most expensive.

**Level 1 — Prompt engineering.** You change what you send to the model. System prompts, few-shot examples, structured output instructions. This costs nothing extra and takes minutes to iterate on. Start here. Always.

**Level 2 — Retrieval-augmented generation (RAG).** You give the model access to external knowledge at query time. A vector database stores your documents; relevant chunks are injected into the prompt before the model responds. This is the right move when the model lacks domain knowledge — your company's internal docs, recent events, proprietary data.

**Level 3 — Fine-tuning.** You modify the model's weights by training on your own data. This changes how the model behaves at a fundamental level — its tone, output format, reasoning style, or domain-specific patterns.

```javascript
// Decision tree as code — given a problem, pick the right approach
function chooseApproach(problem) {
  const needs = analyzeProblem(problem);

  // Level 1: Can you solve it by changing the prompt?
  if (needs.type === "formatting" && needs.complexity === "low") {
    return {
      approach: "prompt-engineering",
      cost: "free",
      timeToShip: "hours",
      example: "Add a system prompt: 'Respond in JSON with keys: summary, confidence'",
    };
  }

  // Level 2: Does the model lack knowledge it needs?
  if (needs.type === "knowledge-gap") {
    return {
      approach: "rag",
      cost: "moderate",
      timeToShip: "days",
      example: "Index company docs in a vector DB, retrieve relevant chunks per query",
    };
  }

  // Level 3: Do you need to change the model's behavior pattern?
  if (needs.type === "style-change" || needs.type === "consistent-format") {
    return {
      approach: "fine-tuning",
      cost: "high",
      timeToShip: "weeks",
      example: "Train on 500+ examples of your desired input/output pattern",
    };
  }

  return { approach: "prompt-engineering", note: "Start simple, escalate if needed" };
}

function analyzeProblem(problem) {
  // In practice, this is your engineering judgment
  return {
    type: problem.category,
    complexity: problem.complexity,
    dataAvailable: problem.hasTrainingData,
  };
}

// Example problems
const problems = [
  { name: "JSON output", category: "formatting", complexity: "low", hasTrainingData: false },
  {
    name: "Answer from internal docs",
    category: "knowledge-gap",
    complexity: "medium",
    hasTrainingData: true,
  },
  {
    name: "Match brand voice consistently",
    category: "style-change",
    complexity: "high",
    hasTrainingData: true,
  },
];

problems.forEach((p) => {
  const result = chooseApproach(p);
  console.log(`${p.name} → ${result.approach} (${result.timeToShip})`);
});
```

## Fine-tuning is for style, not knowledge

This is the single most misunderstood concept in applied AI. Fine-tuning does not reliably inject new factual knowledge into a model. If you train a model on your company's product catalog, it will not memorize every SKU — it will hallucinate details it did not see enough times in training.

What fine-tuning excels at is changing **how** the model responds: its tone, structure, reasoning approach, and formatting patterns. Real examples where fine-tuning is the right call:

- Converting a general model into one that always responds in a specific medical report format
- Teaching a model to write code comments in your team's exact style
- Making a model that classifies support tickets into your company's custom categories
- Training a model to follow a multi-step reasoning chain specific to your domain

## The cost equation

Before fine-tuning, calculate whether the per-token cost savings justify the upfront investment. Fine-tuned smaller models can replace expensive API calls to larger models — a fine-tuned 7B parameter model might match GPT-4 quality on your specific narrow task at a fraction of the inference cost.

```javascript
// Rough cost comparison for 100,000 monthly requests
function compareCosts() {
  const approaches = {
    promptEngineering: {
      setupCost: 0,
      perRequestCost: 0.03, // Large model API, long prompts with examples
      monthlyInference: 0.03 * 100000,
      maintenance: "low",
    },
    rag: {
      setupCost: 500, // Vector DB setup, embedding pipeline
      perRequestCost: 0.02, // Embedding lookup + smaller prompt
      monthlyInference: 0.02 * 100000,
      maintenance: "medium", // Keep index updated
    },
    fineTuning: {
      setupCost: 2000, // Data prep, training compute, evaluation
      perRequestCost: 0.005, // Smaller fine-tuned model, no few-shot needed
      monthlyInference: 0.005 * 100000,
      maintenance: "high", // Retrain as requirements change
    },
  };

  Object.entries(approaches).forEach(([name, data]) => {
    const sixMonthTotal = data.setupCost + data.monthlyInference * 6;
    console.log(
      `${name}: $${sixMonthTotal.toFixed(0)} over 6 months (${data.maintenance} maintenance)`
    );
  });
}

compareCosts();
```

The break-even point matters. Fine-tuning only wins economically at high volume on narrow, well-defined tasks where a smaller model can replace a larger one.

## Signs you are NOT ready to fine-tune

If any of these are true, go back to prompt engineering or RAG: you have fewer than 200 high-quality training examples; the task definition changes frequently; you have not measured baseline performance with a good system prompt; you cannot articulate exactly what output format or style you want; you do not have an evaluation pipeline to measure whether the fine-tuned model is actually better.

## What you learned

- **Prompt engineering, RAG, and fine-tuning** form a cost-escalating hierarchy — always start with the cheapest approach and escalate only when it fails.
- Fine-tuning changes **how** a model responds (style, format, reasoning patterns), not **what** it knows — use RAG for knowledge injection.
- The economics of fine-tuning only work at **high volume on narrow, well-defined tasks** where a smaller trained model can replace a larger general one.
