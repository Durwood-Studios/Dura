## Why this matters

Tools let models take actions. Resources let models read data. When a model needs to understand your project structure, read a configuration file, or examine database records, resources are the mechanism. They work like read-only endpoints — the model requests a URI and gets content back. Resources keep your MCP server organized by separating "things the model reads" from "things the model does."

## Resource URIs

Every MCP resource is identified by a URI. The scheme is flexible — you can use standard schemes like `file://` and `https://`, or define custom schemes that make sense for your domain.

```javascript
// Examples of resource URIs
const resources = [
  "file:///project/README.md", // a local file
  "file:///project/src/config.ts", // source code
  "db://users/schema", // database schema (custom scheme)
  "db://users/recent?limit=10", // recent records
  "https://api.example.com/status", // external API
  "config://app/settings", // application config (custom scheme)
];
```

The URI is what the client sends when it wants to read a resource. Your server maps URIs to handler functions that return the content.

## Registering static resources

The simplest resources return fixed content. Here is how you register a resource with the MCP SDK:

```javascript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";

const server = new McpServer({
  name: "project-server",
  version: "1.0.0",
});

// A static resource that returns the project's README
server.resource(
  "project-readme",
  "file:///project/README.md",
  "The project README with setup instructions and architecture overview",
  async () => {
    const content = await fs.readFile("/project/README.md", "utf-8");
    return {
      contents: [
        {
          uri: "file:///project/README.md",
          mimeType: "text/markdown",
          text: content,
        },
      ],
    };
  }
);
```

The handler returns a `contents` array. Each entry has a `uri`, a `mimeType`, and either `text` (for text content) or `blob` (base64-encoded binary data).

## Resource templates with variables

Often you need a family of related resources — every user profile, every database table, every file in a directory. Resource templates use URI patterns with variables:

```javascript
// A template that resolves any user profile by ID
server.resource(
  "user-profile",
  "db://users/{userId}",
  "User profile data by user ID",
  async (uri, { userId }) => {
    const user = await db.get("SELECT * FROM users WHERE id = ?", [userId]);

    if (!user) {
      return {
        contents: [
          {
            uri: uri.href,
            mimeType: "application/json",
            text: JSON.stringify({ error: "User not found" }),
          },
        ],
      };
    }

    return {
      contents: [
        {
          uri: uri.href,
          mimeType: "application/json",
          text: JSON.stringify(user, null, 2),
        },
      ],
    };
  }
);

// A template for any table's schema
server.resource(
  "table-schema",
  "db://schema/{tableName}",
  "Database table schema including columns and types",
  async (uri, { tableName }) => {
    const columns = await db.all(`PRAGMA table_info(${tableName})`);
    return {
      contents: [
        {
          uri: uri.href,
          mimeType: "application/json",
          text: JSON.stringify(columns, null, 2),
        },
      ],
    };
  }
);
```

When the client requests `db://users/42`, the SDK matches the template, extracts `userId = "42"`, and calls your handler.

## Dynamic resource content

Resources do not have to be static files. They can compute content on the fly — aggregating data, formatting reports, or reflecting system state:

```javascript
server.resource(
  "system-status",
  "status://system/health",
  "Current system health including uptime, memory, and active connections",
  async () => {
    const uptime = process.uptime();
    const memory = process.memoryUsage();
    const activeConnections = await db.get(
      "SELECT COUNT(*) as count FROM connections WHERE active = true"
    );

    const report = {
      uptime: `${Math.floor(uptime / 3600)}h ${Math.floor((uptime % 3600) / 60)}m`,
      memoryMB: Math.round(memory.heapUsed / 1024 / 1024),
      activeConnections: activeConnections.count,
      timestamp: new Date().toISOString(),
    };

    return {
      contents: [
        {
          uri: "status://system/health",
          mimeType: "application/json",
          text: JSON.stringify(report, null, 2),
        },
      ],
    };
  }
);
```

## Listing resources

When a client calls `resources/list`, the server returns all available resources with their URIs and descriptions. Templates appear with their variable patterns so the client knows what URIs are valid.

```javascript
// Response to resources/list
{
  "resources": [
    {
      "uri": "file:///project/README.md",
      "name": "project-readme",
      "description": "The project README",
      "mimeType": "text/markdown"
    }
  ],
  "resourceTemplates": [
    {
      "uriTemplate": "db://users/{userId}",
      "name": "user-profile",
      "description": "User profile data by user ID",
      "mimeType": "application/json"
    }
  ]
}
```

The model uses this listing to know what data it can access. Good names and descriptions help the model pick the right resource, just like with tools.

## Resources vs. tools

Use resources when the model needs to **read** data without side effects. Use tools when the model needs to **act** — create, update, delete, or compute something. If the operation is idempotent and read-only, make it a resource. If it changes state or takes parameters that are not part of a URI pattern, make it a tool.

```javascript
// Resource: reading data (no side effects)
"db://users/{userId}"; // read a user profile
"file:///project/package.json"; // read a config file

// Tool: taking action (side effects)
"create_user"; // creates a database record
"send_email"; // sends an email
"run_query"; // executes arbitrary SQL
```

## What you learned

- MCP resources expose **read-only data** via URIs with schemes like `file://`, `db://`, or custom schemes you define.
- **Resource templates** use `{variable}` patterns in URIs to serve families of related resources from a single handler.
- Resources return **contents** arrays with `uri`, `mimeType`, and `text` (or `blob` for binary) — keeping data reads separate from action-taking tools.
