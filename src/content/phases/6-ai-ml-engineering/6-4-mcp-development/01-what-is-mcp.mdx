## Why this matters

Large language models are powerful reasoners, but they are trapped in a box. They cannot read your database, check the weather, or search your codebase — unless you build the plumbing to connect them. Before MCP, every tool integration was bespoke: different protocols, different schemas, different auth patterns. Each AI client reinvented the wheel.

The **Model Context Protocol (MCP)** is an open standard created by Anthropic that solves this. It defines one protocol for connecting AI models to external tools and data, the same way HTTP standardized how browsers talk to servers. If you build an MCP server once, every MCP-compatible client — Claude Desktop, VS Code extensions, CLI agents, custom apps — can use it immediately.

## The problem MCP solves

Imagine you have five AI tools and ten data sources. Without a standard, you need 50 custom integrations. With MCP, you build 10 servers and 5 clients — 15 components total. That is the power of a shared protocol.

```javascript
// Without MCP: every client has custom tool logic
const claudeTool = { type: "claude_tool", run: fetchWeather };
const gptTool = { type: "function", execute: fetchWeather };
const customTool = { action: "weather", handler: fetchWeather };

// With MCP: one server definition works everywhere
const mcpTool = {
  name: "get_weather",
  description: "Get current weather for a city",
  inputSchema: {
    type: "object",
    properties: {
      city: { type: "string", description: "City name" },
    },
    required: ["city"],
  },
};
```

## The spec: JSON-RPC 2.0

MCP is built on **JSON-RPC 2.0**, a lightweight remote procedure call protocol. Every message is a JSON object with a standard shape. There are three types: requests, responses, and notifications.

```javascript
// A JSON-RPC request from client to server
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": { "city": "Portland" }
  }
}

// The server's response
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "Portland: 58°F, partly cloudy"
      }
    ]
  }
}
```

The `id` field ties requests to responses. The `method` field names the operation. The `params` field carries the payload. Every MCP interaction follows this shape.

## What MCP servers expose

An MCP server can expose three types of capabilities:

**Tools** are functions the model can call. "Search the database," "create a file," "send a notification." The model decides when to call them based on the user's request.

**Resources** are data the model can read. Think of them like GET endpoints — files, database records, API responses. They use URIs like `file:///path/to/doc` or `https://api.example.com/users`.

**Prompts** are reusable prompt templates the server makes available. They accept arguments and generate structured messages that guide the model's behavior.

```javascript
// A server might expose all three
const server = {
  tools: ["search_docs", "create_ticket", "run_query"],
  resources: ["file:///project/README.md", "db://users/schema"],
  prompts: ["summarize_code", "review_pr", "explain_error"],
};
```

## How clients connect

MCP clients — Claude Desktop, Claude Code, IDE extensions, custom applications — discover servers through configuration. They connect, negotiate capabilities, and then relay tool calls between the user (or model) and the server.

```javascript
// Typical MCP client configuration (e.g., claude_desktop_config.json)
{
  "mcpServers": {
    "weather": {
      "command": "node",
      "args": ["./weather-server.js"]
    },
    "database": {
      "command": "npx",
      "args": ["-y", "@myorg/db-mcp-server"],
      "env": { "DB_URL": "postgresql://localhost/mydb" }
    }
  }
}
```

When Claude Desktop launches, it starts each configured server as a subprocess, connects via standard I/O, and asks what tools, resources, and prompts are available. From that point on, the model can use them as naturally as calling a function.

## The MCP ecosystem in 2026

MCP has grown from Anthropic's internal tooling into a broad ecosystem. There are hundreds of community-built servers on registries like npm and GitHub. Servers exist for databases (PostgreSQL, SQLite), developer tools (Git, GitHub, Linear), file systems, web scraping, and more. The protocol is open-source and transport-agnostic — it works over stdio, HTTP, and server-sent events.

The key insight: MCP turns "how do I connect my AI to X?" from a custom engineering project into a configuration problem. Build the server once, use it everywhere.

## What you learned

- **MCP** is an open protocol that standardizes how AI models connect to external tools and data, eliminating bespoke integrations.
- The protocol uses **JSON-RPC 2.0** messages with a consistent structure: `jsonrpc`, `id`, `method`, and `params`.
- MCP servers expose **tools** (callable functions), **resources** (readable data), and **prompts** (reusable templates) that any compatible client can use.
