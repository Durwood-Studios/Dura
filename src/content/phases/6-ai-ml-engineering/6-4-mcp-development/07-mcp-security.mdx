## Why this matters

An MCP server is an API that an AI model calls autonomously. The model decides when to invoke tools and what arguments to pass — sometimes based on untrusted user input. If your server blindly executes whatever it receives, you have built an injection vector with a natural language interface. Security is not optional. Every MCP server that touches real data or systems needs input validation, authentication, rate limiting, and audit logging from day one.

## Input validation

The inputSchema in your tool definition provides structural validation — the SDK rejects requests that do not match the JSON Schema. But structural validation is not enough. You must validate the semantics of inputs in your handler.

```javascript
server.tool(
  "query_database",
  "Execute a read-only SQL query",
  {
    sql: { type: "string", description: "A SELECT query" },
  },
  async ({ sql }) => {
    // Structural validation: SDK ensures sql is a string
    // Semantic validation: you ensure it is safe
    const normalized = sql.trim().toUpperCase();

    // Block anything that is not a SELECT
    const forbidden = ["INSERT", "UPDATE", "DELETE", "DROP", "ALTER", "CREATE", "TRUNCATE"];
    for (const keyword of forbidden) {
      if (normalized.includes(keyword)) {
        return {
          content: [{ type: "text", text: `Blocked: ${keyword} statements are not allowed.` }],
          isError: true,
        };
      }
    }

    // Block multiple statements (SQL injection via semicolons)
    if (sql.includes(";")) {
      return {
        content: [{ type: "text", text: "Blocked: multiple statements not allowed." }],
        isError: true,
      };
    }

    try {
      const rows = await db.all(sql);
      return { content: [{ type: "text", text: JSON.stringify(rows, null, 2) }] };
    } catch (error) {
      return {
        content: [{ type: "text", text: `Query failed: ${error.message}` }],
        isError: true,
      };
    }
  }
);
```

For file system tools, validate paths to prevent directory traversal. For URL-based tools, validate against an allowlist. Never trust that the model will only send reasonable inputs — it might be following a prompt injection.

```javascript
// Path validation to prevent directory traversal
function isPathSafe(requestedPath, allowedRoot) {
  const resolved = path.resolve(allowedRoot, requestedPath);
  return resolved.startsWith(path.resolve(allowedRoot));
}

// URL validation against an allowlist
function isUrlAllowed(url, allowedDomains) {
  try {
    const parsed = new URL(url);
    return allowedDomains.includes(parsed.hostname);
  } catch {
    return false;
  }
}
```

## Authentication

For remote MCP servers (SSE and HTTP transports), you need to verify that the connecting client is authorized. The most common patterns are bearer tokens and OAuth.

```javascript
import express from "express";

const app = express();

// Middleware: validate bearer token on every request
function authenticate(req, res, next) {
  const authHeader = req.headers.authorization;

  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    return res.status(401).json({ error: "Missing or invalid authorization header" });
  }

  const token = authHeader.slice(7);

  // Validate against your auth system
  if (!isValidToken(token)) {
    return res.status(403).json({ error: "Invalid or expired token" });
  }

  req.userId = getUserIdFromToken(token);
  next();
}

// Apply to MCP endpoints
app.post("/mcp", authenticate, async (req, res) => {
  // Only authenticated requests reach the MCP server
  await handleMcpRequest(req, res);
});
```

For stdio servers, authentication is implicit — the server runs as the user's own process with their permissions. But you still need to validate inputs, because the model might forward untrusted content from the conversation.

## Rate limiting

MCP tools that call external APIs or databases should be rate limited. A model in a loop can fire hundreds of tool calls per minute if something goes wrong.

```javascript
// Simple in-memory rate limiter
class RateLimiter {
  constructor(maxRequests, windowMs) {
    this.maxRequests = maxRequests;
    this.windowMs = windowMs;
    this.requests = new Map();
  }

  isAllowed(key) {
    const now = Date.now();
    const windowStart = now - this.windowMs;
    const timestamps = (this.requests.get(key) || []).filter((t) => t > windowStart);

    if (timestamps.length >= this.maxRequests) {
      return false;
    }

    timestamps.push(now);
    this.requests.set(key, timestamps);
    return true;
  }
}

const limiter = new RateLimiter(30, 60000); // 30 requests per minute

server.tool(
  "search_api",
  "Search external API",
  {
    query: { type: "string" },
  },
  async ({ query }) => {
    if (!limiter.isAllowed("search_api")) {
      return {
        content: [{ type: "text", text: "Rate limit exceeded. Try again in a moment." }],
        isError: true,
      };
    }

    const results = await externalApi.search(query);
    return { content: [{ type: "text", text: JSON.stringify(results) }] };
  }
);
```

## Audit logging

Every tool invocation should be logged. When something goes wrong — or when you need to understand what the model did and why — the audit log is your forensic trail.

```javascript
function auditLog(toolName, args, result, metadata) {
  const entry = {
    timestamp: new Date().toISOString(),
    tool: toolName,
    arguments: args,
    success: !result.isError,
    userId: metadata.userId || "anonymous",
    duration: metadata.durationMs,
  };

  // Write to structured log (stdout as JSON for production log pipelines)
  console.log(JSON.stringify(entry));
}

// Wrap tool handlers with audit logging
function withAuditLog(toolName, handler) {
  return async (args, metadata) => {
    const start = Date.now();
    try {
      const result = await handler(args);
      auditLog(toolName, args, result, {
        ...metadata,
        durationMs: Date.now() - start,
      });
      return result;
    } catch (error) {
      auditLog(
        toolName,
        args,
        { isError: true },
        {
          ...metadata,
          durationMs: Date.now() - start,
          error: error.message,
        }
      );
      throw error;
    }
  };
}
```

## The principle of least privilege

Give each MCP server the minimum permissions it needs. A server that reads files should not have write access. A server that queries a database should use a read-only database user. A server that calls one API should not have credentials for others.

```javascript
// Bad: one server with god-mode access
const godServer = {
  tools: ["read_file", "write_file", "delete_file", "query_db", "modify_db", "send_email"],
  dbUser: "admin",
  fileAccess: "/",
};

// Good: separate servers with minimal permissions
const fileReader = {
  tools: ["read_file", "list_files"],
  fileAccess: "/project/docs", // scoped to one directory
  readOnly: true,
};

const dbQuery = {
  tools: ["query_database"],
  dbUser: "readonly_user", // read-only database role
  allowedTables: ["users", "logs"], // scoped to specific tables
};
```

Separate concerns into separate servers. If a file-reading server is compromised, it cannot touch the database. If the database server has a bug, it cannot delete files. Defense in depth starts at architecture.

## What you learned

- **Input validation** must go beyond JSON Schema — validate semantics (safe SQL, allowed paths, permitted URLs) because models may forward untrusted content.
- **Authentication** and **rate limiting** protect remote MCP servers from unauthorized access and runaway model loops.
- The **principle of least privilege** means separating concerns into scoped servers with minimal permissions — defense in depth at the architecture level.
