## Why this matters

LLMs know what was in their training data. They do not know your company's internal docs, yesterday's support tickets, or the product spec you wrote this morning. RAG solves this: instead of hoping the model knows the answer, you **retrieve** the relevant documents and **stuff** them into the prompt alongside the question. The model generates a response **grounded in real context**.

## The core idea

RAG has three steps:

1. **Retrieve** — search a knowledge base and find the most relevant documents for the user's question.
2. **Augment** — insert those documents into the prompt as context.
3. **Generate** — the LLM produces an answer using the provided context.

```text
User: "What is our refund policy?"

→ Retrieve: search docs → find refund-policy.md
→ Augment: "Given this context: [refund policy text], answer: What is our refund policy?"
→ Generate: "Our refund policy allows returns within 30 days..."
```

The model did not "know" the refund policy. It read it from the context you provided. This is why RAG dramatically reduces hallucination — the model has source material to work from.

## Why RAG beats fine-tuning for most use cases

|                | RAG                          | Fine-tuning                     |
| -------------- | ---------------------------- | ------------------------------- |
| Data freshness | Real-time (re-index anytime) | Frozen at training time         |
| Setup cost     | Hours                        | Days to weeks                   |
| Compute cost   | Retrieval + inference        | GPU training + inference        |
| Best for       | Factual Q&A, docs, support   | Style, tone, specialized skills |

RAG is the default first choice. Fine-tuning is a last resort for cases where the model needs to change **how** it responds, not just **what** it knows.

## The RAG pipeline architecture

```text
Documents → Chunk → Embed → Store in vector DB
                                    ↓
User query → Embed query → Search vector DB → Top-K results
                                    ↓
              Stuff results into prompt → LLM → Response
```

Each step has decisions: how to chunk, which embedding model, which vector DB, how many results to retrieve, how to format the prompt. The next 8 lessons cover each one.

## When RAG fails

RAG is not magic. It fails when:

- The retrieval step returns irrelevant documents (garbage in, garbage out).
- The chunks are too small (missing context) or too large (diluting relevance).
- The user's question does not match any document's phrasing.
- The model ignores the context and hallucinates anyway (rare but possible).

Good RAG engineering is mostly about making retrieval better.

  solution={`function simpleRAG(question, chunks) {
const qWords = new Set(question.toLowerCase().split(/\\W+/));
let bestChunk = chunks[0];
let bestScore = 0;
for (const chunk of chunks) {
const words = chunk.toLowerCase().split(/\\W+/);
const score = words.filter((w) => qWords.has(w)).length;
if (score > bestScore) { bestScore = score; bestChunk = chunk; }
}
return \`Given this context:\\n"\${bestChunk}"\\n\\nAnswer: \${question}\`;
}

const chunks = [
"Our refund policy allows returns within 30 days of purchase.",
"Shipping takes 3-5 business days for domestic orders.",
"Contact support at help@example.com for urgent issues.",
];

## What you learned

- **RAG** = Retrieve relevant documents + Augment the prompt with them + Generate a grounded response.
- RAG beats fine-tuning for most factual tasks because it uses real-time data without retraining.
- RAG fails when retrieval is poor. Good RAG engineering is mostly about making retrieval better.
