## Why this matters

You cannot embed an entire 50-page document as a single vector. Embedding models have token limits (typically 512-8192 tokens), and even if you could embed the whole thing, a single vector cannot capture every topic in a long document. Chunking splits documents into smaller pieces so each chunk represents one coherent idea. Get this wrong, and your retrieval step returns fragments that are either too vague to be useful or too small to make sense.

## The chunking problem

Imagine searching for "How do I reset my password?" in a knowledge base. If a chunk contains the full password reset instructions, retrieval works perfectly. If the instructions got split across two chunks — one with the steps and another with the prerequisites — neither chunk alone answers the question. Chunking strategy directly controls retrieval quality.

### Fixed-size chunking

The simplest approach: split text every N characters (or tokens), with an optional overlap.

```javascript
function fixedSizeChunk(text, chunkSize = 500, overlap = 50) {
  const chunks = [];
  let start = 0;

  while (start < text.length) {
    const end = Math.min(start + chunkSize, text.length);
    chunks.push(text.slice(start, end));
    start += chunkSize - overlap;
  }

  return chunks;
}

const text = "A".repeat(1200);
const chunks = fixedSizeChunk(text, 500, 50);
console.log(chunks.length); // 3 chunks
console.log(chunks[0].length); // 500
```

**Pros:** Dead simple, predictable chunk sizes, easy to reason about.
**Cons:** Splits sentences mid-word, breaks paragraphs at arbitrary points. A chunk might start with "...tion process requires admin access" — useless without the beginning.

### Overlap

Overlap is the safety net. By repeating the last N characters of one chunk at the start of the next, you reduce the chance that a critical sentence gets split and lost. Typical overlap: 10-20% of chunk size.

```javascript
// Without overlap: chunk boundary splits a key sentence
// Chunk 1: "...To reset your password, first go to"
// Chunk 2: "the settings page and click Security."

// With 100-char overlap:
// Chunk 1: "...To reset your password, first go to"
// Chunk 2: "...password, first go to the settings page and click Security."
// Now both chunks contain the full instruction
```

### Recursive character splitting

The most popular strategy in production (used by LangChain and similar frameworks). Instead of splitting at a fixed size, it tries a hierarchy of separators in order of preference:

1. Split on `\n\n` (paragraph breaks) — best boundary
2. If chunks are still too large, split on `\n` (line breaks)
3. If still too large, split on `. ` (sentences)
4. Last resort: split on ` ` (words)

```javascript
function recursiveChunk(text, maxSize = 500, overlap = 50) {
  const separators = ["\n\n", "\n", ". ", " "];

  function split(text, sepIndex) {
    if (text.length <= maxSize) return [text];
    if (sepIndex >= separators.length) {
      // Last resort: hard cut
      return fixedSizeChunk(text, maxSize, overlap);
    }

    const sep = separators[sepIndex];
    const parts = text.split(sep);
    const chunks = [];
    let current = "";

    for (const part of parts) {
      const candidate = current ? current + sep + part : part;
      if (candidate.length > maxSize && current) {
        chunks.push(current);
        current = part;
      } else {
        current = candidate;
      }
    }

    if (current) chunks.push(current);

    // Recurse on chunks that are still too large
    return chunks.flatMap((chunk) =>
      chunk.length > maxSize ? split(chunk, sepIndex + 1) : [chunk]
    );
  }

  return split(text, 0);
}
```

This produces chunks that respect natural boundaries. Paragraphs stay together, sentences stay whole, and hard character splits only happen as a last resort.

### Semantic chunking

The most sophisticated approach: use the content's meaning to decide where to split. One method computes sentence embeddings and starts a new chunk when the similarity between consecutive sentences drops below a threshold — indicating a topic shift.

```javascript
// Conceptual — requires an embedding function
function semanticChunkBoundaries(sentences, embeddings, threshold = 0.75) {
  const boundaries = [0];

  for (let i = 1; i < sentences.length; i++) {
    const similarity = cosineSimilarity(embeddings[i - 1], embeddings[i]);
    if (similarity < threshold) {
      boundaries.push(i); // Topic shift detected
    }
  }

  return boundaries;
}
```

Semantic chunking produces the best retrieval results but costs more (you need embeddings just for chunking) and adds complexity. Use it when retrieval quality justifies the overhead.

### Chunk size tradeoffs

| Chunk size              | Retrieval                             | Context                                   |
| ----------------------- | ------------------------------------- | ----------------------------------------- |
| Small (100-200 chars)   | High precision, finds exact sentences | Low — may lack surrounding context        |
| Medium (500-1000 chars) | Good balance for most use cases       | Usually enough to be self-contained       |
| Large (1500-3000 chars) | Lower precision, more noise           | Rich context, but dilutes relevance score |

The sweet spot for most RAG systems is **500-1000 characters** (roughly 100-250 tokens). Start there and adjust based on eval results.

### Preserving metadata through chunking

Each chunk should inherit metadata from its parent document plus its own position:

```javascript
function chunkWithMetadata(document, chunks) {
  return chunks.map((text, index) => ({
    text,
    metadata: {
      ...document.metadata,
      chunkIndex: index,
      totalChunks: chunks.length,
    },
  }));
}
```

## What you learned

- Chunking splits documents into embeddable pieces — too small loses context, too large dilutes relevance. The sweet spot is 500-1000 characters.
- Recursive character splitting is the production default: it prefers paragraph breaks, falls back to sentences, and hard-splits only as a last resort.
- Overlap (10-20% of chunk size) prevents important sentences from being lost at chunk boundaries.
