## Why this matters

You have learned each stage of a RAG pipeline in isolation. This lesson connects them end-to-end. Building the full pipeline reveals how each decision — chunk size, embedding model, retrieval strategy, prompt template — affects the final answer quality. This is the architecture you will build, debug, and iterate on in production.

## The complete architecture

```text
Documents → Load → Chunk → Embed → Store (Vector DB)
                                         ↓
User query → Embed query → Retrieve → Rerank → Build prompt → LLM → Answer
```

Seven stages, each with clear inputs and outputs. Let's build each one.

## Stage 1: Document loading

Load raw documents and extract text with metadata. In a real system, you would support multiple formats. Here we work with plain text documents.

## Stage 2: Chunking

Split each document into smaller chunks. We use a simple sentence-based splitter with a maximum character limit.

```javascript
function chunkDocument(document, maxChunkSize = 150) {
  const sentences = document.text.match(/[^.!?]+[.!?]+/g) || [document.text];
  const chunks = [];
  let current = "";

  for (const sentence of sentences) {
    const trimmed = sentence.trim();
    if (current.length + trimmed.length > maxChunkSize && current) {
      chunks.push({
        text: current.trim(),
        metadata: {
          ...document.metadata,
          chunkIndex: chunks.length,
        },
      });
      current = trimmed;
    } else {
      current = current ? current + " " + trimmed : trimmed;
    }
  }

  if (current.trim()) {
    chunks.push({
      text: current.trim(),
      metadata: { ...document.metadata, chunkIndex: chunks.length },
    });
  }

  return chunks;
}

const allChunks = documents.flatMap((doc) => chunkDocument(doc));
console.log(`Total chunks: ${allChunks.length}`);
```

## Stage 3: Embedding

In production, you would call OpenAI or Cohere. For this project, we use a deterministic fake embedder that produces consistent vectors based on word content — enough to demonstrate the pipeline mechanics.

```javascript
function fakeEmbed(text, dimensions = 8) {
  // Deterministic pseudo-embedding based on character codes
  const vector = new Array(dimensions).fill(0);
  const words = text.toLowerCase().split(/\W+/);
  for (const word of words) {
    for (let i = 0; i < word.length; i++) {
      vector[i % dimensions] += word.charCodeAt(i) / 1000;
    }
  }
  // Normalize
  const magnitude = Math.sqrt(vector.reduce((s, v) => s + v * v, 0));
  return vector.map((v) => v / (magnitude || 1));
}

// Embed all chunks
const embeddedChunks = allChunks.map((chunk) => ({
  ...chunk,
  embedding: fakeEmbed(chunk.text),
}));
```

## Stage 4: Vector storage

A simple in-memory vector store with cosine similarity search and metadata filtering.

```javascript
class VectorStore {
  constructor() {
    this.entries = [];
  }

  upsert(chunks) {
    for (const chunk of chunks) {
      this.entries.push(chunk);
    }
  }

  query(queryEmbedding, topK = 3) {
    return this.entries
      .map((entry) => ({
        ...entry,
        score: this.cosine(queryEmbedding, entry.embedding),
      }))
      .sort((a, b) => b.score - a.score)
      .slice(0, topK);
  }

  cosine(a, b) {
    let dot = 0,
      nA = 0,
      nB = 0;
    for (let i = 0; i < a.length; i++) {
      dot += a[i] * b[i];
      nA += a[i] * a[i];
      nB += b[i] * b[i];
    }
    return dot / (Math.sqrt(nA) * Math.sqrt(nB));
  }
}

const store = new VectorStore();
store.upsert(embeddedChunks);
console.log(`Stored ${store.entries.length} vectors`);
```

## Stage 5: Retrieval

Embed the user's query with the same embedding function, then search the store.

```javascript
function retrieve(query, store, topK = 3) {
  const queryEmbedding = fakeEmbed(query);
  const results = store.query(queryEmbedding, topK);

  return results.map((r) => ({
    text: r.text,
    source: r.metadata.source,
    title: r.metadata.title,
    score: r.score,
  }));
}

const results = retrieve("How do I get a refund?", store);
console.log("Retrieved:", results);
```

## Stage 6: Prompt construction

Assemble the retrieved context and the user's question into a prompt the LLM can answer from.

```javascript
function buildPrompt(question, retrievedChunks) {
  const context = retrievedChunks
    .map((chunk, i) => `[Source ${i + 1}: ${chunk.title}]\n${chunk.text}`)
    .join("\n\n");

  return (
    `Answer the question based ONLY on the provided context. ` +
    `If the context does not contain the answer, say "I don't have ` +
    `information about that."\n\n` +
    `Context:\n${context}\n\n` +
    `Question: ${question}\n\n` +
    `Answer:`
  );
}

const prompt = buildPrompt("How do I get a refund?", results);
console.log(prompt);
```

The prompt template is critical. Telling the model to answer **only** from context reduces hallucination. Including source labels enables citations in the response.

## Putting it all together

```javascript
function ragPipeline(question, store) {
  // Retrieve
  const chunks = retrieve(question, store, 3);

  // Build prompt
  const prompt = buildPrompt(question, chunks);

  // Generate
  const answer = simulateGeneration(prompt);

  return {
    answer,
    sources: chunks.map((c) => ({ title: c.title, source: c.source })),
    prompt,
  };
}

// Run it
const response = ragPipeline("How do I get a refund?", store);
console.log("Answer:", response.answer);
console.log("Sources:", response.sources);
```

This is a complete RAG pipeline. Every production system follows this same seven-stage structure — the difference is using real embedding APIs, real vector databases, and real LLMs instead of the simulation code here.

## What you learned

- A complete RAG pipeline has seven stages: document loading, chunking, embedding, vector storage, retrieval, prompt construction, and generation.
- The prompt template is a critical control point — grounding instructions reduce hallucination and source labels enable citations.
- Every stage uses the same embedding function, and each stage's output feeds directly into the next, making the pipeline easy to debug one stage at a time.
