## Why this matters

Almost every modern web application talks to a server — fetching posts, submitting orders, loading user profiles. The browser's `fetch` API is how you make that happen. Understanding `fetch` also means understanding Promises and `async/await`, because network requests are inherently asynchronous — they take time to complete, and your code needs to handle that without freezing the page.

## How fetch works

`fetch()` returns a Promise that resolves when the server sends back headers. At that point you have a `Response` object, but you don't have the body yet. You then call `.json()` (or `.text()`, `.blob()`) to parse the body — that's a second Promise.

```javascript
// The two-step pattern
fetch("https://api.example.com/posts")
  .then((response) => response.json()) // Step 1: get headers → Step 2: parse body
  .then((data) => console.log(data)) // Step 3: use the data
  .catch((error) => console.error("Network error:", error));
```

With `async/await`, which reads more like synchronous code:

```javascript
async function loadPosts() {
  const response = await fetch("https://api.example.com/posts");
  const data = await response.json();
  console.log(data);
}
```

## Checking for errors

Here is one of the most common `fetch` mistakes: assuming a non-200 status code throws an error. It doesn't. `fetch` only rejects its Promise on **network failures** (no connection, DNS failure). A 404 or 500 response resolves normally — you have to check `response.ok` yourself.

```javascript
async function loadPost(id) {
  const response = await fetch(`/api/posts/${id}`);

  if (!response.ok) {
    // response.ok is true for 200-299, false for everything else
    throw new Error(`Failed to load post: ${response.status} ${response.statusText}`);
  }

  return await response.json();
}

// Wrap the call in try/catch to handle both network errors and bad statuses
try {
  const post = await loadPost(42);
  renderPost(post);
} catch (error) {
  showError(error.message);
}
```

## Sending data — POST requests

GET requests are for reading data. POST (and PUT/PATCH/DELETE) are for sending data. Set the method, add `Content-Type: application/json`, and pass `JSON.stringify(body)` as the body:

```javascript
async function createPost(postData) {
  const response = await fetch("/api/posts", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
    },
    body: JSON.stringify(postData),
  });

  if (!response.ok) {
    throw new Error(`Server error: ${response.status}`);
  }

  return await response.json(); // Server returns the created post
}

await createPost({ title: "Hello World", body: "My first post." });
```

Common HTTP methods and what they mean:

| Method | Purpose                     |
| ------ | --------------------------- |
| GET    | Read a resource             |
| POST   | Create a new resource       |
| PUT    | Replace a resource entirely |
| PATCH  | Update part of a resource   |
| DELETE | Remove a resource           |

## Request headers

Headers are metadata attached to every HTTP request. Two you'll send often:

```javascript
const response = await fetch("/api/data", {
  headers: {
    "Content-Type": "application/json", // Body is JSON
    Authorization: `Bearer ${accessToken}`, // Auth token
  },
});
```

And two you'll read from the response:

```javascript
response.headers.get("Content-Type"); // "application/json; charset=utf-8"
response.headers.get("X-Rate-Limit-Remaining"); // Custom header from server
```

## Cancelling requests with AbortController

If the user navigates away, or types a new search before the previous one finishes, you should cancel the in-flight request. `AbortController` does this:

```javascript
let currentController = null;

async function search(query) {
  // Cancel any request that's still in flight
  if (currentController) {
    currentController.abort();
  }

  currentController = new AbortController();

  try {
    const response = await fetch(`/api/search?q=${query}`, {
      signal: currentController.signal, // Attach the abort signal
    });

    if (!response.ok) throw new Error("Search failed");
    const results = await response.json();
    renderResults(results);
  } catch (error) {
    if (error.name === "AbortError") return; // Intentional cancellation — ignore it
    showError(error.message);
  }
}
```

AbortController matters most in search-as-you-type and React/component-based apps where a component unmounts while a request is still pending.

## A complete fetch utility

A reusable helper function eliminates repetition and enforces consistent error handling across your app:

```javascript
async function apiFetch(url, options = {}) {
  const response = await fetch(url, {
    headers: {
      "Content-Type": "application/json",
      ...options.headers,
    },
    ...options,
  });

  if (!response.ok) {
    const error = await response.json().catch(() => ({}));
    throw new Error(error.message ?? `HTTP ${response.status}`);
  }

  // Handle 204 No Content (no body to parse)
  if (response.status === 204) return null;

  return await response.json();
}

// Usage
const user = await apiFetch("/api/users/1");
const newPost = await apiFetch("/api/posts", {
  method: "POST",
  body: JSON.stringify({ title: "Hello" }),
});
```

## What you learned

- `fetch` returns a two-step Promise: the first resolves with a `Response` when headers arrive, and you call `.json()` for a second Promise that resolves with the parsed body
- `response.ok` is `false` for 4xx and 5xx status codes — you must check it explicitly because `fetch` only rejects on network failure, not HTTP errors
- `AbortController` lets you cancel in-flight requests by passing its `.signal` to fetch, which prevents stale responses from rendering after the user has moved on
