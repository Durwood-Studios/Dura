## Why this matters

Most browser storage tutorials reach for a server database the moment data needs to persist. But many features — user preferences, draft content, recently viewed items, offline progress — don't belong in a database at all. LocalStorage is the simplest form of browser persistence, and understanding it well means knowing not just how to use it, but when to use it and what its limits are.

## The localStorage API

`localStorage` stores key-value pairs that survive page reloads and browser restarts. Each key and value must be a string.

```javascript
// Store a value
localStorage.setItem("theme", "dark");

// Read it back — returns null if the key doesn't exist
const theme = localStorage.getItem("theme"); // "dark"
const missing = localStorage.getItem("nope"); // null

// Update (same as setting — just overwrites)
localStorage.setItem("theme", "light");

// Delete a single key
localStorage.removeItem("theme");

// Delete everything in localStorage for this origin
localStorage.clear();

// Check how many keys are stored
localStorage.length; // Number

// Iterate all keys
for (let i = 0; i < localStorage.length; i++) {
  const key = localStorage.key(i);
  const value = localStorage.getItem(key);
  console.log(key, value);
}
```

## Storing complex data with JSON

Because localStorage only stores strings, you must serialize objects and arrays to JSON before storing them, and parse them back when reading:

```javascript
// Storing an object
const settings = {
  theme: "dark",
  fontSize: 16,
  reducedMotion: false,
};
localStorage.setItem("settings", JSON.stringify(settings));

// Reading it back
const raw = localStorage.getItem("settings");
const parsed = raw ? JSON.parse(raw) : null;
console.log(parsed?.theme); // "dark"
```

The `raw ? JSON.parse(raw) : null` pattern handles the case where the key doesn't exist — `JSON.parse(null)` returns `null`, which is sometimes fine, but `JSON.parse` can also throw on malformed data.

A robust read helper:

```javascript
function storageGet(key, fallback = null) {
  try {
    const raw = localStorage.getItem(key);
    return raw !== null ? JSON.parse(raw) : fallback;
  } catch {
    // Corrupted data — return the fallback
    return fallback;
  }
}

function storageSet(key, value) {
  try {
    localStorage.setItem(key, JSON.stringify(value));
  } catch (error) {
    // Storage quota exceeded — handle gracefully
    console.error("localStorage write failed:", error);
  }
}
```

## sessionStorage vs localStorage

`sessionStorage` has an identical API but different lifetime: it's scoped to the **browser tab** and cleared when the tab closes. Two tabs don't share sessionStorage, even on the same origin.

| Feature    | localStorage                | sessionStorage         |
| ---------- | --------------------------- | ---------------------- |
| Lifetime   | Until explicitly cleared    | Until tab closes       |
| Scope      | All tabs, same origin       | This tab only          |
| Size limit | ~5-10 MB per origin         | ~5-10 MB per tab       |
| Use case   | Preferences, saved progress | Per-session form state |

```javascript
// Good sessionStorage use: preserve form data if the user accidentally refreshes
sessionStorage.setItem("checkout-draft", JSON.stringify(formData));
```

## What NOT to store

LocalStorage is synchronous and blocking. Large reads block the main thread. Sensitive data is readable by any JavaScript on the page (including third-party scripts). Binary data is slow to encode/decode as strings.

**Never store:**

- Authentication tokens or session secrets — use `httpOnly` cookies set by the server
- Sensitive user data (credit cards, passwords, health data)
- Files or binary data — use IndexedDB
- More than a few megabytes — use IndexedDB
- Anything you'd be upset to lose without warning — use a server

**Good use cases:**

- UI preferences (theme, language, layout)
- Completed lesson IDs
- Draft text that doesn't exist yet on the server
- Cached responses that don't need to be perfectly fresh

## The storage event

When one tab changes localStorage, other open tabs on the same origin receive a `storage` event. This is how you can sync state across tabs without a server:

```javascript
window.addEventListener("storage", (event) => {
  event.key; // Which key changed
  event.oldValue; // Previous value (as string)
  event.newValue; // New value (as string), null if deleted
  event.url; // URL of the page that made the change
  event.storageArea; // localStorage or sessionStorage

  if (event.key === "theme") {
    applyTheme(event.newValue);
  }
});
```

Note: the storage event fires in **other** tabs, not in the tab that made the change.

## IndexedDB for more complex needs

localStorage is synchronous and string-only. For structured data, binary files, or more than a few megabytes, IndexedDB is the right tool. It's more complex to use directly, but libraries like `idb` wrap it with a clean Promise-based API.

Think of localStorage as a small notepad — fast, simple, and always accessible. IndexedDB is a full local database — powerful, but you set it up when you need it.

## What you learned

- `localStorage` stores string key-value pairs that persist across reloads; use `JSON.stringify` and `JSON.parse` for objects, and always handle `null` and parse errors defensively
- `sessionStorage` has the same API but is scoped to the tab — cleared when the tab closes and not shared with other tabs
- Never store tokens, passwords, or sensitive data in localStorage; the storage event lets other tabs react to changes, and IndexedDB is the right tool when you need binary data or more than a few megabytes
