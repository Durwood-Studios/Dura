## Why this matters

Forms are how users give you data — account registration, search queries, checkout flows, support tickets. Validation done well protects your server from garbage data and protects your users from confusing failures. Done poorly, it frustrates people with cryptic errors at the wrong time. This lesson covers reading form values correctly and validating them both with HTML5 constraints and with your own JavaScript logic.

## Reading form values

Every form input exposes its current value through JavaScript properties. The property you use depends on the input type.

```javascript
// Text, email, password, number, url, tel, textarea
const input = document.querySelector("#username");
input.value; // The current string value

// Checkbox
const checkbox = document.querySelector("#agree");
checkbox.checked; // true or false

// Radio buttons (find the selected one)
const selected = document.querySelector('input[name="color"]:checked');
selected?.value; // "red", "blue", etc. — null if none selected

// Select dropdown
const select = document.querySelector("#country");
select.value; // The value of the currently selected option

// Multiple select — returns a FileList, use Array.from
const multiSelect = document.querySelector("#tags");
Array.from(multiSelect.selectedOptions).map((opt) => opt.value);
```

### Listening for changes

Use `input` for live feedback (fires on every keystroke), or `change` for feedback after the user finishes and moves on:

```javascript
const emailInput = document.querySelector("#email");

// Fires on every keystroke — good for live character counts
emailInput.addEventListener("input", (event) => {
  charCount.textContent = event.target.value.length;
});

// Fires when focus leaves the field — good for format validation
emailInput.addEventListener("change", (event) => {
  validateEmail(event.target.value);
});
```

## Intercepting form submission

The `submit` event fires when the user submits a form, whether by clicking a submit button or pressing Enter. Always use `event.preventDefault()` in a JavaScript-driven form to stop the browser from sending an HTTP request and reloading the page.

```javascript
const form = document.querySelector("#signup-form");

form.addEventListener("submit", (event) => {
  event.preventDefault(); // Stop the page reload

  // Read all values
  const data = {
    username: form.elements["username"].value.trim(),
    email: form.elements["email"].value.trim(),
    password: form.elements["password"].value,
  };

  // Validate
  const errors = validate(data);
  if (Object.keys(errors).length > 0) {
    displayErrors(errors);
    return;
  }

  // Submit with fetch
  submitSignup(data);
});
```

`form.elements` is a convenient way to access all named inputs in a form without querying by ID.

## Client-side validation

### HTML5 constraint attributes

HTML provides built-in validation through element attributes. These work without JavaScript:

```html
<input type="email" required />
<input type="text" minlength="3" maxlength="50" required />
<input type="number" min="1" max="100" />
<input type="text" pattern="[A-Za-z0-9]+" title="Letters and numbers only" />
<input type="url" required />
```

When a user submits a form with invalid inputs, the browser blocks submission and shows a native error bubble. The styling and wording varies by browser and can't be fully controlled — that's why most production apps use the Constraint Validation API instead.

### The Constraint Validation API

Every form element exposes validation methods you can call programmatically:

```javascript
const input = document.querySelector("#username");

input.checkValidity(); // Returns true/false based on constraints
input.validationMessage; // Browser's built-in error message string
input.validity.valueMissing; // true if required + empty
input.validity.tooShort; // true if shorter than minlength
input.validity.patternMismatch; // true if pattern doesn't match

// Override the browser's message with your own:
input.setCustomValidity("Username must be at least 3 characters");
input.setCustomValidity(""); // Clear the custom message (marks as valid)
```

### Writing your own validation

For full control over timing, messages, and UX, write a pure validation function that checks your data and returns an error map:

```javascript
function validateSignup(data) {
  const errors = {};

  if (!data.username || data.username.length < 3) {
    errors.username = "Username must be at least 3 characters.";
  }

  if (!data.email.includes("@")) {
    errors.email = "Please enter a valid email address.";
  }

  if (data.password.length < 8) {
    errors.password = "Password must be at least 8 characters.";
  }

  return errors; // Empty object means valid
}
```

Keeping validation in a pure function makes it testable without any browser involvement.

## Validation UX best practices

Good validation UX follows a few rules that reduce user frustration:

**Validate on blur, not on every keystroke.** Don't tell a user their email is invalid while they're still typing it. Wait until they leave the field.

**Show errors inline, near the field.** Don't put a generic "form has errors" banner at the top. Place the message right under the problematic input.

**Clear the error as soon as they start fixing it.** Give immediate positive feedback.

**Never disable the submit button to "prevent invalid submissions."** Users don't know what's wrong. Let them try; show the errors.

```javascript
// Good pattern: validate on blur, clear on input
input.addEventListener("blur", () => {
  const error = validateField(input.name, input.value);
  showError(input, error);
});

input.addEventListener("input", () => {
  clearError(input); // Give them a clean slate while they type
});
```

## What you learned

- Read text input values with `.value`, checkboxes with `.checked`, and selected options with `select.value`; the `input` event fires live, `change` fires on blur
- Always call `event.preventDefault()` in a form's submit handler to intercept submission and handle it with JavaScript
- Keep validation logic in pure functions that return an error map — they're testable, reusable, and separate from any DOM concerns
