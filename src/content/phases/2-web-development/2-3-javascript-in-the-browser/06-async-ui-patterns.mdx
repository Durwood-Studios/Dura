## Why this matters

Async code introduces time — and during that time, the user is watching. A page that silently freezes for three seconds loses trust instantly. Getting async UI right is the difference between an app that feels professional and one that feels broken. The patterns here — loading states, error recovery, debounce, optimistic updates — are the vocabulary of production-quality interfaces.

## The three states of every async operation

Every operation that involves a network request or delay has three possible states. Your UI must handle all three explicitly:

- **Idle** — No operation in progress (initial state or after completion)
- **Loading** — Operation in progress; user is waiting
- **Error** — Operation failed; user needs to know and needs a path forward

```javascript
let state = "idle"; // "idle" | "loading" | "error"
let data = null;
let errorMessage = null;

async function loadData() {
  state = "loading";
  render();

  try {
    data = await apiFetch("/api/items");
    state = "idle";
  } catch (error) {
    state = "error";
    errorMessage = error.message;
  }

  render();
}

function render() {
  if (state === "loading") showSpinner();
  else if (state === "error") showError(errorMessage);
  else if (data) showContent(data);
  else showEmptyState();
}
```

Always call `render()` after state changes, not in the middle of transitions.

## Loading states

A loading state should tell the user something is happening and give them a sense of what's coming.

**Spinner** — good for short waits where you don't know the structure of what's loading:

```javascript
function showSpinner() {
  container.innerHTML = `<div class="spinner" aria-label="Loading…"></div>`;
}
```

**Skeleton UI** — placeholder elements in the shape of the content that will appear. Reduces perceived wait time significantly because the layout doesn't jump:

```javascript
function showSkeleton() {
  container.innerHTML = `
    <div class="skeleton-card">
      <div class="skeleton skeleton-title"></div>
      <div class="skeleton skeleton-body"></div>
      <div class="skeleton skeleton-body skeleton-body--short"></div>
    </div>
  `;
}
```

**Disabled button during submission** — prevents double-submits:

```javascript
async function handleSubmit() {
  submitButton.disabled = true;
  submitButton.textContent = "Saving…";

  try {
    await saveData(formData);
    submitButton.textContent = "Saved!";
  } catch {
    submitButton.textContent = "Try again";
  } finally {
    submitButton.disabled = false;
  }
}
```

## Error states

An error state must give users three things: what went wrong, whether it was their fault, and what they can do. "An error occurred" is not helpful. "Couldn't load your posts — check your connection and try again." is.

```javascript
function showError(message, onRetry) {
  container.innerHTML = `
    <div class="error-state" role="alert">
      <p>${message}</p>
      <button id="retry-btn">Try again</button>
    </div>
  `;
  document.querySelector("#retry-btn").addEventListener("click", onRetry);
}
```

Note the `role="alert"` — screen readers will announce this immediately.

## Optimistic updates

An optimistic update changes the UI before the server confirms success, then rolls back if the request fails. This makes the app feel instant even on slow connections.

```javascript
async function toggleLike(postId) {
  // Optimistically update the UI now
  const wasLiked = post.liked;
  post.liked = !wasLiked;
  post.likeCount += wasLiked ? -1 : 1;
  renderPost(post);

  try {
    // Confirm with the server in the background
    await apiFetch(`/api/posts/${postId}/like`, { method: "POST" });
  } catch {
    // Rollback if the server rejected it
    post.liked = wasLiked;
    post.likeCount += wasLiked ? 1 : -1;
    renderPost(post);
    showToast("Couldn't update — please try again.");
  }
}
```

Optimistic updates are appropriate when failure is rare and reversible. Don't use them for destructive actions (deleting data) or payments.

## Debouncing

When a user types in a search box, you don't want to fire a network request on every single keystroke. Debouncing delays the call until the user pauses.

```javascript
function debounce(fn, delayMs) {
  let timerId = null;

  return function (...args) {
    // Cancel the previous scheduled call
    clearTimeout(timerId);

    // Schedule a new call after the delay
    timerId = setTimeout(() => {
      fn(...args);
    }, delayMs);
  };
}

const search = debounce(async (query) => {
  const results = await apiFetch(`/api/search?q=${query}`);
  renderResults(results);
}, 300); // 300ms after the last keystroke

searchInput.addEventListener("input", (event) => {
  search(event.target.value);
});
```

With a 300ms debounce, a user typing "javascript" at normal speed triggers one request instead of ten.

## Race conditions

A race condition happens when two async operations compete and the wrong one wins. The classic case: user types "cat", request fires. They then type "cats", another request fires. If the "cats" response arrives first and "cat" arrives second, the UI shows "cat" results for a "cats" query.

The fix is to cancel the previous request before starting a new one:

```javascript
let searchController = null;

const search = debounce(async (query) => {
  if (searchController) searchController.abort();
  searchController = new AbortController();

  try {
    const results = await apiFetch(`/api/search?q=${query}`, {
      signal: searchController.signal,
    });
    renderResults(results);
  } catch (error) {
    if (error.name !== "AbortError") showError(error.message);
  }
}, 300);
```

Debounce reduces the number of requests; AbortController prevents stale results from appearing.

## What you learned

- Every async operation has three states — idle, loading, error — and your UI needs to handle all three explicitly with meaningful feedback, not silence
- Optimistic updates apply the change immediately and roll back on failure; debouncing delays execution until the user pauses to reduce unnecessary requests
- Race conditions occur when slow responses arrive after faster ones have already rendered — combine debounce with AbortController to prevent stale data from winning
