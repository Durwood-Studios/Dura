## Why this matters

Knowing individual browser APIs is like knowing vocabulary. Building a complete project is grammar — it's where you learn how the pieces fit together into something coherent. A todo app is the traditional "hello world" of UI projects because it requires everything at once: state management, rendering, events, persistence, and filtering. Finishing this teaches you more than any individual lesson can.

## The architecture of a vanilla JS app

Before writing a single line of code, understand the architecture. Every mistake in vanilla JS apps comes from the same place: the DOM becoming out of sync with the program's actual state.

The solution is **state-first development**:

1. **State lives in JavaScript** — a plain array or object is your source of truth
2. **Render derives from state** — the DOM is always generated from the current state, never edited in-place
3. **Events update state** — user actions mutate the state object, then call `render()`

```
User Action → Update State → render() → DOM reflects new state
```

This is the same pattern React, Vue, and Svelte all follow under the hood — they just automate the re-render step.

## The state shape

Define your state before anything else:

```javascript
let state = {
  todos: [
    { id: 1, text: "Learn JavaScript", completed: false },
    { id: 2, text: "Build something", completed: false },
  ],
  filter: "all", // "all" | "active" | "completed"
  nextId: 3,
};
```

Every piece of data the UI needs to display lives in state. The DOM displays what state says. The DOM doesn't store state.

## State mutation functions

Write pure functions to update state — don't scatter mutations throughout event handlers:

```javascript
function addTodo(text) {
  state.todos.push({
    id: state.nextId++,
    text: text.trim(),
    completed: false,
  });
}

function toggleTodo(id) {
  const todo = state.todos.find((t) => t.id === id);
  if (todo) todo.completed = !todo.completed;
}

function removeTodo(id) {
  state.todos = state.todos.filter((t) => t.id !== id);
}

function setFilter(filter) {
  state.filter = filter;
}
```

After any mutation, you call `render()`. The UI always reflects the current state.

## The render function

The render function clears the relevant portion of the DOM and rebuilds it from state. This sounds slow, but for small apps with a few hundred items it's imperceptibly fast.

```javascript
function getFilteredTodos() {
  if (state.filter === "active") return state.todos.filter((t) => !t.completed);
  if (state.filter === "completed") return state.todos.filter((t) => t.completed);
  return state.todos;
}

function render() {
  const list = document.querySelector("#todo-list");
  const todos = getFilteredTodos();

  // Rebuild the list from scratch
  list.innerHTML = "";

  if (todos.length === 0) {
    list.innerHTML = `<li class="empty-state">No items to show.</li>`;
    return;
  }

  const fragment = document.createDocumentFragment();
  todos.forEach((todo) => {
    const li = document.createElement("li");
    li.className = `todo-item${todo.completed ? " todo-item--done" : ""}`;
    li.dataset.id = todo.id;

    li.innerHTML = `
      <button class="todo-toggle" aria-label="Mark complete">
        ${todo.completed ? "✓" : "○"}
      </button>
      <span class="todo-text">${escapeHtml(todo.text)}</span>
      <button class="todo-delete" aria-label="Delete">×</button>
    `;

    fragment.appendChild(li);
  });

  list.appendChild(fragment);

  // Update the counter
  const activeCount = state.todos.filter((t) => !t.completed).length;
  document.querySelector("#item-count").textContent =
    `${activeCount} item${activeCount === 1 ? "" : "s"} left`;
}
```

Note `escapeHtml()` — always escape user-provided text before inserting into `innerHTML`. A simple implementation:

```javascript
function escapeHtml(text) {
  const div = document.createElement("div");
  div.textContent = text;
  return div.innerHTML;
}
```

## Event handling

Use event delegation on the list for toggles and deletes — one listener handles everything, including items added later. Use a direct listener on the form for adding new items.

```javascript
// Add new todo
const form = document.querySelector("#todo-form");
form.addEventListener("submit", (event) => {
  event.preventDefault();
  const input = form.elements["todo-input"];
  const text = input.value.trim();
  if (!text) return;

  addTodo(text);
  persist();
  render();

  input.value = "";
  input.focus();
});

// Toggle and delete via delegation
const list = document.querySelector("#todo-list");
list.addEventListener("click", (event) => {
  const item = event.target.closest(".todo-item");
  if (!item) return;

  const id = Number(item.dataset.id);

  if (event.target.closest(".todo-toggle")) {
    toggleTodo(id);
    persist();
    render();
  }

  if (event.target.closest(".todo-delete")) {
    removeTodo(id);
    persist();
    render();
  }
});

// Filter buttons
document.querySelector("#filters").addEventListener("click", (event) => {
  const btn = event.target.closest("[data-filter]");
  if (!btn) return;

  setFilter(btn.dataset.filter);
  render();
});
```

## Persisting to localStorage

Saving and loading state to localStorage rounds out the feature set — todos survive a page refresh:

```javascript
const STORAGE_KEY = "dura-todos";

function persist() {
  localStorage.setItem(STORAGE_KEY, JSON.stringify(state));
}

function loadPersistedState() {
  try {
    const saved = localStorage.getItem(STORAGE_KEY);
    if (saved) {
      state = JSON.parse(saved);
    }
  } catch {
    // Corrupted data — start fresh
    console.warn("Could not load saved todos, starting fresh.");
  }
}

// Initialize
loadPersistedState();
render();
```

Call `persist()` after every mutation that should be saved (add, toggle, remove). Don't persist `filter` — the filter is a UI preference for this session, not part of the data.

## What you've built

This app demonstrates every major concept from the module:

- DOM selection and manipulation to build the UI
- Events and event delegation for interaction
- Form handling with `preventDefault`
- Pure state functions that are easy to test
- `DocumentFragment` for efficient bulk rendering
- localStorage for persistence across reloads

The architecture scales. When a real app gets complex, you move the state to Zustand, add a proper router, and start rendering components instead of raw HTML strings. But the mental model — state first, render derives from state, events update state — stays exactly the same.

## What you learned

- State-first architecture keeps a JavaScript object as the single source of truth; the DOM is always a rendered reflection of that state, never a place to store data
- A `render()` function that rebuilds from state eliminates whole categories of bugs where the UI and the program's internal state get out of sync
- Event delegation, `DocumentFragment`, and localStorage tie the patterns from every previous lesson together into a complete, production-quality application architecture
