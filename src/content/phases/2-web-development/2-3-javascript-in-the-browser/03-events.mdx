## Why this matters

A web page without events is a brochure. Events are the mechanism that turns static HTML into interactive software. Every button click, form submission, keyboard shortcut, and drag interaction is powered by the browser's event system. Mastering events — including the non-obvious behavior of bubbling — will make you significantly more effective at building anything interactive.

## addEventListener

`addEventListener` attaches a handler function to an element. The handler runs every time the specified event fires on that element.

```javascript
const button = document.querySelector("#save-btn");

button.addEventListener("click", function (event) {
  console.log("Button clicked!");
  console.log(event.target); // The element that was clicked
});
```

Arrow functions are the common style for concise handlers:

```javascript
button.addEventListener("click", (event) => {
  console.log("Clicked:", event.target.textContent);
});
```

To remove a listener later, you need a named reference:

```javascript
function handleClick(event) {
  console.log("clicked");
}

button.addEventListener("click", handleClick);
// Later:
button.removeEventListener("click", handleClick);
```

## Common event types

```javascript
// Mouse
element.addEventListener("click", handler); // Single click
element.addEventListener("dblclick", handler); // Double click
element.addEventListener("mouseenter", handler); // Cursor enters element
element.addEventListener("mouseleave", handler); // Cursor leaves element

// Keyboard (fires on the focused element, or document for global shortcuts)
document.addEventListener("keydown", handler); // Key pressed
document.addEventListener("keyup", handler); // Key released

// Input and forms
input.addEventListener("input", handler); // Value changes (fires live)
input.addEventListener("change", handler); // Value committed (on blur)
form.addEventListener("submit", handler); // Form submitted

// Document lifecycle
document.addEventListener("DOMContentLoaded", handler); // HTML parsed, DOM ready
window.addEventListener("load", handler); // Everything loaded (images too)
window.addEventListener("resize", handler); // Viewport size changed
```

## The event object

Every handler receives an **event object** — a snapshot of what happened. Its properties vary by event type.

```javascript
document.addEventListener("keydown", (event) => {
  event.key; // "Enter", "Escape", "a", "ArrowUp", etc.
  event.code; // "KeyA", "Space", "Enter" — physical key position
  event.ctrlKey; // true if Ctrl was held
  event.metaKey; // true if Cmd (Mac) / Win was held
  event.shiftKey; // true if Shift was held
});

document.addEventListener("click", (event) => {
  event.target; // The exact element that was clicked
  event.currentTarget; // The element the listener is attached to
  event.clientX; // X position relative to the viewport
  event.clientY; // Y position relative to the viewport
});
```

### preventDefault

Some events have default browser behavior. A click on an `<a>` navigates to the href. A form submit sends an HTTP request. `preventDefault()` stops that default action.

```javascript
form.addEventListener("submit", (event) => {
  event.preventDefault(); // Stop the page from reloading
  // Handle submission with JavaScript instead
  submitWithFetch(form);
});

link.addEventListener("click", (event) => {
  event.preventDefault(); // Stop navigation
  openModal(link.href);
});
```

## Event bubbling

When an event fires on an element, it doesn't stay there. It **bubbles up** through every ancestor element, triggering any listeners along the way.

```
User clicks on <button> inside <li> inside <ul> inside <main>
→ click fires on <button>
→ click fires on <li>
→ click fires on <ul>
→ click fires on <main>
→ click fires on <body>
→ click fires on <html>
→ click fires on document
```

This is why `event.target` (the element actually clicked) differs from `event.currentTarget` (the element your listener is on).

To stop an event from continuing to bubble:

```javascript
button.addEventListener("click", (event) => {
  event.stopPropagation(); // Don't bubble to parent elements
});
```

Use `stopPropagation` sparingly — it can break other listeners higher in the tree that legitimately need to hear the event.

## Event delegation

Bubbling enables a powerful pattern: instead of attaching a listener to every item in a list, attach **one listener to the parent** and inspect `event.target` to determine what was clicked.

```javascript
const todoList = document.querySelector("#todo-list");

// One listener handles clicks on ALL list items — even future ones
todoList.addEventListener("click", (event) => {
  const item = event.target.closest(".todo-item");
  if (!item) return; // Click was not on a todo item

  const id = item.dataset.id;
  toggleTodo(id);
});
```

`closest()` walks up the DOM from `event.target` until it finds an ancestor matching the selector — or returns `null` if none exists. This handles clicks on child elements inside `.todo-item` (like an icon or span).

Event delegation is the right approach whenever you have dynamic lists — items added after page load automatically get the behavior because the listener is on the stable parent.

## What you learned

- `addEventListener(type, handler)` attaches a handler that receives an event object with properties like `target`, `key`, and `clientX`; use `removeEventListener` with the same named function to clean up
- Events bubble upward through ancestor elements after firing on the target; `stopPropagation()` halts the bubble, and `preventDefault()` stops the browser's default action
- Event delegation attaches one listener to a parent and uses `event.target` (or `closest()`) to route behavior — the right approach for dynamic lists and large collections of interactive elements
