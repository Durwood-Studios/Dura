## Why this matters

Every real application fetches data. In traditional React, that meant `useEffect` + `useState` + loading flags + error handling — a waterfall of boilerplate before you showed a single piece of content. In the App Router, you `await` data directly in the component. The framework handles caching, loading states, and revalidation. The result is less code that does more.

## Fetching in server components

Because Server Components can be `async`, you fetch data the same way you'd write any Node.js code — just `await` a promise:

```typescript
// app/products/page.tsx
export default async function ProductsPage() {
  // No useEffect. No useState. Just await.
  const products = await fetch("https://api.example.com/products").then(
    (res) => res.json()
  );

  return (
    <ul>
      {products.map((p) => (
        <li key={p.id}>{p.name}</li>
      ))}
    </ul>
  );
}
```

This runs entirely on the server. The browser receives a finished HTML list — not a loading spinner that becomes a list. The user never sees the data-fetching happen.

## Next.js extends fetch with caching

Next.js patches the native `fetch` function in server environments to add caching behavior. You control it with the `cache` option and the `next` option:

### `cache: 'force-cache'` (default behavior in older Next.js)

```typescript
const data = await fetch("https://api.example.com/data", {
  cache: "force-cache",
});
// Result is cached indefinitely. Identical requests return the cached value.
// Useful for data that never changes: country lists, static config.
```

### `cache: 'no-store'`

```typescript
const data = await fetch("https://api.example.com/data", {
  cache: "no-store",
});
// Never cache. Always fetch fresh on every request.
// Useful for real-time data: prices, live counts, personalized content.
```

### `next: { revalidate: N }` — Incremental Static Regeneration

```typescript
const data = await fetch("https://api.example.com/data", {
  next: { revalidate: 60 }, // revalidate at most every 60 seconds
});
// Cached, but refreshed in the background after N seconds.
// Useful for content that changes occasionally: blog posts, product listings.
```

This third option — **ISR (Incremental Static Regeneration)** — is one of Next.js's most powerful features. A page is served from cache (fast), but the cache refreshes automatically in the background on a schedule. You get the speed of static with the freshness of dynamic.

## React's `cache()` for deduplication

When multiple components on a page need the same data, you don't want three separate network requests. React's `cache()` function deduplicates calls within a single render:

```typescript
import { cache } from "react";

// Wrap your fetch in cache() — identical calls in the same render
// return the same promise, resulting in one actual network request.
export const getUser = cache(async (id: string) => {
  const res = await fetch(`https://api.example.com/users/${id}`);
  return res.json();
});

// Now both components can call getUser(userId) safely.
// Only one fetch happens per request cycle.
```

This pattern is the recommended way to share data across a page without prop drilling — define a cached fetcher in a shared module and call it wherever needed.

## Parallel vs sequential fetching

The order in which you `await` matters for performance.

**Sequential** (slow — second fetch can't start until first completes):

```typescript
// Each await blocks the next — total time = A + B
const user = await getUser(userId);
const posts = await getPostsByUser(user.username); // needs user first
```

Sequential fetching is sometimes necessary when the second request depends on the first result. But when requests are independent, it's wasteful.

**Parallel** (fast — both fetches start simultaneously):

```typescript
// Both fetches start at the same time — total time = max(A, B)
const [user, posts] = await Promise.all([getUser(userId), getPosts()]);
```

Always reach for `Promise.all` when your fetches are independent. The savings compound: if fetching a user takes 200ms and fetching posts takes 300ms, sequential takes 500ms. Parallel takes 300ms.

## Loading states with `loading.tsx` and Suspense

A `loading.tsx` file in the same directory as a page automatically wraps that page in a Suspense boundary. While the async page component is fetching, the loading component renders:

```typescript
// app/products/loading.tsx
export default function Loading() {
  return (
    <div className="grid grid-cols-3 gap-4">
      {Array.from({ length: 6 }).map((_, i) => (
        <div key={i} className="h-40 bg-gray-800 animate-pulse rounded-xl" />
      ))}
    </div>
  );
}
```

For more granular control, you can use Suspense directly in your components to progressively reveal parts of a page:

```typescript
import { Suspense } from "react";

export default async function Page() {
  return (
    <div>
      <h1>Dashboard</h1>
      {/* This section shows immediately */}
      <StaticStats />
      {/* This section shows a skeleton while LiveFeed fetches */}
      <Suspense fallback={<FeedSkeleton />}>
        <LiveFeed />
      </Suspense>
    </div>
  );
}
```

This pattern — called **streaming** — lets you show a partial page immediately and fill in the slow parts as they load, rather than waiting for everything before showing anything.

## Dynamic metadata

Pages can export `generateMetadata` to produce dynamic `<title>` and `<meta>` tags that depend on fetched data:

```typescript
// app/blog/[slug]/page.tsx
import type { Metadata } from "next";

export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>;
}): Promise<Metadata> {
  const { slug } = await params;
  const post = await fetchPost(slug);
  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      title: post.title,
      images: [post.coverImage],
    },
  };
}
```

Next.js calls `generateMetadata` in parallel with the page component and injects the result into the `<head>`. You get per-page SEO without any third-party library.

## What you learned

- Async Server Components fetch data with plain `await` — no `useEffect`, no loading state boilerplate
- The `cache` option on `fetch` controls caching: `'no-store'` for always-fresh, `next: { revalidate: N }` for ISR (cached but refreshed on a schedule)
- React's `cache()` deduplicates identical fetch calls within a single render, preventing redundant network requests
- Use `Promise.all` for independent fetches (parallel is faster); sequential awaits only when a request depends on a previous result
- `loading.tsx` provides automatic Suspense boundaries; streaming with explicit `<Suspense>` lets you progressively reveal slow page sections
