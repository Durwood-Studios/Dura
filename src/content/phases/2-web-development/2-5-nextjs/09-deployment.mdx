## Why this matters

Code that lives only on your laptop isn't a product. Deployment is the moment your work becomes real — when users can reach it, when performance matters, when your choices about caching and server rendering pay off. Understanding what happens during a build and deploy cycle makes you a better developer: you write code that builds well, you catch issues before production, and you understand the costs of every page and API route.

## The `npm run build` output

Running `npm run build` is the first step in every deployment. It compiles your TypeScript, bundles your code, and pre-renders your static pages. The output tells you exactly what you're shipping:

```
Route (app)                              Size     First Load JS
┌ ○ /                                   5.2 kB   98.5 kB
├ ○ /about                              2.1 kB   95.4 kB
├ ● /blog                               3.8 kB   97.1 kB
├ ƒ /blog/[slug]                        1.2 kB   94.5 kB
└ ƒ /api/contact                        0 B      0 B

○  (Static)   prerendered as static content
●  (SSG)      prerendered as static HTML (uses generateStaticParams)
ƒ  (Dynamic)  server-rendered on demand
```

The symbols tell the rendering strategy:

- `○` Static — pure HTML, no server needed, served from CDN
- `●` SSG — static HTML generated from data at build time
- `ƒ` Dynamic — rendered per-request on the server

**First Load JS** is the JavaScript downloaded on first visit. Keep this number small. Large bundles mean slow Time to Interactive on mobile. If a route's first load JS is over 200kB, investigate what you're importing.

The build will fail with a clear error message if TypeScript types are wrong, ESLint rules are violated, or imports can't be resolved. This is intentional — catching errors at build time beats catching them in production.

## Environment variables

Next.js has a clear split for environment variables based on where they need to be accessible.

**Server-only variables** — no prefix required:

```bash
# .env.local
DATABASE_URL=postgres://user:password@host:5432/db
STRIPE_SECRET_KEY=sk_live_...
RESEND_API_KEY=re_...
```

These are available in Server Components, Route Handlers, Server Actions, and `next.config.ts`. They are **never** sent to the browser. If you accidentally reference a server-only variable in a Client Component, Next.js throws a build error.

**Client-accessible variables** — must start with `NEXT_PUBLIC_`:

```bash
# .env.local
NEXT_PUBLIC_SUPABASE_URL=https://yourproject.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIs...
NEXT_PUBLIC_GA_MEASUREMENT_ID=G-XXXXXXXXXX
```

`NEXT_PUBLIC_` variables are inlined into the JavaScript bundle at build time. Anyone who downloads your JavaScript can read them — treat them as public. The Supabase anon key is designed for public use; RLS policies protect the data.

**Environment files load in order** (later files override earlier):

```
.env            ← checked into git, safe defaults
.env.local      ← never committed, your secrets
.env.production ← production-specific values
.env.development ← development-specific values
```

## Deploying to Vercel

Vercel is the fastest path from code to production for Next.js.

1. **Push your code** to a GitHub (or GitLab/Bitbucket) repository.
2. **Connect the repo** on vercel.com — Vercel detects Next.js automatically.
3. **Add environment variables** in the Vercel dashboard under Settings → Environment Variables.
4. **Push a commit** — Vercel runs `npm run build`, deploys the output, and gives you a URL.

Every subsequent `git push` to your main branch triggers a new production deployment automatically. Every pull request gets a **preview deployment** — a unique URL like `my-app-git-feat-new-page.vercel.app` that shows exactly what your code does before it merges. Team members can review the live PR before approving it.

Key Vercel features worth knowing:

- **Zero-config** — Vercel reads your `next.config.ts` and handles SSR, ISR, and static assets automatically
- **Edge Functions** — middleware and Route Handlers marked `runtime: 'edge'` run on Vercel's global edge network
- **Analytics** — Vercel Analytics provides Core Web Vitals and page view data with no additional configuration
- **Domain management** — connect a custom domain in the dashboard; Vercel handles SSL automatically

## Other deployment targets

You're not locked into Vercel. Next.js is open source and runs anywhere Node.js runs.

### `next start` (self-hosted Node.js)

Build and run on any Linux server:

```bash
npm run build
npm start  # runs next start on port 3000
```

Use `nginx` as a reverse proxy in front for SSL termination and port 80/443 routing.

### Standalone output for Docker

Set `output: 'standalone'` in `next.config.ts` to generate a self-contained directory with only the files needed to run the app — no `node_modules` installation required in production:

```typescript
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: "standalone",
};

export default nextConfig;
```

After building, `.next/standalone` contains everything needed. Copy it to a Docker container and run `node server.js`. This produces lean Docker images — often 60-80% smaller than copying the full project.

### Other platforms

- **Railway** — git push deploys, managed databases, generous free tier
- **Render** — similar to Railway, zero-config for Node.js apps
- **Fly.io** — deploys Docker containers globally, excellent for apps that need geographic distribution
- **AWS** — Amplify for managed Next.js, or EC2/ECS/Lambda for full control

## Monitoring production

Once deployed, you need visibility into what's happening:

**Vercel Analytics** shows real user Core Web Vitals — Largest Contentful Paint, Cumulative Layout Shift, First Input Delay. These are the numbers Google uses for search ranking and the numbers that correlate with user bounce rates.

**Structured logging** from your Route Handlers and Server Actions goes to Vercel's function logs, accessible in the dashboard. Use `console.error` for errors, `console.log` for diagnostic info.

**Error tracking** — services like Sentry integrate with Next.js in a few lines. They capture stack traces, affected users, and reproduction steps automatically.

## Try it

<SandboxExercise language="javascript">
{`// Write parseEnvVars(envFile, context) function.
// envFile: a multi-line string of KEY=VALUE pairs (like a .env file).
// context: "client" | "server"
//
// Rules:
//   - "server" context gets ALL variables
//   - "client" context gets ONLY variables prefixed with NEXT_PUBLIC_
//   - Skip blank lines and lines starting with #
//   - Trim whitespace around keys and values
//
// Returns an object of the accessible key/value pairs.

function parseEnvVars(envFile, context) {
const result = {};

const lines = envFile.split("\\n");

for (const rawLine of lines) {
const line = rawLine.trim();

    // Skip blank lines and comments
    if (!line || line.startsWith("#")) continue;

    // Parse KEY=VALUE
    const eqIndex = line.indexOf("=");
    if (eqIndex === -1) continue;

    const key = line.slice(0, eqIndex).trim();
    const value = line.slice(eqIndex + 1).trim();

    // Client context only gets NEXT_PUBLIC_ vars
    if (context === "client" && !key.startsWith("NEXT_PUBLIC_")) {
      continue;
    }

    result[key] = value;

}

return result;
}

// Test data
const envFile = \`

# Database

DATABASE_URL=postgres://user:pass@host:5432/db

# Supabase (public — safe for client)

NEXT_PUBLIC_SUPABASE_URL=https://xyz.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIs

# Stripe

STRIPE_SECRET_KEY=sk_live_supersecret
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_live_publickey

# Analytics

NEXT_PUBLIC_GA_ID=G-XXXXXXXXXX
\`;

const serverVars = parseEnvVars(envFile, "server");
console.log("Server vars:", Object.keys(serverVars));
// ["DATABASE_URL", "NEXT_PUBLIC_SUPABASE_URL", "NEXT_PUBLIC_SUPABASE_ANON_KEY",
// "STRIPE_SECRET_KEY", "NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY", "NEXT_PUBLIC_GA_ID"]

const clientVars = parseEnvVars(envFile, "client");
console.log("Client vars:", Object.keys(clientVars));
// ["NEXT_PUBLIC_SUPABASE_URL", "NEXT_PUBLIC_SUPABASE_ANON_KEY",
// "NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY", "NEXT_PUBLIC_GA_ID"]

console.log("Secret hidden from client:", !("STRIPE_SECRET_KEY" in clientVars)); // true
console.log("DB hidden from client:", !("DATABASE_URL" in clientVars)); // true`}

## What you learned

- The `npm run build` output shows each route's rendering strategy (static, SSG, dynamic) and JavaScript bundle size — large first-load JS is a warning sign
- Variables without a prefix are server-only; variables prefixed with `NEXT_PUBLIC_` are inlined into the bundle and accessible on both server and client
- Vercel provides zero-config deployment: connect a GitHub repo, add env vars in the dashboard, and every push deploys automatically with preview URLs per PR
- `output: 'standalone'` in `next.config.ts` enables lean Docker deployments; Next.js also runs on Railway, Render, Fly.io, and self-hosted Node.js servers
