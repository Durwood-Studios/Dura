## Why this matters

React Server Components are the biggest change to React in years, and the App Router is built entirely around them. Understanding the server/client split — what runs where, what can import what, and why — is the foundation for building fast, secure Next.js applications. Get this mental model right and everything else clicks into place.

## What server components are

In the App Router, every component is a **Server Component by default**. That means:

1. The component runs on the server when a request comes in.
2. It renders to HTML (or a serialized React tree).
3. That output is sent to the browser.
4. **No component code reaches the browser.** Zero JavaScript is shipped to the client for this component.

This is fundamentally different from everything you learned about React before. In traditional React, all your component code ships to the browser and runs there. With Server Components, the server does the work and sends the result — like a template engine, but with the full power of React's composition model.

```typescript
// app/blog/[slug]/page.tsx
// This entire file runs ONLY on the server. None of this code
// is ever sent to the browser as JavaScript.

import { readFile } from "fs/promises"; // Node.js API — fine here!
import { db } from "@/lib/database";    // Direct DB access — fine here!

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  // This runs on the server. The browser sees the result, not this code.
  const post = await db.posts.findOne({ slug });

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </article>
  );
}
```

## What server components CAN do

Server Components have capabilities that no browser-side component can match:

- **`async`/`await` at the top level** — no `useEffect`, no loading state boilerplate. Just `await` your data directly in the component body.
- **Direct database access** — import your database client, query it, use the result. No API layer needed between your component and your data.
- **Read environment variables** — server-only secrets (database passwords, API keys) are safe because this code never reaches the browser.
- **Access the filesystem** — `fs.readFile`, `path.join`, reading local MDX files. All of Node.js is available.
- **Import large server-only libraries** — heavy parsers, database drivers, cryptography libraries — none of their code ships to the client bundle.

## What server components CANNOT do

The trade-off is real. Server Components have no access to the browser or to React's interactive features:

- No `useState` — state lives in the browser; the server doesn't maintain state per-component between renders
- No `useEffect` — effects run after mount in the browser; the server renders once and is done
- No event handlers — `onClick`, `onChange`, `onSubmit` all require browser event listeners
- No browser APIs — no `window`, `document`, `localStorage`, `navigator`
- No hooks that require browser context — `useRouter`, `usePathname`, `useSearchParams`

If you try to use any of these in a Server Component, Next.js will throw an error at build time or runtime. This is intentional — it prevents you from accidentally shipping secrets or server code to the client.

## The `"use client"` directive

When a component needs interactivity — a state variable, a click handler, a hook — you mark it as a **Client Component** by adding `"use client"` at the top of the file:

```typescript
"use client";

import { useState } from "react";

export function LikeButton({ postId }: { postId: string }) {
  const [liked, setLiked] = useState(false);

  return (
    <button onClick={() => setLiked(!liked)}>
      {liked ? "Liked" : "Like"}
    </button>
  );
}
```

`"use client"` is a boundary declaration, not a per-component setting. It means: "this file and everything it imports is part of the client bundle." Every component you import from a `"use client"` file is also a Client Component — the directive propagates down the import tree.

## The boundary rule

Here is the rule that trips up most developers: **Client Components cannot import Server Components.**

Once you cross into client territory, you're in the browser. There's no way to run a Server Component inside a component that lives in the browser — those two environments don't coexist at render time.

```typescript
// ❌ This breaks — LikeButton is "use client"
// It cannot import ServerOnlyWidget which is a Server Component
import { ServerOnlyWidget } from "./ServerOnlyWidget"; // Error!
```

However, **Server Components CAN pass Server Components to Client Components as props** — specifically as `children`. This is the pattern that unlocks composition:

```typescript
// app/blog/[slug]/page.tsx — Server Component
import { LikeButton } from "@/components/LikeButton"; // Client Component
import { PostContent } from "@/components/PostContent"; // Server Component

export default async function BlogPost({ params }) {
  const { slug } = await params;
  const post = await db.posts.findOne({ slug });

  return (
    // LikeButton is client, PostContent is server —
    // pass server output as children, not as an import inside LikeButton
    <LikeButton postId={post.id}>
      <PostContent content={post.content} />
    </LikeButton>
  );
}
```

The Server Component renders its output (HTML) before the Client Component runs. The client receives that pre-rendered output as `children` — it never needs to execute `PostContent` itself.

## Hydration

When the browser receives the HTML from a Server Component render, it displays it immediately — no JavaScript needed. Then React downloads the JavaScript for your Client Components, "connects" it to the existing HTML, and makes those parts interactive. This process is called **hydration**.

Hydration is why interactive elements like buttons and forms appear fast: the browser shows the HTML instantly, then quietly attaches event listeners in the background. Users can read content before hydration completes; they just can't interact with buttons until it does.

## The practical split

A good mental model: make everything a Server Component unless you specifically need the browser. In a typical Next.js app:

- **Server Components:** page shells, layouts, data fetching, static content display
- **Client Components:** buttons, forms, dropdowns, modals, anything with state or event handlers

Push `"use client"` as far down the component tree as possible. A page that fetches data on the server and renders a list is a Server Component. The "add to cart" button on each list item is a Client Component. The page itself never needs `"use client"`.

<SandboxExercise language="javascript">
{`// Simulate the server/client component validation rules.
// validateComponentTree(nodes) takes an array of component nodes where
// each node is: { name: string, isClient: boolean, imports: string[] }
//
// Return an array of violation strings for any case where a
// client component is imported (not passed as children) by another
// client component that also imports a server component directly.
//
// Simplified rule: a server component (isClient: false) may NOT appear
// in the imports[] array of a client component (isClient: true).
// That would be a direct server-in-client import violation.

function validateComponentTree(nodes) {
const nodeMap = {};
nodes.forEach((node) => {
nodeMap[node.name] = node;
});

const violations = [];

nodes.forEach((node) => {
if (!node.isClient) return; // Only check client components

    node.imports.forEach((importedName) => {
      const imported = nodeMap[importedName];
      if (imported && !imported.isClient) {
        violations.push(
          \`Violation: Client component "\${node.name}" directly imports Server Component "\${importedName}". Pass it as children instead.\`
        );
      }
    });

});

return violations;
}

// Test cases
const components = [
{ name: "Page", isClient: false, imports: ["UserCard", "LikeButton"] },
{ name: "UserCard", isClient: false, imports: ["Avatar"] },
{ name: "Avatar", isClient: false, imports: [] },
{ name: "LikeButton", isClient: true, imports: ["UserCard"] }, // ❌ violation
{ name: "Modal", isClient: true, imports: ["LikeButton"] }, // ✅ client imports client
];

const violations = validateComponentTree(components);
console.log("Violations found:", violations.length); // 1
console.log(violations[0]);
// "Violation: Client component "LikeButton" directly imports Server Component "UserCard". Pass it as children instead."

// Clean tree — no violations
const cleanComponents = [
{ name: "Page", isClient: false, imports: ["LikeButton"] },
{ name: "LikeButton", isClient: true, imports: ["Icon"] },
{ name: "Icon", isClient: true, imports: [] },
];
console.log("Clean violations:", validateComponentTree(cleanComponents).length); // 0`}

## What you learned

- Server Components run only on the server — they can use async/await, Node.js APIs, database clients, and secrets; none of their code ships to the browser
- Client Components require `"use client"` at the top of the file; they gain state, effects, and event handlers but lose server-only capabilities
- Client Components cannot directly import Server Components — but Server Components can pass rendered Server Component output as `children` to Client Components
- Hydration is the process of attaching interactivity to server-rendered HTML; users see content immediately, then interact after hydration completes
