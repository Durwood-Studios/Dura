## Why this matters

The App Router is the brain of every Next.js application. Once you understand what each special filename does, you can build any page structure — shared navigation, per-section layouts, graceful loading states, and crash-proof error boundaries — without writing any routing logic. The filesystem is the router.

## The `app/` directory structure

The App Router lives inside the `app/` directory at the root of your project. Every folder inside `app/` that contains a `page.tsx` file becomes a URL route. Folders without a `page.tsx` are just organizational — they affect layout nesting but don't create routes on their own.

A typical project looks like this:

```
app/
├── layout.tsx          ← root layout (wraps every page)
├── page.tsx            ← the / route
├── about/
│   └── page.tsx        ← the /about route
├── blog/
│   ├── layout.tsx      ← layout for /blog and all nested blog routes
│   ├── page.tsx        ← the /blog route
│   └── [slug]/
│       ├── page.tsx    ← the /blog/hello-world route
│       └── loading.tsx ← shown while this page's data loads
└── dashboard/
    ├── error.tsx       ← catches errors in this route subtree
    └── page.tsx        ← the /dashboard route
```

## `page.tsx` — what renders at a route

A `page.tsx` file is the public face of a route. It exports a default React component that Next.js renders when someone navigates to that URL.

```typescript
// app/about/page.tsx
export default function AboutPage() {
  return (
    <main>
      <h1>About Us</h1>
      <p>We build things.</p>
    </main>
  );
}
```

Pages are Server Components by default, which means they can be `async` and fetch data directly:

```typescript
// app/blog/[slug]/page.tsx
interface Props {
  params: Promise<{ slug: string }>;
}

export default async function BlogPost({ params }: Props) {
  const { slug } = await params;
  const post = await fetchPostBySlug(slug); // runs on the server
  return <article>{post.content}</article>;
}
```

## `layout.tsx` — the persistent shell

A `layout.tsx` wraps its sibling `page.tsx` and all descendant routes. Unlike a page, **a layout persists across navigations** — it mounts once and stays mounted, even as the page content beneath it changes. This is what makes navigation feel instant: the shell (navbar, sidebar) never unmounts.

The root layout at `app/layout.tsx` is required. It wraps your entire app and must include the `<html>` and `<body>` tags:

```typescript
// app/layout.tsx
export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en">
      <body>
        <nav>My Site Nav</nav>
        {children}
        <footer>My Site Footer</footer>
      </body>
    </html>
  );
}
```

A nested layout at `app/blog/layout.tsx` wraps only the blog section. When a user navigates from `/blog` to `/blog/hello-world`, the blog layout stays mounted and only the inner page swaps:

```typescript
// app/blog/layout.tsx
export default function BlogLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="blog-container">
      <aside>Blog Sidebar</aside>
      <main>{children}</main>
    </div>
  );
}
```

## `loading.tsx` — automatic Suspense

A `loading.tsx` file is an automatic Suspense boundary. When the `page.tsx` in the same directory is an async Server Component fetching data, Next.js shows the `loading.tsx` content immediately while the data loads.

```typescript
// app/blog/[slug]/loading.tsx
export default function Loading() {
  return <div className="skeleton">Loading post...</div>;
}
```

You don't need to write any Suspense code yourself. Next.js wraps the page in `<Suspense fallback={<Loading />}>` automatically. The user sees your loading UI in milliseconds while the real content is being fetched on the server.

## `error.tsx` — route-level error boundaries

A `error.tsx` file catches JavaScript errors that occur anywhere in its route subtree, preventing a single crash from taking down your entire application. It must be a Client Component:

```typescript
// app/dashboard/error.tsx
"use client";

interface Props {
  error: Error;
  reset: () => void;
}

export default function Error({ error, reset }: Props) {
  return (
    <div>
      <h2>Something went wrong.</h2>
      <p>{error.message}</p>
      <button onClick={reset}>Try again</button>
    </div>
  );
}
```

The `reset` function re-renders the route segment, giving the user a recovery path without a full page reload.

## Other special files

**`not-found.tsx`** renders when `notFound()` is called in a page or when a route simply doesn't exist. Use it to build branded 404 pages instead of the default Next.js error screen.

**`template.tsx`** works like `layout.tsx` but re-mounts on every navigation. Use it when you need animations or effects that must reset each time a user enters a section — layouts are better for most cases because they're more efficient.

## Route groups — organize without affecting URLs

Wrapping a folder name in parentheses creates a **route group**. The folder name is ignored for URL purposes but lets you share layouts among a subset of routes.

```
app/
├── (marketing)/
│   ├── layout.tsx    ← shared layout for marketing pages only
│   ├── page.tsx      ← still maps to /
│   └── about/
│       └── page.tsx  ← still maps to /about
└── (app)/
    ├── layout.tsx    ← shared layout for logged-in pages only
    └── dashboard/
        └── page.tsx  ← maps to /dashboard
```

This pattern is common for separating marketing pages (no sidebar) from application pages (with sidebar) without the folder names bleeding into your URLs.

## Try it

<SandboxExercise language="javascript">
{`// Simulate the layout nesting system.
// resolveLayouts(layoutPaths, requestPath) takes:
//   - layoutPaths: array of layout file paths like
//     ["app/layout.tsx", "app/blog/layout.tsx", "app/blog/[slug]/layout.tsx"]
//   - requestPath: the requested URL like "/blog/hello"
//
// Returns the ordered array of layouts that would wrap the page,
// from outermost (root) to innermost (closest ancestor).

function resolveLayouts(layoutPaths, requestPath) {
// Convert a layout file path to the URL prefix it covers.
// "app/layout.tsx" -> "/"
// "app/blog/layout.tsx" -> "/blog"
// "app/blog/[slug]/layout.tsx" -> "/blog/[slug]"
function layoutToPrefix(path) {
return path
.replace(/^app/, "")
.replace(/\/layout\\.tsx$/, "")
|| "/";
}

// Check if a layout prefix matches the request path.
// Dynamic segments ([slug]) match any value.
function prefixMatches(prefix, url) {
if (prefix === "/") return true;
const prefixParts = prefix.split("/").filter(Boolean);
const urlParts = url.split("/").filter(Boolean);
if (prefixParts.length > urlParts.length) return false;
return prefixParts.every((part, i) =>
part.startsWith("[") ? true : part === urlParts[i]
);
}

// Collect matching layouts and sort root-first.
const matching = layoutPaths
.map((path) => ({ path, prefix: layoutToPrefix(path) }))
.filter(({ prefix }) => prefixMatches(prefix, requestPath))
.sort((a, b) => {
// Root layout always first, then by depth
if (a.prefix === "/") return -1;
if (b.prefix === "/") return 1;
return a.prefix.split("/").length - b.prefix.split("/").length;
});

return matching.map((m) => m.path);
}

// Tests
const layouts = [
"app/layout.tsx",
"app/blog/layout.tsx",
"app/blog/[slug]/layout.tsx",
"app/dashboard/layout.tsx",
];

console.log(resolveLayouts(layouts, "/"));
// ["app/layout.tsx"]

console.log(resolveLayouts(layouts, "/blog"));
// ["app/layout.tsx", "app/blog/layout.tsx"]

console.log(resolveLayouts(layouts, "/blog/hello-world"));
// ["app/layout.tsx", "app/blog/layout.tsx", "app/blog/[slug]/layout.tsx"]

console.log(resolveLayouts(layouts, "/dashboard"));
// ["app/layout.tsx", "app/dashboard/layout.tsx"]`}

</SandboxExercise>

## What you learned

- `page.tsx` defines what renders at a URL; `layout.tsx` defines a persistent shell that wraps a route and all its children
- `loading.tsx` is an automatic Suspense boundary shown while the page fetches data; `error.tsx` catches crashes in its route subtree
- Layouts persist across navigations — they mount once, making navigation feel instant
- Route groups (folder names in parentheses) organize routes and share layouts without affecting URLs
