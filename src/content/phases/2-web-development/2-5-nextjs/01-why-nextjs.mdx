## Why this matters

React is a UI library — it answers one question: how do I build component trees? Everything else — routing between pages, loading data from a database, optimizing images, running server-side code — React leaves to you. In production, that gap becomes real work. Next.js closes it, giving you a complete framework so you can build a real application instead of assembling tools from scratch.

## What React alone lacks

A bare React app, created with `create-react-app` or a hand-rolled Vite setup, gives you components and a build pipeline. That's it. The moment you need more, you're on your own:

- **Routing** — React has no router. You install React Router or TanStack Router separately, wire it up, and manage route definitions by hand.
- **Server-side rendering** — React's default is client-side rendering (CSR): the browser downloads a mostly-empty HTML file, downloads your JavaScript bundle, runs it, and then renders your UI. For users on slow connections or slow devices, they see nothing until all of that completes.
- **Data fetching** — `useEffect` + `fetch` works, but it fires after the component mounts on the client, creating a loading flash and exposing your API calls.
- **Build config** — Code splitting, image optimization, font loading, environment variables: you configure each one yourself.

Next.js solves all of this, and it does so with conventions instead of configuration. You don't configure a router — you create files in the right folders. You don't set up an image optimizer — you import from `next/image`.

## What Next.js adds

**File-based routing** is Next.js's most visible feature. Create a file at `app/about/page.tsx` and the URL `/about` automatically works. Nested folders create nested routes. No router config file exists because the filesystem is the config file.

**Server Components** render your UI on the server and send finished HTML to the browser. The browser receives a complete, readable page immediately — no JavaScript required to see content. Search engines can index it without running JavaScript. Users on slow connections see something right away.

**API Routes** (called Route Handlers in the App Router) let you write server endpoints in the same project as your frontend. A file at `app/api/users/route.ts` becomes a real HTTP endpoint. One repo, one deploy — no separate backend server.

**Image optimization** happens automatically when you use the `<Image>` component from `next/image`. Next.js resizes, converts to WebP, lazy-loads, and serves images from a CDN — none of which you configure.

**Font optimization** works similarly: `next/font/google` downloads and self-hosts Google Fonts at build time, eliminating the layout shift caused by fonts loading after the page.

## CSR, SSR, and SSG

These three acronyms describe _when_ and _where_ your page HTML gets built.

**Client-Side Rendering (CSR):** The server sends a bare HTML shell with a `<div id="root">` and a JavaScript bundle. The browser downloads and runs the bundle, which renders your components into that div. Fast server, slow initial paint on the client.

**Server-Side Rendering (SSR):** For each request, the server renders your components to HTML and sends the full page. The browser displays content immediately. Then JavaScript "hydrates" — attaches event listeners to the existing HTML so interactivity works. Good for pages that need fresh, per-request data.

**Static Site Generation (SSG):** Pages are rendered to HTML at _build time_, not request time. The server just serves a pre-built file — as fast as serving a `.txt` file. Perfect for content that doesn't change often: blog posts, docs, marketing pages.

Next.js lets you use all three in the same project. A blog post can be SSG; a dashboard can be SSR; a live comment feed can be CSR. You choose per-page, not per-project.

## App Router vs Pages Router

Next.js has two routing systems. The **Pages Router** is the original (`pages/` directory), present in every Next.js project before version 13. It still works and millions of sites use it.

The **App Router** is the current system (`app/` directory), introduced in Next.js 13 and the default since Next.js 14. It is built around React Server Components — the new model where components render on the server by default. It unlocks features the Pages Router cannot support, and it's what the Next.js team recommends for new projects.

**This course uses the App Router.** You will see the `app/` directory. If you encounter older Next.js tutorials using `pages/`, recognize the difference but don't be alarmed — the core concepts (routing, data fetching, deployment) are the same, the file conventions differ.

## Where Next.js runs

A Next.js app runs on a **Node.js server** — either one you manage yourself or one Vercel manages for you. The server handles incoming requests, renders Server Components, executes Route Handlers, and serves static assets.

**Vercel** is the company that created Next.js, and their platform is purpose-built for it. You connect your GitHub repo, push a commit, and Vercel builds and deploys your app automatically. Every pull request gets a preview URL. It's the fastest path from code to production.

You are not locked into Vercel. Next.js can be self-hosted on any Linux server using Docker and `next start`. It deploys to Railway, Render, Fly.io, and AWS. The `output: 'standalone'` config option in `next.config.ts` bundles everything you need into a self-contained directory.

## Try it

<SandboxExercise language="javascript">
{`// Simulate file-based routing.
// createRouter(fileTree) takes an object where keys are file paths
// like "app/page.tsx", "app/about/page.tsx", "app/blog/[slug]/page.tsx"
// and returns a match(url) function.
//
// match(url) should return an object with the matched route and
// any extracted params, or null if no route matches.

function createRouter(fileTree) {
// Convert file paths to route patterns.
// "app/page.tsx" -> "/"
// "app/about/page.tsx" -> "/about"
// "app/blog/[slug]/page.tsx" -> "/blog/[slug]"
const routes = Object.keys(fileTree).map((filePath) => {
let pattern = filePath
.replace(/^app/, "")
.replace(/\/page\\.tsx$/, "")
      .replace(/\/page\\.ts$/, "") || "/";
return { pattern, filePath };
});

return function match(url) {
for (const route of routes) {
const params = matchRoute(route.pattern, url);
if (params !== null) {
return { route: route.pattern, params, filePath: route.filePath };
}
}
return null;
};
}

function matchRoute(pattern, url) {
const patternParts = pattern === "/" ? [""] : pattern.split("/");
const urlParts = url === "/" ? [""] : url.split("/");

if (patternParts.length !== urlParts.length) return null;

const params = {};
for (let i = 0; i < patternParts.length; i++) {
const p = patternParts[i];
const u = urlParts[i];
if (p.startsWith("[") && p.endsWith("]")) {
const key = p.slice(1, -1);
params[key] = u;
} else if (p !== u) {
return null;
}
}
return params;
}

// Test it
const router = createRouter({
"app/page.tsx": true,
"app/about/page.tsx": true,
"app/blog/[slug]/page.tsx": true,
"app/docs/[...path]/page.tsx": true,
});

console.log(router.match("/")); // { route: "/", params: {}, ... }
console.log(router.match("/about")); // { route: "/about", params: {}, ... }
console.log(router.match("/blog/hello")); // { route: "/blog/[slug]", params: { slug: "hello" }, ... }
console.log(router.match("/contact")); // null`}

</SandboxExercise>

## What you learned

- React handles UI components; Next.js adds routing, server rendering, API routes, image/font optimization, and a build pipeline
- CSR renders in the browser, SSR renders per-request on the server, SSG renders once at build time — Next.js supports all three in one project
- The App Router (`app/` directory) is the current system and uses React Server Components; this course uses App Router exclusively
- Next.js runs on Node.js and deploys to Vercel or any server via Docker
