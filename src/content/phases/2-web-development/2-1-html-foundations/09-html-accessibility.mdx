## Why this matters

Over one billion people worldwide have some form of disability. Many of them use the web daily through screen readers, keyboard navigation, voice control, or switch devices. Accessibility is not a feature you add later — it is a quality standard baked into every element you write. If your HTML is not accessible, it is broken for a significant portion of your users.

## How screen readers work

A screen reader converts the page into spoken text. It reads content linearly and relies entirely on HTML structure to navigate. Here is what a screen reader announces for different elements:

- `<h2>` — "Heading level 2: Learning HTML"
- `<nav>` — "Navigation"
- `<button>` — "Sign up, button"
- `<img alt="Chart showing growth">` — "Image: Chart showing growth"
- `<div class="button">` — nothing useful. It is just a `<div>`.

This is why semantic HTML matters so much. Semantic elements give screen readers the vocabulary to describe your page.

## ARIA roles and attributes

ARIA (Accessible Rich Internet Applications) fills gaps when HTML semantics are not enough. The first rule of ARIA is: **do not use ARIA if a native HTML element works**.

```html
<!-- Bad: ARIA on a div -->
<div role="button" tabindex="0" onclick="submit()">Submit</div>

<!-- Good: native button -->
<button onclick="submit()">Submit</button>
```

The native `<button>` gives you focus management, keyboard activation (Enter and Space), and screen reader announcement for free. The ARIA version requires you to manually recreate all of that.

When ARIA is necessary:

```html
<!-- Live region: screen reader announces updates -->
<div aria-live="polite">3 items in your cart</div>

<!-- Labeling a region -->
<nav aria-label="Primary navigation">
  <a href="/">Home</a>
  <a href="/lessons">Lessons</a>
</nav>

<!-- Describing the state of a toggle -->
<button aria-expanded="false" aria-controls="menu">Menu</button>
<ul id="menu" hidden>
  <li><a href="/about">About</a></li>
</ul>
```

Key ARIA attributes:

- **`aria-label`** — provides a text label when visible text is not sufficient.
- **`aria-live`** — announces dynamic content changes. `"polite"` waits for a pause; `"assertive"` interrupts.
- **`aria-expanded`** — tells screen readers whether a collapsible section is open or closed.
- **`aria-hidden="true"`** — hides an element from screen readers (use for decorative icons).

## Focus management

Keyboard users navigate with Tab (forward) and Shift+Tab (backward). Only interactive elements receive focus by default: links, buttons, inputs, and textareas.

```html
<!-- These are focusable by default -->
<a href="/about">About</a>
<button>Click me</button>
<input type="text" />

<!-- This is NOT focusable — keyboard users cannot reach it -->
<div onclick="doSomething()">Click me</div>
```

If you must make a non-interactive element focusable, add `tabindex="0"`. But ask yourself first: should this just be a `<button>`?

**Never use positive tabindex values** (`tabindex="1"`, `tabindex="5"`). They override the natural document order and create an unpredictable experience.

## Skip links

A skip link lets keyboard users jump past the navigation directly to the main content. It is usually the first focusable element on the page, visually hidden until focused.

```html
<body>
  <a href="#main-content" class="skip-link">Skip to main content</a>
  <header>
    <nav><!-- 20 navigation links --></nav>
  </header>
  <main id="main-content">
    <h1>Lesson Content</h1>
  </main>
</body>
```

```css
.skip-link {
  position: absolute;
  left: -9999px;
}
.skip-link:focus {
  position: static;
  left: auto;
}
```

Without a skip link, a keyboard user presses Tab through every navigation link before reaching the content. On a page with 30 nav links, that is 30 key presses just to start reading.

## Focus indicators

Never remove focus outlines without providing an alternative. The default browser outline tells keyboard users where they are.

```css
/* Bad: removes the indicator entirely */
*:focus {
  outline: none;
}

/* Good: custom focus indicator */
*:focus-visible {
  outline: 2px solid #10b981;
  outline-offset: 2px;
}
```

`:focus-visible` only shows the outline for keyboard users, not mouse clicks — the best of both worlds.

## What you learned

- Screen readers rely on **semantic HTML** to describe page structure — ARIA is a supplement, not a replacement.
- **Skip links** let keyboard users bypass navigation, and **focus indicators** show where they are.
- The first rule of ARIA: **do not use it if a native HTML element already provides the behavior**.
