## Why this matters

Knowing individual HTML elements is like knowing individual words — useful, but you only communicate when you combine them into sentences. This lesson puts everything together. You will build a complete, semantic, accessible HTML page and understand how each piece connects to the whole.

## The full page

Here is a complete HTML page for a developer portfolio. Read through it and notice how every concept from this module appears.

```html
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Alex Chen - Web Developer</title>
  </head>
  <body>
    <a href="#main-content" class="skip-link">Skip to main content</a>

    <header>
      <h1>Alex Chen</h1>
      <nav aria-label="Primary navigation">
        <a href="#about">About</a>
        <a href="#projects">Projects</a>
        <a href="#contact">Contact</a>
      </nav>
    </header>

    <main id="main-content">
      <section id="about">
        <h2>About Me</h2>
        <img
          src="portrait.jpg"
          alt="Alex Chen smiling at a desk with two monitors"
          width="400"
          height="400"
          loading="lazy"
        />
        <p>
          I am a web developer focused on building
          <strong>accessible</strong> and <em>performant</em>
          applications. I believe the web should work for everyone.
        </p>
      </section>

      <section id="projects">
        <h2>Projects</h2>
        <article>
          <h3>Task Tracker</h3>
          <p>A keyboard-accessible to-do app built with vanilla HTML, CSS, and JavaScript.</p>
          <ul>
            <li>Semantic HTML structure</li>
            <li>Full keyboard navigation</li>
            <li>Works offline</li>
          </ul>
          <a href="https://github.com/alexchen/tasks" target="_blank" rel="noopener noreferrer">
            View on GitHub
          </a>
        </article>
        <article>
          <h3>Weather Dashboard</h3>
          <p>Real-time weather data displayed in an accessible table format.</p>
          <table>
            <thead>
              <tr>
                <th>City</th>
                <th>Temperature</th>
                <th>Conditions</th>
              </tr>
            </thead>
            <tbody>
              <tr>
                <td>Portland</td>
                <td>62F</td>
                <td>Cloudy</td>
              </tr>
              <tr>
                <td>Austin</td>
                <td>85F</td>
                <td>Sunny</td>
              </tr>
            </tbody>
          </table>
        </article>
      </section>

      <section id="contact">
        <h2>Contact</h2>
        <form action="/contact" method="POST">
          <label for="name">Name</label>
          <input type="text" id="name" name="name" required minlength="2" />

          <label for="email">Email</label>
          <input type="email" id="email" name="email" required />

          <label for="message">Message</label>
          <textarea id="message" name="message" required minlength="10"></textarea>

          <button type="submit">Send Message</button>
        </form>
      </section>
    </main>

    <footer>
      <p>2026 Alex Chen. Built with semantic HTML.</p>
      <nav aria-label="Social links">
        <a href="https://github.com/alexchen" target="_blank" rel="noopener noreferrer">GitHub</a>
        <a href="mailto:alex@example.com">Email</a>
      </nav>
    </footer>
  </body>
</html>
```

## Breaking it down

**Document setup**: `<!DOCTYPE html>`, `<html lang="en">`, charset, viewport meta, and a descriptive title. Every page starts here.

**Skip link**: The first focusable element. Keyboard users hit Tab and see "Skip to main content" — one press and they are past the nav.

**Header with nav**: The `<header>` contains the site name and an `<nav>` with `aria-label`. Fragment links (`#about`, `#projects`) point to section IDs on the same page.

**Main with sections**: `<main>` wraps all primary content. Each `<section>` has a heading. Projects use `<article>` because each project is self-contained.

**Mixed content**: The page uses images with `alt` text, an unordered list for features, a table for data, and inline elements like `<strong>` and `<em>` — all in the right context.

**Accessible form**: Every input has a `<label>` with matching `for`/`id`. Inputs use appropriate types (`email`, `text`) and validation attributes (`required`, `minlength`).

**Footer with secondary nav**: The footer has its own `<nav>` with a distinct `aria-label` so screen readers can differentiate it from the primary navigation.

## Common mistakes to avoid

- **Missing `lang` attribute on `<html>`** — screen readers need this to select the correct pronunciation engine.
- **Missing viewport meta** — without it, mobile browsers render at desktop width and zoom out.
- **Heading levels that skip** — going from `<h2>` to `<h4>` breaks the document outline.
- **Forms without labels** — the single most common accessibility failure on the web.
- **Links that say "click here"** — link text must describe the destination.

## What you learned

- A complete HTML page combines **document structure, semantic elements, accessible forms, and proper media handling** into a cohesive whole.
- Every page needs a `lang` attribute, viewport meta, skip link, and **one `<main>` element**.
- Building with semantic HTML from the start is easier than retrofitting accessibility later.
