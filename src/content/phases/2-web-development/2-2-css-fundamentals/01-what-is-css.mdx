## Why this matters

Every website you visit uses CSS to turn raw HTML into something people actually want to look at. Without CSS, the web is a wall of unstyled text and images stacked vertically. Understanding selectors and specificity prevents the frustrating "why won't this style apply?" debugging sessions that burn hours in real projects.

## CSS connects to HTML

CSS (Cascading Style Sheets) is a language that describes how HTML elements should look. There are three ways to attach CSS to an HTML document.

**External stylesheet** — the standard approach for production code:

```html
<link rel="stylesheet" href="styles.css" />
```

**Style tag** — useful for prototyping, but keeps styles coupled to a single page:

```html
<style>
  h1 {
    color: navy;
  }
</style>
```

**Inline styles** — highest specificity, hardest to maintain. Avoid in production:

```html
<h1 style="color: navy;">Hello</h1>
```

External stylesheets win because they separate concerns, cache across pages, and keep your HTML clean.

## Selectors target elements

A CSS rule has two parts: a **selector** (what to style) and a **declaration block** (how to style it).

```css
/* Element selector — targets every <p> */
p {
  color: #333;
  line-height: 1.6;
}

/* Class selector — targets elements with class="card" */
.card {
  background: white;
  border-radius: 8px;
}

/* ID selector — targets the single element with id="header" */
#header {
  position: sticky;
  top: 0;
}

/* Attribute selector — targets inputs with type="email" */
input[type="email"] {
  border: 2px solid #ccc;
}
```

You can combine selectors for precision:

```css
/* Descendant: any <a> inside a <nav> */
nav a {
  color: white;
}

/* Direct child: only <li> that are immediate children of .menu */
.menu > li {
  list-style: none;
}

/* Multiple selectors share the same rules */
h1,
h2,
h3 {
  font-family: "DM Sans", sans-serif;
}
```

## Specificity determines which rule wins

When two rules target the same element, the browser uses **specificity** to decide which one applies. Think of specificity as a scoring system with three slots: `(ID, Class, Element)`.

| Selector         | Specificity | Score |
| ---------------- | ----------- | ----- |
| `p`              | (0, 0, 1)   | 1     |
| `.card`          | (0, 1, 0)   | 10    |
| `#header`        | (1, 0, 0)   | 100   |
| `p.card`         | (0, 1, 1)   | 11    |
| `#header .nav a` | (1, 1, 1)   | 111   |

Higher specificity wins. If specificity is equal, the rule that appears **last** in the stylesheet wins. This is the "cascade" in Cascading Style Sheets.

```css
/* Specificity: (0, 1, 0) — this applies */
.title {
  color: blue;
}

/* Specificity: (0, 0, 1) — lower, loses */
h1 {
  color: red;
}
```

Inline styles beat everything except `!important`. Using `!important` is almost always a sign that your selector strategy needs rethinking.

## Properties and values

CSS has hundreds of properties. Every property accepts specific value types:

```css
.example {
  /* Color values */
  color: #1a1a2e;
  background-color: rgb(240, 240, 240);

  /* Length values */
  font-size: 16px;
  margin: 1.5rem;
  width: 50%;

  /* Keyword values */
  display: flex;
  font-weight: bold;
  text-align: center;
}
```

Properties that don't apply to an element are silently ignored. A `color` property on an element with no text content does nothing visible. The browser never throws errors for invalid CSS — it just skips the rule. This is by design for forward compatibility, but it means typos fail silently.

## What you learned

- CSS selectors (element, class, id, attribute) target HTML elements for styling, and can be combined for precision
- Specificity is a scoring system that determines which CSS rule wins when multiple rules target the same element
- The cascade resolves ties by applying the last rule in source order, making stylesheet organization matter
