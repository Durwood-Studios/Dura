## Why this matters

Over half of web traffic comes from mobile devices. A layout that only works on a desktop monitor fails the majority of your users. Responsive design isn't an add-on feature — it's the baseline expectation. Every layout you build must adapt from a 320px phone screen to a 2560px ultrawide.

## Media queries

Media queries let you apply CSS rules only when certain conditions are true — usually screen width:

```css
/* Base styles (mobile) */
.grid {
  display: grid;
  grid-template-columns: 1fr;
  gap: 16px;
}

/* When the viewport is at least 768px wide */
@media (min-width: 768px) {
  .grid {
    grid-template-columns: 1fr 1fr;
  }
}

/* When the viewport is at least 1024px wide */
@media (min-width: 1024px) {
  .grid {
    grid-template-columns: 1fr 1fr 1fr;
  }
}
```

### Mobile-first approach

Write your base CSS for the smallest screen, then add complexity at wider breakpoints with `min-width`. This is **mobile-first**. The alternative — writing desktop CSS first and overriding with `max-width` — leads to more CSS and more overrides.

Mobile-first works because small screens need simpler layouts. You start simple and add complexity. Desktop-first starts complex and tries to subtract — that's harder.

### Common breakpoints

These breakpoints align with Tailwind CSS and cover the majority of real devices:

```css
/* sm: Small tablets, large phones in landscape */
@media (min-width: 640px) {
}

/* md: Tablets in portrait */
@media (min-width: 768px) {
}

/* lg: Tablets in landscape, small laptops */
@media (min-width: 1024px) {
}

/* xl: Desktops */
@media (min-width: 1280px) {
}
```

Don't add breakpoints preemptively. Start with mobile, then add a breakpoint only when the layout breaks at a wider size. Let the content determine the breakpoints, not device categories.

### Beyond width

Media queries can test more than width:

```css
/* User prefers dark mode at the OS level */
@media (prefers-color-scheme: dark) {
  body {
    background: #08080d;
    color: #f0f0f0;
  }
}

/* User has requested reduced motion */
@media (prefers-reduced-motion: reduce) {
  * {
    animation: none !important;
    transition-duration: 0.01ms !important;
  }
}

/* Device supports hover (not a touchscreen) */
@media (hover: hover) {
  .card:hover {
    transform: translateY(-4px);
  }
}
```

The `prefers-reduced-motion` query is an accessibility requirement. Users who set this have vestibular disorders or motion sensitivity. Respect it.

## Viewport units

Viewport units size elements relative to the browser window:

```css
.hero {
  height: 100vh; /* 100% of viewport height */
  width: 100vw; /* 100% of viewport width */
}

.heading {
  font-size: 5vw; /* Scales with viewport width */
}
```

**The mobile address bar problem**: On mobile browsers, the address bar appears and disappears as you scroll, changing the viewport height. `100vh` doesn't account for this, causing content to be hidden behind the address bar.

The fix: use **dynamic viewport units**:

```css
.hero {
  height: 100dvh; /* Dynamic viewport height — adjusts with address bar */
}
```

| Unit          | Meaning                                                |
| ------------- | ------------------------------------------------------ |
| `vh` / `vw`   | Static viewport height / width                         |
| `dvh` / `dvw` | Dynamic — changes as browser chrome appears/disappears |
| `svh` / `svw` | Small — viewport with all browser chrome visible       |
| `lvh` / `lvw` | Large — viewport with no browser chrome                |

Use `dvh` for full-screen sections. Use `svh` when you need to guarantee content is always visible.

## Container queries

Media queries respond to the **viewport** width. Container queries respond to a **parent element's** width. This makes components truly self-contained — they adapt based on the space they're given, not the screen size.

```css
/* Define a container */
.card-wrapper {
  container-type: inline-size;
  container-name: card;
}

/* Style based on the container's width */
@container card (min-width: 400px) {
  .card {
    display: flex;
    flex-direction: row;
  }
}

@container card (max-width: 399px) {
  .card {
    display: flex;
    flex-direction: column;
  }
}
```

The same card component displays horizontally when it has 400px+ of space, and vertically when it doesn't — regardless of screen size. This is powerful for reusable components placed in sidebars, modals, or full-width layouts.

Container queries are supported in all modern browsers as of 2023. They're the future of responsive component design.

<SandboxExercise language="javascript">
{`// Simulate a responsive breakpoint system.
// Given a viewport width and a set of breakpoint rules,
// determine which styles are active.

function activeStyles(viewportWidth, breakpoints) {
// breakpoints is an array of { minWidth, styles } objects,
// sorted from smallest to largest minWidth.
// All matching breakpoints apply (later ones override earlier ones).
// Return the merged styles object.

let result = {};
for (const bp of breakpoints) {
if (viewportWidth >= bp.minWidth) {
result = { ...result, ...bp.styles };
}
}
return result;
}

// Mobile-first breakpoints
const breakpoints = [
{ minWidth: 0, styles: { columns: 1, fontSize: 14, padding: 16 } },
{ minWidth: 640, styles: { columns: 2, fontSize: 16 } },
{ minWidth: 1024, styles: { columns: 3, padding: 24 } },
];

// Mobile (375px)
const mobile = activeStyles(375, breakpoints);
console.log(mobile.columns === 1 && mobile.fontSize === 14 && mobile.padding === 16);

// Tablet (768px)
const tablet = activeStyles(768, breakpoints);
console.log(tablet.columns === 2 && tablet.fontSize === 16 && tablet.padding === 16);

// Desktop (1280px)
const desktop = activeStyles(1280, breakpoints);
console.log(desktop.columns === 3 && desktop.fontSize === 16 && desktop.padding === 24);`}

## What you learned

- Mobile-first responsive design writes base styles for small screens and uses `min-width` media queries to add complexity at standard breakpoints (640/768/1024/1280px)
- Dynamic viewport units (`dvh`, `dvw`) solve the mobile address bar problem where `100vh` doesn't reflect the actual visible area
- Container queries make components responsive to their parent's size rather than the screen size, enabling truly reusable responsive components
