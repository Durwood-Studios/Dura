## Why this matters

Normal document flow handles most layouts. But sticky headers, dropdown menus, modal overlays, and tooltips all require pulling elements out of normal flow and placing them precisely. Understanding positioning and stacking context prevents the "why is my modal behind this random div?" bugs that haunt every CSS developer.

## position: static

The default. Elements follow normal document flow. The `top`, `right`, `bottom`, `left`, and `z-index` properties have **no effect** on static elements.

```css
.default {
  position: static; /* This is what every element starts with */
}
```

## position: relative

The element stays in normal flow (occupying its original space), but you can **offset** it from that position:

```css
.nudged {
  position: relative;
  top: 10px; /* Moves down 10px from its natural position */
  left: 20px; /* Moves right 20px from its natural position */
}
```

The surrounding elements don't move — they still see this element in its original position. Relative positioning is most commonly used as a **containing block** for absolutely positioned children, not for visual nudging.

## position: absolute

The element is **removed from normal flow** — surrounding elements act as if it doesn't exist. It positions itself relative to its nearest **positioned ancestor** (any ancestor with `position` set to something other than `static`).

```css
.parent {
  position: relative; /* Creates the positioning context */
}

.tooltip {
  position: absolute;
  top: 100%; /* Just below the parent */
  left: 0;
  background: #1a1a2e;
  padding: 8px 12px;
  border-radius: 4px;
}
```

If no ancestor is positioned, the element positions relative to the initial containing block (the viewport). This is a common bug — always ensure the parent has `position: relative`.

```html
<div class="parent">
  Hover me
  <div class="tooltip">Extra info</div>
</div>
```

## position: fixed

Like `absolute`, but positions relative to the **viewport**. The element stays in place when the user scrolls. Removed from document flow.

```css
.fixed-header {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  height: 64px;
  background: #08080d;
  z-index: 100;
}

/* Don't forget to offset the content below */
main {
  padding-top: 64px;
}
```

Fixed elements are useful for headers, floating action buttons, and cookie banners. Be cautious with fixed positioning on mobile — the virtual keyboard can push fixed elements into unexpected positions.

## position: sticky

A hybrid. The element behaves like `relative` until the user scrolls past a threshold, then it behaves like `fixed` — but only within its parent container.

```css
.sticky-header {
  position: sticky;
  top: 0; /* Sticks when it reaches the top of the viewport */
  background: #08080d;
  z-index: 50;
}
```

Sticky elements stop sticking when they reach the bottom of their parent. This makes them perfect for section headers in long lists.

**Sticky gotcha**: `overflow: hidden` or `overflow: auto` on any ancestor breaks sticky positioning. If sticky isn't working, check every ancestor for overflow properties.

```css
/* Table of contents that sticks while scrolling the article */
.article-layout {
  display: grid;
  grid-template-columns: 250px 1fr;
  gap: 32px;
}

.toc {
  position: sticky;
  top: 80px; /* Below the fixed header */
  align-self: start; /* Critical: prevents grid from stretching it */
  max-height: calc(100vh - 96px);
  overflow-y: auto;
}
```

## Stacking context and z-index

When positioned elements overlap, `z-index` controls which one appears on top. Higher values sit above lower values.

```css
.dropdown {
  z-index: 10;
}
.modal {
  z-index: 50;
}
.toast {
  z-index: 100;
}
```

But `z-index` only works within a **stacking context**. A new stacking context is created by:

- `position: relative/absolute/fixed/sticky` with `z-index` set
- `opacity` less than 1
- `transform`, `filter`, or `will-change` properties

Elements in different stacking contexts can't interleave. A `z-index: 9999` inside a stacking context with `z-index: 1` will still appear behind a sibling stacking context with `z-index: 2`.

```css
/* This modal will appear BEHIND .sidebar despite z-index: 9999 */
.card {
  position: relative;
  z-index: 1; /* Creates a stacking context */
}
.card .modal {
  position: absolute;
  z-index: 9999; /* Trapped inside .card's stacking context */
}

.sidebar {
  position: relative;
  z-index: 2; /* Higher than .card's context — wins */
}
```

The fix: move the modal to a higher level in the DOM, or manage stacking contexts carefully. This is why many frameworks render modals at the document root using a portal.

{`// Simulate z-index stacking within nested stacking contexts.
// Each element has a parentContext (z-index of its stacking context)
// and its own zIndex. Elements are compared first by parentContext,
// then by their own zIndex if contexts are equal.

function topElement(elements) {
// elements: array of { name, parentContext, zIndex }
// Return the name of the element that appears on top.
// Sort by parentContext (desc), then zIndex (desc).

const sorted = [...elements].sort((a, b) => {
if (a.parentContext !== b.parentContext) {
return b.parentContext - a.parentContext;
}
return b.zIndex - a.zIndex;
});
return sorted[0].name;
}

// The modal has z-index 9999 but its parent context is only 1
console.log(topElement([
{ name: "modal", parentContext: 1, zIndex: 9999 },
{ name: "sidebar", parentContext: 2, zIndex: 1 },
]) === "sidebar");

console.log(topElement([
{ name: "dropdown", parentContext: 10, zIndex: 5 },
{ name: "tooltip", parentContext: 10, zIndex: 20 },
{ name: "header", parentContext: 5, zIndex: 100 },

## What you learned

- The five position values (static, relative, absolute, fixed, sticky) each have distinct behaviors for how elements relate to document flow and their positioning reference point
- Stacking contexts created by positioned elements, opacity, and transforms determine the scope of `z-index` — a high z-index trapped in a low-context parent still loses
- Sticky positioning requires no `overflow: hidden` or `overflow: auto` on any ancestor, and `align-self: start` in grid layouts to prevent stretching
