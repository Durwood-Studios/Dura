## Why this matters

Motion communicates cause and effect. A button that smoothly changes color on hover tells the user "you're interacting with me." A card that slides in from below draws attention to new content. But poor animation causes jank, motion sickness, and wasted battery. The difference between good and bad motion is understanding what the browser can animate cheaply.

## CSS transitions

A transition smoothly animates a property change over time. You define which property to animate, how long it takes, and the acceleration curve.

```css
.button {
  background: #10b981;
  color: white;
  transition: background 200ms ease;
}

.button:hover {
  background: #059669;
}
```

The transition only fires when the property changes — on hover in, and again on hover out (back to the original value).

### Transition properties

```css
.card {
  transition-property: transform, box-shadow;
  transition-duration: 200ms;
  transition-timing-function: ease;
  transition-delay: 0ms;

  /* Shorthand: property duration timing-function delay */
  transition:
    transform 200ms ease,
    box-shadow 200ms ease;
}

.card:hover {
  transform: translateY(-4px);
  box-shadow: 0 8px 24px rgba(0, 0, 0, 0.3);
}
```

### Timing functions

The timing function controls acceleration:

```css
.element {
  transition: transform 300ms ease; /* Slow start and end (default) */
  transition: transform 300ms ease-in; /* Slow start, fast end */
  transition: transform 300ms ease-out; /* Fast start, slow end — feels natural */
  transition: transform 300ms ease-in-out; /* Slow start and end */
  transition: transform 300ms linear; /* Constant speed — robotic */
  transition: transform 300ms cubic-bezier(0.34, 1.56, 0.64, 1); /* Custom bounce */
}
```

For UI interactions, `ease-out` is the best default. Elements that enter the screen should decelerate (ease-out). Elements exiting should accelerate (ease-in).

## CSS transforms

Transforms change an element's visual position, size, or rotation without affecting document flow. They're the most performant way to move things on screen.

```css
.element {
  /* Move */
  transform: translateX(100px);
  transform: translateY(-50%);
  transform: translate(10px, 20px);

  /* Scale */
  transform: scale(1.1);
  transform: scale(0.95);

  /* Rotate */
  transform: rotate(45deg);
  transform: rotate(-90deg);

  /* Combine multiple transforms */
  transform: translateY(-4px) scale(1.02);
}
```

Transforms don't trigger layout recalculations. The browser composites them on the GPU, making them smooth at 60fps. This is why you should always use `transform: translateY()` instead of changing `top` or `margin-top` for animations.

## @keyframes animations

For animations that run automatically or have multiple steps, use `@keyframes`:

```css
@keyframes fadeIn {
  from {
    opacity: 0;
    transform: translateY(20px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

.card {
  animation: fadeIn 400ms ease-out forwards;
}
```

### Multi-step animations

```css
@keyframes pulse {
  0% {
    transform: scale(1);
  }
  50% {
    transform: scale(1.05);
  }
  100% {
    transform: scale(1);
  }
}

.notification-dot {
  animation: pulse 2s ease-in-out infinite;
}
```

### Animation properties

```css
.element {
  animation-name: fadeIn;
  animation-duration: 400ms;
  animation-timing-function: ease-out;
  animation-delay: 100ms;
  animation-iteration-count: 1; /* Or 'infinite' */
  animation-fill-mode: forwards; /* Keeps final state after animation */
  animation-direction: normal; /* Or 'reverse', 'alternate' */

  /* Shorthand */
  animation: fadeIn 400ms ease-out 100ms 1 forwards;
}
```

`animation-fill-mode: forwards` is important — without it, the element snaps back to its pre-animation state when the animation completes.

## Performance: what to animate

The browser handles different properties at different costs:

**Cheap (compositor-only, GPU-accelerated):**

- `transform` (translate, scale, rotate)
- `opacity`

**Expensive (triggers layout or paint):**

- `width`, `height`, `top`, `left`, `right`, `bottom`
- `margin`, `padding`
- `border-width`, `font-size`

Stick to `transform` and `opacity` for animations. If you need to "move" an element, use `transform: translateX/Y()` instead of changing `left` or `margin-left`. The visual result is the same; the performance is dramatically better.

## Accessibility: prefers-reduced-motion

Some users experience motion sickness, vertigo, or seizures from screen animation. Every project must respect the `prefers-reduced-motion` media query:

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

This doesn't disable animations entirely — it makes them instant, preserving the state change without the motion. Place this in your global CSS.

// Linear interpolation: start + (end - start) \* progress

// Simulate an ease-out curve using a simple quadratic function.
// easeOut(t) = t _ (2 - t) where t is 0-1 progress.
function easeOut(t) {
return t _ (2 - t);
}

// Simulate a CSS transition from opacity 0 to 1 over 5 frames
// using ease-out timing.
const frames = 5;
const results = [];
for (let i = 0; i <= frames; i++) {
const linearProgress = i / frames;
const easedProgress = easeOut(linearProgress);
const opacity = lerp(0, 1, easedProgress);
results.push(Math.round(opacity \* 100) / 100);
}

console.log(results[0] === 0); // Frame 0: start
console.log(results[frames] === 1); // Frame 5: end

## What you learned

- CSS transitions animate property changes smoothly with control over duration, timing function, and delay — use `ease-out` for UI interactions
- Transforms (translate, scale, rotate) and opacity are GPU-accelerated and should be preferred over animating layout properties like width, height, or top
- The `prefers-reduced-motion` media query is an accessibility requirement that disables or minimizes animation for users with motion sensitivity
