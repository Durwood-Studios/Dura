## Why this matters

A React component without state is just a template — it renders the same output every time for the same props. State is what makes components dynamic. A counter that counts, a form that collects input, a toggle that switches — all of that is state. Understanding how React's state model works saves you from the most common React bugs: stale values, missing re-renders, and accidental mutations.

## What state is

**State** is data that belongs to a component and can change over time. When state changes, React re-renders the component — calls the function again with the new data — and updates the DOM to reflect the new UI.

Two key properties distinguish state from regular variables:

1. **It persists across renders.** A regular `let count = 0` inside a component resets to `0` every time the component renders. State survives re-renders.
2. **Changing it triggers a re-render.** React watches state values. When you call the setter, React schedules a re-render.

## useState syntax

`useState` is a React hook — a special function that connects your component to React's internal machinery.

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);
  //     ^^^^^  ^^^^^^^^   ^^^^^^^^^
  //   current  setter fn  initial value

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}
```

`useState(0)` returns an array of exactly two items. Array destructuring gives them names. `count` holds the current value. `setCount` is the function you call to update it.

## Why you must use the setter

This is the most common React mistake:

```jsx
// WRONG — React doesn't know count changed, no re-render
function Counter() {
  let count = 0;
  return (
    <button
      onClick={() => {
        count++;
      }}
    >
      Clicks: {count}
    </button>
  );
}

// ALSO WRONG — mutating state directly, same problem
const [count, setCount] = useState(0);
count = count + 1; // React won't re-render
```

React only knows to re-render when you call the setter function. Direct mutation bypasses React's tracking. The DOM stays stale.

## The re-render cycle

When you call `setCount(5)`, React:

1. Schedules a re-render (does not happen immediately).
2. Calls your component function again.
3. `useState(0)` now returns `[5, setCount]` — the new value.
4. React diffs the new virtual DOM against the old one.
5. React applies minimal DOM updates.

Your component function runs again from top to bottom. This is why component functions must be pure — no side effects in the render body itself. Everything re-runs on every render.

## Immutable updates for objects and arrays

State values should be treated as **immutable**. When state is an object or array, do not mutate it — replace it:

```jsx
const [user, setUser] = useState({ name: "Dustin", role: "admin" });

// WRONG — mutating the state object directly
user.name = "Alex"; // React won't re-render

// CORRECT — spread into a new object
setUser({ ...user, name: "Alex" });
```

```jsx
const [items, setItems] = useState(["a", "b", "c"]);

// WRONG
items.push("d"); // mutates, no re-render

// CORRECT — spread into a new array
setItems([...items, "d"]);

// Remove an item
setItems(items.filter((item) => item !== "b"));

// Update an item
setItems(items.map((item) => (item === "a" ? "A" : item)));
```

The reason is referential equality. React checks whether the new state value is a different reference than the old one. If you mutate and pass the same object reference, React sees no change.

## Multiple state values

A component can call `useState` multiple times:

```jsx
function LoginForm() {
  const [email, setEmail] = useState("");
  const [password, setPassword] = useState("");
  const [isLoading, setIsLoading] = useState(false);

  async function handleSubmit(e) {
    e.preventDefault();
    setIsLoading(true);
    await login(email, password);
    setIsLoading(false);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input value={email} onChange={(e) => setEmail(e.target.value)} />
      <input value={password} onChange={(e) => setPassword(e.target.value)} />
      <button disabled={isLoading}>{isLoading ? "Logging in..." : "Log in"}</button>
    </form>
  );
}
```

## Lifting state up

When two sibling components need to share state, move that state to their **closest common parent** and pass it down as props. This is called lifting state up.

```jsx
function App() {
  const [count, setCount] = useState(0); // lifted — shared by both children

  return (
    <>
      <Display count={count} />
      <Controls onIncrement={() => setCount(count + 1)} />
    </>
  );
}
```

State lives in the lowest common ancestor. Data flows down via props. Events bubble up via callback props.

## What you learned

- **State** is data owned by a component that persists across renders and triggers re-renders when changed via the setter function.
- Call `useState(initialValue)` to get `[currentValue, setterFunction]` — always use the setter, never mutate directly.
- React re-renders a component by calling its function again; every call to `useState` returns the current value, not the initial one.
- Objects and arrays must be **replaced, not mutated** — spread into new references so React detects the change.
- **Lift state up** to the closest common ancestor when siblings need to share it.
