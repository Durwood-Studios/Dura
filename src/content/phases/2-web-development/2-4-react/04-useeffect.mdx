## Why this matters

Almost every real React component needs to do something beyond rendering HTML. Fetch data from an API. Set up a WebSocket. Start a timer. Sync to localStorage. These operations are called **side effects**, and `useEffect` is how React handles them safely. Misunderstanding the dependency array is the root cause of some of the most confusing React bugs — infinite loops, stale data, memory leaks.

## What a side effect is

A **side effect** is any operation that reaches outside the render itself. Pure rendering — taking props and state and returning JSX — has no side effects. Everything else does:

- Fetching data from an API
- Setting or reading from `localStorage`
- Setting up a timer (`setTimeout`, `setInterval`)
- Subscribing to a WebSocket or event listener
- Directly manipulating the DOM (rare in React, but sometimes necessary)

React requires these to happen _after_ rendering, not during. If you fetch data directly in your component body, it runs on every render — including the renders triggered by the data arriving. That is an infinite loop.

## useEffect syntax

```jsx
import { useEffect } from "react";

useEffect(
  () => {
    // Side effect code here — runs after render

    return () => {
      // Optional cleanup — runs before the next effect or on unmount
    };
  },
  [
    /* dependency array */
  ]
);
```

Three parts:

1. **Effect function** — the code to run after rendering.
2. **Cleanup function** — returned from the effect; cleans up subscriptions, timers, etc.
3. **Dependency array** — controls _when_ the effect re-runs.

## When it runs — the dependency array

The dependency array is the most important part to understand:

```jsx
// No array: runs after EVERY render (rarely what you want)
useEffect(() => {
  console.log("Rendered");
});

// Empty array: runs ONCE after the first render (like componentDidMount)
useEffect(() => {
  fetchUserData();
}, []);

// With dependencies: runs when those values change
useEffect(() => {
  fetchUserPosts(userId);
}, [userId]); // Re-runs whenever userId changes
```

The rule: **include every value from the component that the effect uses**. If your effect reads `userId`, `userId` goes in the array. Forgetting a dependency means the effect runs with a stale value — one of the most common React bugs.

## Fetching data

```jsx
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setIsLoading(true);
    setError(null);

    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then((data) => {
        setUser(data);
        setIsLoading(false);
      })
      .catch((err) => {
        setError(err.message);
        setIsLoading(false);
      });
  }, [userId]); // Re-fetch whenever userId changes

  if (isLoading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;
  return <h1>{user.name}</h1>;
}
```

## Cleanup — preventing memory leaks

When a component unmounts, its effect's cleanup function runs. This is critical for subscriptions and timers:

```jsx
useEffect(() => {
  const timer = setInterval(() => {
    setSeconds((s) => s + 1);
  }, 1000);

  // Cleanup: clear the interval when the component unmounts
  return () => clearInterval(timer);
}, []); // Empty array — timer starts once, clears on unmount
```

Without cleanup, the interval keeps firing even after the component is gone. It tries to call `setSeconds` on an unmounted component — a memory leak and a React warning.

For API requests, cleanup means cancelling in-flight requests if the component unmounts before they finish:

```jsx
useEffect(() => {
  const controller = new AbortController();

  fetch(`/api/posts/${id}`, { signal: controller.signal })
    .then((res) => res.json())
    .then(setPost)
    .catch((err) => {
      if (err.name !== "AbortError") setError(err.message);
    });

  return () => controller.abort(); // Cancel if component unmounts or id changes
}, [id]);
```

## When NOT to use useEffect

React's documentation has evolved on this. Many things developers reach for `useEffect` for can be done more simply:

**Deriving state** — if you can compute a value from existing state or props, just compute it during render:

```jsx
// Don't do this
const [fullName, setFullName] = useState("");
useEffect(() => {
  setFullName(`${firstName} ${lastName}`);
}, [firstName, lastName]);

// Do this instead — compute during render
const fullName = `${firstName} ${lastName}`;
```

**Handling events** — if something happens because of a user action, put the logic in the event handler, not an effect.

Use `useEffect` when you genuinely need to **synchronize with something external** — an API, a browser API, a third-party library.

{`// Simulate useEffect's dependency tracking.
// createEffect(effectFn, deps) runs effectFn immediately,
// then only re-runs it when deps change.

function createEffect(effectFn, deps) {
let prevDeps = null;
let cleanup = null;

function run(newDeps) {
// First run: always execute
// Subsequent runs: only execute if any dep changed
const shouldRun =
prevDeps === null ||
newDeps.some((dep, i) => dep !== prevDeps[i]);

    if (shouldRun) {
      // Run the cleanup from the previous effect, if any
      if (cleanup) {
        cleanup();
        cleanup = null;
      }

      // Run the effect and store any cleanup it returns
      cleanup = effectFn() || null;
      prevDeps = [...newDeps];
      return true; // effect ran
    }

    return false; // effect skipped

}

return { run };
}

// --- Simulate a component that fetches user data when userId changes ---

fetchCount++;
console.log(\`Fetching user \${currentUserId}... (fetch #\${fetchCount})\`);

// Simulate cleanup: cancel the "request"
return () => console.log(\`Cancelling request for user \${currentUserId}\`);
}, []);

let currentUserId = 1;

// Simulate renders
console.log("Render 1 — userId: 1");
userEffect.run([currentUserId]); // should run

console.log("\\nRender 3 — userId: 2 (changed)");
userEffect.run([currentUserId]); // should run

currentUserId = 2;
console.log("\\nRender 4 — userId: 2 (same)");
userEffect.run([currentUserId]); // should NOT run

## What you learned

- A **side effect** is any operation that reaches outside rendering: fetching data, timers, subscriptions, DOM manipulation.
- `useEffect(fn, deps)` runs `fn` after rendering; React re-runs it only when a value in `deps` changes.
- **Cleanup** — the function returned from an effect — runs before the next effect and when the component unmounts, preventing memory leaks.
- The **dependency array** must include every value from the component the effect uses; forgetting one causes stale-value bugs.
- Many things developers reach for `useEffect` for (derived state, event logic) are better handled in render or event handlers.
