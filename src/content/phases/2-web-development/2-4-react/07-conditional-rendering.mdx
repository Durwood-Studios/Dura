## Why this matters

Real UIs are almost never static. A loading spinner shows while data fetches. An error message appears when something goes wrong. A button changes label when clicked. A panel hides when collapsed. Conditional rendering is the mechanism behind all of it. Getting the patterns right — and knowing the one gotcha that burns every React developer — makes your components clean and bug-free.

## Ternary operator

The **ternary operator** is the most versatile tool for conditional rendering. It works inside JSX because it is an expression (not a statement):

```jsx
function UserGreeting({ isLoggedIn }) {
  return <header>{isLoggedIn ? <p>Welcome back!</p> : <p>Please sign in.</p>}</header>;
}
```

Use ternaries when you have two branches — something to show when true, something different when false. If the false case should render nothing, use `null`:

```jsx
{
  isAdmin ? <AdminPanel /> : null;
}
```

Returning `null` from a ternary (or from a component) renders nothing and adds no DOM node. It is the standard way to conditionally include a component.

## Short-circuit with &&

When you only want to render something if a condition is true — and nothing otherwise — the `&&` short-circuit operator is cleaner than a ternary:

```jsx
function Notifications({ count }) {
  return (
    <div>
      <h1>Inbox</h1>
      {count > 0 && <span className="badge">{count}</span>}
    </div>
  );
}
```

`condition && <Component />` works because:

- If `condition` is truthy, JavaScript evaluates the right side and returns it — React renders the component.
- If `condition` is falsy, JavaScript short-circuits and returns the falsy value — React renders nothing.

## The falsy zero gotcha

Here is the bug that trips up every React developer at least once:

```jsx
// WRONG — renders "0" in the DOM when items.length is 0
{
  items.length && <ItemList items={items} />;
}
```

`0` is falsy, so `&&` short-circuits — but it returns `0`, not `false` or `null`. And React renders `0` as a text node. You see a lone "0" on screen.

Fix it by converting to a boolean:

```jsx
// Correct — renders nothing when items.length is 0
{
  items.length > 0 && <ItemList items={items} />;
}
{
  !!items.length && <ItemList items={items} />;
}
```

The rule: always use a boolean expression on the left side of `&&` in JSX, never a number.

## Early returns

For complex conditions — a loading state, an error state, and a success state — early returns keep the main render body clean:

```jsx
function UserProfile({ userId }) {
  const { user, isLoading, error } = useUser(userId);

  if (isLoading) {
    return <LoadingSpinner />;
  }

  if (error) {
    return <ErrorMessage message={error} />;
  }

  if (!user) {
    return <p>No user found.</p>;
  }

  // Happy path — rendered only when data is ready
  return (
    <div className="profile">
      <h1>{user.name}</h1>
      <p>{user.bio}</p>
    </div>
  );
}
```

Early returns eliminate deeply nested ternaries. Each condition exits the function before reaching the main content. This is sometimes called the "guard clause" pattern, and it makes the happy path — the most important case — visually prominent at the bottom.

## Extracting conditionals into variables

When a conditional expression gets complex, extract it into a variable before the return:

```jsx
function TaskItem({ task }) {
  // Build the status badge outside JSX for readability
  const statusBadge = task.isComplete ? (
    <span className="badge badge-green">Done</span>
  ) : task.isOverdue ? (
    <span className="badge badge-red">Overdue</span>
  ) : (
    <span className="badge badge-gray">Pending</span>
  );

  return (
    <div className="task">
      <p>{task.title}</p>
      {statusBadge}
    </div>
  );
}
```

## The switch pattern for multiple states

When a component can be in one of several distinct states (loading, error, empty, with data), a switch statement reads more clearly than a chain of ternaries:

```jsx
function DataView({ status, data, error }) {
  switch (status) {
    case "loading":
      return <LoadingSpinner />;
    case "error":
      return <ErrorMessage message={error} />;
    case "empty":
      return <EmptyState />;
    case "success":
      return <DataTable rows={data} />;
    default:
      return null;
  }
}
```

This pattern pairs well with a `status` field in your state object — a single string that describes what the component should show.

## What you learned

- **Ternaries** (`condition ? <A /> : <B />`) handle two-branch conditions — use `null` for the empty branch.
- **Short-circuit** (`condition && <Component />`) is cleaner when there is only one branch, but always use a boolean on the left side to avoid the falsy-zero bug.
- **Early returns** eliminate nested ternaries — check each guard condition and return early, leaving the happy path at the bottom.
- Extracting complex conditionals into **variables** before the return statement keeps JSX readable.
- A **switch** on a status string cleanly handles components with many distinct states (loading, error, empty, success).
