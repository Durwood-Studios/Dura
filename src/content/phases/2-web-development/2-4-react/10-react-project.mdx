## Why this matters

Reading about React is not the same as building with React. This lesson ties together everything from the module — state, effects, lists, forms, context, and custom hooks — into a single cohesive project. The flashcard app is deliberately simple enough to finish in one session and complex enough to expose every concept you need for real work.

## App architecture

Before writing a line of code, design the component tree and state shape. This is the most important step in any React project — the decisions you make here determine how painful the next hour will be.

```
FlashcardApp
├── DeckList          (shows all decks, lets you select one)
├── StudyMode         (shows cards one at a time, tracks score)
│   └── FlashCard     (shows front/back, animates flip)
└── AddCard           (form to add a new card to a deck)
```

**State design** — what data does the app need, and where does it live?

```js
// Global state (lives in App, passed via Context)
{
  decks: [
    {
      id: "deck-1",
      title: "JavaScript Basics",
      cards: [
        { id: "card-1", front: "What is a closure?", back: "A function that captures its surrounding scope." }
      ]
    }
  ],
  activeDeckId: null  // which deck is being studied
}

// Local to StudyMode
{
  currentIndex: 0,    // which card we're on
  isFlipped: false,   // showing front or back
  score: { correct: 0, incorrect: 0 }
}
```

The rule: state that only one component needs stays local. State shared across siblings or distant components goes in Context.

## Implementing card flip with state

The flip animation is driven entirely by a boolean in state. CSS does the visual work:

```jsx
function FlashCard({ front, back }) {
  const [isFlipped, setIsFlipped] = useState(false);

  return (
    <div
      className={`card ${isFlipped ? "card-flipped" : ""}`}
      onClick={() => setIsFlipped((prev) => !prev)}
    >
      <div className="card-inner">
        <div className="card-front">{front}</div>
        <div className="card-back">{back}</div>
      </div>
    </div>
  );
}
```

Note `setIsFlipped((prev) => !prev)` — the **functional update form**. When new state depends on the previous value, use this form. It is safe in concurrent mode and prevents stale-value bugs in rapid-click scenarios.

## StudyMode component

```jsx
function StudyMode({ deck, onExit }) {
  const [currentIndex, setCurrentIndex] = useState(0);
  const [isFlipped, setIsFlipped] = useState(false);
  const [score, setScore] = useState({ correct: 0, incorrect: 0 });

  const card = deck.cards[currentIndex];
  const isLastCard = currentIndex === deck.cards.length - 1;

  function advance() {
    setIsFlipped(false); // reset flip for next card
    setCurrentIndex((i) => i + 1);
  }

  function markCorrect() {
    setScore((s) => ({ ...s, correct: s.correct + 1 }));
    if (!isLastCard) advance();
  }

  function markIncorrect() {
    setScore((s) => ({ ...s, incorrect: s.incorrect + 1 }));
    if (!isLastCard) advance();
  }

  if (!card) {
    return (
      <div className="results">
        <h2>Session complete!</h2>
        <p>
          Correct: {score.correct} / {deck.cards.length}
        </p>
        <button onClick={onExit}>Back to decks</button>
      </div>
    );
  }

  return (
    <div className="study-mode">
      <p>
        {currentIndex + 1} / {deck.cards.length}
      </p>
      <FlashCard front={card.front} back={card.back} />
      {isFlipped && (
        <>
          <button onClick={markCorrect}>Got it</button>
          <button onClick={markIncorrect}>Missed it</button>
        </>
      )}
    </div>
  );
}
```

Notice how `isFlipped` is reset to `false` on `advance` — this is state coordination. Changing cards always shows the front of the next card.

## AddCard form with controlled inputs

```jsx
function AddCard({ deckId, onAdd }) {
  const [form, setForm] = useState({ front: "", back: "" });
  const [error, setError] = useState("");

  function handleChange(e) {
    setForm((prev) => ({ ...prev, [e.target.name]: e.target.value }));
  }

  function handleSubmit(e) {
    e.preventDefault();
    if (!form.front.trim() || !form.back.trim()) {
      setError("Both sides of the card are required.");
      return;
    }
    onAdd({ id: `card-${Date.now()}`, ...form });
    setForm({ front: "", back: "" }); // reset
    setError("");
  }

  return (
    <form onSubmit={handleSubmit}>
      <textarea
        name="front"
        value={form.front}
        onChange={handleChange}
        placeholder="Front of card"
      />
      <textarea name="back" value={form.back} onChange={handleChange} placeholder="Back of card" />
      {error && <p className="error">{error}</p>}
      <button type="submit">Add Card</button>
    </form>
  );
}
```

## Persisting decks with useEffect and localStorage

Without persistence, all decks vanish on refresh. A `useEffect` syncs state to `localStorage` on every change:

```jsx
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    // Read from localStorage on first render only
    try {
      const stored = localStorage.getItem(key);
      return stored ? JSON.parse(stored) : initialValue;
    } catch {
      return initialValue;
    }
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];
}

// Usage in the root component
function FlashcardApp() {
  const [decks, setDecks] = useLocalStorage("flashcard-decks", []);
  // ...rest of the app
}
```

The `useState` initializer (the function form) runs once on mount, reading from localStorage. The `useEffect` runs whenever `decks` changes, writing back. This is the complete read/write cycle.

## What you learned

- **State design** comes before code — decide what state exists, who owns it, and where it lives before writing components.
- **Functional updates** (`setState(prev => newValue)`) are safer than closures over state values when the new state depends on the previous one.
- **`useEffect` + localStorage** is the minimal pattern for client-side persistence — read in the initializer, write in the effect.
- Every concept in this module — state, effects, lists, forms, conditional rendering, custom hooks, composition — appears together in real applications.
- A well-architected React app is a tree of small, focused components, each responsible for one thing, composed into something larger.
