## Why this matters

Almost every real UI involves lists — comments, products, notifications, users, messages. React renders arrays of elements elegantly, but there is one non-obvious requirement: every element in a list needs a unique `key` prop. Without it, React cannot track which items changed when the list updates, leading to bugs that are genuinely hard to debug — animations that play on the wrong item, form inputs that show stale values, or DOM updates that are far slower than they should be.

## Rendering arrays with .map()

In JSX, a JavaScript array of React elements renders as a list of nodes. The idiomatic way to produce that array is `Array.prototype.map()`:

```jsx
const fruits = ["Apple", "Banana", "Cherry"];

function FruitList() {
  return (
    <ul>
      {fruits.map((fruit) => (
        <li key={fruit}>{fruit}</li>
      ))}
    </ul>
  );
}
```

The expression inside `{}` is evaluated as JavaScript. `fruits.map(...)` returns an array of `<li>` elements, and React renders them all. This works for any array — strings, numbers, or objects fetched from an API.

## Why keys are required

When React re-renders a list, it must figure out which DOM nodes to update, which to add, and which to remove. Without keys, React uses position — item at index 0 maps to the first DOM node, index 1 to the second, and so on.

This breaks when items move, are inserted in the middle, or are removed from anywhere but the end:

```jsx
// Before: [Apple, Banana, Cherry]
// After inserting Avocado at the start: [Avocado, Apple, Banana, Cherry]

// Without keys, React thinks:
// - position 0: was "Apple", now "Avocado" → update text
// - position 1: was "Banana", now "Apple" → update text
// - position 2: was "Cherry", now "Banana" → update text
// - position 3: new → add "Cherry"
// Four DOM operations instead of one insert.
```

With a stable key, React knows "this is the Apple node — just move it":

```jsx
// With keys, React correctly identifies:
// - "Avocado" is new → insert
// - "Apple", "Banana", "Cherry" are existing → just move them
// One DOM insert instead of four updates.
```

The performance difference is real. More importantly, component state attached to a list item (like a focused input or an animation in progress) will move with the correct item instead of staying at the wrong position.

## What makes a good key

**Good:** A stable, unique identifier from your data — typically a database ID.

```jsx
users.map((user) => <UserCard key={user.id} user={user} />);
```

**Bad:** The array index. Using the index as a key gives every item a "stable" key that is actually positional — you get all the same bugs as having no key at all whenever items are reordered or inserted.

```jsx
// Avoid this when items can move
users.map((user, index) => <UserCard key={index} user={user} />);
```

The index is acceptable only when the list is truly static and will never be reordered, filtered, or have items added or removed.

**Bad:** Math.random() or any generated value. Keys must be stable across renders:

```jsx
// Never do this — generates a new key every render
items.map((item) => <li key={Math.random()}>{item}</li>);
```

## Filtering and mapping together

A common pattern: filter the data, then map it to elements.

```jsx
function CompletedTasks({ tasks }) {
  return (
    <ul>
      {tasks
        .filter((task) => task.isComplete)
        .map((task) => (
          <li key={task.id}>{task.title}</li>
        ))}
    </ul>
  );
}
```

You can also sort before mapping — just remember that `sort()` mutates the array, so spread first:

```jsx
{
  [...tasks]
    .sort((a, b) => a.title.localeCompare(b.title))
    .map((task) => <li key={task.id}>{task.title}</li>);
}
```

## Handling empty state

Always consider what to render when the array is empty. An empty list that renders nothing is a confusing UX:

```jsx
function TaskList({ tasks }) {
  if (tasks.length === 0) {
    return <p className="empty-state">No tasks yet. Add one above!</p>;
  }

  return (
    <ul>
      {tasks.map((task) => (
        <li key={task.id}>{task.title}</li>
      ))}
    </ul>
  );
}
```

## Keys with nested components

The `key` prop is special — it does not get passed to the component as a prop. If you need the ID inside the child component, pass it separately:

```jsx
<UserCard
  key={user.id}
  id={user.id} // passed as a prop — available inside UserCard
  user={user}
/>
```

Inside `UserCard`, you cannot access `props.key`. But `props.id` works fine.

## Try it

<SandboxExercise language="javascript">
{`// Implement diffLists — the function that powers React's key-based reconciliation.
// Given two arrays of items and a key extractor function,
// categorize every item as added, removed, updated, or unchanged.

function diffLists(oldItems, newItems, keyFn) {
const result = {
added: [],
removed: [],
updated: [],
unchanged: [],
};

// Build lookup maps by key for O(n) performance
const oldMap = new Map(oldItems.map((item) => [keyFn(item), item]));
const newMap = new Map(newItems.map((item) => [keyFn(item), item]));

// Find removed items: in old, not in new
for (const [key, item] of oldMap) {
if (!newMap.has(key)) {
result.removed.push(item);
}
}

// Find added and changed items
for (const [key, newItem] of newMap) {
if (!oldMap.has(key)) {
result.added.push(newItem);
} else {
const oldItem = oldMap.get(key);
// Detect updates: any property changed?
const hasChanged = JSON.stringify(oldItem) !== JSON.stringify(newItem);
if (hasChanged) {
result.updated.push({ old: oldItem, new: newItem });
} else {
result.unchanged.push(newItem);
}
}
}

return result;
}

// Test with a task list
const oldTasks = [
{ id: 1, title: "Write tests", done: false },
{ id: 2, title: "Fix bug #42", done: false },
{ id: 3, title: "Deploy to prod", done: false },
];

const newTasks = [
{ id: 1, title: "Write tests", done: true }, // updated
{ id: 3, title: "Deploy to prod", done: false }, // unchanged
{ id: 4, title: "Write docs", done: false }, // added
// id 2 is gone — removed
];

const diff = diffLists(oldTasks, newTasks, (t) => t.id);

console.log("Added:", diff.added.map((t) => t.title));
console.log("Removed:", diff.removed.map((t) => t.title));
console.log("Updated:", diff.updated.map((d) => \`\${d.old.title}: done=\${d.new.done}\`));
console.log("Unchanged:", diff.unchanged.map((t) => t.title));`}

</SandboxExercise>

## What you learned

- Render arrays using `.map()` inside JSX — each mapped element needs a `key` prop.
- **Keys** let React track which items changed, moved, added, or were removed during reconciliation.
- Good keys are **stable IDs from your data** (like a database ID) — not array indices or randomly generated values.
- The `key` prop is consumed by React internally; pass the ID again as a separate prop if the child component needs it.
- Always handle the **empty state** — what renders when the list has zero items is part of the feature.
