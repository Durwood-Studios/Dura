## Why this matters

Every major web application you use today — GitHub, Airbnb, Netflix, Discord — is built with React or a React-influenced framework. Understanding why React exists tells you what problems it solves, which tells you when it is the right tool and when a simpler approach will serve you better.

## The problem with vanilla JavaScript at scale

When you build a small webpage, vanilla JavaScript works fine. You query elements, update their text, toggle classes. Simple. But as an application grows — dozens of interactive components, data that changes over time, multiple places in the UI that reflect the same piece of information — vanilla JS becomes painful.

Imagine a shopping cart. The cart count appears in the header, in a sidebar, and on a checkout button. Every time the user adds an item, you must find all three DOM elements and update them manually. Miss one and the UI is out of sync. Now multiply that by a hundred features and a team of ten developers.

```js
// Vanilla JS approach — imperative, brittle at scale
function addToCart(item) {
  cart.push(item);
  // Must manually find and update every element that shows cart state
  document.querySelector("#header-count").textContent = cart.length;
  document.querySelector("#sidebar-count").textContent = cart.length;
  document.querySelector("#checkout-btn").textContent = `Checkout (${cart.length})`;
  // Forgot the mobile nav count? Bug. Nobody knows until a user reports it.
}
```

The core problem is that the UI and the data are separate. When data changes, you must manually keep the DOM in sync. This is called **imperative** programming — you tell the browser _how_ to update, step by step.

## What React's component model solves

React flips this model. Instead of telling the browser _how_ to update the DOM, you describe _what_ the UI should look like for any given state. This is called **declarative** programming.

```jsx
// React approach — declarative, composable
function CartButton({ count }) {
  return <button>Checkout ({count})</button>;
}
```

Whenever `count` changes, React figures out what the DOM should look like and updates it automatically. You never touch the DOM directly. The UI is a **function of state** — `UI = f(state)`. Change the state, and the UI follows.

Components are the building blocks. A React component is a function that takes data in (called **props**) and returns a description of UI to display. Components can contain other components, forming a tree — just like HTML elements nest inside each other.

## What the virtual DOM is

React does not update the real DOM on every state change directly — that would be slow. Instead, it maintains a **virtual DOM**: a lightweight JavaScript object tree that mirrors the real DOM.

When state changes:

1. React builds a new virtual DOM tree for the new state.
2. It **diffs** the new tree against the previous one, finding only what changed.
3. It applies only those minimal changes to the real DOM.

This process is called **reconciliation**. The diffing algorithm is why React can handle complex UIs efficiently. You describe the whole UI on every render; React surgically updates only what changed.

## React's position in the ecosystem

React is a **library**, not a framework. It provides exactly one thing: a way to build and render component trees. Everything else — routing, data fetching, state management, server rendering — comes from the ecosystem around it.

This is intentional. React can be added to a single page of an existing app or power the entire frontend. Frameworks like Next.js (which DURA uses) layer on top of React to add routing, server components, and full-stack capabilities.

## When NOT to use React

React is not always the right tool. Skip it when:

- You are building a **simple static page** — a blog post, a landing page, a form that posts to a server. Vanilla HTML + minimal JavaScript is faster to ship and easier to maintain.
- **Performance is critical and the UI is simple** — React's runtime adds kilobytes of JavaScript your users must download.
- **Your team does not know React** — the learning curve is real, and forcing it into a simple project slows delivery without adding value.

React shines when you have **dynamic, interconnected UI** that changes in response to user actions or server data. The more complex your UI state, the more React earns its keep.

// TODO: push an "update" change for this path (use path + ".tag") with value newTree.tag

## What you learned

- Vanilla JS at scale leads to **state drift** — the DOM falls out of sync with your data because you must update every UI element manually.
- React is **declarative**: you describe `UI = f(state)` and React keeps the DOM in sync automatically via reconciliation.
- The **virtual DOM** is a lightweight JS object tree React diffs before touching the real DOM, making updates efficient.
- React is a **library**, not a framework — it handles rendering only, and is the right tool when your UI state is complex and dynamic.
