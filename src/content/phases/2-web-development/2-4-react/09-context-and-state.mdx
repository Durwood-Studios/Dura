## Why this matters

As React apps grow, you encounter a problem: state that many parts of the UI need — the current user, the selected theme, the user's locale — has to travel through every component between where it lives and where it is used. This is **prop drilling**, and it couples unrelated components together. Context and `useReducer` are React's built-in tools for solving this. Understanding when to use them — and when not to — separates good React architecture from tangled state.

## The prop drilling problem

Prop drilling happens when props must pass through intermediate components that do not use them:

```jsx
// App knows the user. ProfilePage needs the user. Avatar needs the user.
// But Header is in the middle and has no use for it — it just passes it through.

function App() {
  const [user, setUser] = useState({ name: "Dustin", avatar: "/me.jpg" });
  return <Header user={user} />;
}

function Header({ user }) {
  // Header does not use user — it just forwards it
  return (
    <nav>
      <ProfilePage user={user} />
    </nav>
  );
}

function ProfilePage({ user }) {
  // ProfilePage does not use user either — just forwards it
  return (
    <div>
      <Avatar user={user} />
    </div>
  );
}

function Avatar({ user }) {
  // Avatar finally uses user
  return <img src={user.avatar} alt={user.name} />;
}
```

Two intermediate components are coupled to a prop they do not need. Rename the `user` prop and you must update four files.

## React Context

Context provides a way to share a value with any component in the tree without explicitly passing it through every level.

```jsx
import { createContext, useContext, useState } from "react";

// 1. Create the context
const UserContext = createContext(null);

// 2. Provide it near the top of the tree
function App() {
  const [user, setUser] = useState({ name: "Dustin", avatar: "/me.jpg" });

  return (
    <UserContext.Provider value={{ user, setUser }}>
      <Header />
    </UserContext.Provider>
  );
}

// 3. Consume it anywhere in the tree — no prop threading needed
function Avatar() {
  const { user } = useContext(UserContext);
  return <img src={user.avatar} alt={user.name} />;
}
```

`Header` and `ProfilePage` no longer need to know about `user`. They just render their children. `Avatar` reaches into the context directly.

## When to use Context

Context is the right tool for **slowly-changing, broadly-needed data**:

- Authenticated user (`currentUser`)
- Theme (`"dark"` | `"light"`)
- Locale (`"en-US"`, `"es-ES"`)
- Feature flags

## When NOT to use Context

Context has a performance cost: **every consumer re-renders when the context value changes**. This makes it a poor fit for frequently-updating state like:

- A counter that increments on every keystroke
- Mouse position (updates dozens of times per second)
- Form field values in a large form

For frequently-updating state, keep it local (`useState`) or use a dedicated state library like Zustand that handles subscriptions more granularly.

## useReducer for complex state

When state has multiple related values and the update logic is complex, `useReducer` is cleaner than multiple `useState` calls. It separates the **what** (actions) from the **how** (the reducer function):

```jsx
const initialState = {
  count: 0,
  step: 1,
  history: [],
};

function counterReducer(state, action) {
  switch (action.type) {
    case "increment":
      return {
        ...state,
        count: state.count + state.step,
        history: [...state.history, state.count + state.step],
      };
    case "decrement":
      return {
        ...state,
        count: state.count - state.step,
        history: [...state.history, state.count - state.step],
      };
    case "setStep":
      return { ...state, step: action.payload };
    case "reset":
      return initialState;
    default:
      return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(counterReducer, initialState);

  return (
    <div>
      <p>
        Count: {state.count} (step: {state.step})
      </p>
      <button onClick={() => dispatch({ type: "increment" })}>+</button>
      <button onClick={() => dispatch({ type: "decrement" })}>-</button>
      <button onClick={() => dispatch({ type: "setStep", payload: 5 })}>Step = 5</button>
    </div>
  );
}
```

The reducer is a **pure function** — same input always produces same output, no side effects. This makes it trivially testable and easy to reason about.

## Combining useReducer with Context

The power move: use a reducer for complex state and Context to share it across the tree. This gives you global state management without any external library:

```jsx
const AuthContext = createContext(null);

function authReducer(state, action) {
  switch (action.type) {
    case "login":
      return { ...state, user: action.payload, isAuth: true };
    case "logout":
      return { user: null, isAuth: false };
    default:
      return state;
  }
}

function AuthProvider({ children }) {
  const [state, dispatch] = useReducer(authReducer, { user: null, isAuth: false });

  return <AuthContext.Provider value={{ ...state, dispatch }}>{children}</AuthContext.Provider>;
}

// Custom hook for clean consumption
function useAuth() {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error("useAuth must be used inside AuthProvider");
  return ctx;
}

// Any component can now call useAuth()
function LoginButton() {
  const { isAuth, dispatch } = useAuth();
  return isAuth ? (
    <button onClick={() => dispatch({ type: "logout" })}>Log out</button>
  ) : (
    <button onClick={() => dispatch({ type: "login", payload: { name: "Dustin" } })}>Log in</button>
  );
}
```

Wrapping `useContext` in a custom hook (`useAuth`) also adds a useful safety check — if a component accidentally renders outside the Provider, you get a clear error instead of a silent `null`.

## What you learned

- **Prop drilling** couples intermediate components to data they do not use — Context breaks that coupling.
- `createContext` + `Provider` + `useContext` shares values to any component in the tree without threading props.
- Context is best for **slowly-changing, broadly-needed data** (user, theme, locale) — not frequently-updating state.
- **`useReducer`** separates state update logic into a pure function, making complex state easier to manage and test.
- Combining `useReducer` + Context gives you robust global state management without external dependencies.
