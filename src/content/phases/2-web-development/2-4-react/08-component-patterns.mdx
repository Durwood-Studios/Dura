## Why this matters

React applications start simple and grow complex. Without intentional patterns, components become monoliths: hundreds of lines, tangled logic, impossible to test or reuse. Composition, custom hooks, and the render-prop pattern are the tools that keep components small, focused, and reusable. These are the patterns you will see in every professional React codebase.

## Composition over inheritance

React's model is composition — components wrap other components. This is the opposite of classical object-oriented inheritance where you extend a base class to add behavior.

The `children` prop is the entry point to composition:

```jsx
function Modal({ title, children, onClose }) {
  return (
    <div className="modal-overlay">
      <div className="modal">
        <div className="modal-header">
          <h2>{title}</h2>
          <button onClick={onClose}>×</button>
        </div>
        <div className="modal-body">{children}</div>
      </div>
    </div>
  );
}

// Usage — any JSX slots into the modal body
<Modal title="Confirm Delete" onClose={() => setOpen(false)}>
  <p>Are you sure you want to delete this item?</p>
  <button onClick={handleDelete}>Yes, delete</button>
</Modal>;
```

`Modal` does not know what goes inside it. It provides the shell — the overlay, the header, the close button. The caller provides the content. This is the slot model, and it is far more flexible than inheritance.

## Multiple slots with named props

When a component needs more than one content area, use named props instead of `children`:

```jsx
function PageLayout({ sidebar, main, footer }) {
  return (
    <div className="layout">
      <aside className="sidebar">{sidebar}</aside>
      <main className="main">{main}</main>
      <footer className="footer">{footer}</footer>
    </div>
  );
}

// Usage
<PageLayout sidebar={<Navigation />} main={<ArticleList />} footer={<SiteFooter />} />;
```

Each slot accepts any JSX. `PageLayout` handles the structure; the caller handles the content.

## Render props

A **render prop** is a prop whose value is a function that returns JSX. It gives the child component control over what gets rendered while the parent component controls the data or behavior:

```jsx
function MouseTracker({ render }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  function handleMouseMove(e) {
    setPosition({ x: e.clientX, y: e.clientY });
  }

  return (
    <div onMouseMove={handleMouseMove} style={{ height: "200px" }}>
      {render(position)}
    </div>
  );
}

// Usage — caller decides how to display the position
<MouseTracker
  render={({ x, y }) => (
    <p>
      Mouse is at {x}, {y}
    </p>
  )}
/>;
```

Render props are less common now that hooks exist — most render-prop patterns can be replaced by a custom hook. But you will encounter them in older codebases and some third-party libraries.

## Custom hooks — extracting logic

A **custom hook** is a function whose name starts with `use` and that can call other hooks. Custom hooks let you extract stateful logic from a component and reuse it elsewhere:

```jsx
// Without a custom hook — logic mixed into the component
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setIsLoading(true);
    fetch(`/api/users/${userId}`)
      .then((r) => r.json())
      .then((data) => {
        setUser(data);
        setIsLoading(false);
      })
      .catch((err) => {
        setError(err.message);
        setIsLoading(false);
      });
  }, [userId]);

  // ...render
}

// With a custom hook — logic extracted, component is clean
function useUser(userId) {
  const [user, setUser] = useState(null);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setIsLoading(true);
    fetch(`/api/users/${userId}`)
      .then((r) => r.json())
      .then((data) => {
        setUser(data);
        setIsLoading(false);
      })
      .catch((err) => {
        setError(err.message);
        setIsLoading(false);
      });
  }, [userId]);

  return { user, isLoading, error };
}

function UserProfile({ userId }) {
  const { user, isLoading, error } = useUser(userId);
  // Clean render logic only
}
```

`useUser` can now be used in any component that needs user data. Test the hook in isolation. Swap the fetch implementation without touching any component.

## The rules of hooks

React hooks have two rules that are not optional:

**1. Call hooks only at the top level.** Never inside conditionals, loops, or nested functions:

```jsx
// WRONG
if (isLoggedIn) {
  const [name, setName] = useState(""); // breaks the rules
}

// CORRECT — always call unconditionally at the top level
const [name, setName] = useState("");
```

React depends on the order of hook calls being stable across renders to track which state belongs to which hook. A conditional call breaks that order.

**2. Call hooks only in React function components or custom hooks.** Not in regular JavaScript functions, class components, or event handlers.

ESLint's `react-hooks` plugin catches both violations automatically. Enable it in every project.

## Separating UI from logic

A useful heuristic: if a component contains both complex logic AND complex JSX, split it. The logic hook handles state and effects. The UI component handles rendering.

```jsx
// Logic — no JSX
function useTimer(initialSeconds) {
  const [seconds, setSeconds] = useState(initialSeconds);
  const [isRunning, setIsRunning] = useState(false);

  useEffect(() => {
    if (!isRunning) return;
    const interval = setInterval(() => setSeconds((s) => s - 1), 1000);
    return () => clearInterval(interval);
  }, [isRunning]);

  return { seconds, isRunning, start: () => setIsRunning(true), stop: () => setIsRunning(false) };
}

// UI — no logic
function TimerDisplay({ userId }) {
  const { seconds, isRunning, start, stop } = useTimer(60);

  return (
    <div>
      <p>{seconds}s</p>
      <button onClick={isRunning ? stop : start}>{isRunning ? "Pause" : "Start"}</button>
    </div>
  );
}
```

const handler = actions[action.type];

listeners.push(fn);
return () => {

if (i > -1) listeners.splice(i, 1);

}

## What you learned

- **Composition** via `children` gives parent components a flexible slot they can fill with any JSX, avoiding the rigidity of inheritance.
- **Named prop slots** extend composition to multiple content areas in a single layout component.
- **Render props** pass a function as a prop to give callers control over what gets rendered given some data — useful but largely replaced by hooks.
- **Custom hooks** extract stateful logic from components into reusable `useX` functions that can call other hooks.
- **Hook rules**: always call hooks at the top level, always in function components or other hooks — ESLint's `react-hooks` plugin enforces this automatically.
