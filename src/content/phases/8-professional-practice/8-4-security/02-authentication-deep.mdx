## Why this matters

Authentication is the front door of every application. Get it wrong and nothing else matters — your authorization, encryption, and monitoring are all protecting a building with an open entrance. Modern apps rarely build auth from scratch. They use protocols like OAuth 2.0 and OpenID Connect. Understanding these protocols at the flow level (not just "add this library") separates engineers who can debug auth failures from those who are stuck when the magic stops working.

## Authentication vs. authorization

Authentication answers "who are you?" Authorization answers "what can you do?" They are separate concerns and should be implemented separately. OAuth 2.0 handles authorization. OpenID Connect (OIDC) adds an authentication layer on top of OAuth 2.0.

## OAuth 2.0 — the authorization framework

OAuth 2.0 lets a user grant a third-party app limited access to their resources without sharing their password. Four roles exist: resource owner (user), client (your app), authorization server (Google, GitHub), and resource server (the API).

```javascript
// The Authorization Code flow — most secure for server-side apps
// Step 1: Redirect user to authorization server
function startOAuthFlow(res) {
  const params = new URLSearchParams({
    response_type: "code",
    client_id: process.env.OAUTH_CLIENT_ID,
    redirect_uri: "https://myapp.com/callback",
    scope: "openid profile email",
    state: crypto.randomUUID(), // CSRF protection
  });

  res.redirect(`https://auth.provider.com/authorize?${params}`);
}

// Step 2: Exchange the authorization code for tokens
async function handleCallback(req, res) {
  const { code, state } = req.query;

  // Verify state matches what we stored in the session
  if (state !== req.session.oauthState) {
    return res.status(403).json({ error: "State mismatch — possible CSRF" });
  }

  const tokenResponse = await fetch("https://auth.provider.com/token", {
    method: "POST",
    headers: { "Content-Type": "application/x-www-form-urlencoded" },
    body: new URLSearchParams({
      grant_type: "authorization_code",
      code,
      redirect_uri: "https://myapp.com/callback",
      client_id: process.env.OAUTH_CLIENT_ID,
      client_secret: process.env.OAUTH_CLIENT_SECRET,
    }),
  });

  const tokens = await tokenResponse.json();
  // tokens contains: access_token, refresh_token, id_token (if OIDC)
  req.session.accessToken = tokens.access_token;
  res.redirect("/dashboard");
}
```

## PKCE — protecting public clients

Single-page apps and mobile apps cannot keep a client secret. PKCE (Proof Key for Code Exchange, pronounced "pixy") adds a challenge-response mechanism so that even if an attacker intercepts the authorization code, they cannot exchange it for tokens.

```javascript
// PKCE flow for a browser-based SPA
async function generatePKCE() {
  // Step 1: Generate a random code_verifier (43-128 characters)
  const array = new Uint8Array(32);
  crypto.getRandomValues(array);
  const codeVerifier = btoa(String.fromCharCode(...array))
    .replace(/\+/g, "-")
    .replace(/\//g, "_")
    .replace(/=+$/, "");

  // Step 2: Hash it to create code_challenge
  const encoder = new TextEncoder();
  const digest = await crypto.subtle.digest("SHA-256", encoder.encode(codeVerifier));
  const codeChallenge = btoa(String.fromCharCode(...new Uint8Array(digest)))
    .replace(/\+/g, "-")
    .replace(/\//g, "_")
    .replace(/=+$/, "");

  return { codeVerifier, codeChallenge };
}

// Include code_challenge in the authorization request
// Include code_verifier in the token exchange
// The server verifies: SHA256(code_verifier) === code_challenge
```

## OpenID Connect — adding identity

OIDC extends OAuth 2.0 with an `id_token` — a JWT that contains claims about the authenticated user.

```javascript
// Decoding an OIDC id_token (JWT)
function decodeJWT(token) {
  const [headerB64, payloadB64, signature] = token.split(".");

  const payload = JSON.parse(atob(payloadB64));
  // payload contains claims:
  // {
  //   iss: "https://auth.provider.com",   — who issued it
  //   sub: "user-123",                     — unique user ID
  //   aud: "my-client-id",                 — intended audience
  //   exp: 1700000000,                     — expiration timestamp
  //   iat: 1699996400,                     — issued at
  //   email: "user@example.com",
  //   name: "Jane Doe"
  // }
  return payload;
}

// CRITICAL: Never trust a JWT without verifying the signature
// In production, use a library that validates:
// 1. Signature against the provider's public key (JWKS)
// 2. exp has not passed
// 3. iss matches expected issuer
// 4. aud matches your client ID
```

## Session management

Sessions bridge stateless HTTP and stateful user experiences. Two dominant patterns exist.

```javascript
// Pattern 1: Server-side sessions (traditional, more secure)
const session = require("express-session");

app.use(
  session({
    secret: process.env.SESSION_SECRET,
    resave: false,
    saveUninitialized: false,
    cookie: {
      httpOnly: true, // JavaScript cannot read this cookie
      secure: true, // Only sent over HTTPS
      sameSite: "lax", // CSRF protection
      maxAge: 3600000, // 1 hour
    },
  })
);

// Pattern 2: Token-based (JWTs stored in httpOnly cookies)
function setAuthCookies(res, tokens) {
  // Store the access token in an httpOnly cookie — NOT localStorage
  res.cookie("access_token", tokens.access_token, {
    httpOnly: true,
    secure: true,
    sameSite: "strict",
    maxAge: 15 * 60 * 1000, // 15 minutes
  });

  // Refresh token in a separate cookie with longer expiry
  res.cookie("refresh_token", tokens.refresh_token, {
    httpOnly: true,
    secure: true,
    sameSite: "strict",
    path: "/api/auth/refresh", // Only sent to the refresh endpoint
    maxAge: 7 * 24 * 60 * 60 * 1000, // 7 days
  });
}
```

## Token rotation and refresh

Access tokens should be short-lived. Refresh tokens get new access tokens without requiring the user to log in again.

```javascript
async function refreshAccessToken(req, res) {
  const refreshToken = req.cookies.refresh_token;
  if (!refreshToken) {
    return res.status(401).json({ error: "No refresh token" });
  }

  try {
    const response = await fetch("https://auth.provider.com/token", {
      method: "POST",
      headers: { "Content-Type": "application/x-www-form-urlencoded" },
      body: new URLSearchParams({
        grant_type: "refresh_token",
        refresh_token: refreshToken,
        client_id: process.env.OAUTH_CLIENT_ID,
      }),
    });

    if (!response.ok) {
      // Refresh token expired or revoked — force re-login
      res.clearCookie("access_token");
      res.clearCookie("refresh_token");
      return res.status(401).json({ error: "Session expired" });
    }

    const tokens = await response.json();
    setAuthCookies(res, tokens);
    res.json({ message: "Token refreshed" });
  } catch (error) {
    console.error("Token refresh failed:", error);
    res.status(500).json({ error: "Internal error" });
  }
}
```

## What you learned

- OAuth 2.0 is an authorization framework with four roles; OIDC adds an identity layer on top via the id_token JWT
- PKCE protects public clients (SPAs, mobile apps) by binding the authorization code to a cryptographic verifier that only the original client possesses
- Session security depends on httpOnly cookies, short-lived access tokens, refresh token rotation, and proper expiration handling
