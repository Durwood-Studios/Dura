## Why this matters

Cryptography is the foundation of trust on the internet. Every HTTPS connection, every password stored safely, every signed JWT, every encrypted database column relies on cryptographic primitives. You do not need to implement these algorithms. You need to understand which primitive to use, when to use it, and how to avoid the common mistakes that turn strong cryptography into a false sense of security.

## Hashing — one-way fingerprints

A hash function takes input of any size and produces a fixed-size output. It is one-way (you cannot reverse it), deterministic (same input always produces same output), and collision-resistant (different inputs should not produce the same hash).

```javascript
const crypto = require("crypto");

// SHA-256 — general purpose hashing (file integrity, checksums)
function hashSHA256(data) {
  return crypto.createHash("sha256").update(data).digest("hex");
}

console.log(hashSHA256("hello"));
// 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824

console.log(hashSHA256("hello!"));
// Completely different hash — even one character change avalanches
// ce06092fb948d9ffac7d1a376f7b656b7da1e5cf1b4335ebed6f09b8ef4da59a

// NEVER use MD5 or SHA-1 for security — both have known collision attacks
// Use SHA-256 or SHA-3 for integrity checks
// Use bcrypt, scrypt, or Argon2 for passwords (see below)
```

## Password hashing — deliberately slow

General-purpose hashes are fast. That is bad for passwords because attackers can try billions of guesses per second. Password hashing algorithms are intentionally slow and include a salt (random data) to prevent precomputed attacks.

```javascript
const bcrypt = require("bcrypt");

// Hashing a password — cost factor 12 means 2^12 iterations
async function hashPassword(plaintext) {
  // bcrypt automatically generates a random salt and embeds it in the hash
  const hash = await bcrypt.hash(plaintext, 12);
  return hash;
  // "$2b$12$LJ3m4ys3Lz0rOFzGqKqKaeKF4RVNWvJg..."
  // Contains: algorithm ($2b$), cost (12), salt, and hash
}

// Verifying a password
async function verifyPassword(plaintext, storedHash) {
  // bcrypt extracts the salt from the stored hash and re-hashes
  const isMatch = await bcrypt.compare(plaintext, storedHash);
  return isMatch;
}

// WHY a salt matters:
// Without salt: hash("password123") is always the same
// Attackers precompute hashes for millions of common passwords (rainbow tables)
// With salt: hash("password123" + randomSalt) is unique per user
// Rainbow tables become useless — each user requires separate brute force
```

## Symmetric encryption — one key for both sides

Symmetric encryption uses the same key to encrypt and decrypt. Fast and efficient for large amounts of data. The challenge is getting the key to both parties securely.

```javascript
const crypto = require("crypto");

// AES-256-GCM — authenticated encryption (confidentiality + integrity)
function encrypt(plaintext, key) {
  const iv = crypto.randomBytes(12); // 96-bit IV for GCM
  const cipher = crypto.createCipheriv("aes-256-gcm", key, iv);

  let encrypted = cipher.update(plaintext, "utf8", "hex");
  encrypted += cipher.final("hex");

  const authTag = cipher.getAuthTag(); // Integrity tag

  return {
    iv: iv.toString("hex"),
    encrypted,
    authTag: authTag.toString("hex"),
  };
}

function decrypt(encryptedData, key) {
  const decipher = crypto.createDecipheriv(
    "aes-256-gcm",
    key,
    Buffer.from(encryptedData.iv, "hex")
  );

  decipher.setAuthTag(Buffer.from(encryptedData.authTag, "hex"));

  let decrypted = decipher.update(encryptedData.encrypted, "hex", "utf8");
  decrypted += decipher.final("utf8");
  return decrypted;
}

// Usage
const key = crypto.randomBytes(32); // 256-bit key
const result = encrypt("sensitive data", key);
console.log(decrypt(result, key)); // "sensitive data"

// CRITICAL: Never reuse an IV with the same key
// CRITICAL: Always use authenticated encryption (GCM, not CBC without HMAC)
```

## Asymmetric encryption — public and private keys

Asymmetric encryption uses a key pair: a public key (anyone can have it) and a private key (kept secret). Encrypt with the public key, decrypt with the private key. Slower than symmetric, used for key exchange and digital signatures.

```javascript
const crypto = require("crypto");

// Generate an RSA key pair
const { publicKey, privateKey } = crypto.generateKeyPairSync("rsa", {
  modulusLength: 2048,
  publicKeyEncoding: { type: "spki", format: "pem" },
  privateKeyEncoding: { type: "pkcs8", format: "pem" },
});

// Encrypt with public key — anyone can do this
function encryptWithPublic(data, pubKey) {
  return crypto.publicEncrypt(pubKey, Buffer.from(data)).toString("base64");
}

// Decrypt with private key — only the key holder
function decryptWithPrivate(encrypted, privKey) {
  return crypto.privateDecrypt(privKey, Buffer.from(encrypted, "base64")).toString("utf8");
}

const encrypted = encryptWithPublic("secret message", publicKey);
const decrypted = decryptWithPrivate(encrypted, privateKey);
console.log(decrypted); // "secret message"
```

## Digital signatures — proving authenticity

Signing proves that data was created by the holder of a private key and has not been modified. Sign with the private key, verify with the public key.

```javascript
const crypto = require("crypto");

function signData(data, privateKey) {
  const signer = crypto.createSign("SHA256");
  signer.update(data);
  return signer.sign(privateKey, "base64");
}

function verifySignature(data, signature, publicKey) {
  const verifier = crypto.createVerify("SHA256");
  verifier.update(data);
  return verifier.verify(publicKey, signature, "base64");
}

// JWTs use this pattern:
// header.payload is signed with the server's private key
// Anyone with the public key can verify the token is authentic
// But only the server can create valid tokens
```

## HMAC — keyed hashing for message authentication

HMAC combines a hash function with a secret key. Used for verifying that a message was not tampered with and came from someone who knows the key.

```javascript
const crypto = require("crypto");

// Webhook verification — Stripe, GitHub, etc. use this pattern
function verifyWebhookSignature(payload, signature, secret) {
  const expected = crypto.createHmac("sha256", secret).update(payload).digest("hex");

  // Use timingSafeEqual to prevent timing attacks
  const isValid = crypto.timingSafeEqual(
    Buffer.from(signature, "hex"),
    Buffer.from(expected, "hex")
  );

  return isValid;
}

// When the webhook arrives:
const payload = JSON.stringify(req.body);
const signature = req.headers["x-signature-256"];
if (!verifyWebhookSignature(payload, signature, process.env.WEBHOOK_SECRET)) {
  return res.status(401).json({ error: "Invalid signature" });
}
```

## Key management — the hardest part

The cryptography is usually fine. The keys are where things go wrong.

```javascript
// BAD — hardcoded key in source code
const ENCRYPTION_KEY = "my-super-secret-key-12345678901"; // in git history forever

// BAD — key derived from predictable input
const key = crypto.createHash("sha256").update("password").digest();

// GOOD — key from environment variable, rotated regularly
const key = Buffer.from(process.env.ENCRYPTION_KEY, "hex");

// GOOD — key derived with proper KDF when you must derive from a password
async function deriveKey(password, salt) {
  return new Promise((resolve, reject) => {
    // scrypt: memory-hard KDF resistant to GPU attacks
    crypto.scrypt(password, salt, 32, { N: 16384, r: 8, p: 1 }, (err, key) => {
      if (err) reject(err);
      resolve(key);
    });
  });
}

// KEY ROTATION: encrypt data with the new key, keep the old key for decryption
// during the transition period. Store a key version identifier with encrypted data.
```

## What you learned

- Hashing (SHA-256) creates one-way fingerprints for integrity; password hashing (bcrypt, Argon2) adds deliberate slowness and salts to resist brute force
- Symmetric encryption (AES-GCM) uses one key for both sides and provides both confidentiality and integrity; asymmetric encryption uses a public/private key pair for key exchange and signatures
- Key management is the hardest part of cryptography — keys must never be hardcoded, should be rotated regularly, and must be stored in environment variables or dedicated secret management systems
