## Why this matters

The OWASP Top Ten is the industry-standard awareness document for web application security. Hiring managers, compliance audits, and penetration testers all reference it. If you ship a production application without understanding these ten categories, you are gambling with user data, company reputation, and legal liability. Every year, breaches that cost millions trace back to vulnerabilities described in this list.

## A01 — Broken Access Control

The most common category. Users act outside their intended permissions.

```javascript
// VULNERABLE — relies on client-side route hiding
app.get("/admin/users", (req, res) => {
  // No server-side check. Anyone who knows the URL gets in.
  const users = db.query("SELECT * FROM users");
  res.json(users);
});

// FIXED — enforce authorization on every request
app.get("/admin/users", authenticate, authorize("admin"), (req, res) => {
  const users = db.query("SELECT * FROM users");
  res.json(users);
});

function authorize(role) {
  return (req, res, next) => {
    if (req.user.role !== role) {
      return res.status(403).json({ error: "Forbidden" });
    }
    next();
  };
}
```

## A02 — Cryptographic Failures

Formerly "Sensitive Data Exposure." Data in transit or at rest is not properly protected.

```javascript
// VULNERABLE — storing passwords in plaintext
async function createUser(email, password) {
  await db.query("INSERT INTO users (email, password) VALUES ($1, $2)", [
    email,
    password, // plaintext!
  ]);
}

// FIXED — hash with bcrypt, cost factor 12+
const bcrypt = require("bcrypt");

async function createUser(email, password) {
  const hash = await bcrypt.hash(password, 12);
  await db.query("INSERT INTO users (email, password_hash) VALUES ($1, $2)", [email, hash]);
}
```

## A03 — Injection

Untrusted data is sent to an interpreter as part of a command or query. SQL injection remains the classic example but command injection, LDAP injection, and NoSQL injection all belong here.

```javascript
// VULNERABLE — string concatenation in query
app.get("/users", (req, res) => {
  const name = req.query.name;
  const result = db.query(`SELECT * FROM users WHERE name = '${name}'`);
  res.json(result);
});
// Attacker sends: ?name=' OR '1'='1

// FIXED — parameterized query
app.get("/users", (req, res) => {
  const name = req.query.name;
  const result = db.query("SELECT * FROM users WHERE name = $1", [name]);
  res.json(result);
});
```

## A04 — Insecure Design

Security flaws baked into the architecture itself, not just the implementation. No amount of perfect code fixes a design that never considered abuse cases.

```javascript
// VULNERABLE — unlimited password reset attempts
app.post("/reset-password", (req, res) => {
  const code = generateSixDigitCode();
  sendEmail(req.body.email, code);
  res.json({ message: "Code sent" });
});
// Attacker brute-forces all 1,000,000 combinations

// FIXED — rate limiting + account lockout + longer codes
const rateLimit = require("express-rate-limit");

const resetLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 3, // 3 attempts per window
  message: { error: "Too many attempts. Try again later." },
});

app.post("/reset-password", resetLimiter, (req, res) => {
  const code = crypto.randomBytes(16).toString("hex"); // 128-bit token
  sendEmail(req.body.email, code);
  res.json({ message: "Code sent" });
});
```

## A05 — Security Misconfiguration

Default credentials, open cloud storage buckets, verbose error messages, unnecessary features enabled.

```javascript
// VULNERABLE — stack traces in production
app.use((err, req, res, next) => {
  res.status(500).json({
    error: err.message,
    stack: err.stack, // leaks internals to attackers
  });
});

// FIXED — generic message in production, details in logs only
app.use((err, req, res, next) => {
  console.error(err); // internal logging
  res.status(500).json({
    error: process.env.NODE_ENV === "production" ? "Internal server error" : err.message,
  });
});
```

## A08 — Software and Data Integrity Failures

Trusting unsigned updates, CI/CD pipeline compromises, insecure deserialization.

```javascript
// VULNERABLE — deserializing untrusted input
const userData = JSON.parse(req.body.data);
// If this were a language with object deserialization (Java, Python pickle),
// arbitrary code execution is possible.

// FIXED — validate structure with a schema
const { z } = require("zod");
const UserSchema = z.object({
  name: z.string().max(100),
  email: z.string().email(),
});

const parsed = UserSchema.safeParse(JSON.parse(req.body.data));
if (!parsed.success) {
  return res.status(400).json({ error: parsed.error.issues });
}
```

## A09 — Security Logging and Monitoring Failures

When breaches happen and nobody notices for months, this is the category.

```javascript
// Minimum: log authentication events
function logSecurityEvent(event) {
  const entry = {
    timestamp: new Date().toISOString(),
    type: event.type, // "LOGIN_SUCCESS", "LOGIN_FAILURE", "ACCESS_DENIED"
    userId: event.userId,
    ip: event.ip,
    userAgent: event.userAgent,
  };
  // Ship to centralized logging (not just console.log)
  securityLogger.info(entry);
}
```

## A10 — Server-Side Request Forgery (SSRF)

The server fetches a URL provided by the user, allowing attackers to access internal services.

```javascript
// VULNERABLE — fetch any URL the user provides
app.get("/preview", async (req, res) => {
  const response = await fetch(req.query.url);
  const html = await response.text();
  res.send(html);
});
// Attacker sends: ?url=http://169.254.169.254/latest/meta-data/ (AWS metadata)

// FIXED — allowlist of permitted domains
const ALLOWED_HOSTS = new Set(["api.example.com", "cdn.example.com"]);

app.get("/preview", async (req, res) => {
  const url = new URL(req.query.url);
  if (!ALLOWED_HOSTS.has(url.hostname)) {
    return res.status(400).json({ error: "Domain not allowed" });
  }
  const response = await fetch(url.toString());
  const html = await response.text();
  res.send(html);
});
```

## What you learned

- The OWASP Top Ten categorizes the most critical web security risks, from broken access control to SSRF, providing a shared vocabulary for security discussions
- Each category has concrete defensive patterns: parameterized queries for injection, schema validation for integrity failures, allowlists for SSRF
- Security is not a feature you add at the end — it is a design concern that must be considered from the first architecture decision
