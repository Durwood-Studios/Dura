## Why this matters

Knowing architecture patterns is necessary but not sufficient. The real skill is choosing the right patterns for a specific set of requirements, constraints, and tradeoffs — then documenting that reasoning so future developers (including future you) understand why the system is shaped the way it is. Architecture Decision Records (ADRs) are the tool teams use to capture these decisions. Without them, every new team member asks "why did we do it this way?" and nobody remembers.

This lesson ties together everything from the module: Clean Architecture, design patterns, microservices vs. monoliths, event-driven systems, API styles, and DDD. You'll practice the full arc — from requirements to architecture to decision documentation.

## From requirements to architecture

Architecture starts with requirements, not technology. Before choosing any pattern, you need to understand what the system must do (functional requirements) and how well it must do it (non-functional requirements).

```javascript
// Step 1: Capture requirements as structured data.
// This is a real technique — requirements as code makes them testable.

const systemRequirements = {
  functional: [
    "Users can create and manage study plans",
    "Users can take quizzes and see scores",
    "Content authors can publish lessons without deploying code",
    "Progress syncs across devices when online",
    "App works fully offline",
  ],
  nonFunctional: {
    availability: "99.9% uptime",
    latency: "Page load under 2 seconds on 3G",
    scale: "100,000 concurrent users",
    security: "User data encrypted at rest, GDPR compliant",
    deployment: "Zero-downtime deploys, rollback within 5 minutes",
  },
  constraints: [
    "Team of 4 developers",
    "Must launch MVP in 3 months",
    "Budget: $500/month infrastructure",
    "Must support mobile browsers — no native app",
  ],
};
```

The constraints matter as much as the requirements. A team of 4 with a 3-month deadline should not architect for microservices. A $500/month budget rules out multi-region Kubernetes clusters.

## Tradeoff analysis

Every architectural decision is a tradeoff. The job isn't finding the "perfect" solution — it's finding the best solution given the constraints, and being explicit about what you're giving up.

```javascript
// Structured tradeoff analysis.
function analyzeTradeoffs(options) {
  return options.map((option) => ({
    name: option.name,
    score: option.pros.length - option.cons.length,
    fits: option.pros,
    costs: option.cons,
    risksIfChosen: option.risks,
    mitigations: option.mitigations,
  }));
}

const architectureOptions = [
  {
    name: "Monolith + Static Export (Next.js)",
    pros: [
      "Single codebase — easy for a 4-person team",
      "Fast offline support via service worker",
      "Low infrastructure cost (static hosting + serverless)",
      "Deployable in under 3 months",
    ],
    cons: ["Scaling requires scaling the entire app", "Content updates require a rebuild"],
    risks: ["Rebuild times grow with content volume"],
    mitigations: ["Incremental Static Regeneration for content pages"],
  },
  {
    name: "Microservices (Content, Auth, Progress, Quiz)",
    pros: ["Independent scaling per service", "Services can use different technologies"],
    cons: [
      "Operational complexity far exceeds team capacity",
      "4 services = 4 deployment pipelines, 4 monitoring dashboards",
      "Distributed debugging with a 4-person team",
      "3+ months just for infrastructure",
    ],
    risks: ["Team burns out on ops before shipping features"],
    mitigations: ["None realistic at this team size"],
  },
];

const analysis = analyzeTradeoffs(architectureOptions);
// Option 1 scores higher for this set of constraints.
```

The monolith wins here not because microservices are bad, but because the constraints (team size, timeline, budget) make microservices a net negative. This is the core of architecture: matching solutions to contexts.

## Architecture Decision Records

An ADR is a short document that captures one significant architectural decision. The format is simple and consistent:

```javascript
// ADR as structured data — many teams store these as markdown files
// in the repo alongside the code they describe.

const adr001 = {
  id: "ADR-001",
  title: "Use monolithic Next.js architecture for MVP",
  date: "2025-06-15",
  status: "accepted", // proposed | accepted | deprecated | superseded

  context:
    "We are building a learning management system with offline support. " +
    "The team has 4 developers and a 3-month deadline. Infrastructure " +
    "budget is $500/month. The app must work on mobile browsers.",

  decision:
    "We will build a monolithic Next.js application with static export " +
    "for content pages and serverless API routes for dynamic features. " +
    "Offline support will use a service worker with IndexedDB for local " +
    "state. Supabase will handle auth and cross-device sync.",

  consequences: [
    "Positive: Single codebase, single deployment, fast iteration",
    "Positive: Low ops burden — Vercel handles infrastructure",
    "Positive: Offline-first architecture from day one",
    "Negative: Cannot scale subsystems independently",
    "Negative: Content updates trigger a full rebuild (mitigated by ISR)",
    "Risk: If we exceed 100K users, we may need to extract services",
  ],

  alternatives: [
    "Microservices — rejected due to team size and timeline constraints",
    "Traditional SPA + separate API — rejected due to offline complexity",
  ],
};
```

### Why ADRs matter

Six months from now, a new developer will ask: "Why didn't we use microservices?" Without an ADR, the answer is tribal knowledge that may have already left the company. With an ADR, the answer is in the repo, versioned alongside the code, with full context about what was considered and why.

ADRs also prevent re-litigation. When someone proposes the same alternative that was already evaluated and rejected, the ADR shows the reasoning. If the constraints have changed (team grew to 20, budget increased 10x), the ADR shows that too — and it's time to write a new ADR that supersedes the old one.

## Putting it all together

Here's the full process, from requirements through architecture to documentation:

```javascript
// A complete architecture exercise: design a notification system.

const notificationSystemDesign = {
  requirements: {
    functional: [
      "Send notifications via email, SMS, and push",
      "Users configure which channels they want",
      "Support scheduled notifications (send at 9am user's timezone)",
      "Retry failed deliveries up to 3 times",
    ],
    nonFunctional: {
      throughput: "10,000 notifications per minute at peak",
      latency: "Delivered within 60 seconds of trigger",
      reliability: "No notification silently dropped",
    },
  },

  architecturalChoices: {
    pattern: "Event-driven with Strategy pattern for delivery channels",
    reasoning:
      "Events decouple notification triggers from delivery. " +
      "Strategy pattern makes adding new channels trivial. " +
      "A queue with retry handles the reliability requirement.",

    components: {
      triggerService: "Accepts notification requests, validates, publishes events",
      deliveryWorker: "Consumes events, selects channel strategy, delivers",
      scheduler: "Holds scheduled notifications, publishes at the right time",
      deadLetterQueue: "Captures permanently failed notifications for review",
    },
  },

  adr: {
    id: "ADR-007",
    title: "Use event-driven delivery with pluggable channel strategies",
    status: "accepted",
    context:
      "Notifications must support multiple channels with different reliability " +
      "characteristics. Email has high latency, push is fire-and-forget, " +
      "SMS requires vendor-specific APIs. Adding channels should not " +
      "require modifying existing delivery code.",
    decision:
      "Each delivery channel is a Strategy. A message queue decouples " +
      "triggers from delivery. Failed deliveries retry with exponential " +
      "backoff. After 3 failures, messages go to a dead letter queue.",
    consequences: [
      "Positive: New channels are a single Strategy implementation",
      "Positive: Retry logic is centralized in the delivery worker",
      "Negative: Eventual consistency — notification is 'sent' before confirmed delivered",
      "Risk: Queue backlog during peak — mitigated with autoscaling workers",
    ],
  },
};
```

## What you learned

- Architecture starts with requirements and constraints, not technology — match solutions to the specific context before choosing patterns
- Tradeoff analysis compares options explicitly, weighing pros, cons, and risks against the team's real constraints (size, budget, timeline)
- Architecture Decision Records capture the context, decision, consequences, and rejected alternatives so future developers understand why the system is shaped the way it is
