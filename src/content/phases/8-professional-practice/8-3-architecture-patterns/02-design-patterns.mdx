## Why this matters

Design patterns are named solutions to recurring design problems. They give teams a shared vocabulary — saying "use a Strategy here" communicates an entire structural idea in three words. But patterns are tools, not mandates. Misapplied patterns create more complexity than they solve. The skill isn't knowing patterns — it's knowing when a pattern earns its weight and when a plain function does the job better.

We'll examine four patterns that appear constantly in production JavaScript: Factory, Strategy, Observer, and Decorator. For each, you'll see what problem it solves, how to implement it, and the specific signal that tells you the pattern is overkill.

## Factory: creating objects without `new`

A Factory encapsulates object creation so the caller doesn't need to know the exact class or configuration. This matters when construction logic is complex, conditional, or likely to change.

```javascript
// Without a factory — every caller must know the construction details.
const logger = new Logger({ level: "info", transport: new ConsoleTransport() });

// With a factory — callers describe what they need, not how to build it.
function createLogger(environment) {
  if (environment === "production") {
    return new Logger({ level: "warn", transport: new CloudTransport() });
  }
  if (environment === "test") {
    return new Logger({ level: "silent", transport: new NullTransport() });
  }
  return new Logger({ level: "debug", transport: new ConsoleTransport() });
}

const logger = createLogger(process.env.NODE_ENV);
```

**Use when:** Object creation depends on runtime conditions, you need to centralize default configuration, or you want to hide third-party constructors behind your own interface.

**Skip when:** Construction is a single `new` call with obvious parameters. A factory for `new Date()` is ceremony, not design.

## Strategy: swapping algorithms at runtime

Strategy extracts a family of interchangeable algorithms behind a common interface. The context object delegates to whichever strategy is plugged in.

```javascript
// Pricing strategies — each follows the same shape: (basePrice) => finalPrice.
const regularPricing = (price) => price;
const memberPricing = (price) => price * 0.9;
const bulkPricing = (price) => (price > 1000 ? price * 0.8 : price * 0.95);

// The checkout function doesn't know which pricing logic it runs.
function calculateTotal(items, pricingStrategy) {
  return items.reduce((total, item) => {
    return total + pricingStrategy(item.price * item.quantity);
  }, 0);
}

const items = [
  { name: "Widget", price: 25, quantity: 10 },
  { name: "Gadget", price: 75, quantity: 2 },
];

console.log(calculateTotal(items, regularPricing)); // 400
console.log(calculateTotal(items, memberPricing)); // 360
console.log(calculateTotal(items, bulkPricing)); // 380
```

In JavaScript, Strategy often reduces to "pass a function." That's fine — you don't need a `PricingStrategy` class with an `apply()` method when a plain function does the same thing with less ceremony.

**Use when:** You have multiple algorithms for the same task, and the choice varies by context, configuration, or user input.

**Skip when:** There's only one algorithm and no foreseeable variation. Wrapping a single behavior in Strategy is premature abstraction.

## Observer: reacting to changes

Observer lets an object (the subject) notify multiple dependents (observers) when its state changes, without the subject knowing who those observers are.

```javascript
// A minimal event emitter — the core of the Observer pattern.
function createEventEmitter() {
  const listeners = new Map();

  return {
    on(event, callback) {
      if (!listeners.has(event)) listeners.set(event, []);
      listeners.get(event).push(callback);
    },

    off(event, callback) {
      const cbs = listeners.get(event);
      if (cbs)
        listeners.set(
          event,
          cbs.filter((cb) => cb !== callback)
        );
    },

    emit(event, data) {
      const cbs = listeners.get(event) || [];
      cbs.forEach((cb) => cb(data));
    },
  };
}

// Usage: a shopping cart that notifies any interested part of the app.
const cart = createEventEmitter();

cart.on("itemAdded", (item) => console.log(`Analytics: tracked ${item.name}`));
cart.on("itemAdded", (item) => console.log(`UI: refreshing cart badge`));

cart.emit("itemAdded", { name: "Keyboard", price: 89 });
// Analytics: tracked Keyboard
// UI: refreshing cart badge
```

You've already used this pattern — `addEventListener` in the browser, `.on()` in Node's `EventEmitter`, React's `useEffect` reacting to state changes. Observer is everywhere.

**Use when:** Multiple parts of a system need to react to the same event, and the source shouldn't be coupled to the receivers.

**Skip when:** There's exactly one consumer. A direct function call is simpler and easier to trace than an event you have to search the codebase to find.

## Decorator: adding behavior without modifying the original

Decorator wraps an object to extend its behavior, leaving the original intact. In JavaScript, this is often implemented as higher-order functions.

```javascript
// A base API client.
function createApiClient(baseUrl) {
  return {
    async get(path) {
      const response = await fetch(`${baseUrl}${path}`);
      return response.json();
    },
  };
}

// Decorator: add automatic retry logic.
function withRetry(client, maxRetries = 3) {
  return {
    async get(path) {
      let lastError;
      for (let attempt = 0; attempt < maxRetries; attempt++) {
        try {
          return await client.get(path);
        } catch (error) {
          lastError = error;
        }
      }
      throw lastError;
    },
  };
}

// Decorator: add request logging.
function withLogging(client) {
  return {
    async get(path) {
      console.log(`GET ${path}`);
      const result = await client.get(path);
      console.log(`GET ${path} — complete`);
      return result;
    },
  };
}

// Compose decorators — order matters.
const api = withLogging(withRetry(createApiClient("https://api.example.com")));
// Logging wraps retry, which wraps the base client.
```

**Use when:** You need to layer cross-cutting concerns (logging, caching, retry, auth) onto existing objects without modifying their source.

**Skip when:** The "decoration" is a one-off `if` statement inside the original function. Don't create a wrapper class for a three-line change.

## The pattern smell test

Before applying any pattern, ask: "If I delete this abstraction and inline the code, does the result get simpler?" If yes, the pattern is adding cost without adding value. Patterns should reduce the total complexity a reader must hold in their head, not increase it.

## What you learned

- Factory centralizes object creation behind a function, Strategy swaps algorithms via interchangeable functions, Observer decouples event producers from consumers, and Decorator layers behavior by wrapping objects
- Each pattern has a clear "use when" signal — apply it only when the problem genuinely matches, not as a default
- In JavaScript, many classic OOP patterns reduce to plain functions and closures, keeping the benefit while dropping the ceremony
