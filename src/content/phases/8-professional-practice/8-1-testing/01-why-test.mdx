## Why this matters

Every developer ships bugs. The question is whether you find them before your users do. Testing is not paperwork you file after the real work is done — it is the mechanism that lets you change code with confidence. Without tests, every refactor is a gamble, every deploy a prayer. With tests, you build a safety net that catches regressions the moment they appear.

Teams that test well ship faster, not slower. That sounds counterintuitive until you factor in the hours spent debugging production issues, rolling back broken deploys, and manually clicking through flows you have already verified once. Testing front-loads a small cost to eliminate a large, recurring one.

## The testing pyramid

The testing pyramid is a model that tells you how many tests of each type to write.

```
        /  E2E  \           Few — slow, expensive, brittle
       /----------\
      / Integration \       Some — moderate speed, real boundaries
     /----------------\
    /    Unit Tests     \   Many — fast, cheap, isolated
   /____________________\
```

**Unit tests** verify a single function or module in isolation. They run in milliseconds. You should have hundreds or thousands of them.

**Integration tests** verify that multiple modules work together — an API route that reads from a database, a form that calls a service. They take seconds. You should have dozens to hundreds.

**End-to-end (E2E) tests** verify complete user flows through the real UI. They take minutes. You should have a handful covering critical paths.

The pyramid shape matters because cost and fragility increase as you move up. A unit test that fails always points to the exact broken function. An E2E test that fails could mean anything — network, timing, browser rendering, test infrastructure.

## What to test

Not everything deserves a test. Here is a practical framework.

```javascript
// ALWAYS test: business logic
function calculateDiscount(price, memberYears) {
  if (memberYears >= 5) return price * 0.8;
  if (memberYears >= 2) return price * 0.9;
  return price;
}

// ALWAYS test: data transformations
function formatUserForAPI(user) {
  return {
    full_name: `${user.firstName} ${user.lastName}`,
    email: user.email.toLowerCase().trim(),
    joined: user.createdAt.toISOString(),
  };
}

// ALWAYS test: edge cases and error paths
function divide(a, b) {
  if (b === 0) throw new Error("Cannot divide by zero");
  return a / b;
}

// SKIP testing: trivial getters, framework boilerplate,
// third-party library internals, CSS styling
```

The rule of thumb: if it has logic, test it. If it is pure configuration or delegation to a framework, let the framework's own tests cover it.

## The ROI of testing

Testing has diminishing returns. The first 20% of test coverage catches 80% of bugs — those are the tests on your core business logic, data transformations, and error handling. Chasing 100% coverage leads to brittle tests on trivial code that cost more to maintain than the bugs they would catch.

```javascript
// High ROI: tests on a checkout flow
// This function processes real money — a bug here is a revenue bug
function processOrder(cart, paymentMethod) {
  const subtotal = cart.items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  const tax = subtotal * cart.taxRate;
  const total = subtotal + tax;

  if (total <= 0) {
    throw new Error("Order total must be positive");
  }

  return { subtotal, tax, total, paymentMethod };
}

// Low ROI: testing that a component renders "Hello"
// If this breaks, you will notice immediately
function Greeting({ name }) {
  return <h1>Hello, {name}</h1>;
}
```

Invest your testing time where bugs are expensive. A broken checkout calculation costs revenue. A missing greeting costs nothing.

## What you learned

- The **testing pyramid** guides how many tests of each type to write: many unit, some integration, few E2E.
- **Test what has logic** — business rules, data transformations, error paths — and skip trivial boilerplate.
- Testing ROI follows the **80/20 rule**: focus coverage on code where bugs are expensive.
