## Why this matters

Unit tests are the fastest feedback loop you have. A good unit test runs in under 50 milliseconds, tells you exactly which function broke, and gives you the confidence to refactor without fear. Every professional codebase relies on hundreds or thousands of unit tests that run on every commit. When you understand how to write them well, you write better code — because testable code is modular code.

## Anatomy of a test

Every test has three phases: **arrange**, **act**, **assert**. This pattern is sometimes called AAA, and it keeps your tests readable.

```javascript
// math.js
function add(a, b) {
  return a + b;
}

// math.test.js
import { describe, it, expect } from "vitest";
import { add } from "./math.js";

describe("add", () => {
  it("returns the sum of two positive numbers", () => {
    // Arrange
    const a = 3;
    const b = 7;

    // Act
    const result = add(a, b);

    // Assert
    expect(result).toBe(10);
  });

  it("handles negative numbers", () => {
    expect(add(-2, 5)).toBe(3);
    expect(add(-3, -4)).toBe(-7);
  });

  it("returns zero when both inputs are zero", () => {
    expect(add(0, 0)).toBe(0);
  });
});
```

`describe` groups related tests. `it` defines a single test case. `expect` makes an assertion about the result. If the assertion fails, the test fails — and the error message tells you exactly what went wrong.

## Common assertions

Vitest (and Jest) provide matchers for every common comparison.

```javascript
import { expect } from "vitest";

// Exact equality
expect(2 + 2).toBe(4);

// Deep equality for objects and arrays
expect({ name: "Ada" }).toEqual({ name: "Ada" });

// Truthiness
expect(true).toBeTruthy();
expect(null).toBeFalsy();
expect(undefined).toBeUndefined();

// Numbers
expect(0.1 + 0.2).toBeCloseTo(0.3);
expect(10).toBeGreaterThan(5);

// Strings
expect("team").toContain("tea");
expect("Hello World").toMatch(/hello/i);

// Arrays
expect([1, 2, 3]).toContain(2);
expect([1, 2, 3]).toHaveLength(3);

// Errors
expect(() => {
  throw new Error("fail");
}).toThrow("fail");
```

Choose the most specific matcher. `toBe(true)` is clearer than `toBeTruthy()` when you expect exactly `true`.

## Mocking dependencies

Unit tests isolate the function under test. When that function calls an external service, you replace the dependency with a mock.

```javascript
import { describe, it, expect, vi } from "vitest";

// The module we want to test
function fetchUserName(userId, apiClient) {
  return apiClient.get(`/users/${userId}`).then((res) => res.name);
}

describe("fetchUserName", () => {
  it("returns the user name from the API response", async () => {
    // Arrange: create a mock API client
    const mockClient = {
      get: vi.fn().mockResolvedValue({ name: "Grace Hopper" }),
    };

    // Act
    const name = await fetchUserName(42, mockClient);

    // Assert
    expect(name).toBe("Grace Hopper");
    expect(mockClient.get).toHaveBeenCalledWith("/users/42");
  });

  it("propagates API errors", async () => {
    const mockClient = {
      get: vi.fn().mockRejectedValue(new Error("Network error")),
    };

    await expect(fetchUserName(1, mockClient)).rejects.toThrow("Network error");
  });
});
```

`vi.fn()` creates a mock function. `.mockResolvedValue()` makes it return a resolved promise. `.toHaveBeenCalledWith()` verifies the mock was called with the right arguments. This approach tests your logic without hitting a real network.

## Test structure best practices

```javascript
import { describe, it, expect, beforeEach } from "vitest";

// Use beforeEach for shared setup
describe("ShoppingCart", () => {
  let cart;

  beforeEach(() => {
    cart = { items: [], total: 0 };
  });

  // Test names describe the behavior, not the implementation
  it("starts empty", () => {
    expect(cart.items).toHaveLength(0);
    expect(cart.total).toBe(0);
  });

  // One assertion per concept — multiple expects are fine
  // if they test the same behavior
  it("calculates total after adding items", () => {
    cart.items.push({ name: "Book", price: 15 });
    cart.items.push({ name: "Pen", price: 3 });
    cart.total = cart.items.reduce((sum, item) => sum + item.price, 0);

    expect(cart.items).toHaveLength(2);
    expect(cart.total).toBe(18);
  });
});
```

Name your tests so that when they fail, you can read the test name and know what broke — without reading the test body.

## What you learned

- Unit tests follow the **Arrange, Act, Assert** pattern for clarity and consistency.
- **Assertions** like `toBe`, `toEqual`, and `toThrow` verify specific behaviors with precise error messages.
- **Mocking** with `vi.fn()` isolates the function under test from external dependencies like APIs and databases.
