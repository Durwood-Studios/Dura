## Why this matters

Test-driven development flips the usual order of writing code. Instead of writing the function first and testing it after, you write the test first and let the failing test guide your implementation. This sounds backwards until you try it — then you realize the test forces you to think about the interface before the internals, the requirements before the details. TDD does not suit every situation, but when it does, it produces cleaner code with fewer bugs and better design.

## Red-green-refactor

TDD follows a strict three-step cycle.

**Red**: Write a test for behavior that does not exist yet. Run it. Watch it fail.

```javascript
// Step 1: RED — write the test first
import { describe, it, expect } from "vitest";
import { slugify } from "./slugify.js";

describe("slugify", () => {
  it("converts a title to a URL-safe slug", () => {
    expect(slugify("Hello World")).toBe("hello-world");
  });
});

// Running this test fails because slugify does not exist yet.
// That failing test is the "red" phase.
```

**Green**: Write the minimum code to make the test pass. Nothing more.

```javascript
// Step 2: GREEN — make it pass with the simplest code
function slugify(text) {
  return text.toLowerCase().replace(/\s+/g, "-");
}

export { slugify };

// The test passes. We are "green."
```

**Refactor**: Clean up the code without changing behavior. The test protects you.

```javascript
// Step 3: REFACTOR — improve without breaking
function slugify(text) {
  return text
    .toLowerCase()
    .trim()
    .replace(/[^a-z0-9\s-]/g, "") // remove special chars
    .replace(/\s+/g, "-") // spaces to hyphens
    .replace(/-+/g, "-"); // collapse multiple hyphens
}

export { slugify };

// The original test still passes.
// Now add more tests for the new edge cases.
```

Then you repeat. Add another test for the next behavior. Watch it fail. Make it pass. Refactor.

## A full TDD cycle

Let us build a `PasswordValidator` with TDD, one test at a time.

```javascript
import { describe, it, expect } from "vitest";
import { validatePassword } from "./password.js";

// Cycle 1: passwords must be at least 8 characters
describe("validatePassword", () => {
  it("rejects passwords shorter than 8 characters", () => {
    const result = validatePassword("short");
    expect(result.isValid).toBe(false);
    expect(result.errors).toContain("Must be at least 8 characters");
  });

  it("accepts passwords with 8 or more characters", () => {
    const result = validatePassword("longEnough");
    expect(result.isValid).toBe(true);
  });
});
```

```javascript
// password.js — minimum implementation for cycle 1
function validatePassword(password) {
  const errors = [];

  if (password.length < 8) {
    errors.push("Must be at least 8 characters");
  }

  return { isValid: errors.length === 0, errors };
}

export { validatePassword };
```

```javascript
// Cycle 2: add a new requirement — must contain a number
describe("validatePassword", () => {
  // ...previous tests still here...

  it("rejects passwords without a number", () => {
    const result = validatePassword("NoNumbersHere");
    expect(result.isValid).toBe(false);
    expect(result.errors).toContain("Must contain at least one number");
  });

  it("accepts passwords with numbers", () => {
    const result = validatePassword("hasNumber1");
    expect(result.isValid).toBe(true);
  });
});
```

```javascript
// password.js — extend to satisfy cycle 2
function validatePassword(password) {
  const errors = [];

  if (password.length < 8) {
    errors.push("Must be at least 8 characters");
  }

  if (!/\d/.test(password)) {
    errors.push("Must contain at least one number");
  }

  return { isValid: errors.length === 0, errors };
}

export { validatePassword };
```

Each cycle adds one behavior. You never write code that is not required by a failing test.

## When TDD helps

TDD is at its best when:

- **Requirements are clear.** You know what the function should do before you start. TDD translates requirements directly into tests.
- **The logic is algorithmic.** Calculations, validations, parsers, and state machines benefit from the tight feedback loop.
- **You are building a library or utility.** Pure functions with defined inputs and outputs are ideal TDD targets.
- **You need to refactor legacy code.** Write tests that capture current behavior first, then refactor under their protection.

## When TDD does not help

TDD is the wrong tool when:

- **You are exploring.** If you do not know what you are building yet — a prototype, a spike, an experiment — writing tests for undefined behavior wastes time. Explore first, test after.
- **The work is mostly visual.** CSS layouts, animations, and design tweaks are better verified by eye and screenshot tests than by TDD.
- **You are integrating with an unfamiliar API.** You need to understand the API before you can write meaningful tests for it. Spike first, then TDD the integration layer.
- **The test is harder to write than the code.** If your test requires elaborate mocking that dwarfs the implementation, TDD adds cost without proportional value.

The pragmatic approach: use TDD for business logic and algorithmic code. Use test-after for UI, integrations, and exploratory work.

## What you learned

- TDD follows the **red-green-refactor** cycle: write a failing test, make it pass with minimal code, then clean up.
- TDD works best for **business logic, algorithms, and validation** where requirements are clear and functions are pure.
- TDD is **not a universal rule** — skip it during prototyping, visual work, and API exploration where test-after is more practical.
