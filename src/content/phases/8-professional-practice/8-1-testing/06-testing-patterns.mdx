## Why this matters

As your test suite grows, the patterns you use to organize test data, generate inputs, and verify outputs matter as much as the tests themselves. Bad patterns lead to brittle tests that break on every refactor and take longer to maintain than the code they protect. Good patterns — fixtures, factories, snapshots, and property-based testing — keep your tests readable, reliable, and scalable. These are the techniques that separate a test suite you trust from one you dread.

## Fixtures

A fixture is a reusable piece of test data. Instead of constructing objects from scratch in every test, you define them once and reuse them.

```javascript
// fixtures/users.js — centralized test data
const fixtures = {
  validUser: {
    id: "user-1",
    name: "Ada Lovelace",
    email: "ada@example.com",
    role: "member",
    createdAt: new Date("2024-01-15"),
  },
  adminUser: {
    id: "user-2",
    name: "Grace Hopper",
    email: "grace@example.com",
    role: "admin",
    createdAt: new Date("2023-06-01"),
  },
  incompleteUser: {
    id: "user-3",
    name: null,
    email: "no-name@example.com",
    role: "member",
    createdAt: new Date("2024-03-10"),
  },
};

// In your test file
import { describe, it, expect } from "vitest";

describe("formatUserDisplay", () => {
  it("shows name and role for valid users", () => {
    const result = formatUserDisplay(fixtures.validUser);
    expect(result).toBe("Ada Lovelace (member)");
  });

  it("shows email as fallback when name is missing", () => {
    const result = formatUserDisplay(fixtures.incompleteUser);
    expect(result).toBe("no-name@example.com (member)");
  });
});
```

Fixtures make tests self-documenting. `fixtures.adminUser` is immediately clear. An inline object with 8 properties is not.

## Factories

Factories generate test data dynamically. They are more flexible than fixtures because you can override specific fields per test.

```javascript
// factories/user.js
let idCounter = 0;

function createUser(overrides = {}) {
  idCounter += 1;
  return {
    id: `user-${idCounter}`,
    name: `User ${idCounter}`,
    email: `user${idCounter}@example.com`,
    role: "member",
    isActive: true,
    createdAt: new Date(),
    ...overrides,
  };
}

// In your tests — override only what matters for each test
describe("deactivateUser", () => {
  it("sets isActive to false", () => {
    const user = createUser({ isActive: true });
    const result = deactivateUser(user);
    expect(result.isActive).toBe(false);
  });

  it("preserves all other fields", () => {
    const user = createUser({ name: "Original", role: "admin" });
    const result = deactivateUser(user);
    expect(result.name).toBe("Original");
    expect(result.role).toBe("admin");
  });
});
```

The factory pattern shines when you have many tests that need slightly different versions of the same data structure. Each test only specifies the fields it cares about.

## Snapshot testing

Snapshot tests capture the output of a function and compare it against a stored reference. If the output changes, the test fails — and you decide if the change is intentional or a regression.

```javascript
import { describe, it, expect } from "vitest";

function generateReport(orders) {
  return {
    totalOrders: orders.length,
    totalRevenue: orders.reduce((sum, o) => sum + o.amount, 0),
    averageOrder: orders.length ? orders.reduce((sum, o) => sum + o.amount, 0) / orders.length : 0,
    currencies: [...new Set(orders.map((o) => o.currency))],
  };
}

describe("generateReport", () => {
  it("matches the expected output structure", () => {
    const orders = [
      { id: 1, amount: 100, currency: "USD" },
      { id: 2, amount: 200, currency: "USD" },
      { id: 3, amount: 150, currency: "EUR" },
    ];

    const report = generateReport(orders);

    // Snapshot test — Vitest stores the result in a .snap file
    // On first run, it creates the snapshot.
    // On subsequent runs, it compares against the stored version.
    expect(report).toMatchSnapshot();
  });
});
```

Use snapshots for complex output structures where writing individual assertions would be tedious. Avoid them for simple values — `expect(x).toBe(5)` is clearer than a snapshot of `5`.

**When to update snapshots:** If you intentionally change the output, run `vitest --update` to regenerate snapshots. Always review snapshot diffs in code review — a snapshot update can hide bugs if rubber-stamped.

## Property-based testing

Instead of testing specific inputs, property-based testing generates random inputs and checks that properties always hold.

```javascript
import { describe, it, expect } from "vitest";

// A property-based approach using manual random generation
function testProperty(fn, times = 100) {
  for (let i = 0; i < times; i++) {
    fn();
  }
}

function reverseString(str) {
  return str.split("").reverse().join("");
}

describe("reverseString properties", () => {
  it("reversing twice returns the original", () => {
    testProperty(() => {
      // Generate a random string
      const length = Math.floor(Math.random() * 50);
      const chars = "abcdefghijklmnopqrstuvwxyz0123456789";
      let input = "";
      for (let i = 0; i < length; i++) {
        input += chars[Math.floor(Math.random() * chars.length)];
      }

      // Property: reverse(reverse(x)) === x
      expect(reverseString(reverseString(input))).toBe(input);
    });
  });

  it("preserves string length", () => {
    testProperty(() => {
      const length = Math.floor(Math.random() * 100);
      const input = "a".repeat(length);

      // Property: reverse does not change length
      expect(reverseString(input).length).toBe(input.length);
    });
  });
});
```

Property-based testing finds edge cases you would never think of — empty strings, strings with unicode, extremely long inputs. Libraries like `fast-check` generate inputs more intelligently than random, shrinking failing cases to the simplest reproduction.

**Good properties to test:**

- Roundtrip: `decode(encode(x)) === x`
- Idempotence: `format(format(x)) === format(x)`
- Invariants: sorting never changes array length
- Commutativity: `add(a, b) === add(b, a)`

## What you learned

- **Fixtures** provide reusable, named test data that makes tests self-documenting and consistent.
- **Factories** generate data with sensible defaults and per-test overrides, scaling better than fixtures for large test suites.
- **Property-based testing** generates random inputs to verify invariants — roundtrip, idempotence, length preservation — catching edge cases you would never write by hand.
