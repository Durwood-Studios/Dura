## Why this matters

Knowing testing concepts is not the same as writing tests. The gap between understanding the testing pyramid and actually producing a well-structured test suite is where most developers stall. This lesson walks through writing a complete test suite for a real-world module — a task manager — covering unit tests for pure logic, integration tests for data flow, and edge cases that would bite you in production. By the end, you will have a repeatable process you can apply to any codebase.

## The module under test

Here is a task manager module with enough complexity to warrant a serious test suite.

```javascript
// taskManager.js
function createTaskManager() {
  const tasks = [];
  let nextId = 1;

  function addTask(title, priority = "medium") {
    if (!title || typeof title !== "string" || !title.trim()) {
      throw new Error("Task title is required");
    }

    const validPriorities = ["low", "medium", "high"];
    if (!validPriorities.includes(priority)) {
      throw new Error("Priority must be low, medium, or high");
    }

    const task = {
      id: nextId++,
      title: title.trim(),
      priority,
      completed: false,
      createdAt: new Date(),
    };

    tasks.push(task);
    return task;
  }

  function completeTask(id) {
    const task = tasks.find((t) => t.id === id);
    if (!task) throw new Error("Task not found");
    if (task.completed) throw new Error("Task already completed");
    task.completed = true;
    task.completedAt = new Date();
    return task;
  }

  function getStats() {
    const total = tasks.length;
    const completed = tasks.filter((t) => t.completed).length;
    const pending = total - completed;
    const completionRate = total > 0 ? Math.round((completed / total) * 100) : 0;
    return { total, completed, pending, completionRate };
  }

  function getTasksByPriority(priority) {
    return tasks.filter((t) => t.priority === priority && !t.completed);
  }

  return { addTask, completeTask, getStats, getTasksByPriority };
}
```

Before writing tests, study the code. Identify the public API, the validation rules, the edge cases, and the possible error states. That analysis drives your test plan.

## Step 1: Test the happy paths

Start with the straightforward cases — the behavior users expect when everything goes right.

```javascript
import { describe, it, expect, beforeEach } from "vitest";

describe("TaskManager", () => {
  let manager;

  beforeEach(() => {
    manager = createTaskManager();
  });

  describe("addTask", () => {
    it("creates a task with the given title and default priority", () => {
      const task = manager.addTask("Write tests");

      expect(task.id).toBe(1);
      expect(task.title).toBe("Write tests");
      expect(task.priority).toBe("medium");
      expect(task.completed).toBe(false);
      expect(task.createdAt).toBeInstanceOf(Date);
    });

    it("accepts a custom priority", () => {
      const task = manager.addTask("Fix bug", "high");
      expect(task.priority).toBe("high");
    });

    it("assigns incrementing IDs", () => {
      const first = manager.addTask("Task 1");
      const second = manager.addTask("Task 2");
      expect(second.id).toBe(first.id + 1);
    });
  });

  describe("completeTask", () => {
    it("marks a task as completed", () => {
      const task = manager.addTask("Ship feature");
      const completed = manager.completeTask(task.id);

      expect(completed.completed).toBe(true);
      expect(completed.completedAt).toBeInstanceOf(Date);
    });
  });
});
```

## Step 2: Test the error paths

Every `throw` in the source code should have a corresponding test.

```javascript
describe("addTask — validation", () => {
  it("rejects empty string titles", () => {
    expect(() => manager.addTask("")).toThrow("Task title is required");
  });

  it("rejects null titles", () => {
    expect(() => manager.addTask(null)).toThrow("Task title is required");
  });

  it("rejects whitespace-only titles", () => {
    expect(() => manager.addTask("   ")).toThrow("Task title is required");
  });

  it("rejects invalid priorities", () => {
    expect(() => manager.addTask("Task", "urgent")).toThrow(
      "Priority must be low, medium, or high"
    );
  });
});

describe("completeTask — errors", () => {
  it("throws when task ID does not exist", () => {
    expect(() => manager.completeTask(999)).toThrow("Task not found");
  });

  it("throws when completing an already completed task", () => {
    const task = manager.addTask("Done task");
    manager.completeTask(task.id);

    expect(() => manager.completeTask(task.id)).toThrow("Task already completed");
  });
});
```

## Step 3: Test computed state

Functions that aggregate data — like `getStats` — need tests for empty, partial, and full states.

```javascript
describe("getStats", () => {
  it("returns zeros when no tasks exist", () => {
    const stats = manager.getStats();

    expect(stats).toEqual({
      total: 0,
      completed: 0,
      pending: 0,
      completionRate: 0,
    });
  });

  it("calculates stats with a mix of completed and pending", () => {
    manager.addTask("Task 1");
    manager.addTask("Task 2");
    manager.addTask("Task 3");
    manager.completeTask(1);

    const stats = manager.getStats();

    expect(stats.total).toBe(3);
    expect(stats.completed).toBe(1);
    expect(stats.pending).toBe(2);
    expect(stats.completionRate).toBe(33);
  });

  it("shows 100% when all tasks are completed", () => {
    manager.addTask("Only task");
    manager.completeTask(1);

    expect(manager.getStats().completionRate).toBe(100);
  });
});
```

## Step 4: Test filtering

```javascript
describe("getTasksByPriority", () => {
  it("returns only pending tasks of the given priority", () => {
    manager.addTask("Low 1", "low");
    manager.addTask("High 1", "high");
    manager.addTask("Low 2", "low");

    const lowTasks = manager.getTasksByPriority("low");

    expect(lowTasks).toHaveLength(2);
    expect(lowTasks.every((t) => t.priority === "low")).toBe(true);
  });

  it("excludes completed tasks", () => {
    const task = manager.addTask("Done", "high");
    manager.completeTask(task.id);
    manager.addTask("Pending", "high");

    const highTasks = manager.getTasksByPriority("high");

    expect(highTasks).toHaveLength(1);
    expect(highTasks[0].title).toBe("Pending");
  });

  it("returns an empty array when no tasks match", () => {
    manager.addTask("Only medium");

    expect(manager.getTasksByPriority("high")).toEqual([]);
  });
});
```

## The test plan checklist

When writing tests for any module, follow this checklist:

1. **Happy paths** — Does it work correctly with valid input?
2. **Error paths** — Does it reject invalid input with clear errors?
3. **Edge cases** — Empty collections, zero values, boundary conditions
4. **State transitions** — Does state change correctly across multiple operations?
5. **Computed values** — Do aggregations and calculations produce correct results?

## What you learned

- Start every test suite with **happy paths**, then add **error paths**, **edge cases**, and **state transition** tests systematically.
- Use `beforeEach` to create a **fresh instance** for every test, preventing state leakage between test cases.
- Follow the **five-point checklist** — happy paths, error paths, edge cases, state transitions, computed values — to build comprehensive coverage for any module.
