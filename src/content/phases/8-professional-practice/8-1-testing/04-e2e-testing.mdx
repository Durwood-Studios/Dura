## Why this matters

End-to-end tests are the closest thing to watching a real user interact with your application. They launch an actual browser, click actual buttons, fill actual forms, and verify what appears on screen. When an E2E test passes, you know the entire stack — frontend, backend, database, routing — works together. They are slow, expensive, and sometimes flaky. But for critical user journeys like sign-up, checkout, and authentication, nothing else provides the same level of confidence.

## Playwright fundamentals

Playwright is a testing framework that controls real browsers. It supports Chromium, Firefox, and WebKit from a single API.

```javascript
// tests/login.spec.js
import { test, expect } from "@playwright/test";

test.describe("Login flow", () => {
  test("allows a user to log in with valid credentials", async ({ page }) => {
    // Navigate to the login page
    await page.goto("/login");

    // Fill in the form
    await page.fill('[data-testid="email-input"]', "user@example.com");
    await page.fill('[data-testid="password-input"]', "securePass123");

    // Click the submit button
    await page.click('[data-testid="login-button"]');

    // Wait for navigation and verify the result
    await expect(page).toHaveURL("/dashboard");
    await expect(page.getByRole("heading", { name: "Welcome back" })).toBeVisible();
  });

  test("shows an error for invalid credentials", async ({ page }) => {
    await page.goto("/login");
    await page.fill('[data-testid="email-input"]', "wrong@example.com");
    await page.fill('[data-testid="password-input"]', "badPassword");
    await page.click('[data-testid="login-button"]');

    await expect(page.getByText("Invalid email or password")).toBeVisible();
    await expect(page).toHaveURL("/login");
  });
});
```

Playwright waits for elements automatically — no manual `sleep()` calls. It retries assertions until they pass or time out. This makes tests more reliable than older tools like Selenium.

## Selecting elements

How you select elements determines how fragile your tests are.

```javascript
// BAD: coupled to CSS structure — breaks when styling changes
await page.click(".btn.btn-primary.submit-form");

// BAD: coupled to DOM hierarchy — breaks on any restructure
await page.click("div > form > div:nth-child(3) > button");

// GOOD: data-testid is stable and intentional
await page.click('[data-testid="submit-button"]');

// GOOD: role-based selectors match how users perceive the page
await page.getByRole("button", { name: "Submit" }).click();

// GOOD: text selectors for visible content
await page.getByText("Sign up for free").click();
```

Use `data-testid` for elements that have no accessible role or unique text. Use role-based selectors (`getByRole`) when possible — they also verify your accessibility markup is correct.

## Testing user flows

E2E tests should model complete user journeys, not isolated clicks.

```javascript
import { test, expect } from "@playwright/test";

test("new user can sign up and create their first project", async ({ page }) => {
  // Step 1: Sign up
  await page.goto("/signup");
  await page.fill('[data-testid="name-input"]', "Ada Lovelace");
  await page.fill('[data-testid="email-input"]', "ada@example.com");
  await page.fill('[data-testid="password-input"]', "StrongPass!99");
  await page.click('[data-testid="signup-button"]');

  // Step 2: Verify redirect to onboarding
  await expect(page).toHaveURL("/onboarding");
  await expect(page.getByText("Welcome, Ada")).toBeVisible();

  // Step 3: Create a project
  await page.click('[data-testid="create-project-button"]');
  await page.fill('[data-testid="project-name"]', "My First App");
  await page.click('[data-testid="save-project"]');

  // Step 4: Verify project appears in dashboard
  await expect(page).toHaveURL(/\/projects\//);
  await expect(page.getByText("My First App")).toBeVisible();
});
```

Keep E2E tests focused on critical paths. If your application has a checkout flow that generates revenue, that needs an E2E test. If a button changes color on hover, that does not.

## Running E2E in CI

E2E tests belong in your CI pipeline so they run on every pull request.

```javascript
// playwright.config.js
export default {
  testDir: "./tests",
  timeout: 30000,
  retries: 2, // Retry flaky tests in CI
  use: {
    baseURL: "http://localhost:3000",
    screenshot: "only-on-failure",
    trace: "retain-on-failure",
  },
  webServer: {
    command: "npm run start",
    port: 3000,
    reuseExistingServer: !process.env.CI,
  },
  projects: [
    { name: "chromium", use: { browserName: "chromium" } },
    { name: "firefox", use: { browserName: "firefox" } },
  ],
};
```

Key CI settings: `retries: 2` handles transient failures. `screenshot: "only-on-failure"` captures evidence when something breaks. `trace: "retain-on-failure"` records a full timeline you can replay in the Playwright Trace Viewer to debug failures without re-running the test.

## Avoiding flaky tests

Flaky tests — tests that sometimes pass and sometimes fail — are worse than no tests. They erode trust in the test suite until the team ignores failures entirely.

```javascript
// FLAKY: hardcoded wait — fails if server is slow, wastes time if fast
await page.click('[data-testid="save"]');
await page.waitForTimeout(3000);
await expect(page.getByText("Saved")).toBeVisible();

// STABLE: wait for the specific condition you care about
await page.click('[data-testid="save"]');
await expect(page.getByText("Saved")).toBeVisible({ timeout: 10000 });

// FLAKY: depends on animation timing
await page.click('[data-testid="menu-toggle"]');
await expect(page.getByRole("menu")).toBeVisible();

// STABLE: wait for the final state, not the animation
await page.click('[data-testid="menu-toggle"]');
await expect(page.getByRole("menu")).toBeVisible({ timeout: 5000 });
```

Never use `waitForTimeout`. Always wait for a specific condition — a visible element, a URL change, or a network response.

## What you learned

- **Playwright** controls real browsers to test complete user flows — sign-up, checkout, authentication — end to end.
- Use **data-testid and role-based selectors** instead of CSS classes or DOM hierarchy to avoid brittle tests.
- Run E2E tests in **CI with retries and failure artifacts** (screenshots, traces) to catch regressions on every pull request.
