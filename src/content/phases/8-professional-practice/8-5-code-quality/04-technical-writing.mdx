## Why this matters

Developers write more prose than they expect. Pull request descriptions, design documents, incident postmortems, Slack messages explaining a tricky bug, onboarding guides, commit messages, code comments — the written word is everywhere in software engineering. Yet most computer science programs never teach writing.

Bad technical writing wastes time at scale. A confusing design document forces a 30-minute meeting that should have been unnecessary. A vague incident report means the next on-call engineer learns nothing. A poorly written error message sends a user to support instead of to the fix. Every unclear sentence multiplies across every reader.

Good technical writing is not about elegant prose. It is about clarity, precision, and respect for the reader's time. You can learn it with a small set of concrete principles and deliberate practice.

## Principles of clear technical writing

```javascript
// PRINCIPLE 1: Lead with the conclusion
// Bad: "After investigating the issue for several hours, examining logs,
//       and testing various hypotheses, we determined that the root cause
//       was a race condition in the session middleware."
// Good: "Root cause: race condition in session middleware."
//       Then explain the investigation if the reader wants detail.

// PRINCIPLE 2: Use active voice
// Bad:  "The request is validated by the middleware."
// Good: "The middleware validates the request."
// Active voice is shorter, clearer, and names the actor.

// PRINCIPLE 3: One idea per sentence
// Bad:  "The function parses the input, validates it against the schema,
//        transforms the data, and if any step fails it throws an error
//        that the caller should catch."
// Good: "The function parses the input and validates it against the schema.
//        It then transforms the valid data. If any step fails, it throws
//        an error. The caller must catch this error."
```

## Structure for scannability

Developers scan before they read. Structure your writing so scanning works.

```javascript
// TECHNIQUE 1: Use headings as a table of contents
// A reader should understand the document's scope from headings alone.

const GOOD_HEADINGS = [
  "Problem", // What is broken or missing
  "Proposed solution", // What you want to build
  "Alternatives", // What you considered and rejected
  "Migration plan", // How to get from here to there
  "Open questions", // What you still do not know
];

// TECHNIQUE 2: Front-load paragraphs
// The first sentence of each paragraph should be the point.
// Supporting details follow. A reader who only reads first sentences
// should still understand the document.

// TECHNIQUE 3: Use lists for parallel items
// Bad prose:  "The system supports JSON and XML input formats.
//              It also handles CSV for bulk imports and can process
//              YAML for configuration files."
// Good list:  "Supported input formats:
//              - JSON (all endpoints)
//              - XML (legacy endpoints only)
//              - CSV (bulk import)
//              - YAML (configuration)"
```

## Writing error messages

Error messages are technical writing under extreme constraints. The user is frustrated, the message is short, and it must guide them to a fix.

```javascript
// PATTERN: What happened + Why + What to do

// Bad error messages:
const BAD_ERRORS = [
  "Error", // Says nothing
  "Something went wrong", // Says almost nothing
  "Invalid input", // Which input? Why invalid?
  "Error code 0x8004005E", // Only meaningful to the developer
];

// Good error messages:
const GOOD_ERRORS = [
  "Email address is required. Please enter a valid email to continue.",
  "Password must be at least 8 characters. Yours has 5.",
  "Could not connect to the server. Check your internet connection and try again.",
  "This file is too large (52 MB). The maximum upload size is 25 MB.",
];

// Implementation pattern:
function validateAge(age) {
  if (typeof age !== "number") {
    throw new TypeError(`Age must be a number. Received ${typeof age}.`);
  }
  if (age < 0 || age > 150) {
    throw new RangeError(`Age must be between 0 and 150. Received ${age}.`);
  }
  return age;
}
```

## Writing commit messages and PR descriptions

```javascript
// Commit messages: what and why, not how
// Bad:  "fix bug"
// Bad:  "Updated the user validation function to check for null"
// Good: "fix(auth): reject null email before database query"
// Good: "feat(shipping): add international rate multiplier for EU destinations"

// PR descriptions: context for the reviewer
const PR_TEMPLATE = `
## What
Add rate limiting to the /api/login endpoint.

## Why
Brute-force attacks were hitting this endpoint at 500 req/s.
Rate limiting to 10 req/min/IP blocks automated attacks
while allowing normal users through.

## How to test
1. Start the dev server
2. Run: for i in {1..15}; do curl -X POST localhost:3000/api/login; done
3. Observe 429 responses after the 10th request

## Notes
Rate limit config is in env vars so ops can tune without a deploy.
`;
```

## Eliminating filler

Technical writing improves most by subtraction. Remove words that add no information.

```javascript
// Cut filler phrases:
const FILLER_MAP = {
  "In order to": "To",
  "Due to the fact that": "Because",
  "At this point in time": "Now",
  "It should be noted that": "", // Just state the thing
  Basically: "", // If it is basic, it does not need the word
  "As you can see": "", // Let the reader see for themselves
  "It is important to note": "", // If it is important, the content shows it
};

// Before: "In order to fix this issue, it is important to note that
//          the database connection pool is basically exhausted due to
//          the fact that connections are not being released."
//
// After:  "The connection pool is exhausted because connections are
//          not released after use."
// 35 words → 13 words. Same information. Twice as clear.
```

## What you learned

- Lead with the conclusion, use active voice, and keep one idea per sentence.
- Structure documents for scanning with headings, front-loaded paragraphs, and lists.
- Write error messages that tell the user what happened, why, and what to do next.
