## Why this matters

Code review is the single highest-leverage activity on a software team. Studies at Google and Microsoft consistently show that reviewed code has 60-80% fewer defects than unreviewed code. But code review is not just about catching bugs. It spreads knowledge across the team, enforces consistency, mentors junior developers, and creates a shared ownership culture where nobody is the sole expert on any system.

Most developers learn to write code in isolation. School projects, tutorials, and side projects rarely involve another human reading your work. Then you join a team and discover that every line you write will be scrutinized by a peer before it ships. That transition is jarring if you have never practiced giving and receiving feedback on code.

The skill of reviewing is distinct from the skill of writing. A great writer can be a poor reviewer if they focus on style nitpicks instead of logic errors. A mediocre writer can be an excellent reviewer if they ask the right questions and catch the right assumptions. This lesson teaches both sides: how to review someone else's code, and how to respond when someone reviews yours.

## What to look for in a review

Professional reviewers use a mental checklist. The order matters — catch the structural problems before the cosmetic ones.

```javascript
// 1. CORRECTNESS — Does it work?
// Look for logic errors, off-by-one bugs, missing awaits
async function fetchUser(id) {
  // Bug: no error handling. What if the fetch fails?
  const response = await fetch(`/api/users/${id}`);
  // Bug: no status check. 404 returns a valid Response object.
  const data = await response.json();
  return data;
}

// Reviewed version:
async function fetchUser(id) {
  const response = await fetch(`/api/users/${id}`);
  if (!response.ok) {
    throw new Error(`Failed to fetch user ${id}: ${response.status}`);
  }
  return response.json();
}
```

```javascript
// 2. READABILITY — Can you understand it in six months?
// Bad: clever but obscure
const r = xs.reduce((a, x) => (x.s === "a" ? { ...a, c: a.c + 1 } : a), { c: 0 });

// Better: clear names, obvious intent
const activeCount = users.filter((user) => user.status === "active").length;
```

```javascript
// 3. EDGE CASES — What breaks with unexpected input?
function divide(a, b) {
  return a / b; // What if b is 0? What if a or b is NaN?
}

function safeDivide(a, b) {
  if (typeof a !== "number" || typeof b !== "number") {
    throw new TypeError("Both arguments must be numbers");
  }
  if (b === 0) {
    throw new RangeError("Cannot divide by zero");
  }
  return a / b;
}
```

```javascript
// 4. DESIGN — Is the abstraction right?
// Review question: does this function do too many things?
function processOrder(order) {
  validateOrder(order);
  const tax = calculateTax(order);
  const total = order.subtotal + tax;
  sendConfirmationEmail(order.email, total);
  updateInventory(order.items);
  logAnalytics("order_completed", { total });
  return { total, tax };
}
// This function validates, calculates, emails, updates inventory,
// and logs analytics. That is five responsibilities. Each should
// be a separate step the caller orchestrates.
```

## Giving feedback that lands

The best review comments follow a pattern: observation, reasoning, suggestion.

```javascript
// Weak feedback:
// "Don't use var."

// Strong feedback:
// "Line 12: `var discount` is function-scoped, which means the variable
// exists even when neither branch executes — causing NaN on line 18.
// Consider `let discount = 0` so it has a safe default."

// Weak feedback:
// "This is confusing."

// Strong feedback:
// "The triple-nested ternary on line 45 is hard to follow.
// Could this be an if/else or a lookup object instead?
// Example: const ROLE_LABELS = { admin: 'Admin', user: 'User', guest: 'Guest' };"
```

## Receiving feedback gracefully

When someone critiques your code, they are not critiquing you. Separate your identity from your output. Ask clarifying questions. If you disagree, explain your reasoning — but be willing to change your mind. The goal is better code, not winning the argument.

## What you learned

- Review code in priority order: correctness, readability, edge cases, then design.
- Give feedback with the observation-reasoning-suggestion pattern.
- Receive feedback as a conversation about the code, not a judgment of you.
