## Why this matters

Open source is not charity work. It is how the modern software industry operates. Linux runs most servers. React, Next.js, and Node.js are all open source. The packages in your node_modules folder represent millions of hours of volunteer and sponsored labor. Understanding how open source works — how to contribute, how to maintain, how to choose a license — is a core professional skill.

Contributing to open source is also the most accessible form of real-world experience. You do not need a job to submit a pull request to a popular project. You do not need permission to fix a bug in a tool you use daily. Your contribution history is a public portfolio that speaks louder than any resume bullet point because it shows actual code, actual collaboration, and actual impact.

But open source has norms that differ from company code. Maintainers are often volunteers with limited time. Communication style matters. Understanding licensing protects you and your employer. Knowing how to file a useful issue, write a clean PR, and respond to feedback gracefully will determine whether your contributions are welcomed or ignored.

## Making your first contribution

```javascript
// Step 1: Find a project you actually use
// Your best first contribution is to a tool you already depend on.
// You understand its purpose, you can reproduce its bugs, and
// you are motivated to see it improve.

// Step 2: Start with documentation or tests
// Maintainers love contributions that improve docs or test coverage.
// These are lower risk than code changes and easier to review.

// Step 3: Read the CONTRIBUTING.md before writing any code
const CONTRIBUTING_CHECKLIST = [
  "Read CONTRIBUTING.md and CODE_OF_CONDUCT.md",
  "Check existing issues — is someone already working on this?",
  "Open an issue describing what you want to change BEFORE coding",
  "Fork the repo and create a feature branch",
  "Follow the project's code style and commit conventions",
  "Write tests for your changes",
  "Keep the PR small and focused on one change",
  "Respond to review feedback promptly and politely",
];
```

## Writing a good issue

```javascript
// A well-structured bug report saves the maintainer hours

const GOOD_ISSUE = {
  title: "TypeError when passing empty array to processItems()",
  sections: {
    description: "processItems crashes with TypeError instead of returning []",
    stepsToReproduce: [
      "Call processItems([])",
      "Observe: TypeError: Cannot read property 'map' of undefined",
      "Expected: returns an empty array",
    ],
    environment: {
      version: "2.4.1",
      node: "20.11.0",
      os: "Ubuntu 22.04",
    },
    possibleCause:
      "Line 42 in processor.js accesses items.data.map() " +
      "but items.data is undefined when the input array is empty.",
  },
};

// A bad issue: "It's broken. Please fix."
// That tells the maintainer nothing actionable.
```

## Writing a clean pull request

```javascript
// A PR that maintainers want to review

const CLEAN_PR = {
  title: "Fix TypeError on empty array input in processItems",
  body: `
    ## Problem
    processItems([]) throws TypeError because items.data is
    undefined when input is empty (fixes #234).

    ## Solution
    Add an early return for empty arrays before accessing
    items.data. Added two test cases for the empty input path.

    ## Testing
    - Added unit tests in processor.test.js
    - Tested manually with empty, single-item, and large arrays
    - All existing tests pass
  `,
  checklist: [
    "Single focused change",
    "Tests included",
    "Follows project code style",
    "PR description links to the issue",
  ],
};
```

## Understanding licenses

Licenses determine what others can do with your code and what you can do with theirs. Choosing wrong can have legal consequences.

```javascript
const COMMON_LICENSES = {
  MIT: {
    permissions: "Use, modify, distribute, commercial use",
    conditions: "Include license text",
    restrictions: "None — most permissive",
    bestFor: "Libraries you want maximum adoption for",
  },
  "Apache-2.0": {
    permissions: "Same as MIT plus patent grant",
    conditions: "Include license, state changes",
    restrictions: "Cannot use trademarks",
    bestFor: "Enterprise-friendly projects with patent concerns",
  },
  "GPL-3.0": {
    permissions: "Use, modify, distribute",
    conditions: "Derivative works must also be GPL (copyleft)",
    restrictions: "Cannot make proprietary derivatives",
    bestFor: "Projects that must stay open forever",
  },
  "AGPL-3.0": {
    permissions: "Same as GPL",
    conditions: "Network use triggers copyleft (even SaaS)",
    restrictions: "Server-side use must share source",
    bestFor: "Preventing SaaS companies from closing your code",
  },
};

// DURA uses AGPL-3.0: anyone can use and modify it,
// but they must share their changes — even over a network.
```

## Maintaining a project

```javascript
// If you create an open-source project, maintain it responsibly

const MAINTAINER_PRACTICES = {
  documentation: [
    "README with quick start and examples",
    "CONTRIBUTING.md with setup and PR guidelines",
    "CODE_OF_CONDUCT.md to set community expectations",
    "CHANGELOG.md tracking notable changes per version",
  ],
  communityHealth: [
    "Respond to issues within 48 hours, even if just to acknowledge",
    "Label issues (bug, feature, good-first-issue, help-wanted)",
    "Close stale issues with an explanation",
    "Thank contributors publicly",
  ],
  sustainability: [
    "Document bus factor — can someone else maintain this?",
    "Automate testing, linting, and releases with CI/CD",
    "Set boundaries — it is OK to say no to feature requests",
    "If you cannot maintain it, archive it honestly",
  ],
};
```

## What you learned

- Start contributing with documentation or tests in projects you already use.
- Write issues with reproduction steps and PRs with focused, tested changes.
- Understand licensing implications: MIT is permissive, GPL/AGPL enforce copyleft.
