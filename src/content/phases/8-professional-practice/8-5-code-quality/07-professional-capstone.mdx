## Why this matters

Individual skills are useful. Combined skills are powerful. A developer who can review code, document decisions, and present ideas clearly is worth far more than one who can only write functions. This capstone pulls together everything from the module — code review, refactoring, documentation, technical writing, open-source contribution, and professional communication — into three integrated exercises that mirror real professional work.

In your career, you will be asked to do all three of these tasks regularly. You will review a colleague's pull request on Monday, write an ADR on Wednesday, and present a technical proposal on Friday. The quality of your work in these areas determines how much influence you have on your team. Engineers who only write code reach a ceiling. Engineers who also communicate, document, and mentor break through it.

These exercises are deliberately open-ended. There is no single correct answer. The goal is to practice judgment — deciding what matters, what to say, and how to say it — in realistic scenarios.

## Exercise 1: Review a pull request

The following code is a pull request that adds a caching layer to an API client. Review it as you would in a professional setting: identify bugs, suggest improvements, and note what is done well.

```javascript
// PR #247: Add caching to API client
// Author: junior-dev
// Description: "Added caching so we don't hit the API so much"

const cache = {};

async function fetchWithCache(url, options = {}) {
  const cacheKey = url;

  if (cache[cacheKey]) {
    return cache[cacheKey];
  }

  const response = await fetch(url, options);
  const data = await response.json();

  cache[cacheKey] = data;

  return data;
}

// Usage
async function getUser(id) {
  return fetchWithCache(`/api/users/${id}`);
}

async function updateUser(id, updates) {
  return fetchWithCache(`/api/users/${id}`, {
    method: "PUT",
    body: JSON.stringify(updates),
  });
}
```

```javascript
// REVIEW COMMENTS you should identify:

// 1. BUG: PUT requests are being cached. The cache key is only the URL,
//    so updateUser() will return stale data from the GET cache instead
//    of actually sending the PUT request.
//    Fix: Only cache GET requests. Check options.method.

// 2. BUG: No cache expiration. Data is cached forever. Stale data
//    will be served indefinitely.
//    Fix: Store timestamps, add a TTL parameter.

// 3. BUG: No error handling. If fetch() fails or response.json() fails,
//    the error propagates unhandled. Worse, a failed response (404, 500)
//    gets cached as if it were valid data.
//    Fix: Check response.ok before caching.

// 4. DESIGN: Global mutable object (`const cache = {}`) will grow
//    without bound. In a long-running app, this is a memory leak.
//    Fix: Use a Map with a max size, or use LRU eviction.

// 5. POSITIVE: The function signature is clean and the caching logic
//    is extracted into a reusable wrapper. Good instinct for abstraction.

// A complete review includes both problems AND praise.
```

## Exercise 2: Write an ADR

Scenario: Your team needs to choose a state management approach for a new React application. The options are React Context, Redux, and Zustand. You chose Zustand.

```javascript
// ADR-012: Use Zustand for client-side state management

const ADR = {
  status: "Accepted",
  date: "2026-04-13",
  deciders: ["Tech Lead", "Senior Frontend Dev", "You"],

  context:
    "The application needs to share state across deeply nested components. " +
    "State includes user preferences, navigation state, and cached API responses. " +
    "The team has 3 frontend developers with React experience. " +
    "Bundle size and developer experience are priorities.",

  options: {
    reactContext: {
      pros: ["Built-in, no dependency", "Simple for small state"],
      cons: [
        "Re-renders all consumers on any state change",
        "Provider nesting becomes unwieldy at scale",
        "No built-in devtools for debugging state",
      ],
    },
    redux: {
      pros: ["Mature ecosystem", "Excellent devtools", "Large community"],
      cons: [
        "Significant boilerplate (actions, reducers, dispatch)",
        "14.4 kB bundle size (with Redux Toolkit)",
        "Steep learning curve for new team members",
      ],
    },
    zustand: {
      pros: [
        "1.1 kB bundle size — 13x smaller than Redux",
        "Minimal boilerplate — define a store in 5 lines",
        "Selective re-renders with selector functions",
        "Works outside React components (e.g., in utility functions)",
      ],
      cons: ["Smaller ecosystem than Redux", "Fewer tutorials and Stack Overflow answers"],
    },
  },

  decision:
    "Use Zustand. Its minimal API reduces onboarding time for the team, " +
    "its selector-based rendering avoids the performance pitfalls of Context, " +
    "and its bundle size aligns with our performance budget. " +
    "The smaller ecosystem is an acceptable tradeoff because our state " +
    "management needs are straightforward.",

  consequences: [
    "Positive: New developers can understand stores in under 30 minutes",
    "Positive: Bundle size stays under our 50 kB JS budget for state management",
    "Negative: Team must write custom middleware instead of using Redux ecosystem",
    "Accepted: If needs grow complex, we reassess — but migrating from Zustand is low cost",
  ],
};
```

## Exercise 3: Outline a tech presentation

A good tech presentation follows a structure: hook the audience, establish the problem, present the solution, demonstrate it, and close with a call to action.

```javascript
// Tech talk: "Why We Stopped Writing Tests Last and Started Writing Them First"

const PRESENTATION_OUTLINE = {
  hook: {
    duration: "2 minutes",
    content:
      "Show a production bug that cost 4 hours to fix. " +
      "Show the 3-line test that would have caught it in seconds. " +
      "Ask: 'How many of you write tests after the code is done?'",
  },

  problem: {
    duration: "5 minutes",
    content:
      "Tests written after code tend to test implementation, not behavior. " +
      "They are fragile, they get skipped under deadline pressure, and they " +
      "miss edge cases because the developer is already anchored to their solution.",
    evidence: "Show team metrics: bug rate before vs. after adopting TDD.",
  },

  solution: {
    duration: "8 minutes",
    content:
      "Walk through TDD: Red (write a failing test), Green (make it pass), " +
      "Refactor (clean up). Live-code a small feature using this cycle.",
    demo:
      "Build a password validator test-first in real time. " +
      "Show how the tests drive the design.",
  },

  objections: {
    duration: "3 minutes",
    content:
      "Address the top 3 pushbacks: 'It's slower' (short-term yes, " +
      "long-term no), 'Not everything is testable' (true — focus on logic), " +
      "'Management won't give us time' (show the bug-cost data).",
  },

  callToAction: {
    duration: "2 minutes",
    content:
      "Challenge: try TDD on your next bug fix. Just one. " +
      "Write the test that reproduces the bug, then fix it. " +
      "Share your experience in the team channel.",
  },
};

// Total: 20 minutes. Focused. Practical. Ends with a specific ask.
```

const cache = new Map();

async function fetchWithCache(url, options = {}) {
const method = (options.method ?? "GET").toUpperCase();
const cacheKey = url;

if (method === "GET" && cache.has(cacheKey)) {
const entry = cache.get(cacheKey);
const isExpired = Date.now() - entry.timestamp > CACHE_TTL_MS;
if (!isExpired) {
return entry.data;
}
cache.delete(cacheKey);
}

const response = await fetch(url, options);

if (!response.ok) {
throw new Error(\`Request failed: \${response.status} \${response.statusText}\`);
}

const data = await response.json();

if (method === "GET") {
cache.set(cacheKey, { data, timestamp: Date.now() });
}

return data;
}

console.log("Cache uses Map:", cache instanceof Map);
console.log("TTL set to 60 seconds:", CACHE_TTL_MS === 60000);`}

## What you learned

- PR reviews should identify bugs, design issues, and positive patterns — then suggest fixes with reasoning.
- ADRs document the context, alternatives, decision, and consequences so future teams understand why.
- Tech presentations follow hook, problem, solution, objections, and call to action for maximum impact.
