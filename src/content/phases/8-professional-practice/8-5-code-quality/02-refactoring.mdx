## Why this matters

Every codebase decays. Features get bolted on. Requirements shift. Shortcuts taken under deadline pressure become permanent fixtures. Refactoring is the discipline of improving code structure without changing its behavior — cleaning up the mess so the next feature is easier to build instead of harder.

Martin Fowler's original catalog lists over 60 named refactorings, but you will use about ten of them 90% of the time. The key insight is that refactoring is not a separate activity from feature work. You refactor continuously, in small steps, as part of your daily workflow. The "big rewrite" is almost always a mistake — it introduces new bugs, loses institutional knowledge embedded in the old code, and takes months longer than estimated.

The hardest part of refactoring is not the mechanics. It is recognizing when code needs it. That recognition comes from learning to smell trouble before it becomes a crisis. Code smells are patterns that suggest deeper structural problems. They are not bugs — the code works — but they make the code harder to change, test, and understand.

## Recognizing code smells

```javascript
// SMELL: Long function — does too many things
function processRegistration(formData) {
  // Validate (15 lines)
  // Hash password (5 lines)
  // Create user record (10 lines)
  // Send welcome email (8 lines)
  // Log analytics event (5 lines)
  // Return response (3 lines)
  // Total: 46 lines doing 6 different things
}

// SMELL: Duplicated logic — same pattern in multiple places
function formatUserName(user) {
  return `${user.firstName.trim()} ${user.lastName.trim()}`;
}
function formatAdminName(admin) {
  return `${admin.firstName.trim()} ${admin.lastName.trim()}`;
}

// SMELL: Deeply nested conditionals
function getAccess(user, resource) {
  if (user) {
    if (user.isActive) {
      if (user.role === "admin") {
        return "full";
      } else {
        if (resource.isPublic) {
          return "read";
        } else {
          return "none";
        }
      }
    }
  }
  return "none";
}
```

## Refactoring patterns

```javascript
// PATTERN 1: Extract Function
// Before: one giant function
function processRegistration(formData) {
  // 46 lines of mixed concerns
}

// After: orchestrator with focused helpers
function processRegistration(formData) {
  const validated = validateRegistration(formData);
  const user = createUserRecord(validated);
  sendWelcomeEmail(user.email);
  logEvent("registration_complete", { userId: user.id });
  return { success: true, userId: user.id };
}
```

```javascript
// PATTERN 2: Replace Nested Conditionals with Guard Clauses
// Before: deeply nested
function getAccess(user, resource) {
  if (user) {
    if (user.isActive) {
      if (user.role === "admin") {
        return "full";
      } else {
        if (resource.isPublic) {
          return "read";
        }
      }
    }
  }
  return "none";
}

// After: flat and scannable
function getAccess(user, resource) {
  if (!user) return "none";
  if (!user.isActive) return "none";
  if (user.role === "admin") return "full";
  if (resource.isPublic) return "read";
  return "none";
}
```

```javascript
// PATTERN 3: Replace Magic Values with Named Constants
// Before
if (password.length < 8) {
  throw new Error("Too short");
}

// After
const MIN_PASSWORD_LENGTH = 8;
if (password.length < MIN_PASSWORD_LENGTH) {
  throw new Error(`Password must be at least ${MIN_PASSWORD_LENGTH} characters`);
}
```

```javascript
// PATTERN 4: Replace Conditionals with a Lookup Object
// Before
function getStatusLabel(status) {
  if (status === "active") return "Active";
  if (status === "inactive") return "Inactive";
  if (status === "pending") return "Pending Review";
  if (status === "banned") return "Banned";
  return "Unknown";
}

// After
const STATUS_LABELS = {
  active: "Active",
  inactive: "Inactive",
  pending: "Pending Review",
  banned: "Banned",
};

function getStatusLabel(status) {
  return STATUS_LABELS[status] ?? "Unknown";
}
```

## When to refactor

Refactor when you are about to add a feature and the current structure makes it hard. Refactor when you find a bug caused by confusing code. Refactor when onboarding a teammate reveals that nobody can explain what a module does. Do not refactor code that works, that nobody needs to change, and that nobody needs to understand. Working ugly code is better than broken pretty code.

## What you learned

- Code smells are structural patterns that signal future maintenance problems.
- Core refactoring moves: extract function, guard clauses, named constants, lookup objects.
- Refactor in small steps during normal work, not as a separate big-bang rewrite.
