## Why this matters

Code without documentation is a puzzle with missing pieces. You might solve it, but it takes far longer than it should. Documentation is not a chore you do after the real work — it is part of the real work. A function without a JSDoc comment, a project without a README, a decision without a recorded rationale: each creates invisible friction that compounds over months and years.

The Divio documentation framework identifies four distinct types of documentation, each serving a different need at a different moment. Most projects fail because they mix these types together or only write one kind. Understanding the framework means you write the right documentation for the right audience at the right time.

Good documentation has a surprising property: writing it improves the code. When you cannot explain a function's purpose in one sentence, the function does too much. When you cannot describe the API without caveats and exceptions, the API design is flawed. Documentation is a mirror that reveals design problems.

## The Divio framework

The framework divides all documentation into four categories based on two axes: learning vs. working, and practical vs. theoretical.

```javascript
// The four types of documentation:
const DIVIO_FRAMEWORK = {
  tutorials: {
    purpose: "Learning-oriented",
    audience: "Newcomers following step-by-step",
    example: "Build your first API endpoint in 10 minutes",
  },
  howToGuides: {
    purpose: "Task-oriented",
    audience: "Practitioners solving a specific problem",
    example: "How to add authentication to an existing route",
  },
  reference: {
    purpose: "Information-oriented",
    audience: "Practitioners who need exact details",
    example: "API endpoint list with parameters and return types",
  },
  explanation: {
    purpose: "Understanding-oriented",
    audience: "Anyone who wants to know WHY",
    example: "Why we chose PostgreSQL over MongoDB",
  },
};
```

## Writing a README

Every project needs a README. It is the first file a new developer reads. A strong README answers five questions in under two minutes.

````javascript
// README structure — the essential five sections
const README_SECTIONS = [
  "What is this?", // One paragraph. What problem does it solve?
  "Quick start", // Clone, install, run — under 5 commands
  "Usage examples", // Show, don't tell. Real code snippets
  "Configuration", // Environment variables, options, defaults
  "Contributing", // How to set up the dev environment and submit changes
];

// Example README opening:
// # Taskflow
// A lightweight task queue for Node.js that persists jobs to SQLite.
//
// ## Quick start
// ```bash
// npm install taskflow
// ```
// ```javascript
// import { createQueue } from "taskflow";
// const queue = createQueue("./jobs.db");
// queue.add("send-email", { to: "user@example.com" });
// ```
````

## Architecture Decision Records

An ADR captures WHY a decision was made, not just what was decided. Six months from now, when someone asks "why did we use SQLite instead of Redis?", the ADR answers them without a meeting.

```javascript
// ADR template — keep it short and honest
const ADR_TEMPLATE = `
# ADR-001: Use SQLite for job persistence

## Status
Accepted

## Context
We need to persist background jobs across server restarts.
The team has experience with PostgreSQL and Redis.
The application runs on a single server.

## Decision
Use SQLite because it requires no external service,
supports our throughput needs (< 1000 jobs/hour),
and simplifies deployment to a single binary.

## Consequences
- Positive: Zero infrastructure overhead, easy backups
- Negative: Cannot scale horizontally without migration
- Accepted risk: If we outgrow single-server, we migrate to PostgreSQL
`;
```

## JSDoc for functions

JSDoc lives next to the code it describes. It documents the contract — what goes in, what comes out, and what can go wrong.

```javascript
/**
 * Calculate the shipping cost for an order based on weight and destination.
 *
 * Domestic orders use a flat rate per kilogram. International orders
 * apply a 2.5x multiplier plus a fixed customs processing fee.
 *
 * @param {Object[]} items - Array of items with a weight property in kg
 * @param {number} items[].weight - Weight in kilograms (must be positive)
 * @param {string} destination - Either "domestic" or "international"
 * @returns {number} Total shipping cost in USD
 * @throws {TypeError} If items is not an array
 * @throws {RangeError} If any item weight is negative
 *
 * @example
 * calculateShipping([{ weight: 2 }, { weight: 3 }], "domestic");
 * // => 2.50
 */
function calculateShipping(items, destination) {
  if (!Array.isArray(items)) {
    throw new TypeError("items must be an array");
  }

  const RATE_PER_KG = 0.5;
  const INTERNATIONAL_MULTIPLIER = 2.5;
  const CUSTOMS_FEE = 15.0;

  const totalWeight = items.reduce((sum, item) => {
    if (item.weight < 0) throw new RangeError("Weight cannot be negative");
    return sum + item.weight;
  }, 0);

  const baseCost = totalWeight * RATE_PER_KG;

  if (destination === "international") {
    return baseCost * INTERNATIONAL_MULTIPLIER + CUSTOMS_FEE;
  }
  return baseCost;
}
```

## API documentation

API docs are reference material. They must be complete, accurate, and scannable. Every endpoint needs its method, path, parameters, request body shape, response shape, and error codes.

```javascript
// Documenting an API endpoint
const API_DOC = {
  method: "POST",
  path: "/api/users",
  description: "Create a new user account",
  headers: { "Content-Type": "application/json" },
  body: {
    email: "string (required) — must be unique",
    password: "string (required) — minimum 8 characters",
    name: "string (optional) — display name",
  },
  responses: {
    201: { id: "string", email: "string", createdAt: "ISO 8601" },
    400: { error: "Validation failed", details: ["email is required"] },
    409: { error: "Email already exists" },
  },
};
```

## What you learned

- The Divio framework organizes documentation into tutorials, how-to guides, reference, and explanation.
- READMEs answer five questions: what, quick start, usage, configuration, contributing.
- ADRs record why decisions were made, not just what was decided.
