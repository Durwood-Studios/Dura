## Why this matters

GitHub Actions is the CI/CD platform built directly into GitHub — the same place your code already lives. There is no separate CI server to configure, no webhook plumbing to wire up. You write a YAML file, push it, and your pipeline runs. Understanding workflows, jobs, steps, matrix strategies, and caching is the difference between a pipeline that takes 15 minutes and one that takes 3.

## Workflows, jobs, and steps

A GitHub Actions pipeline has three levels of organization:

- **Workflow** — a YAML file in `.github/workflows/` that defines when and what to run
- **Job** — a unit of work that runs on a virtual machine (runner). Jobs run in parallel by default.
- **Step** — a single command or action within a job. Steps run sequentially.

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Type check
        run: npm run typecheck

      - name: Test
        run: npm test

      - name: Build
        run: npm run build
```

The `on` block defines triggers. The `jobs` block defines what runs. Each step either `uses` a prebuilt action or `run`s a shell command.

## Matrix builds

A matrix strategy runs the same job across multiple configurations — different Node versions, operating systems, or any variable you define. One YAML block replaces what would otherwise be dozens of duplicated jobs.

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [18, 20, 22]
      fail-fast: false # Don't cancel other jobs if one fails

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}

      - run: npm ci
      - run: npm test
```

This creates 9 parallel jobs (3 OS x 3 Node versions). The `fail-fast: false` flag ensures every combination runs even if one fails — useful for understanding the full compatibility picture.

## Caching dependencies

Without caching, every job downloads and installs every dependency from scratch. For a Node.js project, that means downloading hundreds of megabytes of packages on every run. Caching eliminates this waste.

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js with cache
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm" # Automatically caches ~/.npm

      - name: Install dependencies
        run: npm ci
        # npm ci uses the lockfile — cache hit means near-instant install

      - name: Build
        run: npm run build
```

The `cache: "npm"` option on `setup-node` handles everything. On the first run, it saves the npm cache. On subsequent runs, it restores it. A cache hit can cut install time from 45 seconds to 3 seconds.

## Job dependencies and artifacts

Jobs run in parallel by default. Use `needs` to create dependencies between jobs, and artifacts to pass files between them.

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: "npm" }
      - run: npm ci
      - run: npm run build

      - name: Upload build output
        uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: .next/

  deploy:
    needs: build # Waits for build to finish
    runs-on: ubuntu-latest
    steps:
      - name: Download build output
        uses: actions/download-artifact@v4
        with:
          name: build-output
          path: .next/

      - name: Deploy
        run: echo "Deploying build artifacts..."
```

The `deploy` job only starts after `build` succeeds. The artifact upload/download pattern passes the build output between jobs without rebuilding.

## Secrets and environment variables

Sensitive values like API keys never go in workflow files. GitHub encrypts secrets and injects them at runtime.

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    env:
      NODE_ENV: production
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to hosting
        env:
          DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
        run: |
          echo "Deploying with token..."
          # The token value is masked in logs automatically
```

Secrets are configured in the repository settings, never committed to code. GitHub masks secret values in logs so they cannot leak through build output.

## What you learned

- GitHub Actions organizes CI/CD into workflows (YAML files), jobs (parallel units on runners), and steps (sequential commands)
- Matrix builds test across multiple configurations automatically, and caching avoids redundant dependency downloads
- Secrets are encrypted and injected at runtime — never hardcode sensitive values in workflow files
