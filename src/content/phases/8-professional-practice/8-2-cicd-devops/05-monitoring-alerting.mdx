## Why this matters

Code that works in development can fail in production in ways you never anticipated. A memory leak that surfaces after 10,000 requests. A database query that slows down when the table grows past a million rows. A third-party API that starts timing out at 2 PM every Tuesday. Without monitoring, you find out about these problems when users complain — or leave. With monitoring, you find out before users notice.

## The four golden signals

Google's Site Reliability Engineering handbook identifies four signals that capture the health of any service:

```javascript
const goldenSignals = {
  latency: {
    what: "How long requests take to serve",
    healthy: "p99 under 500ms",
    alert: "p99 exceeds 1000ms for 5 minutes",
  },
  traffic: {
    what: "How many requests the system handles",
    healthy: "Within expected range for time of day",
    alert: "Sudden drop of 50% or spike of 300%",
  },
  errors: {
    what: "Rate of failed requests",
    healthy: "Error rate below 0.1%",
    alert: "Error rate exceeds 1% for 3 minutes",
  },
  saturation: {
    what: "How full the system resources are",
    healthy: "CPU under 70%, memory under 80%, disk under 85%",
    alert: "Any resource exceeds 90% for 10 minutes",
  },
};
```

If you can only monitor four things, monitor these. They cover the vast majority of production problems.

## Metrics and percentiles

Averages lie. If 99 requests take 50ms and one request takes 10 seconds, the average is 150ms — which describes nobody's actual experience. Percentiles tell the real story.

```javascript
// Why percentiles matter more than averages
function calculatePercentiles(latencies) {
  const sorted = [...latencies].sort((a, b) => a - b);
  const len = sorted.length;

  return {
    p50: sorted[Math.floor(len * 0.5)], // Median — typical experience
    p90: sorted[Math.floor(len * 0.9)], // 90th percentile — most users
    p99: sorted[Math.floor(len * 0.99)], // 99th percentile — worst normal case
    max: sorted[len - 1], // Maximum — worst case
  };
}

// Example: 1000 requests
const latencies = Array.from({ length: 999 }, () => Math.random() * 100);
latencies.push(8500); // One slow request

const stats = calculatePercentiles(latencies);
console.log(
  "Average:",
  (latencies.reduce((a, b) => a + b, 0) / latencies.length).toFixed(0) + "ms"
);
console.log("p50:", stats.p50.toFixed(0) + "ms");
console.log("p99:", stats.p99.toFixed(0) + "ms");
// Average hides the outlier. p99 exposes it.
```

Alert on p99, not averages. A rising p99 means your slowest users are getting worse — even if the average looks fine.

## Structured logging

Unstructured logs are strings meant for humans. Structured logs are JSON objects meant for machines. The difference matters at scale because you need to search, filter, and aggregate logs across thousands of requests.

```javascript
// Unstructured — hard to parse, impossible to query
console.log("User 42 placed order 789 for $45.99 at 2024-03-15T10:30:00Z");

// Structured — every field is searchable and filterable
function structuredLog(level, message, context) {
  const entry = {
    timestamp: new Date().toISOString(),
    level,
    message,
    ...context,
    service: "order-api",
    environment: process.env.NODE_ENV,
  };
  console.log(JSON.stringify(entry));
}

structuredLog("info", "Order placed", {
  userId: 42,
  orderId: 789,
  amount: 45.99,
  currency: "USD",
  durationMs: 234,
});

// Now you can query: "show me all orders over $100 that took more than 500ms"
```

Every log entry should include: timestamp, severity level, a human-readable message, and structured context fields. A correlation ID that traces a request across services is essential for debugging distributed systems.

## Dashboards

A good dashboard answers one question at a glance. The worst dashboards try to show everything and communicate nothing.

```yaml
# Dashboard layout for a web application
dashboard:
  title: "Production Health"
  refresh: 30s

  row_1_overview:
    - panel: "Request Rate"
      metric: http_requests_total
      visualization: line_chart
      timerange: 6h

    - panel: "Error Rate"
      metric: http_errors_total / http_requests_total * 100
      visualization: gauge
      thresholds: [0.1, 1.0, 5.0] # green, yellow, red

    - panel: "p99 Latency"
      metric: http_request_duration_p99
      visualization: line_chart
      alert_line: 500ms

  row_2_resources:
    - panel: "CPU Usage"
      metric: system_cpu_percent
      visualization: line_chart

    - panel: "Memory Usage"
      metric: system_memory_percent
      visualization: gauge
      thresholds: [70, 85, 95]

    - panel: "Active Connections"
      metric: db_active_connections
      visualization: line_chart
      max_line: 100 # connection pool limit
```

Three rules for effective dashboards: red means action required now, yellow means investigate soon, green means no attention needed. If a dashboard has no red/yellow/green indicators, it is a decoration.

## Incident response

When monitoring detects a problem, the team needs a clear process — not heroics.

```javascript
// Incident severity levels and response expectations
const severityLevels = {
  SEV1: {
    description: "Service is down for all users",
    responseTime: "5 minutes",
    communicationCadence: "Every 15 minutes",
    example: "Homepage returns 500 for all requests",
  },
  SEV2: {
    description: "Service is degraded for some users",
    responseTime: "30 minutes",
    communicationCadence: "Every hour",
    example: "Search is slow but functional, checkout works",
  },
  SEV3: {
    description: "Non-critical feature is broken",
    responseTime: "4 hours",
    communicationCadence: "Daily update",
    example: "Profile avatar upload fails",
  },
};

// Post-incident: blameless retrospective
const retroTemplate = {
  title: "Incident #142 — Checkout 500 errors",
  timeline: "What happened, minute by minute",
  rootCause: "Why it happened — the technical cause",
  contributing: "What conditions allowed it to happen",
  actionItems: "What we change to prevent recurrence",
  blame: "NEVER — focus on systems, not people",
};
```

The blameless retrospective is critical. If people fear punishment for outages, they hide problems instead of surfacing them. Blame the system, fix the system.

## What you learned

- The four golden signals (latency, traffic, errors, saturation) capture the health of any production system and should be your primary monitoring targets
- Percentiles (especially p99) reveal user experience that averages hide, and structured JSON logging enables searching across thousands of requests
- Effective incident response requires clear severity levels, communication cadences, and blameless retrospectives focused on system improvement
