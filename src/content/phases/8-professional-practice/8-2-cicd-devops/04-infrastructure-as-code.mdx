## Why this matters

Manually configuring servers through a web console or SSH session creates infrastructure that exists only in someone's memory. When that person leaves the team, or when you need to recreate the environment at 3 AM during an outage, the knowledge is gone. Infrastructure as Code (IaC) means your servers, databases, networks, and configurations are defined in files that live in version control — reviewable, reproducible, and recoverable.

## Declarative vs imperative infrastructure

IaC tools come in two flavors. **Imperative** tools describe the steps to reach a desired state (scripts). **Declarative** tools describe the desired state itself and let the tool figure out the steps.

```javascript
// Imperative approach — you describe HOW
const steps = [
  "create a VPC",
  "create a subnet inside the VPC",
  "create a security group allowing port 443",
  "launch an EC2 instance in the subnet with the security group",
  "attach an elastic IP to the instance",
];
// Problem: if you run this twice, you get TWO of everything

// Declarative approach — you describe WHAT
const desiredState = {
  vpc: { cidr: "10.0.0.0/16" },
  subnet: { cidr: "10.0.1.0/24", vpc: "vpc" },
  securityGroup: { ingress: [{ port: 443, protocol: "tcp" }] },
  instance: { type: "t3.micro", subnet: "subnet", sg: "securityGroup" },
  elasticIp: { attachTo: "instance" },
};
// Benefit: running this twice changes nothing — it is idempotent
```

Declarative tools are idempotent — applying the same configuration twice produces the same result. This is why Terraform and Docker Compose dominate modern infrastructure.

## Terraform concepts

Terraform uses HCL (HashiCorp Configuration Language) to define cloud resources. You write `.tf` files, run `terraform plan` to preview changes, and `terraform apply` to make them real.

```yaml
# Terraform workflow (simplified as YAML for readability)
# In practice, Terraform uses HCL syntax in .tf files

# 1. Define a provider (which cloud)
provider:
  name: aws
  region: us-east-1

# 2. Define resources (what you want)
resources:
  web_server:
    type: aws_instance
    ami: ami-0abcdef1234567890
    instance_type: t3.micro
    tags:
      Name: production-web
      Environment: production

  database:
    type: aws_rds_instance
    engine: postgresql
    instance_class: db.t3.micro
    allocated_storage: 20
    # Terraform knows this depends on the VPC being created first

# 3. Plan → shows what will be created/changed/destroyed
# 4. Apply → makes the changes
# 5. State → Terraform tracks what it created in a state file
```

The Terraform workflow has three critical concepts:

- **Plan** — preview every change before it happens. No surprises.
- **State** — Terraform tracks what it manages. If you delete a resource from the config, Terraform destroys it in the cloud.
- **Modules** — reusable packages of infrastructure. Define a "web app" module once, deploy it across staging, production, and disaster recovery.

## Docker Compose for local environments

Docker Compose defines multi-container applications in a single YAML file. Every developer on the team runs `docker compose up` and gets an identical environment — same database version, same Redis version, same configuration.

```yaml
# docker-compose.yml
version: "3.9"

services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/myapp
      REDIS_URL: redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 5s
      timeout: 3s
      retries: 5

  cache:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  pgdata:
```

The `depends_on` with `condition: service_healthy` ensures the app does not start until the database is actually ready to accept connections — not just started, but healthy.

## IaC in the CI pipeline

Infrastructure changes should go through the same CI process as application code. A pull request that modifies Terraform files triggers `terraform plan`, and the plan output is posted as a PR comment for review.

```javascript
// CI pipeline for infrastructure changes
const infraPipeline = {
  trigger: "pull request modifying .tf files",
  steps: [
    {
      name: "terraform-init",
      run: "terraform init",
      purpose: "Download providers and modules",
    },
    {
      name: "terraform-validate",
      run: "terraform validate",
      purpose: "Check syntax and internal consistency",
    },
    {
      name: "terraform-plan",
      run: "terraform plan -out=plan.tfplan",
      purpose: "Preview changes without applying",
    },
    {
      name: "post-plan-to-pr",
      run: "post plan output as PR comment",
      purpose: "Reviewers see exactly what will change",
    },
    {
      name: "terraform-apply",
      run: "terraform apply plan.tfplan",
      purpose: "Only on merge to main — applies reviewed changes",
      condition: "main branch only",
    },
  ],
};
```

## What you learned

- Infrastructure as Code replaces manual server configuration with version-controlled, reviewable, reproducible definition files
- Declarative IaC tools like Terraform are idempotent — applying the same config twice changes nothing, eliminating drift and duplication
- Docker Compose defines multi-container environments in a single file so every developer gets an identical local setup with one command
