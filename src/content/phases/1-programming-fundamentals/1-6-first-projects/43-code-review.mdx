## Why this matters

You will spend more time reading code than writing it. Code review — reading someone else's code and giving feedback — is how professional teams maintain quality, share knowledge, and catch bugs before they reach users.

## How to read code

1. **Start with the purpose.** What does this code do? Read the PR description, the function name, the commit message.
2. **Trace the data.** Follow the inputs through the logic to the outputs. Where does data enter? Where does it exit?
3. **Look for patterns you recognize.** Is this a loop? A filter? A state machine? Naming the pattern tells you what to expect.
4. **Spot what's missing.** Is there error handling? Input validation? Edge case coverage?

## What to look for in a review

- **Correctness.** Does it do what it claims? Are there off-by-one errors, missing awaits, or wrong conditions?
- **Readability.** Could you understand this in six months? Are names clear? Is the logic flat (guard clauses) or deeply nested?
- **Edge cases.** What happens with empty input, null, zero, very large values?
- **Tests.** Are there tests? Do they cover the interesting cases?

## How to give feedback

- **Be specific.** "This could be cleaner" is not useful. "This nested if could be a guard clause (see line 42)" is.
- **Explain why.** Don't just say "use `===`" — say "use `===` because `==` coerces types and can produce surprising results here."
- **Suggest, don't demand.** "Consider using..." is better than "You must..." unless it's a real bug.
- **Praise good things.** "Good use of early returns here" costs nothing and builds trust.

## Reading code as a learning tool

When you read a well-written open-source project, you absorb patterns you would never discover by writing alone. Find a project you admire, open a recent PR, and read the diff. This is free mentorship.

discount = 0.1;
}
return price - price \* discount;
}

console.log(getDiscount(100, "gold"));

## What you learned

- Read for purpose first, then trace data through the logic.
- Review for correctness, readability, edge cases, and test coverage.
- Give specific, reasoned feedback. Praise the good parts.
