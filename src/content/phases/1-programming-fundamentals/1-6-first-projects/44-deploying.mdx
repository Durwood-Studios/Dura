## Why this matters

Code on your laptop helps nobody. Deployed code helps everyone. The gap between "it works on my machine" and "it's live on the internet" is smaller than you think — and closing it is one of the most satisfying moments in programming.

## What deployment means

Deployment is copying your built code to a server that serves it to users. For a static site or a simple app, modern platforms do this automatically when you push to GitHub.

## The simplest deployment: Vercel

1. Push your project to GitHub.
2. Sign up for Vercel (free tier).
3. Import your repository.
4. Vercel detects the framework, builds, and deploys.
5. You get a URL. Your project is live.

```text
Push to GitHub → Vercel auto-builds → Live at your-project.vercel.app
```

Every subsequent push triggers a new build. This is **continuous deployment** — the same concept from lesson 16 (The Developer Workflow), now real.

## Other free options

- **GitHub Pages** — static HTML/CSS/JS. No server-side code.
- **Netlify** — similar to Vercel, supports static sites and serverless functions.
- **Railway** — for apps that need a backend (Node.js, Python, databases).

For Phase 1 projects (pure JavaScript), any of these work.

## Environment variables

Some projects need secrets (API keys, database URLs). These go in environment variables, not in your source code.

```text
# .env.local (never committed to git)
API_KEY=sk-abc123
```

Your hosting platform has a settings panel where you add these. The build process injects them.

## The checklist before deploying

1. `npm run build` passes locally.
2. `npm run lint` has no errors.
3. No secrets in source code.
4. README explains what the project is and how to run it.
5. Git commit history is clean (no "fix fix fix" messages).

## What you learned

- Deployment = your code running on the internet. Vercel/Netlify make it free and automatic.
- Continuous deployment means every push to main is live within minutes.
- Never commit secrets. Use environment variables.
