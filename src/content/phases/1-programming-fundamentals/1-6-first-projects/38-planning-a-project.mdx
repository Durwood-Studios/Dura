## Why this matters

Beginners open an editor and start typing. Professionals open a notepad and start thinking. The twenty minutes you spend planning saves hours of rework. This lesson teaches the minimum viable planning process — enough structure to stay on track without enough overhead to feel like bureaucracy.

## Step 1: Define the outcome

Write one sentence: "When this is done, a user can \_\_\_."

- "When this is done, a user can enter two numbers and see their sum."
- "When this is done, a user can add items to a list, check them off, and delete them."

If you cannot finish that sentence, you do not know what you are building yet.

## Step 2: List the pieces

Break the outcome into the smallest working pieces:

1. A function that adds two numbers.
2. An input mechanism (prompt, form, CLI argument).
3. An output mechanism (console.log, DOM update).

Each piece should be buildable and testable on its own.

## Step 3: Pick the order

Build the core logic first, then the interface. A calculator that can add but has no UI is useful for testing. A UI that cannot add is not.

Order for a calculator:

1. `add(a, b)` function — test with console.log.
2. Input: read two numbers from the user.
3. Output: display the result.
4. Polish: handle bad input, add more operations.

## Step 4: Ship the smallest thing

Get version 1 working — even if it's ugly. Then improve. The biggest project-killer is polishing features you haven't finished yet.

## What you learned

- Define the outcome in one sentence. Break it into pieces. Order by dependency. Ship v1 fast.
- Build core logic first, interface second.
- Twenty minutes of planning saves hours of rework.
