## Why this matters

A chain of `else if` comparing the same variable to many values gets hard to read past three or four cases. `switch` is designed for exactly that situation: one value, many possible matches.

## The basic shape

```javascript
const day = "Monday";

switch (day) {
  case "Monday":
  case "Tuesday":
  case "Wednesday":
  case "Thursday":
  case "Friday":
    console.log("Weekday");
    break;
  case "Saturday":
  case "Sunday":
    console.log("Weekend");
    break;
  default:
    console.log("Unknown day");
}
```

`switch` compares the value in `()` with each `case` using `===`. When it finds a match, it runs everything below that case **until it hits a `break`**. Without `break`, execution falls through to the next case.

## Fall-through: feature or footgun?

Fall-through is intentional in the weekday example above — Monday through Friday share the same block. But accidentally forgetting a `break` is one of the most common `switch` bugs.

```javascript
const fruit = "apple";
switch (fruit) {
  case "apple":
    console.log("Fruit is apple");
  // oops — no break. falls through!
  case "banana":
    console.log("Fruit is banana");
    break;
}
// Prints BOTH lines!
```

Some teams ban `switch` for this reason and use objects or maps instead. DURA teaches both approaches.

## The object-map alternative

```javascript
const statusMessages = {
  200: "OK",
  404: "Not found",
  500: "Server error",
};

const message = statusMessages[404] ?? "Unknown status";
console.log(message); // "Not found"
```

This pattern replaces a switch when every case just returns a value. Cleaner, no fall-through risk, and the object can be built dynamically.

## What you learned

- `switch` compares one value against many cases using `===`.
- `break` exits the switch; forgetting it causes fall-through.
- For simple value lookups, an object map is often cleaner than switch.
