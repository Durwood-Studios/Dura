## Why this matters

Beginners nest `if` inside `if` inside `if`. The result is a rightward-drifting pyramid that is hard to read and harder to change. Experienced developers flatten that pyramid with **early returns** and **guard clauses**. This lesson teaches the technique.

## The problem

```javascript
function processOrder(order) {
  if (order) {
    if (order.items.length > 0) {
      if (order.paid) {
        return "Ship it.";
      } else {
        return "Payment needed.";
      }
    } else {
      return "No items.";
    }
  } else {
    return "No order.";
  }
}
```

Four levels of nesting. Every `else` is far from the `if` it belongs to. This gets worse fast.

## The fix: guard clauses

Invert the conditions, return early, and keep the happy path at the top level.

```javascript
function processOrder(order) {
  if (!order) return "No order.";
  if (order.items.length === 0) return "No items.";
  if (!order.paid) return "Payment needed.";
  return "Ship it.";
}
```

Same logic, zero nesting. Each guard clause handles one edge case and exits. The final line is always the "everything is fine" path.

## When to nest

Nesting is fine at one level. The problem starts at two or three. If you find yourself indenting more than twice, stop and ask: "Can I return early instead?"

## Logical operators as conditions

You can combine conditions with `&&` and `||` instead of nesting.

```javascript
if (user && user.isAdmin && user.isActive) {
  grantAccess();
}
```

Or with optional chaining:

```javascript
if (user?.isAdmin && user?.isActive) {
  grantAccess();
}
```

## What you learned

- Deep nesting makes code hard to read. **Guard clauses** (early returns) flatten it.
- Handle edge cases first, exit early, and keep the happy path un-indented.
- One level of nesting is fine; three is a smell.
