## Why this matters

Networks fail. Files are missing. Users enter garbage. A program that does not handle errors crashes. A program that handles them badly swallows information and makes debugging impossible. `try/catch` is JavaScript's structured way to deal with the unexpected.

## The basic shape

```javascript
try {
  const data = JSON.parse('{"valid": true}');
  console.log(data.valid); // true
} catch (error) {
  console.error("Parsing failed:", error.message);
}
```

The code inside `try` runs normally. If any line throws an error, execution immediately jumps to `catch`. The `error` parameter holds the thrown value — usually an `Error` object with a `.message` property.

## finally

```javascript
try {
  doSomethingRisky();
} catch (error) {
  console.error(error.message);
} finally {
  cleanup(); // always runs
}
```

`finally` runs whether or not an error occurred. Use it for cleanup: closing connections, stopping timers, resetting state.

## Throwing your own errors

```javascript
function divide(a, b) {
  if (b === 0) throw new Error("Cannot divide by zero.");
  return a / b;
}

try {
  console.log(divide(10, 0));
} catch (error) {
  console.error(error.message); // "Cannot divide by zero."
}
```

`throw` creates an exception. It can be any value, but convention is to throw an `Error` object so the stack trace is captured.

## Never swallow errors silently

```javascript
// BAD: the error disappears
try {
  riskyOperation();
} catch (error) {
  // do nothing
}

// GOOD: at minimum, log it
try {
  riskyOperation();
} catch (error) {
  console.error("[app] riskyOperation failed:", error);
}
```

Every swallowed error is a future debugging nightmare. If you catch an error, at least log it.

## When to catch

Catch at **boundaries**: where user input enters, where network calls return, where file I/O happens. Do not wrap every function in try/catch — let errors propagate until they reach a boundary that knows how to handle them.

## What you learned

- `try/catch` handles errors at runtime. `finally` runs regardless.
- `throw new Error(message)` creates an exception with a stack trace.
- Catch at **boundaries**. Never swallow errors — always log.
