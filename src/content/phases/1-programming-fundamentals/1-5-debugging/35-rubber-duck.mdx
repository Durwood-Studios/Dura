## Why this matters

Sometimes you stare at code for twenty minutes and see nothing wrong. Then you start explaining it to a colleague and the answer hits you before you finish the first sentence. Rubber duck debugging formalizes that process — you explain the code to a rubber duck (or any inanimate object) line by line, and the mismatch between what you think the code does and what it actually does surfaces.

## The method

1. Place a rubber duck (or coffee mug, or stuffed animal) on your desk.
2. Explain what the code is supposed to do, at a high level.
3. Walk through the code line by line, out loud, explaining what each line does.
4. At some point, you will say something that does not match what you expected. That is the bug.

This works because:

- **Explaining forces precision.** You cannot hand-wave in a sentence the way you can while scanning code visually.
- **Your brain runs a different path.** Reading and speaking activate different cognitive processes. The speaking path catches things the reading path missed.

## When to use it

- You have been stuck for more than 10 minutes.
- You have already read the error message and checked the obvious.
- You do not want to ask a colleague yet because you suspect you are missing something simple.

## It scales up

Senior engineers do this constantly — they just do it faster and internally. Code reviews work partly because explaining your code to a reviewer triggers the same effect. Writing commit messages, documenting functions, and even writing tests all force the same line-by-line accounting.

## What you learned

- Rubber duck debugging: explain the code line by line, out loud. The mismatch reveals the bug.
- It works because speaking forces precision that scanning does not.
- Code reviews, commit messages, and documentation all trigger the same effect.
