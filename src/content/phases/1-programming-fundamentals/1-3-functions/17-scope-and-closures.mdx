## Why this matters

Scope is the invisible boundary that decides which variables you can see from where. Closures are what happens when a function remembers its scope after the scope should have been gone. If you understand closures, you understand event handlers, callbacks, React hooks, and module patterns.

## Block scope

`let` and `const` are block-scoped. A variable declared inside `{}` is invisible outside.

```javascript
{
  const secret = 42;
}
console.log(secret); // ReferenceError
```

Each `if`, `for`, and `while` block creates its own scope.

## Function scope

A function creates a scope too. Variables declared inside are invisible outside.

```javascript
function demo() {
  const local = "only here";
}
console.log(local); // ReferenceError
```

## Nested scope (the scope chain)

Inner scopes can read outer scopes, but not the other way around.

```javascript
const outer = "visible everywhere";

function inner() {
  const innerOnly = "just in here";
  console.log(outer); // works
  console.log(innerOnly); // works
}

inner();
console.log(innerOnly); // ReferenceError
```

When JavaScript looks up a variable name, it checks the current scope first, then the parent scope, then the grandparent, all the way up to the global scope. This is the **scope chain**.

## Closures

A closure happens when a function captures variables from an outer scope and retains access to them — even after the outer function has returned.

```javascript
function makeCounter() {
  let count = 0;
  return function () {
    count++;
    return count;
  };
}

const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3
```

`makeCounter` has returned. Its local `count` should be gone. But the inner function still holds a reference to it through the closure. Each call to `counter()` updates the same `count`.

## Why closures matter

1. **Callbacks** — every `setTimeout`, `addEventListener`, and `.then` callback closes over its surrounding variables.
2. **Data privacy** — the `count` above is unreachable from outside. Only the returned function can modify it.
3. **Factories** — `makeCounter` is a factory. Call it twice and you get two independent counters.

## Common trap: closures in loops

```javascript
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// prints 3, 3, 3 — not 0, 1, 2
```

Because `var` is function-scoped, all three closures share the same `i`. By the time the timeout fires, `i` is 3. Fix: use `let`, which is block-scoped — each iteration gets its own `i`.

```javascript
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// prints 0, 1, 2
```

## What you learned

- **Scope** is where variables are visible. Block scope (let/const) and function scope are the two main kinds.
- A **closure** is a function that remembers its surrounding scope. It powers callbacks, factories, and data privacy.
- Use `let` in loops to avoid the shared-variable-in-closures trap.
