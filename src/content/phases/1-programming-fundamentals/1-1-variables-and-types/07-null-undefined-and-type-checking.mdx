## Why this matters

Every real program encounters missing data. A user leaves a form field blank. An API returns a record with no email. A function is called before a value has been set. JavaScript has two ways to represent "nothing" — `null` and `undefined` — and confusing them causes bugs that are hard to find.

## undefined: no value assigned

A declared variable that has not been given a value is `undefined`. A function parameter that was not passed is `undefined`. A property that does not exist on an object is `undefined`.

```javascript
let x;
console.log(x); // undefined

function greet(name) {
  console.log(name);
}
greet(); // undefined — no argument passed
```

`undefined` means "this exists, but nobody has put a value here yet."

## null: intentionally empty

`null` is the deliberate assignment of "no value." You set `null` explicitly.

```javascript
let selectedUser = null;
// later:
selectedUser = getUser();
```

`null` means "I know this could hold a value, but right now I'm saying it has none."

## typeof quirk

```javascript
console.log(typeof undefined); // "undefined"
console.log(typeof null); // "object"  — this is a famous JavaScript bug
```

`typeof null === "object"` is a bug from JavaScript's first implementation in 1995. It will never be fixed because too much code depends on it. Just know it is wrong and check for null with `value === null`.

## Checking types safely

A pattern you will use constantly:

```javascript
function isString(value) {
  return typeof value === "string";
}

function isNumber(value) {
  return typeof value === "number" && !Number.isNaN(value);
}

function isNullOrUndefined(value) {
  return value === null || value === undefined;
  // or: value == null  (the one legitimate use of ==)
}
```

The `value == null` trick is the single case where `==` is accepted in modern code — it catches both `null` and `undefined` in one check. But `=== null || === undefined` is clearer and just as correct.

## Optional chaining

When you are not sure if an object property exists, optional chaining (`?.`) short-circuits to `undefined` instead of throwing.

```javascript
const user = { name: "Dustin" };
console.log(user.address?.city); // undefined — no error
```

Without `?.`, `user.address.city` would throw a `TypeError` because `address` is `undefined` and you cannot read `.city` on it.

## Nullish coalescing

`??` provides a default when the left side is `null` or `undefined`.

```javascript
const name = null;
console.log(name ?? "Anonymous"); // "Anonymous"
console.log(0 ?? "fallback"); // 0 — 0 is not null/undefined
```

Compare with `||`, which uses the first truthy value — `0 || "fallback"` gives `"fallback"` because `0` is falsy, even though it is a valid number.

## What you learned

- `undefined` = no value assigned. `null` = intentionally empty. Both are valid but mean different things.
- `typeof null === "object"` is a bug. Check with `=== null`.
- Use `?.` (optional chaining) to safely access deep properties and `??` (nullish coalescing) for defaults.
