## Why this matters

At 10 engineers, process is a conversation. At 50 engineers, process is a document. At 200 engineers, process is infrastructure. The scale-up CTO does not write code — they write the rules by which code gets written, reviewed, shipped, and fixed when it breaks. Bad process creates bureaucracy. No process creates chaos. Good process is invisible: engineers follow it naturally because it matches how they already want to work.

## Code review standards

Code review is your highest-leverage quality mechanism. It is also where most organizations waste the most engineering time. The goal is not to catch every bug — that is what tests are for. The goal is knowledge distribution, consistency enforcement, and mentorship.

### Define what a review checks

Write it down. Publish it in your engineering handbook. Without explicit criteria, every reviewer invents their own checklist, and authors have no idea what to expect.

```markdown
## Code Review Checklist

### Must-check (block merge if violated)

- [ ] No security vulnerabilities (SQL injection, XSS, auth bypass)
- [ ] No data loss risk (destructive migrations, missing backups)
- [ ] Tests cover the behavioral change
- [ ] Error handling exists for failure paths
- [ ] No hardcoded secrets or credentials

### Should-check (comment, do not block)

- [ ] Naming clarity
- [ ] Unnecessary complexity
- [ ] Performance implications at current scale
- [ ] Documentation for non-obvious behavior

### Out of scope (do not comment)

- Style preferences covered by automated formatters
- Alternative approaches that are equivalent in quality
- Nits that do not affect correctness or readability
```

### Set turnaround expectations

An unreviewed PR is an engineer blocked. Define a review SLA: first meaningful review within four working hours. Not a rubber-stamp approval — a real review. Track review latency as a team metric.

### Limit review scope

PRs over 400 lines get diminishing review quality. If a PR is too large to review in 30 minutes, it should be split. This is a process rule, not a suggestion.

## Incident response

When production breaks at 3 AM, nobody should be figuring out who to call or what to do. That was decided weeks ago and written in a runbook.

### Severity levels

Define them before you need them:

| Level | Definition                                | Response time     | Example                        |
| ----- | ----------------------------------------- | ----------------- | ------------------------------ |
| P0    | Service down, all users affected          | 15 minutes        | Database unreachable           |
| P1    | Major feature broken, many users affected | 1 hour            | Payment processing failing     |
| P2    | Feature degraded, some users affected     | 4 hours           | Search returning stale results |
| P3    | Minor issue, workaround exists            | Next business day | Tooltip rendering incorrectly  |

### Incident command structure

Every incident needs exactly three roles:

**Incident Commander (IC)** — owns the timeline, coordinates communication, decides when to escalate. Does not debug. Their job is to keep the response organized.

**Technical Lead** — drives the investigation and fix. This is the on-call engineer or the domain expert for the affected system.

**Communications Lead** — posts status updates to the internal channel and the external status page. Shields the Technical Lead from "what's happening?" messages.

### Runbooks

For every service that can page you, write a runbook. A runbook answers: what does this alert mean, what should I check first, what are the common fixes, and when should I escalate?

```markdown
## Runbook: API Response Time > 2s

### What this means

P95 API latency has exceeded 2 seconds for 5 consecutive minutes.

### First checks

1. Check database connection pool — are connections maxed?
2. Check recent deployments — was anything shipped in the last hour?
3. Check external dependency status pages (Stripe, SendGrid)

### Common fixes

- If connection pool exhausted: restart the API pods
- If recent deploy: rollback via `gh workflow run rollback`
- If external dependency: enable circuit breaker, notify users

### Escalation

If unresolved after 30 minutes, page the database team lead.
```

### Postmortems

Every P0 and P1 gets a written postmortem within 48 hours. Blameless. The format:

1. **Timeline** — what happened, when, in chronological order
2. **Root cause** — the actual technical failure, not "human error"
3. **Impact** — users affected, revenue lost, duration
4. **What went well** — detection speed, response coordination
5. **What went poorly** — gaps in monitoring, slow escalation
6. **Action items** — specific, assigned, with deadlines

The postmortem is not punishment. It is the mechanism by which the organization learns. If engineers fear postmortems, they hide incidents. If they hide incidents, you lose the ability to improve.

## On-call rotations

On-call is a tax on engineers' personal lives. Respect it.

### Rotation structure

- Rotate weekly. Longer rotations cause burnout. Shorter rotations lose context.
- Two-tier: primary on-call responds first. Secondary is the escalation path.
- No engineer on-call during their first 30 days. They do not know the systems yet.
- Compensate on-call with time off, extra pay, or both. "It's part of the job" is how you lose engineers.

### On-call health metrics

Track these monthly:

- **Pages per rotation** — if it exceeds 10 per week, your systems are too noisy. Fix alerts before adding more engineers.
- **Off-hours pages** — target zero. Every off-hours page is a process or reliability failure worth investigating.
- **Time to acknowledge** — are engineers actually available, or are pages going unanswered?
- **False positive rate** — noisy alerts train engineers to ignore all alerts. A 50% false positive rate means your alerting is broken.

## The anti-bureaucracy test

Before introducing any new process, ask three questions:

1. **What failure does this prevent?** If you cannot name a specific incident or pattern, you do not need the process.
2. **What is the cost of compliance?** If following the process takes more time than the failures it prevents, the process is net negative.
3. **Can automation replace it?** If a human is checking something a linter could check, automate it and remove the human step.

Process should shrink over time as automation grows. If your process document only gets longer, you are building bureaucracy, not engineering excellence.
