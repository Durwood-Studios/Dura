## Why this matters

Every scale-up sits on top of code written when the company was a different size, solving different problems, with different constraints. That early code enabled growth. Now it constrains it. The CTO's challenge is not eliminating tech debt — that is impossible and not even desirable. The challenge is managing it strategically: knowing which debt to pay, which to accept, and which to prevent from accumulating further.

## Tech debt at scale

Tech debt is not bad code. Tech debt is the gap between the system you have and the system you need. Some of that gap was created deliberately — you shipped fast to capture a market window and accepted shortcuts. Some was created accidentally — what seemed like a good design at 10,000 users buckles at 10,000,000.

### Categorizing debt

Not all debt is equal. Categorize it by impact and urgency:

**Critical debt**: actively causing incidents, blocking features, or creating security vulnerabilities. Fix immediately. Examples: a database schema that causes deadlocks under load, an authentication system with known bypasses, a service that cannot be deployed independently.

**Strategic debt**: not causing problems today but will block known future needs. Schedule it on the roadmap. Examples: a monolith that prevents independent team deployment, a data pipeline that cannot handle projected growth, a vendor dependency with an approaching end-of-life.

**Ambient debt**: annoying but tolerable. Inconsistent naming conventions, outdated libraries that still work, code that is hard to read but correct. Do not schedule dedicated work. Fix opportunistically when you touch the area for other reasons.

**Accepted debt**: debt you have consciously decided not to pay. The cost of fixing it exceeds the cost of living with it. Document the decision and the conditions under which you would revisit it.

### The 20% rule

Many organizations allocate a percentage of engineering capacity to debt reduction — commonly 20%. This is a reasonable starting point, but apply it thoughtfully:

- Do not spread it evenly. Some quarters need 40% on debt (after a rapid growth phase). Some quarters need 5% (during a critical product launch).
- Do not let it become a dumping ground. "Tech debt work" is not "whatever an engineer feels like refactoring." Debt work should be prioritized by impact, just like feature work.
- Track outcomes, not hours. "We spent 20% on debt" means nothing. "We reduced deploy time from 45 minutes to 8 minutes" means everything.

## The rewrite question

At some point, someone will propose rewriting a core system from scratch. This is the most dangerous proposal a scale-up CTO will encounter. Most rewrites fail. The ones that succeed take 2-3x longer than estimated and divert engineering capacity from product work for quarters or years.

### Why rewrites fail

**Feature parity is a trap.** The old system has years of edge cases, workarounds, and implicit behavior baked in. Users depend on behavior that was never intentional. A rewrite must replicate all of it, and nobody has a complete list of what "all of it" includes.

**The old system keeps moving.** While the rewrite team builds the new system, the product team keeps shipping features on the old system. The new system is always behind. This creates a moving target that the rewrite can never catch.

**Morale collapse.** The rewrite team works for 18 months without shipping anything to users. They lose momentum. The product team resents them for consuming headcount without visible output. The rewrite gets quietly canceled, or worse, ships half-complete.

### When a rewrite is actually justified

- The old system's language or framework is truly end-of-life with no migration path
- The old system has fundamental architectural limits (single-threaded, single-tenant, embedded database) that cannot be incrementally changed
- The domain has changed so completely that the old system's data model is wrong at the foundation level
- The old system is small enough that a rewrite is genuinely a months-not-years effort

If none of these apply, refactor.

## The strangler fig pattern

The strangler fig is the safest approach to large-scale migration. Instead of replacing the old system at once, you grow a new system around it, gradually routing traffic from old to new until the old system has no remaining consumers and can be decommissioned.

### How it works

1. **Identify a boundary.** Find a seam in the old system where a subset of functionality can be extracted. A single API endpoint. A single page. A single data domain.
2. **Build the replacement.** Implement that subset in the new system with the new architecture.
3. **Route traffic.** Use a proxy, feature flag, or load balancer to send traffic to the new implementation. Start with a small percentage. Monitor for correctness.
4. **Expand.** Once the new implementation is stable, increase traffic to 100%. Repeat for the next boundary.
5. **Decommission.** When no traffic reaches the old system, remove it.

```
Phase 1:  [Old System] handles 100% of traffic

Phase 2:  [Old System] handles 90%
          [New: Users API] handles 10%

Phase 3:  [Old System] handles 60%
          [New: Users API] handles 15%
          [New: Orders API] handles 25%

Phase 4:  [Old System] handles 0% → decommission
          [New System] handles 100%
```

### Why it works

- You ship value incrementally. Each migrated boundary is a measurable improvement.
- You can stop at any point. If priorities change, the partially migrated system still works.
- You validate the new architecture with real traffic before committing fully.
- The old system shrinks continuously, reducing its maintenance burden.

## Legacy migration planning

Any migration larger than a single team and a single quarter needs a formal plan.

### The migration plan template

**Current state**: what exists, why it is a problem, and what specifically is the pain.

**Target state**: what the world looks like after the migration. Be concrete — diagrams, API contracts, data schemas.

**Migration path**: the ordered sequence of steps from current to target. Each step must leave the system in a working state. No "big bang" cutover unless the blast radius is tiny.

**Rollback plan**: for each step, how do you undo it if something goes wrong? If the answer is "we can't," the step is too large.

**Success criteria**: how do you know the migration is done? Latency targets, error rate targets, feature parity checklist.

**Timeline and staffing**: how many engineers, for how long. Include the migration tax — the ongoing cost of maintaining both old and new systems during the transition.

### The migration tax

During any migration, you are running two systems. Both need monitoring. Both need on-call. Both need bug fixes. This dual-maintenance cost is the migration tax, and teams consistently underestimate it. Budget 30-50% overhead on top of the raw migration effort.

The migration tax is also why speed matters. A migration that takes 6 months costs much less in total than the same migration stretched to 18 months, even if the engineering hours are identical. The tax is a function of duration, not effort.

## Prevention

The best tech debt strategy is prevention. Not zero debt — that is impossible and would mean you never ship anything quickly. But deliberate debt, consciously incurred and tracked.

**Require debt tickets.** When an engineer takes a shortcut, they file a ticket describing the shortcut, the reason, and the estimated cost to fix later. This makes debt visible and trackable.

**Review debt quarterly.** In roadmap planning, review the debt backlog. Prioritize by impact. Some tickets will age out — the system they describe gets replaced for other reasons. Some tickets will escalate — what was minor debt six months ago is now causing weekly incidents.

**Celebrate debt paydown.** When an engineer spends a week reducing deploy time from 30 minutes to 5 minutes, that is as valuable as shipping a feature. Recognize it publicly.
