## Why this matters

A single great hire changes the trajectory of a team. A single bad hire can set it back six months. As an engineering manager, hiring is your highest-leverage activity — more impactful than any architectural decision or process improvement. Yet most managers inherit a broken interview process and never fix it. They wing the phone screen, run inconsistent technical rounds, and make gut-feel decisions in the debrief. The result is slow hiring, biased outcomes, and missed candidates. You can do better by treating hiring as an engineering problem: design the system, measure the signals, iterate on the results.

## Writing job descriptions that work

Most JDs are wishlists. "8+ years of experience, expert in React, Node, AWS, Kubernetes, GraphQL, and machine learning." This describes nobody, and it repels strong candidates who self-select out if they do not match every bullet. Research from Hewlett-Packard found that women apply to jobs when they meet 100% of the listed qualifications, while men apply at 60%.

**Structure that works:**

- **Two sentences on the team's mission.** What are you building and why does it matter?
- **Three to five "you will" statements.** Concrete responsibilities, not vague platitudes.
- **Must-have skills (3-4 max).** The actual hard requirements for day-one effectiveness.
- **Nice-to-have skills (2-3).** Things you can teach in the first quarter.
- **What you offer.** Compensation range (yes, include it), growth opportunities, team culture.

Kill the "years of experience" requirement. A developer with three years of intense, high-ownership experience will outperform someone with ten years of passive maintenance work. Describe the problems they will solve, not the resume they should have.

## Designing the interview loop

A strong loop has four stages, each measuring something different. Redundant signals waste everyone's time.

**Phone screen (30 min).** A hiring manager or senior engineer validates communication skills, motivation, and basic technical fit. This is a filter, not an evaluation. You are answering: "Is it worth spending four hours of the team's time on this person?"

**Technical round (60-90 min).** A focused coding or system design exercise relevant to the actual work. Not LeetCode puzzles — real problems. If you are hiring for a frontend role, have them build a component. If you are hiring for backend, have them design an API. Pair programming formats reveal more than whiteboard algorithms because they show collaboration, not just ability.

**System design (45-60 min).** For senior candidates. Present an open-ended problem ("Design the notification system for a platform with 10M users") and evaluate how they think through tradeoffs, ask clarifying questions, and communicate uncertainty. There is no right answer — you are assessing judgment.

**Culture/values round (45 min).** Not "culture fit" — that is code for "people like us." Use "culture add." Structured behavioral questions based on your team's values. "Tell me about a time you disagreed with a technical decision and how you handled it." "Describe a situation where you had to give difficult feedback to a peer." Score against a rubric, not a vibe.

## Evaluation rubrics

Every interviewer fills out a scorecard before the debrief. This is non-negotiable. Without written evaluations submitted independently, the debrief becomes an anchoring exercise where the loudest voice wins.

**Score each competency 1-4:**

- **1 — Strong no.** Demonstrated a clear gap that cannot be addressed through onboarding.
- **2 — Lean no.** Below the bar in this area. Could grow into it, but not where you need them.
- **3 — Lean yes.** Meets the bar. Solid evidence of competency.
- **4 — Strong yes.** Exceeded expectations. This person raises the team's level.

Avoid 5-point scales — the middle becomes a dumping ground for "I don't know." A 4-point scale forces a lean.

## Bias mitigation

Structured interviews with rubrics are the single most effective bias mitigation tool. Beyond that:

- **Diverse interview panels.** If every interviewer is a white man, your process has a blind spot. Deliberately.
- **Standardized questions.** Ask every candidate for a given role the same core questions. Ad-hoc interviews measure interviewer creativity, not candidate quality.
- **Work samples over credentials.** A GitHub profile or take-home project tells you more than a Stanford degree.
- **Blind resume review.** Remove names, schools, and companies from the initial screen. Tools like Greenhouse and Lever support this.

## Making the offer and knowing when to say no

When you find the right person, move fast. The best candidates have options. Get the offer out within 48 hours of the debrief decision. A slow process loses candidates to companies that move faster.

Saying no is harder but equally important. If the debrief is lukewarm — "they were fine, I guess" — that is a no. The bar is not "can this person do the job." The bar is "will this person raise the average of the team." Every hire either raises or lowers the bar. There is no neutral.

<SandboxExercise language="javascript">
{`// Interview Rubric Scorer
// Models the structured evaluation process for a candidate
// across multiple interview rounds.

function evaluateCandidate(candidate) {
// candidate: { name, rounds: [{ interviewer, scores }] }
// Each round has scores: { technicalSkill, problemSolving,
// communication, collaboration } rated 1-4

const competencies = ["technicalSkill", "problemSolving", "communication", "collaboration"];

// Aggregate scores across all rounds
const aggregated = {};
competencies.forEach(c => {
const allScores = candidate.rounds.map(r => r.scores[c]).filter(Boolean);
aggregated[c] = {
scores: allScores,
average: allScores.reduce((a, b) => a + b, 0) / allScores.length,
hasStrongNo: allScores.some(s => s === 1)
};
});

// Decision logic
const overallAverage = Object.values(aggregated)
.reduce((sum, c) => sum + c.average, 0) / competencies.length;

const hasAnyStrongNo = Object.values(aggregated).some(c => c.hasStrongNo);
const allAbove2 = Object.values(aggregated).every(c => c.average >= 2.5);

let decision;
if (hasAnyStrongNo) {
decision = "NO HIRE — strong-no signal in at least one competency";
} else if (overallAverage >= 3.0 && allAbove2) {
decision = "HIRE — consistent signal above the bar";
} else if (overallAverage >= 2.5) {
decision = "DISCUSS — mixed signals, debrief should focus on specific concerns";
} else {
decision = "NO HIRE — below the bar across multiple dimensions";
}

return {
candidate: candidate.name,
competencies: aggregated,
overallAverage: Math.round(overallAverage \* 100) / 100,
decision
};
}

// Example candidate with three interview rounds
const result = evaluateCandidate({
name: "Jordan",
rounds: [
{
interviewer: "Phone Screen - Sarah",
scores: { technicalSkill: 3, problemSolving: 3, communication: 4, collaboration: 3 }
},
{
interviewer: "Technical - Marcus",
scores: { technicalSkill: 3, problemSolving: 4, communication: 3, collaboration: 3 }
},
{
interviewer: "Culture - Priya",
scores: { technicalSkill: 3, problemSolving: 3, communication: 3, collaboration: 4 }
}
]
});

console.log("Candidate:", result.candidate);
console.log("Overall average:", result.overallAverage);
console.log("Decision:", result.decision);
Object.entries(result.competencies).forEach(([comp, data]) => {
console.log(" " + comp + ": avg " + data.average.toFixed(1) +
(data.hasStrongNo ? " [STRONG NO]" : ""));
});`}

## What you learned

- Job descriptions should describe problems to solve and concrete responsibilities, not credential wishlists — keep must-haves to 3-4 actual requirements
- Structured interviews with rubrics and independently submitted scorecards are the most effective tools for consistent, bias-reduced evaluation
- The hiring bar is "will this person raise the average of the team" — lukewarm consensus is a no
