## Why this matters

Sprint planning is where strategy meets reality. You can have the best product roadmap in the world, but if your team consistently overcommits, gets derailed by scope creep, and treats retros as complaint sessions, nothing ships predictably. The engineering manager does not need to run every sprint ceremony personally, but you own the system that makes them work. When sprints go well, the team has clarity, focus, and a sense of accomplishment. When they go badly, you get burnout, missed deadlines, and the slow erosion of trust between engineering and product.

## Story points vs. time estimates

This debate has been running since the Agile Manifesto was signed. Here is the pragmatic answer: it does not matter which you use, as long as you are consistent and honest about what you are measuring.

**Story points** measure relative complexity. A 5-point story is roughly twice as complex as a 3-point story. They abstract away individual speed differences and focus on the work itself. The advantage is that velocity becomes a team metric, not an individual one. The disadvantage is that stakeholders want to know "when will it ship?" and story points do not answer that directly.

**Time estimates** are intuitive but dangerous. Engineers are optimists. A "two-day task" routinely takes four days because the estimate did not account for code review, testing, context switching, and the unexpected bug that appears when you integrate with the other service. If you use time estimates, build in a multiplier. Martin Fowler's teams historically used 2x as a correction factor, and they were better estimators than most.

**The hybrid approach that works:** Use story points for sprint planning and velocity tracking. Map velocity to a delivery forecast for stakeholders. "Our velocity is 30 points per sprint. This epic is 90 points. Expect 3 sprints, roughly 6 weeks, with a buffer sprint for unknowns."

## Capacity planning

Velocity is not a target. It is a measurement. The moment you set a velocity target — "we need to hit 35 points this sprint" — you have incentivized point inflation, corner-cutting, and burnout. Velocity should emerge from sustainable work.

**Real capacity is lower than you think.** A two-week sprint has 10 working days per engineer. Subtract: 1:1s, sprint ceremonies, code reviews, interviews, on-call rotations, context switching overhead. A realistic capacity for focused development work is 6-7 days per engineer per sprint. Plan to that number, not the theoretical 10.

**Account for the predictable.** Holidays, PTO, on-call rotations, and company all-hands are not surprises. Build them into capacity before sprint planning starts. If two of your six engineers are on PTO during a sprint, your capacity is 67%, not 100%.

## Sprint goals that matter

A sprint goal is not a list of tickets. It is a statement of what the team will accomplish that creates user or business value. "Complete 15 Jira tickets" is not a goal. "Users can reset their password via email" is a goal.

Good sprint goals:

- Are achievable within the sprint
- Create demonstrable value (you can show it to someone)
- Help the team make tradeoff decisions mid-sprint ("does this unplanned work help us hit the sprint goal?")

When an unplanned request comes in — and it will — the sprint goal is your negotiation tool. "We can take that on, but it means we won't hit the sprint goal. Which is more important?" This forces the requester to make a prioritization decision rather than simply adding to the pile.

## Handling scope creep

Scope creep is not a force of nature. It is a failure of process and boundaries. It happens when:

- Requirements were not clear at the start (fix: better grooming)
- Stakeholders add requests mid-sprint without removing anything (fix: sprint goal as a contract)
- Engineers discover complexity they did not anticipate (fix: spike stories for unknowns)
- Nobody says no (fix: you, the EM, say no)

Your job is to protect the sprint. This does not mean being rigid — genuine emergencies override any process. But "the CEO saw a competitor's feature and wants it by Friday" is not an emergency. It is a prioritization conversation.

**The scope creep response template:** "I understand this is important. Here's what we're committed to this sprint. We can either swap it for an existing item of similar size, or we can put it at the top of next sprint. Which works?"

## The retrospective that actually improves things

Most retros follow the same pattern: the team lists things that went well, things that went badly, and action items. The action items are forgotten by the next sprint. The same problems come up again. Everyone becomes cynical about retros.

**Fix the retro by fixing the follow-through:**

- Limit action items to two per retro. More than that, and none get done.
- Assign an owner and a due date to each action item.
- Start the next retro by reviewing last sprint's action items. Did they happen? If not, why?
- Rotate the facilitator. The EM should not always run the retro — it creates a dynamic where the team reports problems to the manager rather than solving them collaboratively.

**Formats that break the rut:**

- **Start/Stop/Continue** — what should we start doing, stop doing, keep doing?
- **4Ls** — what did we Love, Learn, Lack, Long for?
- **Sailboat** — what is the wind (propelling us), the anchor (holding us back), the rocks (risks ahead)?

The format matters less than the psychological safety. If people do not feel safe raising real problems, no format will help. That is on you.

<SandboxExercise language="javascript">
{`// Sprint Capacity Calculator
// Models realistic capacity planning for a two-week sprint,
// accounting for the overhead that most teams ignore.

function calculateSprintCapacity(team) {
// team: [{ name, daysOff, onCall, interviewSlots, meetingsPerDay }]
// Returns capacity plan with realistic available hours

const SPRINT_DAYS = 10;
const HOURS_PER_DAY = 8;
const CEREMONY_HOURS = 4; // planning, standup x10, grooming, retro, review
const CODE_REVIEW_HOURS_PER_DAY = 0.75;

const plan = team.map(member => {
const workingDays = SPRINT_DAYS - member.daysOff;
const grossHours = workingDays \* HOURS_PER_DAY;

    // Subtract fixed overhead
    const ceremonyTime = CEREMONY_HOURS;
    const codeReviewTime = workingDays * CODE_REVIEW_HOURS_PER_DAY;
    const meetingTime = workingDays * member.meetingsPerDay * 0.5; // 30min avg
    const onCallTime = member.onCall ? 8 : 0; // one full day of disruption
    const interviewTime = member.interviewSlots * 1.5; // 1hr interview + 30min debrief
    const contextSwitchTax = workingDays * 0.5; // 30min/day lost to switching

    const totalOverhead = ceremonyTime + codeReviewTime + meetingTime +
      onCallTime + interviewTime + contextSwitchTax;

    const netHours = Math.max(0, grossHours - totalOverhead);
    const utilizationRate = grossHours > 0 ? (netHours / grossHours * 100) : 0;

    return {
      name: member.name,
      workingDays,
      grossHours: Math.round(grossHours),
      overheadHours: Math.round(totalOverhead),
      netDevelopmentHours: Math.round(netHours),
      utilizationRate: Math.round(utilizationRate)
    };

});

const teamTotal = plan.reduce((sum, m) => sum + m.netDevelopmentHours, 0);
const teamGross = plan.reduce((sum, m) => sum + m.grossHours, 0);

return {
members: plan,
teamNetHours: teamTotal,
teamGrossHours: teamGross,
teamUtilization: Math.round(teamTotal / teamGross _ 100),
recommendation: "Plan for " + teamTotal + " development hours (" +
Math.round(teamTotal / teamGross _ 100) + "% of theoretical capacity). " +
"If your velocity assumes " + teamGross + " hours, you are overcommitting by " +
(teamGross - teamTotal) + " hours."
};
}

const sprint = calculateSprintCapacity([
{ name: "Alex", daysOff: 0, onCall: false, interviewSlots: 2, meetingsPerDay: 1 },
{ name: "Jordan", daysOff: 2, onCall: true, interviewSlots: 0, meetingsPerDay: 0.5 },
{ name: "Priya", daysOff: 0, onCall: false, interviewSlots: 0, meetingsPerDay: 1 },
{ name: "Marcus", daysOff: 1, onCall: false, interviewSlots: 1, meetingsPerDay: 1.5 },
{ name: "Sam", daysOff: 0, onCall: false, interviewSlots: 0, meetingsPerDay: 0.5 }
]);

console.log("=== Sprint Capacity Plan ===\\n");
sprint.members.forEach(m => {
console.log(m.name + ": " + m.netDevelopmentHours + "h net / " +
m.grossHours + "h gross (" + m.utilizationRate + "% utilization)");
});
console.log("\\nTeam total:", sprint.teamNetHours + "h / " + sprint.teamGrossHours + "h");
console.log("\\n" + sprint.recommendation);`}

## What you learned

- Velocity is a measurement of sustainable output, not a target — the moment you set a velocity goal, you incentivize point inflation and burnout
- Real sprint capacity is 60-70% of theoretical capacity after accounting for ceremonies, code review, meetings, on-call, and context switching
- Retrospectives fail not because of format but because of follow-through — limit action items to two, assign owners, and review them at the start of the next retro
