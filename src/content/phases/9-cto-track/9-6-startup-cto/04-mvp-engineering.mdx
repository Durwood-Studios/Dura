## Why this matters

The MVP is the highest-stakes code you will ever write. Not because it needs to be perfect — it does not. But because the decisions you make about what to build, what to skip, and what to cut corners on determine whether you ship in time to learn from customers or run out of money building something nobody wants. The art of MVP engineering is knowing exactly where to be sloppy and where sloppiness will kill you.

## The speed-quality spectrum

Speed and quality are not opposites. They are a spectrum, and different parts of your MVP belong at different points.

```
NEVER SKIP                          SAFE TO SKIP
←——————————————————————————————————————→
Auth security    |  Tests   |  Pixel-perfect UI
Data integrity   |  Docs    |  i18n
Input validation |  Logging |  Dark mode
Backups          |  CI/CD   |  Animations
HTTPS            |          |  Performance optimization
```

The left side is non-negotiable. If your MVP leaks user passwords, loses customer data, or accepts SQL injection, you are done. No amount of speed justifies shipping a security vulnerability. The right side is polish. Polish matters, but not more than existing.

## What to never skip

### Authentication and authorization

Do not build your own auth. Use Supabase Auth, Auth0, Clerk, or Firebase Auth. But whatever you use, get it right. A startup that leaks credentials or allows unauthorized access loses trust permanently. There is no recovering from "they lost our data."

### Data integrity

If a user saves something, it must stay saved. If a transaction processes, it must be atomic. If data has a relationship, enforce it with foreign keys. Eventual consistency is fine for read models. It is not fine for "did the customer's payment go through."

```sql
-- Non-negotiable: foreign keys and constraints
CREATE TABLE orders (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id),
  total_cents INTEGER NOT NULL CHECK (total_cents > 0),
  status TEXT NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'paid', 'failed', 'refunded')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Non-negotiable: your data must survive a server crash
-- Use a managed database with automated backups.
-- Verify you can restore from backup BEFORE you need to.
```

### Input validation

Every piece of data that enters your system from the outside world must be validated. User input, webhook payloads, API responses from third parties — trust nothing. Use Zod or a similar schema validator at every boundary.

```typescript
import { z } from "zod";

// Validate at the boundary, not deep in the business logic
const CreateOrderSchema = z.object({
  productId: z.string().uuid(),
  quantity: z.number().int().min(1).max(100),
  // Do NOT trust client-sent prices — look up the price server-side
});

export function createOrder(rawInput: unknown): Order {
  const input = CreateOrderSchema.parse(rawInput);
  // Now input is typed and validated
  // ...
}
```

## What is safe to skip

### Comprehensive test suites

Write tests for your payment flow and your authentication logic. Skip tests for your landing page layout and your settings form. You are not skipping testing permanently — you are prioritizing what to test first based on blast radius.

### Pixel-perfect design

Ship with a component library or a basic Tailwind setup. The first ten customers do not care if your border radius is 8px or 12px. They care if the product solves their problem. You will redesign everything in six months anyway.

### Performance optimization

Do not cache anything until something is slow. Do not add a CDN until your images are actually slow to load. Do not add pagination until someone has enough data to need it. Premature optimization at the MVP stage is time stolen from feature development.

### Documentation

Your codebase documentation can be minimal at the MVP stage. Code comments on non-obvious decisions, a README with setup instructions, and your DECISIONS.md file. That is enough until you have a team.

## The throwaway vs. foundation decision

Every piece of MVP code falls into one of two categories:

**Throwaway code** — you know you will rewrite it. The landing page, the admin dashboard, the first version of the onboarding flow. Build it fast and dirty. Put a `// THROWAWAY:` comment on it so future-you knows this was intentional, not negligent.

**Load-bearing code** — it will be in production for years. The database schema, the authentication system, the core data model, the API contract with your first paying customers. Build this properly. Take the extra hour to get the schema right because migrating a production database is ten times harder than designing it correctly the first time.

```typescript
// THROWAWAY: This admin page is a quick hack for internal use.
// Rebuild with proper RBAC when we have more than 3 admin users.
function AdminDashboard() {
  // Hard-coded admin email check — intentionally quick and dirty
  if (user.email !== "dustin@durwood.dev") {
    return <p>Not authorized</p>;
  }
  // ...
}
```

```typescript
// LOAD-BEARING: This is the core order processing pipeline.
// Changes here affect billing, inventory, and customer notifications.
// Do not modify without understanding the full flow.
async function processOrder(order: ValidatedOrder): Promise<OrderResult> {
  return await db.transaction(async (tx) => {
    const inventory = await tx.checkInventory(order.productId, order.quantity);
    if (!inventory.available) {
      return { success: false, reason: "out_of_stock" };
    }
    await tx.reserveInventory(order.productId, order.quantity);
    const payment = await tx.chargeCustomer(order.userId, order.totalCents);
    if (!payment.success) {
      // Transaction rolls back automatically — inventory unreserved
      return { success: false, reason: "payment_failed" };
    }
    return { success: true, orderId: order.id };
  });
}
```

## The scope trap

The most common MVP failure is scope creep. The CEO wants "just one more feature." The designer wants "just a little more polish." You want "just a bit more architecture." Each addition feels small. Together they push your launch from four weeks to four months.

Fight scope with a hard rule: **if it is not required for the first paying customer to succeed, it is not in the MVP**. Write everything else on a list. You will build it later. Some of it you will never build, because customers will tell you they need something completely different.

## What you learned

- MVP engineering is not about writing bad code — it is about knowing exactly where quality matters (auth, data, validation) and where speed wins (UI polish, tests for non-critical paths, optimization).
- Label code as **throwaway** or **load-bearing** explicitly. Build throwaway code fast. Build load-bearing code right.
- Scope creep is the most common MVP killer. Apply a hard filter: if the first paying customer does not need it, cut it.
- Never skip authentication security, data integrity, input validation, backups, or HTTPS. No amount of shipping speed justifies these shortcuts.
