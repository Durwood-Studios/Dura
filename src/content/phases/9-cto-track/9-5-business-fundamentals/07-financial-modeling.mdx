## Why this matters

The CFO builds the financial model. The CEO presents it to investors. But the CTO's inputs determine whether the model is grounded in reality or wishful thinking.

When the CEO asks "Can we ship the enterprise tier by Q3?", the answer changes the revenue projection. When you propose hiring three more engineers, the burn rate changes. When you estimate that the platform migration saves $400K annually, that flows into the margin forecast. Every commitment you make in a planning meeting becomes a number in a spreadsheet that investors, board members, and lenders will scrutinize.

A CTO who cannot build a basic financial model is a CTO who gets committed to impossible timelines by people who do not understand the constraints. A CTO who can model scenarios brings data to the negotiation instead of opinions.

## Burn Rate

Burn rate is how fast the company spends money. It is the clock ticking toward zero.

```javascript
function calculateBurnRate(financials) {
  const grossBurn = financials.monthlyExpenses;
  const netBurn = financials.monthlyExpenses - financials.monthlyRevenue;

  return {
    grossBurn,
    netBurn,
    grossBurnAnnual: grossBurn * 12,
    netBurnAnnual: netBurn * 12,
    revenueOffsetPercent: Math.round(
      (financials.monthlyRevenue / financials.monthlyExpenses) * 100
    ),
  };
}

// Early-stage startup
console.log(
  calculateBurnRate({
    monthlyExpenses: 350_000,
    monthlyRevenue: 80_000,
  })
);
// Gross burn: $350K/month, Net burn: $270K/month
// Revenue covers 23% of expenses

// The CTO controls a large portion of burn:
const CTO_BURN_LEVERS = {
  headcount: "65-80% of tech budget. Every hire is $15-25K/month fully loaded.",
  infrastructure: "Cloud, tools, licenses. Often 10-20% of tech budget.",
  contractors: "Flexible capacity but expensive per hour. Easy to let costs drift.",
  techDebtInterest:
    "Not on the P&L, but it slows everything down. " +
    "A team running at 60% velocity because of debt is burning 40% of its cost on nothing.",
};
```

## Runway

Runway is how many months of operation the company can fund at the current burn rate. When runway gets short, bad decisions get made — panic hires stop, morale drops, and the company negotiates from weakness.

```javascript
function calculateRunway(cashBalance, monthlyNetBurn, burnTrend) {
  if (monthlyNetBurn <= 0) {
    return {
      months: Infinity,
      status: "CASH FLOW POSITIVE",
      message: "The company generates more than it spends.",
    };
  }

  const currentRunway = cashBalance / monthlyNetBurn;

  // Project runway with burn trend (burn growing or shrinking)
  let projectedRunway = currentRunway;
  if (burnTrend !== 0) {
    let remaining = cashBalance;
    let months = 0;
    let currentBurn = monthlyNetBurn;
    while (remaining > 0 && months < 120) {
      remaining -= currentBurn;
      currentBurn *= 1 + burnTrend; // burnTrend: 0.02 = 2% monthly increase
      months++;
    }
    projectedRunway = months;
  }

  return {
    months: Math.round(currentRunway),
    projectedMonths: Math.round(projectedRunway),
    status:
      currentRunway > 18
        ? "COMFORTABLE"
        : currentRunway > 12
          ? "PLAN FUNDRAISE"
          : currentRunway > 6
            ? "URGENT — fundraise or cut costs now"
            : "CRITICAL — survival mode",
    mustRaiseBy: new Date(Date.now() + Math.max(0, currentRunway - 6) * 30 * 24 * 60 * 60 * 1000)
      .toISOString()
      .slice(0, 7),
  };
}

// $5M in the bank, burning $270K/month net, burn growing 3% monthly
console.log(calculateRunway(5_000_000, 270_000, 0.03));
// ~18.5 months at current rate, but only ~16 months with growing burn
// Status: PLAN FUNDRAISE
```

## Revenue Projections

Revenue projections are where the CTO's input matters most — because feature delivery timelines directly determine when revenue can be recognized.

```javascript
function projectRevenue(inputs) {
  const months = [];
  let currentMRR = inputs.currentMRR;
  let customers = inputs.currentCustomers;

  for (let month = 1; month <= inputs.projectionMonths; month++) {
    // Organic growth: existing customers expand
    const expansion = currentMRR * (inputs.monthlyExpansionRate / 100);

    // New customer acquisition
    const newCustomers = Math.round(inputs.monthlyNewCustomers);
    const newRevenue = newCustomers * inputs.avgNewCustomerMRR;

    // Churn
    const churnedCustomers = Math.round(customers * (inputs.monthlyChurnRate / 100));
    const churnedRevenue = churnedCustomers * (currentMRR / customers);

    // Feature-driven revenue (unlocked by engineering milestones)
    let featureRevenue = 0;
    for (const feature of inputs.featureMilestones) {
      if (feature.launchMonth === month) {
        featureRevenue = feature.estimatedMRRImpact;
      }
    }

    customers = customers + newCustomers - churnedCustomers;
    currentMRR = currentMRR + expansion + newRevenue - churnedRevenue + featureRevenue;

    months.push({
      month,
      mrr: Math.round(currentMRR),
      arr: Math.round(currentMRR * 12),
      customers,
      netNewMRR: Math.round(expansion + newRevenue - churnedRevenue + featureRevenue),
    });
  }

  return months;
}

const projection = projectRevenue({
  currentMRR: 80_000,
  currentCustomers: 200,
  monthlyNewCustomers: 15,
  avgNewCustomerMRR: 300,
  monthlyExpansionRate: 1.5, // existing customers grow 1.5%/month
  monthlyChurnRate: 3, // 3% of customers leave each month
  projectionMonths: 12,
  featureMilestones: [
    { name: "Enterprise SSO", launchMonth: 4, estimatedMRRImpact: 15_000 },
    { name: "API Platform", launchMonth: 8, estimatedMRRImpact: 25_000 },
  ],
});

// Notice: feature launch months directly change the revenue curve.
// When you tell the CEO "SSO ships in Q2", that becomes a revenue assumption.
// If SSO slips to Q3, the model breaks. That is why timeline commitments matter.
```

## Scenario Modeling

Smart CTOs never present a single plan. They present three scenarios and let leadership choose.

```javascript
function scenarioModel(baseInputs) {
  const scenarios = {
    conservative: {
      label: "Conservative — current trajectory, no new hires",
      inputs: {
        ...baseInputs,
        newHires: 0,
        featureDelay: 2, // features slip 2 months
        churnRate: baseInputs.churnRate * 1.1, // slightly worse retention
      },
    },
    base: {
      label: "Base — planned hires, on-time delivery",
      inputs: { ...baseInputs },
    },
    aggressive: {
      label: "Aggressive — accelerated hiring, early delivery",
      inputs: {
        ...baseInputs,
        newHires: baseInputs.newHires + 3,
        featureDelay: -1, // features ship 1 month early
        churnRate: baseInputs.churnRate * 0.9, // better retention from faster features
      },
    },
  };

  const results = {};

  for (const [name, scenario] of Object.entries(scenarios)) {
    const monthlyBurn =
      baseInputs.currentBurn + (scenario.inputs.newHires * baseInputs.costPerHire) / 12;
    const additionalRevenue = (scenario.inputs.newHires * baseInputs.revenuePerEngineer) / 12;
    const endOfYearMRR =
      baseInputs.currentMRR * (1 + baseInputs.growthRate / 100) ** 12 +
      additionalRevenue * (12 - scenario.inputs.featureDelay);

    results[name] = {
      label: scenario.label,
      monthlyBurn: Math.round(monthlyBurn),
      endOfYearMRR: Math.round(endOfYearMRR),
      endOfYearARR: Math.round(endOfYearMRR * 12),
      additionalCost: Math.round(scenario.inputs.newHires * baseInputs.costPerHire),
      runway: Math.round(baseInputs.cashBalance / monthlyBurn),
    };
  }

  return results;
}

// This gives leadership a choice with clear tradeoffs:
// Conservative: lower cost, lower growth, longer runway
// Base: balanced execution
// Aggressive: higher cost, higher growth, shorter runway
```

## What you learned

- Burn rate and runway are the vital signs of a startup — the CTO controls a large portion of both through headcount, infrastructure, and build-vs-buy decisions.
- Revenue projections depend on feature delivery timelines, making every engineering commitment a financial commitment that flows into investor models.
- Scenario modeling (conservative, base, aggressive) gives leadership actionable tradeoffs instead of a single plan that obscures risk.
