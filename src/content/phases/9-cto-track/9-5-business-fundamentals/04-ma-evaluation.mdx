## Why this matters

When your company considers acquiring another company — or gets acquired itself — the CTO becomes the gatekeeper of the most consequential technical decision the business will make. A bad acquisition can burn years of engineering effort and tens of millions of dollars in integration costs that nobody anticipated because the technical assessment was superficial.

The CEO sees revenue synergies. The CFO sees financial arbitrage. You see the codebase that your team will inherit, the architectural decisions that will constrain you for years, and the engineers who may or may not stay through integration. Your assessment can greenlight a deal, reshape its terms, or kill it entirely. That is not a responsibility to take lightly.

## The Technical Due Diligence Framework

```javascript
const TECH_DUE_DILIGENCE = {
  codebase: {
    weight: 0.25,
    dimensions: [
      {
        name: "Code quality and consistency",
        questions: [
          "Is the code well-tested? What is the test coverage?",
          "Is there a consistent coding style and architecture pattern?",
          "How much commented-out code and dead code exists?",
          "Are dependencies up to date and free of known vulnerabilities?",
        ],
      },
      {
        name: "Technical debt inventory",
        questions: [
          "Is tech debt documented or discovered during exploration?",
          "Are there known performance bottlenecks or scaling limits?",
          "How many workarounds and hacks exist in production?",
          "What would a senior engineer's honest assessment be?",
        ],
      },
    ],
  },

  architecture: {
    weight: 0.25,
    dimensions: [
      {
        name: "Compatibility with acquirer's stack",
        questions: [
          "Same language ecosystem or requires translation layer?",
          "Compatible data models or requires ETL migration?",
          "Overlapping infrastructure (AWS to AWS) or cross-cloud?",
          "API-first design or tightly coupled monolith?",
        ],
      },
      {
        name: "Scalability and reliability",
        questions: [
          "Can the system handle the acquirer's traffic volume?",
          "What is the actual uptime history (not the SLA promise)?",
          "Is the system horizontally scalable or vertically constrained?",
          "Disaster recovery: tested or theoretical?",
        ],
      },
    ],
  },

  team: {
    weight: 0.3, // people are the most valuable and fragile asset
    dimensions: [
      {
        name: "Retention risk",
        questions: [
          "Key engineers with cliff vesting that completes post-close?",
          "Cultural compatibility between engineering teams?",
          "Compensation gap between acquirer and target?",
          "Geographic distribution — remote-friendly or office-dependent?",
        ],
      },
      {
        name: "Knowledge concentration",
        questions: [
          "Bus factor for critical systems?",
          "Documentation quality — could a new team operate the system?",
          "Tribal knowledge vs. written runbooks?",
          "Key individuals with non-compete or non-solicit agreements?",
        ],
      },
    ],
  },

  ip: {
    weight: 0.2,
    dimensions: [
      {
        name: "Intellectual property",
        questions: [
          "Who owns the code? Employee agreements assign IP to company?",
          "Open-source license compliance — any copyleft contamination?",
          "Patents filed? Pending? Defensible?",
          "Third-party code or libraries with restrictive licenses?",
        ],
      },
    ],
  },
};
```

## Scoring an Acquisition Target

```javascript
function scoreTechTarget(assessment) {
  const categories = {
    codeQuality: {
      score: 0,
      maxScore: 30,
      factors: [
        { name: "Test coverage", score: scoreTestCoverage(assessment.testCoverage) },
        {
          name: "Dependency health",
          score: scoreDependencies(assessment.outdatedDeps, assessment.knownCVEs),
        },
        { name: "Architecture clarity", score: assessment.hasArchDocs ? 10 : 3 },
      ],
    },
    compatibility: {
      score: 0,
      maxScore: 25,
      factors: [
        { name: "Stack overlap", score: scoreStackOverlap(assessment.stackOverlapPercent) },
        { name: "Data model compatibility", score: assessment.dataModelCompatible ? 10 : 2 },
        { name: "API maturity", score: assessment.hasPublicAPI ? 5 : 1 },
      ],
    },
    teamRetention: {
      score: 0,
      maxScore: 30,
      factors: [
        { name: "Key person retention plan", score: assessment.retentionPackages ? 15 : 3 },
        { name: "Culture compatibility", score: assessment.cultureScore },
        { name: "Compensation alignment", score: assessment.compAligned ? 10 : 4 },
      ],
    },
    integrationRisk: {
      score: 0,
      maxScore: 15,
      factors: [
        {
          name: "Estimated integration months",
          score: scoreIntegrationTime(assessment.integrationMonths),
        },
        {
          name: "Customer migration complexity",
          score: assessment.migrationComplexity <= 3 ? 10 : 3,
        },
      ],
    },
  };

  for (const cat of Object.values(categories)) {
    cat.score = cat.factors.reduce((sum, f) => sum + f.score, 0);
  }

  const totalScore = Object.values(categories).reduce((sum, c) => sum + c.score, 0);
  const maxScore = Object.values(categories).reduce((sum, c) => sum + c.maxScore, 0);

  return {
    categories,
    totalScore,
    maxScore,
    percentage: Math.round((totalScore / maxScore) * 100),
    recommendation:
      totalScore / maxScore >= 0.75
        ? "PROCEED — strong technical fit"
        : totalScore / maxScore >= 0.5
          ? "PROCEED WITH CONDITIONS — negotiate integration budget into deal terms"
          : "RECONSIDER — integration costs may exceed acquisition value",
  };
}

function scoreTestCoverage(coverage) {
  if (coverage >= 70) return 10;
  if (coverage >= 40) return 6;
  return 2;
}

function scoreDependencies(outdated, cves) {
  if (cves > 5) return 1;
  if (outdated > 30) return 3;
  if (outdated > 10) return 6;
  return 10;
}

function scoreStackOverlap(percent) {
  if (percent >= 80) return 10;
  if (percent >= 50) return 6;
  return 2;
}

function scoreIntegrationTime(months) {
  if (months <= 3) return 10;
  if (months <= 6) return 7;
  if (months <= 12) return 4;
  return 1;
}
```

## Integration Strategies

The acquisition closes. Now what? How you integrate the acquired technology determines whether the deal creates value or destroys it.

```javascript
const INTEGRATION_STRATEGIES = {
  absorb: {
    description: "Migrate acquired product into acquirer's platform",
    when: "Feature acquisition — you want the capability, not the codebase",
    timeline: "6-18 months",
    risk: "HIGH — rebuilding features risks breaking what made them valuable",
    teamImpact: "Acquired engineers learn new stack or leave",
    example:
      "Acquire a notification service. Rebuild its logic inside your " +
      "existing platform. Sunset the acquired codebase.",
  },

  preserve: {
    description: "Keep acquired product running independently",
    when: "The product serves a different market or the codebase is too different to merge",
    timeline: "1-3 months for operational integration only",
    risk: "LOW for technology, HIGH for organizational fragmentation",
    teamImpact: "Acquired team stays largely intact",
    example:
      "Acquire a B2B analytics product while you serve B2C. " +
      "Keep it running separately. Share only back-office systems.",
  },

  converge: {
    description: "Gradually merge the best of both platforms over time",
    when: "Both products serve similar markets with complementary strengths",
    timeline: "12-36 months",
    risk: "MEDIUM — requires sustained leadership attention",
    teamImpact: "Teams gradually merge; requires careful org design",
    example:
      "Both companies have e-commerce platforms. Extract the best " +
      "checkout flow from one and the best inventory system from the other. " +
      "Build a unified platform over 2 years.",
  },
};

function estimateIntegrationCost(strategy, acquiredTeamSize, avgEngineerCost) {
  const multipliers = {
    absorb: { engineeringMonths: acquiredTeamSize * 4, attritionRate: 0.4 },
    preserve: { engineeringMonths: acquiredTeamSize * 1, attritionRate: 0.15 },
    converge: { engineeringMonths: acquiredTeamSize * 6, attritionRate: 0.25 },
  };

  const m = multipliers[strategy];
  const engineeringCost = m.engineeringMonths * (avgEngineerCost / 12);
  const attritionCost = Math.round(acquiredTeamSize * m.attritionRate) * avgEngineerCost * 0.5;
  // Attrition cost: replacing departing engineers (recruiting + ramp)

  return {
    strategy,
    engineeringCost: Math.round(engineeringCost),
    expectedAttrition: `${Math.round(m.attritionRate * 100)}%`,
    attritionReplacementCost: Math.round(attritionCost),
    totalEstimate: Math.round(engineeringCost + attritionCost),
  };
}

// 15-person acquired team, $220K avg fully loaded cost
console.log(estimateIntegrationCost("absorb", 15, 220_000));
console.log(estimateIntegrationCost("preserve", 15, 220_000));
console.log(estimateIntegrationCost("converge", 15, 220_000));
```

## The Acqui-Hire Assessment

Sometimes the acquisition is really about the team, not the technology. The CTO's role shifts from evaluating code to evaluating people.

## What you learned

- Technical due diligence evaluates code quality, architecture compatibility, team retention risk, and IP ownership — each can independently kill or reshape a deal.
- Integration strategy choice (absorb, preserve, converge) depends on stack overlap and market overlap, and each carries different cost and attrition profiles.
- The CTO's assessment directly affects deal terms — integration cost estimates should be factored into the acquisition price, not discovered after closing.
