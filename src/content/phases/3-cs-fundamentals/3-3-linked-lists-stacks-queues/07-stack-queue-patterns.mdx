## Why this matters

A handful of advanced patterns built on stacks and queues appear so frequently in technical interviews and real systems code that they deserve their own treatment. The monotonic stack solves "next greater element" and related problems in O(n) where a naive approach would be O(n²). The deque-based sliding window maximum processes streaming data with guaranteed O(n) performance. The queue-from-two-stacks shows how to build one abstract data type from another — a pattern of practical and theoretical importance. Each of these is worth understanding deeply, not just memorizing.

## The monotonic stack: next greater element

Given an array, find for each element the next element to its right that is greater. For `[2, 1, 2, 4, 3]`, the answer is `[4, 2, 4, -1, -1]` — the next greater element after index 2 is 4, after index 4 there is nothing, so -1.

The naive approach compares every element to every element to its right: O(n²).

The monotonic stack approach is O(n). The key insight: maintain a stack of indices of elements that have **not yet found their next greater element**. When you encounter a new element that is larger than what is on top of the stack, the top element's answer is this new element.

```javascript
function nextGreaterElement(arr) {
  const result = new Array(arr.length).fill(-1);
  const stack = []; // stack of indices (not values)

  for (let i = 0; i < arr.length; i++) {
    // Pop everything that is smaller than the current element
    while (stack.length > 0 && arr[stack[stack.length - 1]] < arr[i]) {
      const idx = stack.pop();
      result[idx] = arr[i]; // arr[i] is the next greater element for idx
    }
    stack.push(i);
  }

  // Whatever remains in the stack has no greater element to the right — stays -1
  return result;
}

nextGreaterElement([2, 1, 2, 4, 3]);
// [4, 2, 4, -1, -1]
```

**Why is this O(n)?** Each index is pushed exactly once and popped at most once. No matter how many pops happen at a given step, the total number of pushes across the entire run is n, so the total number of pops is also at most n. The work is O(n) amortized — individual iterations may do many pops, but the total cannot exceed 2n operations.

The stack is "monotonic" in the sense that it always holds indices in increasing order of their values — elements waiting for a greater element are kept in non-increasing value order from bottom to top. The moment an element finds its answer, it leaves.

## Sliding window maximum with a deque

Given an array and a window size `k`, find the maximum value in each window of k consecutive elements. Naive is O(nk); deque-based is O(n).

Maintain a deque of array indices. The invariant: the front of the deque is always the index of the current window's maximum. Elements are added at the back; elements that fall outside the window are evicted from the front.

```javascript
function slidingWindowMax(arr, k) {
  const result = [];
  // Deque stores indices; front is the index of the current window max
  const deque = []; // simulated with array: push = pushBack, deque[0] = front

  for (let i = 0; i < arr.length; i++) {
    // Remove indices that are outside the current window
    while (deque.length > 0 && deque[0] < i - k + 1) {
      deque.shift(); // evict from front
    }

    // Remove indices whose values are smaller than arr[i]
    // — they can never be the maximum of any future window
    while (deque.length > 0 && arr[deque[deque.length - 1]] < arr[i]) {
      deque.pop(); // evict from back
    }

    deque.push(i);

    // The window is full once i >= k - 1
    if (i >= k - 1) {
      result.push(arr[deque[0]]); // front of deque is current window max
    }
  }

  return result;
}

slidingWindowMax([1, 3, -1, -3, 5, 3, 6, 7], 3);
// [3, 3, 5, 5, 6, 7]
```

Note: `deque.shift()` is used once per element at most — each index enters and leaves at most once — so the total work is O(n) even though the loop contains a `shift()`.

## Queue from two stacks

Can you build a queue using only stacks? Yes — and the trick has practical value for implementing queues in languages that only have stacks, or for understanding amortized analysis.

Maintain two stacks: `inbox` and `outbox`. Enqueue pushes to inbox. Dequeue pops from outbox — if outbox is empty, pour all items from inbox into outbox (reversing their order, which converts LIFO to FIFO).

```javascript
class QueueFromTwoStacks {
  constructor() {
    this._inbox = [];
    this._outbox = [];
  }

  enqueue(val) {
    this._inbox.push(val);
  }

  dequeue() {
    if (this._outbox.length === 0) {
      // Pour inbox into outbox — reverses order, making oldest item on top
      while (this._inbox.length > 0) {
        this._outbox.push(this._inbox.pop());
      }
    }
    if (this._outbox.length === 0) throw new Error("Queue underflow");
    return this._outbox.pop();
  }

  isEmpty() {
    return this._inbox.length === 0 && this._outbox.length === 0;
  }
}
```

**Amortized analysis**: each element is pushed to inbox once (O(1)), transferred from inbox to outbox once (O(1)), and popped from outbox once (O(1)). That is three O(1) operations per element across its lifetime. Any single dequeue might do O(n) work if it has to transfer n elements — but only if those n elements have never been transferred before. Averaged across all operations, each dequeue is O(1) amortized.

This is the same amortized reasoning used to explain why JavaScript's dynamic array push is O(1) amortized even though occasional resizes are O(n).

## Postfix expression evaluation

Postfix (reverse Polish notation) evaluates expressions without parentheses by using a stack. Numbers are pushed; operators pop the top two values, apply the operator, and push the result.

```javascript
function evalPostfix(tokens) {
  const stack = [];

  for (const token of tokens) {
    if (!isNaN(Number(token))) {
      stack.push(Number(token));
    } else {
      const b = stack.pop();
      const a = stack.pop();
      if (token === "+") stack.push(a + b);
      else if (token === "-") stack.push(a - b);
      else if (token === "*") stack.push(a * b);
      else if (token === "/") stack.push(Math.trunc(a / b));
    }
  }

  return stack[0];
}

evalPostfix(["2", "1", "+", "3", "*"]); // (2+1)*3 = 9
```

Compilers and calculators use this technique because postfix notation makes operator precedence explicit in the token order — no need to parse parentheses.

## What you learned

- The monotonic stack finds the next greater element for all array elements in O(n) total — each index is pushed and popped at most once.
- The sliding window maximum uses a deque to maintain the current window's maximum index in O(n) without re-scanning the window.
- Queue-from-two-stacks achieves O(1) amortized dequeue by lazily transferring from inbox to outbox only when outbox is empty.
- Postfix expression evaluation uses a stack to compute results left-to-right without parsing parentheses.
- Amortized analysis proves O(1) per operation by bounding total work across n operations, not individual cost.
