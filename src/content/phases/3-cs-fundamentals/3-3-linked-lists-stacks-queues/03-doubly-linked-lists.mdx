## Why this matters

A singly linked list has a painful limitation: to delete a node, you need its predecessor. If someone hands you a direct reference to a node and says "delete this," you still have to traverse from the head to find the node that points to it — O(n). A doubly linked list solves this by giving every node a `prev` pointer. When you have a direct reference to any node, deletion is O(1): update the neighbors and bypass the target. This property is the reason doubly linked lists power LRU caches, text editor undo buffers, and browser history — all cases where you need O(1) removal of an arbitrary node.

## The doubly linked node

Each node holds three things: a value, a pointer to the next node, and a pointer to the previous node.

```javascript
class DNode {
  constructor(val) {
    this.val = val;
    this.next = null;
    this.prev = null;
  }
}
```

When you link two nodes together, you must update both directions:

```javascript
// Link a → b
a.next = b;
b.prev = a;
```

Every insertion and deletion now touches four pointers instead of two, and the cost of a mistake doubles — forgetting one pointer leaves the list in an inconsistent state where forward and backward traversals disagree.

## Sentinel nodes: eliminating edge cases

One of the most useful techniques in doubly linked list implementation is the **sentinel node** (also called a dummy node). Instead of letting `head` and `tail` be `null` for an empty list, you create two permanent placeholder nodes — one at the front (dummy head) and one at the back (dummy tail) — that never hold real data.

```javascript
class DoublyLinkedList {
  constructor() {
    this.head = new DNode(null); // dummy head sentinel
    this.tail = new DNode(null); // dummy tail sentinel
    this.head.next = this.tail;
    this.tail.prev = this.head;
    this._size = 0;
  }
}
```

The sentinel nodes contain the entire list between them: `sentinel_head ↔ [real nodes] ↔ sentinel_tail`. Every real node always has both a valid `prev` and a valid `next` — they are never null. This eliminates the special-case logic for empty lists and boundary conditions, making insertion and deletion code identical regardless of where the operation occurs.

## Insert after a given node

With sentinels in place, inserting a new node after any existing node `node` is four pointer updates:

```javascript
insertAfter(node, val) {
  const newNode = new DNode(val);
  const successor = node.next;

  newNode.prev = node;
  newNode.next = successor;
  node.next = newNode;
  successor.prev = newNode;

  this._size++;
  return newNode; // return the new node so callers can cache a reference
}
```

To add to the front of the real list, call `insertAfter(this.head, val)`. To add to the back, call `insertAfter(this.tail.prev, val)`. No special-casing needed — sentinels handle the boundaries.

## Remove a node in O(1)

Given a direct reference to any real node, removal is O(1):

```javascript
removeNode(node) {
  const predecessor = node.prev;
  const successor = node.next;

  predecessor.next = successor;
  successor.prev = predecessor;

  node.next = null; // help GC sever the removed node from the list
  node.prev = null;

  this._size--;
}
```

Compare this to a singly linked list, where you would need to traverse from head to find the predecessor first. In a doubly linked list, you already have it: `node.prev`.

## Bidirectional traversal

Forward and backward traversal both work:

```javascript
toArray() {
  const result = [];
  let curr = this.head.next; // start after dummy head
  while (curr !== this.tail) {
    result.push(curr.val);
    curr = curr.next;
  }
  return result;
}

toArrayReverse() {
  const result = [];
  let curr = this.tail.prev; // start before dummy tail
  while (curr !== this.head) {
    result.push(curr.val);
    curr = curr.prev;
  }
  return result;
}
```

## Real-world applications

**Browser history.** Clicking the back and forward buttons are O(1) pointer movements in a doubly linked list. The current page is a node; back moves to `curr.prev`, forward moves to `curr.next`. Visiting a new page truncates the forward chain by removing everything after `curr`.

**Text editor undo.** Each edit is a node. Undo moves backward through the list, redo moves forward. Insert and delete (the edit operations) are O(1) pointer updates on the undo node, not O(n) array shifts.

**LRU cache.** A Least Recently Used cache combines a doubly linked list with a hash map. The list maintains access order — most recently used at the front, least recently used at the back. The hash map provides O(1) lookup of any node by key. When a key is accessed, its node is removed from its current position (O(1) because you have a direct reference) and moved to the front (O(1) with sentinels). When the cache is full and you need to evict, you remove from the back — O(1) via `tail.prev`.

This combined structure delivers O(1) get and O(1) put, which is why LRU cache appears in countless systems from CPU caches to CDN eviction policies.

## The cost of the extra pointer

Doubly linked lists use more memory than singly linked lists — each node now carries two references instead of one. For a list of `n` nodes in a language with 8-byte pointers, that is an extra `8n` bytes. For most applications this is negligible. The O(1) deletion guarantee is worth it whenever you need to remove arbitrary nodes by direct reference.

## What you learned

- Doubly linked nodes carry both .next and .prev, enabling O(1) removal of any node given a direct reference.
- Sentinel nodes eliminate null-boundary edge cases, making insertion and deletion code uniform across all positions.
- Insertion after a given node updates exactly four pointers; removal updates two.
- LRU caches, browser history, and undo systems are canonical real-world uses of doubly linked lists.
- The cost is one extra pointer per node — typically worth it when O(1) arbitrary deletion is required.
