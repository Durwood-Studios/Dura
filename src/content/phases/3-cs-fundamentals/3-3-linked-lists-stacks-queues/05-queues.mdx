## Why this matters

Queues model the natural order of the physical world: whoever arrived first gets served first. Every operating system uses queues to schedule CPU time. Every network stack uses queues to buffer packets. Every event loop — including the one running your JavaScript — uses a queue to decide which callback runs next. Understanding how to implement an efficient queue is the difference between an O(1) dequeue and an O(n) one — a gap that becomes catastrophic at scale.

## The FIFO principle

A queue follows **First In, First Out** — items are added at the back and removed from the front. Like a line at a ticket counter: the first person in line is the first to be served.

Four core operations:

- **enqueue(val)**: add an item to the back — O(1)
- **dequeue()**: remove and return the front item — O(1)
- **peek()**: look at the front item without removing it — O(1)
- **isEmpty()**: check if the queue has items — O(1)

The crucial requirement is that both enqueue and dequeue are O(1). This is harder to achieve than it looks.

## The array pitfall: O(n) dequeue

The intuitive JavaScript implementation reaches for an array, using `push()` to enqueue and `shift()` to dequeue:

```javascript
const queue = [];
queue.push("a"); // enqueue
queue.push("b");
queue.shift(); // dequeue — returns "a"
```

This works correctly — but `shift()` is O(n). Every dequeue operation copies every remaining element one position forward in memory. At 100,000 elements, each dequeue takes 100,000 copy operations. If you enqueue and dequeue 100,000 items, the total work is O(n²).

This is a real performance trap. Many production bugs have been traced to `Array.shift()` in a hot loop.

## Efficient queue: offset into an internal array

The fix is to stop shrinking the array. Instead of shifting elements, track a `_head` index that advances forward:

```javascript
class Queue {
  constructor() {
    this._data = {};
    this._head = 0;
    this._tail = 0;
  }

  enqueue(val) {
    this._data[this._tail] = val;
    this._tail++;
  }

  dequeue() {
    if (this.isEmpty()) throw new Error("Queue underflow");
    const val = this._data[this._head];
    delete this._data[this._head];
    this._head++;
    return val;
  }

  peek() {
    if (this.isEmpty()) throw new Error("Queue is empty");
    return this._data[this._head];
  }

  isEmpty() {
    return this._head === this._tail;
  }

  size() {
    return this._tail - this._head;
  }
}
```

Using an object keyed by integer index, each dequeue is O(1): look up the current head index, return its value, advance the index, delete the old key. The object grows as items accumulate but never requires shifting. The trade-off is that consumed slots are deleted but the integer keys keep incrementing — memory is freed by `delete`, so this is fine in practice for most use cases.

## How the event loop uses a queue

JavaScript is single-threaded. When you call `setTimeout(fn, 0)` or resolve a promise, the callback does not run immediately — it is placed in a task queue (or microtask queue). The event loop repeatedly checks: "Is the call stack empty? If so, take the oldest item from the queue and run it."

This is FIFO scheduling. Callbacks are processed in the order they were enqueued. If you enqueue 100 callbacks, they run in exactly the order they arrived — no jumping the line.

Understanding this is why `await` and `setTimeout` behave the way they do: your code yields, a callback enters the queue, and it runs when the stack is clear.

## Breadth-first search: queues in algorithms

The most direct algorithmic use of a queue is **breadth-first search (BFS)**. BFS explores a graph or tree level by level — visit all nodes at distance 1, then all at distance 2, and so on. The queue enforces this order: when you visit a node, enqueue all its unvisited neighbors. Because of FIFO, you will process distance-1 nodes before distance-2 nodes.

```javascript
function bfs(graph, start) {
  const queue = new Queue();
  const visited = new Set();
  const order = [];

  queue.enqueue(start);
  visited.add(start);

  while (!queue.isEmpty()) {
    const node = queue.dequeue();
    order.push(node);

    for (const neighbor of graph[node] || []) {
      if (!visited.has(neighbor)) {
        visited.add(neighbor);
        queue.enqueue(neighbor);
      }
    }
  }

  return order;
}
```

Replace the queue with a stack and you get DFS. The data structure determines the traversal order — same algorithm, different behavior.

## Queue vs stack: opposite disciplines

| Property      | Stack (LIFO)         | Queue (FIFO)    |
| ------------- | -------------------- | --------------- |
| Add to        | Top                  | Back            |
| Remove from   | Top                  | Front           |
| Order         | Most recent first    | Oldest first    |
| Algorithm use | DFS, undo, backtrack | BFS, scheduling |
| Analogy       | Stack of plates      | Line at a store |

Both are abstractions over the same primitive data (a sequence of items). The difference is which end you remove from. This single decision completely changes the behavior of search algorithms and job schedulers.

## Real-world queues

**Print spooling**: documents sent to a printer are queued; they print in order of submission.

**CPU scheduling**: the operating system keeps a ready queue of processes. The scheduler dequeues the next process, runs it for a time slice, and if it is not done, requeues it.

**Message queues** (Kafka, RabbitMQ, SQS): distributed systems use queues to decouple producers from consumers. A web server enqueues jobs; a fleet of workers dequeues them. The queue buffers load spikes.

**Web server request handling**: incoming HTTP requests are queued when all worker threads are busy. A queue that grows without bound is a sign of insufficient capacity — or an O(n) dequeue bug.

## What you learned

- Queues are FIFO: enqueue at the back, dequeue from the front — both must be O(1).
- Array.shift() gives you a correct queue but with O(n) dequeue — use an offset index instead.
- The JavaScript event loop is a real queue: callbacks run in the order they were enqueued.
- BFS uses a queue to explore graphs level by level; swap in a stack and you get DFS.
- Message queues, CPU schedulers, print spoolers, and web servers all use FIFO ordering for fairness and predictability.
