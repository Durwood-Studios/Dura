## Why this matters

Arrays store elements in a contiguous block of memory, which makes index-based access instant but insertion and deletion at the front expensive — everything must shift over. Linked lists solve the opposite problem: insertion at the head is O(1), no matter how large the list, because you only update a single pointer. Understanding the trade-offs between arrays and linked lists is the foundation for every more complex data structure you will study — stacks, queues, trees, and graphs all build on the same node-and-pointer idea.

## What a node is

A linked list is made of **nodes**. Each node is a small object that holds two things: a **value** (whatever data you are storing) and a **pointer to the next node** in the chain. The pointer is just a reference — in JavaScript, it is a property that holds either a reference to another node object, or `null` to signal "this is the end."

```javascript
class Node {
  constructor(val) {
    this.val = val;
    this.next = null; // null means "no next node"
  }
}
```

You build a linked list by connecting nodes together manually:

```javascript
const a = new Node(1);
const b = new Node(2);
const c = new Node(3);

a.next = b; // 1 → 2
b.next = c; // 2 → 3
// c.next is already null: 3 → null
```

The list now looks like: `1 → 2 → 3 → null`.

## The head pointer

A linked list needs exactly one external reference to be useful: a pointer to the **first node**, called the **head**. If you lose the head reference, you lose the entire list. Every operation — traversal, insertion, deletion — starts from the head and works forward.

```javascript
class LinkedList {
  constructor() {
    this.head = null; // empty list: no head yet
  }
}
```

## Building the list: prepend and append

**Prepend** (add to the front) is the fast operation — O(1):

```javascript
prepend(val) {
  const newNode = new Node(val);
  newNode.next = this.head; // new node points to old head
  this.head = newNode;      // new node becomes the head
}
```

**Append** (add to the back) requires traversing to the last node first — O(n):

```javascript
append(val) {
  const newNode = new Node(val);
  if (!this.head) {
    this.head = newNode;
    return;
  }
  let curr = this.head;
  while (curr.next !== null) {
    curr = curr.next; // walk until we hit the tail
  }
  curr.next = newNode; // attach new node after the tail
}
```

## Traversal: following the chain

There is no random access in a linked list. You cannot write `list[3]` and get the fourth element instantly. To reach any node, you must start at the head and follow `.next` pointers until you arrive. This is called **traversal**, and it is always O(n) in the worst case.

```javascript
toArray() {
  const result = [];
  let curr = this.head;
  while (curr !== null) {
    result.push(curr.val);
    curr = curr.next;
  }
  return result;
}
```

The pattern `let curr = this.head; while (curr !== null) { ... curr = curr.next; }` is the backbone of almost every linked list algorithm. Memorize it.

## Finding the length

Because there is no stored size property (unless you add one), finding the length means counting traversal steps:

```javascript
length() {
  let count = 0;
  let curr = this.head;
  while (curr !== null) {
    count++;
    curr = curr.next;
  }
  return count;
}
```

This is O(n). If you need length frequently, maintain a `size` counter that increments on each insert and decrements on each delete.

## Why linked lists exist: the insertion advantage

Consider inserting a value at the front of a 1,000,000-element array. The runtime is O(n) — every existing element must shift one position to the right. At the front of a linked list, insertion is O(1) regardless of size: create a node, update two pointers, done.

| Operation       | Array  | Linked List |
| --------------- | ------ | ----------- |
| Access by index | O(1)   | O(n)        |
| Insert at front | O(n)   | O(1)        |
| Insert at back  | O(1)\* | O(n)†       |
| Delete at front | O(n)   | O(1)        |

\*amortized with dynamic array growth  
†O(1) if you maintain a tail pointer

## The cache locality trade-off

Arrays win on **cache locality**: their elements sit next to each other in memory, so when the CPU loads one element it pulls several neighboring elements into the cache simultaneously. Linked list nodes can be scattered anywhere in memory. Traversing a linked list causes frequent cache misses, which makes it slower in practice than the O(n) comparison suggests. On modern hardware, an array traversal can be 5–10x faster than a linked list traversal of the same length due to this effect.

The practical rule: **use an array by default**. Reach for a linked list when you specifically need O(1) head insertions or deletions, when you are implementing a stack or queue, or when you need to splice a node out of the middle of a collection without shifting everything else.

## What you learned

- A linked list is a chain of nodes, each holding a value and a pointer to the next node.
- The head pointer is the only external reference needed; losing it loses the list.
- Prepend is O(1); append and traversal are O(n) because they require walking the chain.
- Arrays have O(1) index access; linked lists have O(1) head insertion — know which trade-off you need.
- Cache locality makes arrays faster in practice for traversal-heavy workloads even when Big O is the same.
