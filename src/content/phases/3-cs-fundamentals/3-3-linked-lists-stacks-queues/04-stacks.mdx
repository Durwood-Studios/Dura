## Why this matters

The stack is one of the most useful data structures you will ever encounter — and it is already running inside every JavaScript engine you have ever used. Every function call you make pushes a frame onto the call stack. Every return pops one off. Understanding stacks means understanding how your program actually executes, and recognizing the pattern of "remember things in reverse order of arrival" unlocks a whole class of problems that would otherwise seem unrelated.

## The LIFO principle

A stack follows **Last In, First Out** — the most recently added item is always the first to be removed. Think of a stack of plates: you can only add or remove from the top. You cannot pull a plate from the middle without disturbing everything above it.

Stacks have exactly three core operations:

- **push(val)**: add an item to the top — O(1)
- **pop()**: remove and return the top item — O(1)
- **peek()**: look at the top item without removing it — O(1)

Both push and pop are O(1). There is no searching, no traversal, no sorting. A stack is deliberately restrictive — it only exposes the top, nothing else.

## Array-backed implementation

JavaScript arrays make a natural stack. `push()` adds to the end of the array (the "top"), and `pop()` removes from the end. Both are O(1) amortized.

```javascript
class Stack {
  constructor() {
    this._data = [];
  }

  push(val) {
    this._data.push(val);
  }

  pop() {
    if (this.isEmpty()) throw new Error("Stack underflow");
    return this._data.pop();
  }

  peek() {
    if (this.isEmpty()) throw new Error("Stack is empty");
    return this._data[this._data.length - 1];
  }

  isEmpty() {
    return this._data.length === 0;
  }

  size() {
    return this._data.length;
  }
}
```

Always check `isEmpty()` before calling `pop()` or `peek()`. An underflow — popping from an empty stack — is a common runtime error, and in languages without automatic bounds checking it can cause memory corruption.

## The call stack

Every time a JavaScript function is called, the engine creates a **stack frame** containing the function's local variables, arguments, and the return address (where execution should resume after the function returns). This frame is pushed onto the **call stack**.

```javascript
function a() {
  return b(); // push frame for b, then return
}

function b() {
  return c(); // push frame for c, then return
}

function c() {
  return 42; // return 42, pop frame for c
  // then pop frame for b
  // then pop frame for a
}

a();
```

When `a` calls `b`, a frame for `b` sits on top of the frame for `a`. When `c` returns, its frame is popped; when `b` returns, its frame is popped; when `a` returns, its frame is popped. Recursion works exactly the same way — each recursive call pushes a new frame.

**Stack overflow** is a literal name: when recursion is too deep (or an infinite recursive loop runs), the call stack runs out of allocated space and the engine throws a `RangeError: Maximum call stack size exceeded`. The stack is full.

## Recognizing when to use a stack

The mental model: use a stack whenever you need to **access the most recently seen thing first**. Common triggers:

- **Undo/redo**: each action is pushed; undo pops the last action and reverses it.
- **Bracket matching**: push open brackets, pop when you see a closing one — if the popped bracket matches, you are balanced.
- **Depth-first search**: push neighbors to explore; always expand the most recently pushed one.
- **Expression evaluation**: postfix notation uses a stack to compute results without parentheses.
- **Backtracking**: if a path fails, pop back to the last decision point and try another direction.

The signature of a stack problem: "I need to remember things and process them in reverse order of when I saw them."

## Bracket matching: a classic stack application

Given a string of brackets, determine if they are properly matched and nested. The stack approach is elegant:

- Encounter an open bracket (`(`, `[`, `{`): push it.
- Encounter a close bracket (`)`, `]`, `}`): pop from the stack and check if it matches.
- At the end: if the stack is empty, every open bracket was closed — balanced.

```javascript
function isBalanced(str) {
  const stack = new Stack();
  const matching = { ")": "(", "]": "[", "}": "{" };

  for (const char of str) {
    if ("([{".includes(char)) {
      stack.push(char);
    } else if (")]}".includes(char)) {
      if (stack.isEmpty()) return false; // close with nothing open
      if (stack.pop() !== matching[char]) return false; // wrong pair
    }
  }

  return stack.isEmpty(); // true only if all opens were closed
}

console.log(isBalanced("({[]})")); // true
console.log(isBalanced("({[})")); // false — ] closes before [
console.log(isBalanced("(((")); // false — unclosed opens remain
```

Notice how naturally LIFO fits the problem: the most recently opened bracket must be the next one closed. Nesting is exactly LIFO ordering.

## Stack overflow and recursion depth

Every language limits the call stack depth. In Node.js, the limit is typically around 10,000–15,000 frames, depending on the platform and frame size. A recursive function that does not reduce its input toward a base case will overflow.

The fix for stack overflow in recursive code is almost always **iterative rewriting using an explicit stack**: instead of letting the call stack hold the state, you manage your own stack data structure. The logic is identical; you just control the stack explicitly, and it can grow as large as heap memory allows.

## What you learned

- Stacks are LIFO: push adds to the top, pop removes from the top, both O(1).
- The JavaScript call stack is a literal stack of function frames — push on call, pop on return.
- Stack overflow happens when recursion depth exceeds the allocated stack space.
- Bracket matching, undo/redo, DFS, and expression evaluation all have LIFO structure — reach for a stack.
- An explicit stack can replace recursion when call stack depth is a concern.
