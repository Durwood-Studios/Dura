## Why this matters

Dynamic programming has a reputation for being difficult. That reputation is undeserved — DP is just careful brute force with a memory upgrade. The hard part is not the implementation; it is recognizing when a problem has the two required properties. Once you can identify those, you already know 80% of the solution.

## What dynamic programming actually is

Dynamic programming is a technique for solving problems that have two properties:

1. **Overlapping subproblems** — the same smaller computations appear many times when you break the problem down
2. **Optimal substructure** — the optimal solution to the full problem is built from optimal solutions to subproblems

If a problem has these two properties, you can avoid recomputing subproblems by caching their results. That is the entire idea. DP is not a clever algorithm — it is a smarter way to organize brute force.

## Two flavors: top-down vs bottom-up

**Top-down (memoization):** start with the original problem, recurse toward base cases, cache results on the way back up.

```javascript
// Top-down: think recursively, cache results
function fibTopDown(n, memo = {}) {
  if (n <= 1) return n;
  if (n in memo) return memo[n];
  memo[n] = fibTopDown(n - 1, memo) + fibTopDown(n - 2, memo);
  return memo[n];
}
```

**Bottom-up (tabulation):** start from the base cases and fill a table of answers upward until you reach the original problem.

```javascript
// Bottom-up: start at the base, build up
function fibBottomUp(n) {
  if (n <= 1) return n;
  const dp = new Array(n + 1);
  dp[0] = 0;
  dp[1] = 1;
  for (let i = 2; i <= n; i++) {
    dp[i] = dp[i - 1] + dp[i - 2];
  }
  return dp[n];
}
```

Both are O(n) time. Bottom-up avoids recursion overhead and stack depth limits. Top-down is often easier to derive because it mirrors the recursive structure of the problem.

## The DP thinking process

Every DP problem follows the same four-step derivation:

**Step 1 — Define the subproblem.** What does `dp[i]` represent? This is the most important step and the most commonly skipped.

**Step 2 — Write the recurrence.** Express `dp[i]` in terms of smaller dp values.

**Step 3 — Identify the base cases.** What are the smallest subproblems you can answer directly without recursion?

**Step 4 — Choose direction.** Top-down if the recursion is natural; bottom-up if you want to avoid stack depth or want to optimize space.

## Classic problem: Coin Change

Given coin denominations and a target amount, find the minimum number of coins needed. This is a perfect DP problem:

- **Overlapping subproblems?** Yes — to find the minimum coins for amount 15, you need to know the minimum for 14, 10, 4, and many others. These are needed from many different paths.
- **Optimal substructure?** Yes — if the optimal solution for 15 uses a coin of value 5, then the remaining 10 must also use an optimal solution.

**Define the subproblem:** `dp[i]` = minimum number of coins to make amount `i`.

**Recurrence:** `dp[i] = min(dp[i - coin] + 1)` for every coin denomination where `coin <= i`.

**Base case:** `dp[0] = 0` (zero coins needed to make amount 0).

```javascript
function coinChange(coins, amount) {
  // Initialize all amounts as "impossible" (Infinity)
  const dp = new Array(amount + 1).fill(Infinity);
  dp[0] = 0; // base case: 0 coins needed for amount 0

  for (let i = 1; i <= amount; i++) {
    for (const coin of coins) {
      if (coin <= i && dp[i - coin] !== Infinity) {
        dp[i] = Math.min(dp[i], dp[i - coin] + 1);
      }
    }
  }

  return dp[amount] === Infinity ? -1 : dp[amount];
}
```

Trace through `coins=[1,5,11], amount=15`:

- `dp[0]` = 0
- `dp[1]` = dp[0]+1 = 1 (one 1-coin)
- `dp[5]` = dp[0]+1 = 1 (one 5-coin)
- `dp[10]` = dp[5]+1 = 2 (two 5-coins)
- `dp[11]` = dp[0]+1 = 1 (one 11-coin)
- `dp[15]` = min(dp[14]+1, dp[10]+1, dp[4]+1) = min(14+1? no, 2+1=3, ...) = 3 (three 5-coins)

## Classic problem: Longest Common Subsequence

Given two strings, find the length of their longest common subsequence (characters in order, not necessarily contiguous).

**Subproblem:** `dp[i][j]` = length of LCS of `s1[0..i-1]` and `s2[0..j-1]`.

**Recurrence:**

- If `s1[i-1] === s2[j-1]`: `dp[i][j] = dp[i-1][j-1] + 1` (this character is part of the LCS)
- Otherwise: `dp[i][j] = max(dp[i-1][j], dp[i][j-1])` (skip one character from either string)

```javascript
function lcs(s1, s2) {
  const m = s1.length,
    n = s2.length;
  const dp = Array.from({ length: m + 1 }, () => new Array(n + 1).fill(0));

  for (let i = 1; i <= m; i++) {
    for (let j = 1; j <= n; j++) {
      if (s1[i - 1] === s2[j - 1]) {
        dp[i][j] = dp[i - 1][j - 1] + 1;
      } else {
        dp[i][j] = Math.max(dp[i - 1][j], dp[i][j - 1]);
      }
    }
  }
  return dp[m][n];
}
```

Time: O(m × n). Space: O(m × n), optimizable to O(min(m, n)) by keeping only two rows.

## How to spot a DP problem

Look for these signals:

- "Minimum/maximum number of ways/steps to..."
- "Count the number of ways to..."
- "Can you reach X from Y given constraints..."
- The problem involves sequences, sums, or subsets
- A greedy approach feels right but produces wrong answers on edge cases (greedy fails when the locally optimal choice is not globally optimal)

Non-signals (greedy or other techniques are better):

- "Sort by X and pick greedily" — activity selection, fractional knapsack
- Problems without overlapping subproblems — divide and conquer is more appropriate

## What you learned

- Dynamic programming applies when a problem has overlapping subproblems and optimal substructure
- Top-down DP (memoization) adds a cache to recursion; bottom-up DP (tabulation) fills a table from base cases upward
- The four DP steps: define the subproblem, write the recurrence, identify base cases, choose direction
- Coin Change: `dp[i] = min(dp[i - coin] + 1)` over all coin denominations
- Longest Common Subsequence: a 2D DP table where each cell depends on the cell diagonally up-left (match) or the max of the cell above/left (skip)
