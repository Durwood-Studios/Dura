## Why this matters

Every senior engineer has a clear mental model of what recursion is doing at the machine level: function calls going onto the call stack, unwinding as they return. Without that model, recursion is magic — it seems to work, but you cannot reason about why it fails (stack overflow), why it is slow (exponential calls), or how to fix it (memoization or iteration). This lesson builds that model.

## The three laws

Every correct recursive function obeys three rules:

1. **There must be a base case** — a condition where the function returns a value directly without calling itself.
2. **Every recursive call must make the problem smaller** — you must move toward the base case on every call.
3. **The recursion must converge** — with smaller inputs on each call, you must eventually hit the base case.

Violate any of these and you get infinite recursion and a stack overflow.

```javascript
// Correct: 3 laws satisfied
function factorial(n) {
  if (n <= 1) return 1; // law 1: base case
  return n * factorial(n - 1); // law 2 + 3: n-1 is smaller, converges to 1
}

// Broken: law 2 violated — never gets smaller
function broken(n) {
  if (n === 0) return 0;
  return broken(n + 1); // WRONG — grows away from base case
}
```

## Visualizing the call tree

For `factorial(5)`, execution does NOT run top to bottom — it branches down the call tree and then unwinds:

```
factorial(5)
  → 5 * factorial(4)
         → 4 * factorial(3)
                → 3 * factorial(2)
                       → 2 * factorial(1)
                              → 1          ← base case, start returning
                       ← 2 * 1 = 2
                ← 3 * 2 = 6
         ← 4 * 6 = 24
  ← 5 * 24 = 120
```

Every frame on this tree is a separate function call sitting on the call stack. For `factorial(10000)`, you would have 10,000 frames stacked simultaneously — most JavaScript engines limit this to around 10,000–15,000, after which you get a "Maximum call stack size exceeded" error.

## Why naive Fibonacci is exponential

Fibonacci seems like a natural fit for recursion:

```javascript
function fib(n) {
  if (n <= 1) return n;
  return fib(n - 1) + fib(n - 2);
}
```

But `fib(n - 1)` and `fib(n - 2)` both independently compute `fib(n - 2)`, `fib(n - 3)`, and so on. The call tree does not grow linearly — it grows like a binary tree: 2^n nodes for input n.

```
fib(5)
├── fib(4)
│   ├── fib(3)
│   │   ├── fib(2) ← computed here
│   │   └── fib(1)
│   └── fib(2) ← computed AGAIN — wasted work
└── fib(3)       ← ENTIRE fib(3) recomputed — wasted work
    ├── fib(2)   ← computed a third time
    └── fib(1)
```

For n=30, naive Fibonacci makes ~2.7 million calls. For n=50, it makes ~2.25 trillion. This is the classic example of overlapping subproblems — the same computation is repeated exponentially many times.

## Memoization: cache the results

The fix is simple: if you have already computed `fib(k)`, store the result and return it immediately on subsequent calls.

```javascript
function fibMemo(n, memo = new Map()) {
  if (n <= 1) return n;
  if (memo.has(n)) return memo.get(n); // O(1) cache hit
  const result = fibMemo(n - 1, memo) + fibMemo(n - 2, memo);
  memo.set(n, result);
  return result;
}
```

Now the call tree for `fib(n)` is a single chain of length n — each value is computed exactly once. Total calls: 2n - 1, or O(n).

## Tail recursion

A recursive call is **tail position** when it is the very last operation performed — there is no multiplication, addition, or other work waiting after the recursive call returns.

```javascript
// NOT tail recursive — must multiply by n after the call returns
function factorial(n) {
  if (n <= 1) return 1;
  return n * factorial(n - 1); // multiply pending
}

// Tail recursive — the return value IS the recursive call
function factorialTail(n, acc = 1) {
  if (n <= 1) return acc;
  return factorialTail(n - 1, n * acc); // nothing pending
}
```

Some languages (Scheme, Elixir, Scala in some cases) can optimize tail calls to reuse the same stack frame, effectively converting the recursion to iteration and allowing infinite depth. **JavaScript does not do this reliably** — the spec added it in ES6 but most engines never implemented it in practice. Do not rely on tail call optimization in JavaScript.

## When to convert recursion to iteration

Use an explicit stack (a regular array acting as a stack) to eliminate deep recursion:

```javascript
// Recursive tree traversal — risky for deep trees
function dfsRecursive(node) {
  if (!node) return;
  console.log(node.val);
  dfsRecursive(node.left);
  dfsRecursive(node.right);
}

// Iterative equivalent — safe for any depth
function dfsIterative(root) {
  if (!root) return;
  const stack = [root];
  while (stack.length > 0) {
    const node = stack.pop();
    console.log(node.val);
    if (node.right) stack.push(node.right);
    if (node.left) stack.push(node.left);
  }
}
```

The rule of thumb: if recursion depth could exceed ~1,000–5,000 frames in production, convert to iterative with an explicit stack.

## What you learned

- Every recursive function needs a base case, a smaller subproblem on each call, and convergence to the base case
- The call tree for naive Fibonacci is exponential because overlapping subproblems are recomputed independently
- Memoization caches computed results so each subproblem is solved only once, reducing Fibonacci from O(2^n) to O(n)
- Tail recursion places the recursive call in the last position; JavaScript does not reliably optimize this
- Convert recursion to iterative with an explicit stack when call depth could exceed production limits (~1,000–5,000 frames)
