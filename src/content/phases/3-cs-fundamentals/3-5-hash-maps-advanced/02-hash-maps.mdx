## Why this matters

The single most common algorithmic upgrade in technical interviews — and in real codebases — is replacing a nested loop with a hash map. An O(n²) solution that works correctly is a good start; an O(n) solution that works correctly is what gets merged. Learning when and how to apply this trade (time for space) is one of the highest-leverage skills you can build.

## Map vs Object — always prefer Map

JavaScript gives you two hash map options. Plain objects (`{}`) are tempting because they are terse, but they have critical limitations:

```javascript
// Object: hidden pitfalls
const obj = {};
obj["constructor"] = "gotcha"; // shadows Object.prototype.constructor
obj[true] = "value"; // key is coerced to the string "true"
obj[1] = "one";
obj[2] = "two";
Object.keys(obj); // ["1", "2"] — integer keys are stringified

// Map: clean and predictable
const map = new Map();
map.set(true, "boolean key — works fine");
map.set({ id: 1 }, "object key — works fine");
map.set(42, "number key — stays a number");
map.size; // exact count, no .length quirk
```

Use `Map` when:

- Keys are not strings (numbers, booleans, objects)
- You need to track insertion order reliably
- You need `.size` without counting manually
- You are iterating with `.entries()`, `.keys()`, `.values()`

Use `Set` when you only care about membership (does this value exist?) and need O(1) `.has()`.

## The fundamental trick: trade space for time

Nearly every hash map optimization follows this pattern:

> **In one pass, store information you will need later so you can answer future queries in O(1) instead of O(n).**

The classic example is Two-Sum: given an array of numbers, find two that add to a target.

**Naive O(n²):** for every element, scan the rest of the array looking for the complement.

```javascript
// O(n²) — two nested loops
function twoSumSlow(nums, target) {
  for (let i = 0; i < nums.length; i++) {
    for (let j = i + 1; j < nums.length; j++) {
      if (nums[i] + nums[j] === target) return [i, j];
    }
  }
  return [];
}
```

**Optimized O(n):** as you scan left to right, store each value's index in a Map. For each new element, check if its complement is already in the map.

```javascript
// O(n) — one pass
function twoSum(nums, target) {
  const seen = new Map(); // value → index
  for (let i = 0; i < nums.length; i++) {
    const complement = target - nums[i];
    if (seen.has(complement)) {
      return [seen.get(complement), i];
    }
    seen.set(nums[i], i);
  }
  return [];
}
```

The map lets you answer "have I seen the complement before?" in O(1) instead of scanning the entire array again.

## Three fundamental hash map patterns

**Pattern 1: Frequency counting**

Count how many times each element appears. Useful for finding duplicates, checking anagrams, and identifying the most common element.

```javascript
function charFrequency(str) {
  const freq = new Map();
  for (const char of str) {
    freq.set(char, (freq.get(char) ?? 0) + 1);
  }
  return freq;
}

// Anagram check: same characters, same counts
function isAnagram(s, t) {
  if (s.length !== t.length) return false;
  const freq = charFrequency(s);
  for (const char of t) {
    if (!freq.has(char)) return false;
    freq.set(char, freq.get(char) - 1);
    if (freq.get(char) === 0) freq.delete(char);
  }
  return freq.size === 0;
}
```

**Pattern 2: Grouping by key**

Map a computed key to an array of items sharing that property. Useful for grouping anagrams, grouping by category, or building an adjacency list.

```javascript
function groupByFirstLetter(words) {
  const groups = new Map();
  for (const word of words) {
    const key = word[0];
    if (!groups.has(key)) groups.set(key, []);
    groups.get(key).push(word);
  }
  return groups;
}
```

**Pattern 3: Existence / deduplication**

Use a Set to track what you have already seen and skip duplicates in O(1).

```javascript
function hasDuplicate(nums) {
  const seen = new Set();
  for (const n of nums) {
    if (seen.has(n)) return true;
    seen.add(n);
  }
  return false;
}
```

## When hash maps fail

Hash maps are not always the right tool:

- **You need sorted order.** Maps have no ordering by value. Use a sorted array or balanced BST (JavaScript's built-in sort is O(n log n)).
- **You need prefix or range queries.** "All keys starting with 'app'" requires a trie, not a hash map.
- **Memory is severely constrained.** Each entry in V8's Map costs approximately 100–200 bytes of overhead. For 10 million entries, that's up to 2 GB — more than many environments allow.
- **Keys are objects with value equality.** `map.set({id:1}, "a")` and `map.get({id:1})` return `undefined` because object identity, not value, is used for Map keys. You need to serialize the key to a string first.

```javascript
// Serializing object keys
const cache = new Map();
function cacheWithObjectKey(key, value) {
  const serialized = JSON.stringify(key); // risky if key has undefined values
  cache.set(serialized, value);
}
```

## What you learned

- Prefer `Map` over plain objects for hash maps in JavaScript — it handles any key type and avoids prototype quirks
- `Set` provides O(1) membership testing for deduplication and existence checks
- The core hash map optimization is the space-for-time trade: store computed information so future lookups are O(1) instead of O(n)
- Three fundamental patterns: frequency counting, grouping by key, and existence tracking
- Hash maps are not suitable for sorted order queries, prefix queries, or memory-constrained environments
