## Why this matters

Understanding an algorithm in isolation is different from deploying it under time pressure to solve a problem you have never seen before. Interviews test the latter. This capstone presents three problems that each require you to combine multiple techniques — hash maps, DP, and careful complexity analysis — and walk through the problem-solving process explicitly. Treat each section as a 20-minute timed exercise before reading the solution.

## How to approach each problem

Real technical interviews have a structure that rewards process over brute speed:

1. **Clarify constraints (2 minutes):** What are the input sizes? Can values be negative? Is the input guaranteed sorted? Edge cases?
2. **Brute force first (3 minutes):** Describe the naive solution and its complexity. This proves you understand the problem.
3. **Identify the bottleneck (2 minutes):** What makes the brute force slow? What property could you exploit?
4. **Optimize (10 minutes):** Implement the better solution while narrating your reasoning.
5. **Verify (3 minutes):** Trace through at least one example. Check edge cases: empty input, single element, duplicates.

Never jump to code before you understand the problem. Interviewers care more about your reasoning than whether your first draft compiles.

## Problem 1: Word Break

**Problem:** Given a string `s` and a dictionary of words, return `true` if `s` can be segmented into one or more dictionary words (words can be reused).

**Brute force:** try every possible split point recursively. Exponential — O(2^n) because at each position you branch into "cut here" or "don't cut here."

**Bottleneck:** subproblems overlap massively. Whether position 5 can reach the end is the same regardless of how you reached position 5.

**DP approach:**

Define `dp[i]` = `true` if `s[0..i-1]` can be fully segmented. We want `dp[s.length]`.

Recurrence: `dp[i] = true` if there exists a `j < i` such that `dp[j]` is true AND `s[j..i-1]` is in the dictionary.

```javascript
function wordBreak(s, wordDict) {
  const dict = new Set(wordDict); // O(1) lookup per word
  const dp = new Array(s.length + 1).fill(false);
  dp[0] = true; // empty string is always "segmented"

  for (let i = 1; i <= s.length; i++) {
    for (let j = 0; j < i; j++) {
      if (dp[j] && dict.has(s.slice(j, i))) {
        dp[i] = true;
        break; // found one valid split — no need to check more
      }
    }
  }
  return dp[s.length];
}
```

**Complexity:** O(n² × L) time where n = string length, L = average word length for the `.slice()` operation. O(n) space for the dp array plus O(W) for the Set where W = total dictionary characters.

**Trace:** `s = "leetcode"`, `dict = ["leet", "code"]`

- `dp[0] = true`
- `dp[4]`: j=0, `dp[0]=true`, `s[0..3]="leet"` in dict → `dp[4]=true`
- `dp[8]`: j=4, `dp[4]=true`, `s[4..7]="code"` in dict → `dp[8]=true`
- Return `true`

## Problem 2: Longest Increasing Subsequence

**Problem:** Given an array of integers, find the length of the longest strictly increasing subsequence (elements need not be contiguous).

**Brute force:** generate all 2^n subsets, filter for strictly increasing, return the longest length. O(2^n).

**DP approach — O(n²):**

Define `dp[i]` = length of the longest increasing subsequence ending at index i.

```javascript
function lisDP(nums) {
  const n = nums.length;
  if (n === 0) return 0;
  const dp = new Array(n).fill(1); // every element alone is an LIS of length 1

  for (let i = 1; i < n; i++) {
    for (let j = 0; j < i; j++) {
      if (nums[j] < nums[i]) {
        dp[i] = Math.max(dp[i], dp[j] + 1);
      }
    }
  }
  return Math.max(...dp);
}
```

**Optimized approach — O(n log n) with binary search:**

Maintain a `tails` array where `tails[k]` is the smallest tail element of all increasing subsequences of length k+1. For each new number, binary search for the first tail that is >= the number and replace it (or append if all tails are smaller).

```javascript
function lis(nums) {
  const tails = [];

  for (const num of nums) {
    // Binary search for the leftmost position where tails[pos] >= num
    let lo = 0,
      hi = tails.length;
    while (lo < hi) {
      const mid = Math.floor((lo + hi) / 2);
      if (tails[mid] < num) lo = mid + 1;
      else hi = mid;
    }
    tails[lo] = num; // replace or extend
  }

  return tails.length;
}
```

`tails` does NOT contain the actual LIS, but its length is always the LIS length. This is a clever invariant: replacing a tail never shrinks any existing subsequence's validity — it only improves future potential.

**Complexity:** O(n log n) time, O(n) space.

**Trace:** `nums = [10, 9, 2, 5, 3, 7, 101, 18]`

- Process 10: tails=[10]
- Process 9: 9 < 10, replace → tails=[9]
- Process 2: 2 < 9, replace → tails=[2]
- Process 5: 5 > 2, extend → tails=[2,5]
- Process 3: 3 < 5, replace position 1 → tails=[2,3]
- Process 7: 7 > 3, extend → tails=[2,3,7]
- Process 101: extend → tails=[2,3,7,101]
- Process 18: 18 < 101, replace position 3 → tails=[2,3,7,18]
- Return `tails.length` = 4

## Problem 3: Word Break — Interview technique spotlight

Let us go deeper on word break as a model for communicating DP solutions:

**Narration template:**

> "I notice that checking every split recursively leads to overlapping subproblems — whether s[0..4] can be segmented is the same regardless of how we got there. That tells me DP applies. I'll define dp[i] as whether the first i characters can be segmented. My base case is dp[0] = true because an empty prefix is trivially segmented. The recurrence checks all split points j < i: if dp[j] is true and s[j..i-1] is a valid word, then dp[i] is true. I'll store the dictionary in a Set for O(1) word lookups. Total complexity: O(n² × L) time, O(n + W) space."

This narration demonstrates: problem recognition, DP property identification, subproblem definition, recurrence, base case, complexity analysis. That is everything an interviewer is evaluating.

## Complexity tradeoffs summary

| Problem            | Brute Force    | Optimized          | Key Technique               |
| ------------------ | -------------- | ------------------ | --------------------------- |
| Word Break         | O(2^n)         | O(n² × L)          | Bottom-up DP + hash set     |
| LIS                | O(2^n)         | O(n²) → O(n log n) | DP → binary search on tails |
| Activity Selection | O(2^n subsets) | O(n log n)         | Greedy (sort by end time)   |
| Two Sum            | O(n²)          | O(n)               | Hash map: store complement  |
| Subarray Sum = k   | O(n²)          | O(n)               | Prefix sums + hash map      |

The pattern: when brute force is exponential, look for overlapping subproblems (DP) or a greedy ordering. When brute force is O(n²), look for a hash map or sorting trick to push it to O(n) or O(n log n).

## What you learned

- Structured problem-solving (clarify, brute force, bottleneck, optimize, verify) produces better interviews than jumping straight to code
- Word Break uses bottom-up DP where dp[i] = "can the first i characters be segmented" — O(n² × L) with a Set for O(1) word lookup
- LIS has two DP approaches: O(n²) explicit DP and O(n log n) binary search on a maintained tails array
- Narrating your reasoning — problem recognition, DP properties, recurrence, complexity — is what interviews actually evaluate
- The recurring pattern: exponential brute force → overlapping subproblems → DP; quadratic brute force → hash map or sort-based trick → O(n) or O(n log n)
