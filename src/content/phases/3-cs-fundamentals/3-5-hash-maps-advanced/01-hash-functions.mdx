## Why this matters

Hash maps are the most frequently used data structure in real software — they power database indexes, caches, compilers, and dictionaries. But they are not magic. Their O(1) average performance depends entirely on having a good hash function. Understanding how hash functions work, where they fail, and how to handle collisions is what separates engineers who use hash maps from engineers who understand them.

## What a hash function does

A hash function takes an arbitrary key — a string, a number, an object — and maps it to an integer index within a fixed-size array called a hash table. The goal is to spread keys as evenly as possible across the available buckets so that lookups are fast.

```javascript
// Conceptually, every hash map does this:
function get(key) {
  const index = hash(key) % tableSize;
  return table[index];
}
```

Four properties define a good hash function:

1. **Deterministic** — the same key must always produce the same index.
2. **Uniform distribution** — keys should spread evenly across buckets. If 90% of keys land in bucket 0, your O(1) lookup degrades to O(n).
3. **Fast computation** — hashing happens on every insert and lookup. An O(n) hash function defeats the purpose.
4. **Avalanche effect** — a small change to the input should produce a wildly different hash. This prevents clustered collisions.

## Why naive hashing fails

The simplest possible hash for strings is to sum the character codes and mod by table size:

```javascript
function naiveHash(str, size) {
  let total = 0;
  for (const char of str) {
    total += char.charCodeAt(0);
  }
  return total % size;
}
```

This is fast and deterministic — but it violates uniform distribution because **anagrams produce identical hashes**. The strings `"listen"` and `"silent"` both sum to the same character total, so they always land in the same bucket. In a word dictionary application, this causes severe clustering.

## Polynomial rolling hash

A much better approach weights each character by its position, making anagrams extremely unlikely to collide:

```javascript
function polynomialHash(str, size) {
  const BASE = 31;
  let result = 0;
  for (let i = 0; i < str.length; i++) {
    // Use charCode - 96 so 'a'=1, 'b'=2, etc.
    result = (result * BASE + str.charCodeAt(i)) % size;
  }
  return result;
}
```

The prime base `31` is a classic choice — it distributes bits well and is cheap to multiply. Changing a single character changes the multiplied sum dramatically (avalanche effect). This is the same approach used internally by Java's `String.hashCode()`.

## Collisions are inevitable

No matter how good your hash function, **collisions will happen**. By the birthday paradox, with a table of 365 buckets and only 23 random keys, there is a 50% chance of a collision. This is not a flaw to be fixed — it is a mathematical reality to be managed.

Two main strategies exist:

**Separate chaining** — each bucket holds a linked list (or array) of all entries that hash to that index. A collision just appends to the list. Lookup scans the list for the matching key.

```javascript
// Separate chaining lookup
function get(key) {
  const index = hash(key) % this.table.length;
  const chain = this.table[index]; // array of [key, value] pairs
  if (!chain) return undefined;
  const entry = chain.find(([k]) => k === key);
  return entry ? entry[1] : undefined;
}
```

**Open addressing** — all entries live in the main array. On collision, probe for the next available slot (linear probing: try index+1, index+2, etc.). Simpler memory layout but more sensitive to clustering.

## Load factor and rehashing

The **load factor** α = n/m, where n is the number of stored entries and m is the number of buckets. It measures how "full" the table is.

| Load factor | Behavior                                  |
| ----------- | ----------------------------------------- |
| α < 0.5     | Excellent — very few collisions           |
| 0.5 – 0.75  | Good — acceptable collision rate          |
| α > 0.75    | Degrades — chains grow, performance drops |

When α exceeds the threshold (typically 0.75), the hash map **rehashes**: it allocates a new table roughly twice the size, then re-inserts every existing entry. This is O(n) work for that single operation — but because it happens exponentially less often as the table grows, the amortized cost per insert remains O(1).

```javascript
function maybeRehash() {
  const loadFactor = this.count / this.table.length;
  if (loadFactor > 0.75) {
    const oldTable = this.table;
    this.table = new Array(this.table.length * 2).fill(null);
    this.count = 0;
    // Re-insert all existing entries into the new table
    for (const chain of oldTable) {
      if (chain) {
        for (const [key, value] of chain) {
          this.set(key, value); // uses new table size
        }
      }
    }
  }
}
```

The doubling strategy ensures that even if you insert n items, you trigger at most log₂(n) rehashes, and each item participates in at most log₂(n) rehashes total — giving O(n) total rehashing cost for n inserts, or O(1) amortized per insert.

## What you learned

- A hash function maps arbitrary keys to array indices; good ones are deterministic, fast, and uniformly distributed
- Polynomial rolling hashes use positional weights to prevent anagrams and similar strings from colliding
- Collisions are inevitable; separate chaining handles them with per-bucket lists, open addressing probes for the next empty slot
- Load factor α = n/m measures table fullness; rehashing at α > 0.75 keeps performance at O(1) amortized
