## Why this matters

Greedy algorithms are seductive. They are simple to implement, often very fast, and feel obviously correct. The danger is that "feels obviously correct" is not a proof. Many greedy solutions that pass small test cases fail on carefully crafted edge cases. Knowing when to trust greedy — and when to reach for dynamic programming instead — is a critical algorithmic judgment call.

## The greedy choice property

A greedy algorithm makes the locally optimal choice at each step without reconsidering previous choices. For this to produce the globally optimal solution, the problem must have the **greedy choice property**: a locally optimal choice is always part of a globally optimal solution.

This is not true by default. It must be proven for each specific problem.

The standard proof technique is the **exchange argument**: assume an optimal solution does NOT use the greedy choice at some step. Show that you can swap in the greedy choice without making the solution worse. Therefore, a solution using all greedy choices is equally optimal — so greedy works.

## Activity selection: greedy in action

Given a set of activities with start and end times, find the maximum number of non-overlapping activities you can schedule.

**Greedy choice:** always pick the activity that finishes earliest. Intuitively, finishing early leaves the most room for future activities.

**Why this works (exchange argument):** suppose an optimal schedule picks activity X at some slot, but the greedy algorithm would pick activity Y (which finishes earlier). If we swap X for Y, Y finishes earlier, which can only open up more (or the same) future options. The schedule is no worse. Therefore, always picking the earliest-finishing activity is safe.

```javascript
function activitySelection(activities) {
  // Sort by end time — this is the greedy key insight
  const sorted = [...activities].sort((a, b) => a.end - b.end);

  let count = 0;
  let lastEnd = -Infinity;

  for (const activity of sorted) {
    // Only take this activity if it starts after the last one ended
    if (activity.start >= lastEnd) {
      count++;
      lastEnd = activity.end;
    }
  }
  return count;
}
```

Time: O(n log n) dominated by the sort. The greedy scan itself is O(n).

## Huffman encoding: greedy tree construction

Huffman coding assigns shorter bit sequences to more frequent characters. It is optimal and uses a greedy strategy: repeatedly merge the two lowest-frequency nodes into a new parent node.

The greedy choice — always merge the two least frequent — guarantees the rarest characters end up deepest in the tree (longest codes) and the most frequent ones stay near the root (shortest codes). The exchange argument shows that any swap of merge order produces a suboptimal or equal code.

You do not need to implement Huffman from scratch in interviews, but you should know:

- It produces a prefix-free code (no code is a prefix of another)
- It is provably optimal for symbol-by-symbol encoding
- The algorithm requires a min-heap (priority queue) to efficiently find the two smallest nodes each time

## Coin change with standard denominations

US coin denominations — 1, 5, 10, 25 cents — have a special property: greedy (always use the largest coin that fits) produces the optimal (fewest coins) answer.

```javascript
function greedyCoins(amount) {
  const denominations = [25, 10, 5, 1];
  const counts = [];
  for (const d of denominations) {
    counts.push(Math.floor(amount / d));
    amount = amount % d;
  }
  return counts; // [quarters, dimes, nickels, pennies]
}

greedyCoins(41); // [1, 1, 1, 1] → 1 quarter + 1 dime + 1 nickel + 1 penny
```

**But greedy fails with arbitrary coins.** Suppose denominations are [1, 3, 4] and the target is 6:

- Greedy: 4 + 1 + 1 = **3 coins**
- Optimal: 3 + 3 = **2 coins**

Greedy chose 4 because it is the largest coin under 6 — but that local choice blocked the globally better answer. This is exactly the problem where dynamic programming is required.

## The greedy vs DP decision

Ask yourself these questions:

| Question                                          | Greedy  | DP           |
| ------------------------------------------------- | ------- | ------------ |
| Can I prove a local choice is always safe?        | Yes     | No           |
| Do I need to consider multiple future paths?      | No      | Yes          |
| Does making a choice eliminate useful options?    | No      | Yes          |
| Is there a canonical greedy ordering (sort by X)? | Usually | Not required |

In practice: if a sorting strategy (earliest end, smallest weight, highest value/weight ratio) feels right and you can sketch an exchange argument, try greedy. If you keep finding counterexamples, switch to DP.

## Common greedy problems and their keys

- **Jump Game** (can you reach the end?) — track the maximum reachable index greedily
- **Merge Intervals** — sort by start time, merge overlapping pairs
- **Task Scheduler** — always schedule the most frequent remaining task (greedy with a priority queue)
- **Minimum spanning tree** (Kruskal's, Prim's) — greedy edge selection with a union-find
- **Dijkstra's shortest path** — greedy: always relax the nearest unvisited node

## What you learned

- Greedy algorithms make locally optimal choices at each step without reconsidering past decisions
- The greedy choice property must hold: a local choice must always be part of a globally optimal solution
- Exchange arguments prove greedy correctness by showing that swapping in the greedy choice never worsens an optimal solution
- Activity selection (sort by end time) and Huffman coding are canonical correct greedy algorithms
- Greedy coin change works for standard denominations but fails for arbitrary ones — that is where DP is required
