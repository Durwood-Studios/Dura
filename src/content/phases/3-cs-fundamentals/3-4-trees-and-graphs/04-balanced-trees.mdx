## Why this matters

A BST's O(log n) promise depends entirely on the tree staying reasonably short. As you saw in the previous lesson, inserting sorted data creates a linked list — O(n) operations instead of O(log n). Real-world data isn't random, so degeneration is a real risk, not a theoretical edge case. Balanced trees solve this by automatically restructuring themselves after insertions and deletions to keep the height bounded.

## What "balanced" means

A tree is **height-balanced** if no path from root to leaf is dramatically longer than any other. More precisely, balance means the height is guaranteed to be O(log n) regardless of insertion order.

The key insight: in a binary tree with n nodes, the minimum possible height is ⌊log₂(n)⌋ (perfect tree). We want self-balancing trees to stay close to this minimum — specifically, within a constant factor.

The **balance factor** of a node is defined as:

```
balance_factor(node) = height(node.left) - height(node.right)
```

- Balance factor of 0: subtrees are equally tall.
- Balance factor of +1: left subtree is one level taller.
- Balance factor of -1: right subtree is one level taller.
- |balance factor| > 1: the node is **unbalanced** and needs fixing.

## AVL trees: strict balance

An **AVL tree** (Adelson-Velsky and Landis, 1962) maintains the invariant that every node's balance factor is -1, 0, or +1. After every insert or delete, the tree checks balance factors from the modified node back up to the root and performs **rotations** wherever needed.

### Rotations

A rotation is a local restructuring that preserves the BST property while changing the tree's shape. There are four cases:

**Right rotation** (fix a left-heavy node):

```
    z              y
   / \            / \
  y   T4   →    x   z
 / \                / \
x   T3            T3   T4
```

Node z is left-heavy (balance factor +2). We promote y to z's position. z becomes y's right child. y's former right subtree T3 becomes z's new left child. The BST property is maintained because T3 has values between y and z.

**Left rotation** (fix a right-heavy node): mirror image of right rotation.

**Left-Right rotation**: node is left-heavy, but the left child is right-heavy. First rotate the left child left, then rotate the current node right.

**Right-Left rotation**: node is right-heavy, but the right child is left-heavy. Mirror image.

```javascript
function rotateRight(z) {
  const y = z.left;
  const T3 = y.right;

  // Perform rotation
  y.right = z;
  z.left = T3;

  // Update heights (z first, since it's now lower)
  z.height = 1 + Math.max(height(z.left), height(z.right));
  y.height = 1 + Math.max(height(y.left), height(y.right));

  return y; // y is the new root of this subtree
}

function rotateLeft(z) {
  const y = z.right;
  const T2 = y.left;

  y.left = z;
  z.right = T2;

  z.height = 1 + Math.max(height(z.left), height(z.right));
  y.height = 1 + Math.max(height(y.left), height(y.right));

  return y;
}
```

An AVL tree guarantees height ≤ 1.44 log₂(n) — at most 44% taller than the theoretical minimum. In practice this is excellent.

## Red-black trees: looser balance, faster updates

A **red-black tree** is a BST where every node is colored red or black. It maintains five properties:

1. Every node is red or black.
2. The root is black.
3. Every null pointer ("nil leaf") is black.
4. A red node cannot have a red child (no two consecutive red nodes).
5. Every path from any node to its null descendants has the same number of black nodes (**black-height**).

These five rules together guarantee that the tree's height is at most **2 log₂(n+1)** — at most twice the minimum. This is a slightly looser guarantee than AVL, but red-black trees require fewer rotations on average during insert and delete, making them faster for update-heavy workloads.

**Where you already use red-black trees**: JavaScript's `Map` and `Set` in V8 (Chrome's JS engine) use red-black trees internally. When you call `map.get(key)`, that's an O(log n) red-black tree lookup. Same for `set.has(value)`.

## Treaps and skip lists

Two other approaches worth knowing by name:

**Treap**: a BST where each node also has a random priority, and the tree is structured as both a BST (by key) and a max-heap (by priority). Random priorities make degeneration statistically extremely unlikely — expected height is O(log n).

**Skip list**: not a tree at all, but a layered linked list that achieves O(log n) expected operations by maintaining multiple "express lanes" of pointers. Used in Redis and some database indexes.

## The practical takeaway

You will almost never implement an AVL or red-black tree from scratch in production — the language runtime or standard library provides them. What matters is understanding:

- Why plain BSTs aren't safe for production data
- That self-balancing structures exist and what they guarantee
- That your language's built-in Map, Set, and sorted containers use these structures

## What you learned

- Unbalanced BSTs degrade to O(n) — self-balancing trees prevent this
- **Balance factor** = height(left) - height(right); AVL trees keep it in {-1, 0, +1}
- **Rotations** restructure the tree locally while preserving the BST property
- **AVL trees**: stricter balance (≤ 1.44 log n height), more rotations
- **Red-black trees**: looser balance (≤ 2 log n height), fewer rotations — used in JS Map and Set
