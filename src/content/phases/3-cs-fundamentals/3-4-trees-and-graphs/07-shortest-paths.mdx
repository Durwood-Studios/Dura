## Why this matters

BFS finds shortest paths in unweighted graphs by counting hops. But when edges have different costs — like roads with different distances, or flights with different prices — you need an algorithm that accounts for those costs. Dijkstra's algorithm is the classic solution, and it powers the routing logic inside every navigation app. Understanding it teaches you greedy algorithm design, priority queues, and the "relaxation" pattern that appears across many optimization problems.

## Unweighted vs weighted shortest paths

In an unweighted graph, "shortest" means fewest edges. BFS handles this naturally — it explores vertices in order of hop count.

In a weighted graph, a path with fewer edges might have a higher total cost than a path with more edges:

```
A --1-- B --1-- D
 \             /
  ---10--------
```

The path A→B→D has cost 2. The direct path A→D has cost 10. BFS would prefer A→D (1 hop) over A→B→D (2 hops) — but that's the wrong answer. We need a weighted shortest path algorithm.

## Dijkstra's algorithm

Dijkstra's (1956) is a **greedy** algorithm. At each step, it commits to the minimum-cost unprocessed vertex — the one we're most confident we know the true shortest distance to.

**The idea:**

1. Initialize: `dist[source] = 0`, `dist[all others] = Infinity`.
2. Maintain a priority queue (min-heap) of `{vertex, currentCost}` pairs.
3. Extract the vertex with the **minimum distance** from the queue.
4. For each of its neighbors, compute the cost of reaching that neighbor through the current vertex: `newCost = dist[current] + edgeWeight`.
5. If `newCost < dist[neighbor]`, update `dist[neighbor]` and add the neighbor to the queue. This step is called **relaxation**.
6. Repeat until the queue is empty or the target is reached.

A vertex is "settled" once it's extracted from the priority queue — at that point, its shortest distance is final and won't change.

```javascript
function dijkstra(graph, source) {
  const dist = {};
  const previous = {};

  // Initialize all distances to Infinity
  for (const vertex of Object.keys(graph)) {
    dist[vertex] = Infinity;
    previous[vertex] = null;
  }
  dist[source] = 0;

  // Priority queue: sorted array (min at front)
  // Each entry: { node, cost }
  const pq = [{ node: source, cost: 0 }];

  while (pq.length > 0) {
    // Extract minimum — sort to simulate priority queue
    pq.sort((a, b) => a.cost - b.cost);
    const { node: current, cost: currentCost } = pq.shift();

    // If this entry is stale (we found a better path already), skip it
    if (currentCost > dist[current]) continue;

    for (const { node: neighbor, weight } of graph[current] || []) {
      const newCost = dist[current] + weight;

      if (newCost < dist[neighbor]) {
        dist[neighbor] = newCost;
        previous[neighbor] = current;
        pq.push({ node: neighbor, cost: newCost });
      }
    }
  }

  return { distances: dist, previous };
}
```

## Reconstructing the path

The `previous` map lets you trace the shortest path backward from any vertex to the source:

```javascript
function getPath(previous, target) {
  const path = [];
  let current = target;
  while (current !== null) {
    path.unshift(current); // add to front
    current = previous[current];
  }
  return path;
}
```

## Why it fails with negative weights

Dijkstra's greedy assumption is that once a vertex is extracted from the priority queue with a certain cost, that cost is final. This is only true when all edge weights are non-negative — a positive-weight edge can only increase the cost of a path.

If negative weights exist, a later path that goes through a negative edge could produce a lower cost than what Dijkstra already committed to. Example:

```
A --5-- B
A --10-- C
B --(-8)-- C  (negative weight)
```

Dijkstra settles A at cost 0, then B at cost 5, then tries to settle C at cost 10 (via A→C) — but A→B→C costs 5 + (-8) = -3. Dijkstra misses this.

**Bellman-Ford** handles negative weights (and can detect negative cycles) by relaxing all edges V-1 times, at the cost of O(VE) time instead of O((V+E) log V).

## Complexity

With a proper binary heap priority queue: **O((V + E) log V)**.

With a naive sorted array (as in the implementation above): O(V² + E) — acceptable for learning and small graphs, but not production-scale.

With a Fibonacci heap: O(E + V log V) — theoretically optimal but complex to implement.

## What you learned

- BFS finds shortest paths by hop count; **Dijkstra's** finds shortest paths by total weight
- **Relaxation**: update a neighbor's distance if a cheaper path is found through the current vertex
- Dijkstra's greedy approach settles vertices in order of minimum distance
- It fails on **negative weights** — use Bellman-Ford instead
- Time complexity: O((V + E) log V) with a binary heap priority queue
- The `previous` map lets you reconstruct the actual path, not just the minimum cost
