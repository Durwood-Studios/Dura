## Why this matters

Reading about algorithms is necessary but not sufficient. The ability to look at a problem, recognize which tree or graph pattern it matches, and implement a clean solution — that's the skill that actually matters. This lesson works through four problems that appear frequently in interviews and real codebases. For each one, focus on identifying the pattern before writing code.

## Problem 1: Maximum depth of a binary tree

**Problem**: Given the root of a binary tree, return its maximum depth (the number of nodes along the longest path from root to a leaf).

**Pattern**: Postorder recursion — you need both subtrees' depths before computing the current node's depth.

```javascript
function maxDepth(root) {
  if (root === null) return 0;
  return 1 + Math.max(maxDepth(root.left), maxDepth(root.right));
}
```

**Complexity**: O(n) time (visit every node), O(h) space (call stack depth equals tree height).

## Problem 2: Validate a binary search tree

**Problem**: Given the root of a binary tree, determine if it is a valid BST.

**Common mistake**: checking only that `node.left.val < node.val < node.right.val` is wrong. Consider:

```
    5
   / \
  1   4
     / \
    3   6
```

Node 4's children look locally valid (3 < 4 < 6), but 4 itself is in the right subtree of 5 — so all values there must be > 5. The value 3 violates that.

**Pattern**: pass down min/max bounds that tighten as you recurse.

```javascript
function isValidBST(node, min = -Infinity, max = Infinity) {
  if (node === null) return true;
  if (node.val <= min || node.val >= max) return false;
  return isValidBST(node.left, min, node.val) && isValidBST(node.right, node.val, max);
}
```

**Alternative pattern**: inorder traversal — if the resulting array is strictly increasing, it's a valid BST.

**Complexity**: O(n) time, O(h) space.

## Problem 3: Number of islands

**Problem**: Given a 2D grid of `'1'` (land) and `'0'` (water), return the number of islands. An island is a group of adjacent land cells (connected horizontally or vertically).

**Pattern**: treat the grid as an implicit graph — each land cell is a vertex, each pair of adjacent land cells is an edge. Count connected components using DFS flood-fill.

```javascript
function numIslands(grid) {
  if (!grid || grid.length === 0) return 0;

  const rows = grid.length;
  const cols = grid[0].length;
  let count = 0;

  function dfs(r, c) {
    // Out of bounds or water — stop
    if (r < 0 || r >= rows || c < 0 || c >= cols || grid[r][c] !== "1") return;
    // Mark as visited by changing '1' to '0' (in-place visited tracking)
    grid[r][c] = "0";
    // Explore all 4 neighbors
    dfs(r + 1, c);
    dfs(r - 1, c);
    dfs(r, c + 1);
    dfs(r, c - 1);
  }

  for (let r = 0; r < rows; r++) {
    for (let c = 0; c < cols; c++) {
      if (grid[r][c] === "1") {
        count++; // Found a new island
        dfs(r, c); // Flood-fill the entire island
      }
    }
  }

  return count;
}
```

The flood-fill modifies the grid in-place — each `'1'` becomes `'0'` as it's visited, so it won't be counted again. If you need to preserve the original grid, use a separate `visited` boolean matrix instead.

**Complexity**: O(m × n) time (each cell visited at most once), O(m × n) space for the call stack in the worst case (one giant island).

## Problem 4: Topological sort

**Problem**: Given a DAG (Directed Acyclic Graph) representing task dependencies, produce a valid execution order where every task comes before any task that depends on it.

**Pattern**: DFS with a completion stack. When all of a node's neighbors have been fully explored (all dependencies of its dependents are settled), push the node onto the stack. The result is the stack reversed — or equivalently, you prepend to the result.

```javascript
function topologicalSort(graph) {
  const visited = new Set();
  const result = [];

  function dfs(vertex) {
    visited.add(vertex);
    for (const neighbor of graph[vertex] || []) {
      if (!visited.has(neighbor)) {
        dfs(neighbor);
      }
    }
    result.unshift(vertex); // push to front after all dependents are placed
  }

  for (const vertex of Object.keys(graph)) {
    if (!visited.has(vertex)) {
      dfs(vertex);
    }
  }

  return result;
}
```

Why it works: a vertex is added to the front of the result only after all of its successors have been added. So in the final array, it appears before them — which is the correct dependency order.

**Use cases**: build systems (compile B before A if A imports B), course prerequisites, package installation order.

**Complexity**: O(V + E) time, O(V) space.

## What you learned

- **Max depth**: classic postorder recursion — `1 + max(left depth, right depth)`
- **Valid BST**: pass min/max bounds down the recursion — local checks are not enough
- **Number of islands**: grid as an implicit graph, DFS flood-fill counts connected components
- **Topological sort**: DFS with unshift-after-explore produces a valid task ordering for DAGs
- Pattern recognition is the key skill — identify which template applies before writing a single line
