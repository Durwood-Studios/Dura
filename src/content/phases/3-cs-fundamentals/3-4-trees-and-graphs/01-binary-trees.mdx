## Why this matters

Trees are the hidden structure behind nearly everything in computing — from the folders on your hard drive to how a web browser parses HTML to how databases execute queries. Understanding binary trees gives you the mental model to work with all of them. Before you can build or analyze a tree, you need the vocabulary to describe what you're looking at.

## The anatomy of a tree

A **tree** is a collection of **nodes** connected by **edges**, with one special node at the top called the **root**. Every node except the root has exactly one parent. This is what distinguishes a tree from a general graph — there are no cycles and no node has more than one path back to the root.

Key terms:

- **Root**: the topmost node — the one with no parent.
- **Leaf**: a node with no children. It's the bottom of a branch.
- **Parent / Child**: if node A has an edge going down to node B, then A is B's parent and B is A's child.
- **Sibling**: nodes that share the same parent.
- **Edge**: the connection between a parent and a child.
- **Subtree**: any node, plus all of its descendants, forms its own smaller tree.

**Depth** measures how far a node is from the root. The root itself is at depth 0. Its children are at depth 1. Their children at depth 2, and so on. Depth = number of edges on the path from the root to that node.

**Height** measures how far a node is from its deepest leaf. A leaf node has height 0. A node whose deepest descendant is 2 levels below has height 2. The **height of the entire tree** is the height of its root — which equals the longest path from the root to any leaf.

## What makes a tree "binary"

A **binary tree** is a tree where every node has **at most two children** — called the **left child** and the **right child**. The "at most" matters: a node can have 0, 1, or 2 children, but never 3 or more.

Three important subtypes:

**Full binary tree**: every node has exactly 0 or exactly 2 children. No node has just one child. A full binary tree looks "complete" at every level — nothing dangles.

**Complete binary tree**: all levels are completely filled except possibly the last, and the last level is filled left-to-right. If you were to write the nodes into an array level by level, left to right, there would be no gaps until the very end. This property is what makes heaps efficient — a complete binary tree can be stored compactly in an array.

**Perfect binary tree**: every level is completely full. A perfect binary tree with height h has exactly 2^(h+1) - 1 nodes. Every perfect binary tree is also full and complete, but not vice versa.

In practice you'll most often work with binary trees that don't fit any of these special cases — they're just trees with some nodes having 0, 1, or 2 children depending on what data was inserted.

## Representing a binary tree in code

The standard representation uses a linked-node structure, where each node holds a value and two pointers:

```javascript
class TreeNode {
  constructor(val) {
    this.val = val;
    this.left = null;
    this.right = null;
  }
}

// Build the tree:
//       1
//      / \
//     2   3
//    / \
//   4   5

const root = new TreeNode(1);
root.left = new TreeNode(2);
root.right = new TreeNode(3);
root.left.left = new TreeNode(4);
root.left.right = new TreeNode(5);
```

This tree has:

- Root = 1 (depth 0, height 2)
- Node 2 (depth 1, height 1)
- Nodes 3, 4, 5 (depth 2 for 4 and 5, depth 1 for 3 — all leaves except 2)
- Leaves = 3, 4, 5 (3 leaves)
- Height of tree = 2 (longest path: 1→2→4)

## Why binary trees are everywhere

Binary trees appear in so many places because they strike a balance between simplicity (only two choices at each node) and power (branching means logarithmic depth). With a balanced binary tree of n nodes, you can reach any node in O(log n) steps — far better than the O(n) required for a linked list.

Applications you'll encounter soon:

- **Binary Search Trees (BSTs)**: fast lookup, insertion, and deletion
- **Heaps**: priority queues for scheduling and Dijkstra's algorithm
- **Expression trees**: compilers parse `3 + 4 * 2` into a tree to evaluate it correctly
- **File systems**: directories form trees (though with arbitrary branching, not just 2)
- **HTML DOM**: your browser builds a tree from HTML to render a page

## What you learned

- A binary tree is a tree where every node has at most two children: left and right
- **Root** = no parent; **leaf** = no children; **depth** = distance from root; **height** = distance to deepest leaf
- Full, complete, and perfect binary trees are progressively stricter subtypes
- A binary tree node in code is a simple object: `{ val, left, right }`
- Tree height can be computed recursively: `1 + max(height(left), height(right))`
