## Why this matters

Trees are a restricted kind of graph — connected, acyclic, hierarchical. Real-world relationships don't always work that way. Road networks have cycles. Social networks have bidirectional connections. Dependency graphs have direction. To model any of these accurately, you need the general concept of a **graph**. Once you understand graphs, you unlock an entire family of algorithms — shortest paths, network flow, cycle detection — that drive GPS, social feeds, build systems, and package managers.

## Formal definition

A graph G is defined as a pair **(V, E)** where:

- **V** is a set of **vertices** (also called nodes) — the things in your system
- **E** is a set of **edges** — the relationships between things

Each edge connects two vertices. In graph notation: edge (u, v) connects vertex u to vertex v.

A tree is a graph with two additional constraints: it must be **connected** (you can reach any vertex from any other) and **acyclic** (no path from a vertex leads back to itself).

## Types of graphs

**Directed vs undirected**

In an **undirected** graph, edges have no direction — if (u, v) exists, you can travel from u to v or from v to u. Think of Facebook friendships: if Alice is friends with Bob, Bob is friends with Alice.

In a **directed** graph (digraph), edges have direction. Edge (u, v) means you can go from u to v, but not necessarily from v to u. Think of Twitter follows: Alice can follow Bob without Bob following Alice.

**Weighted vs unweighted**

In a **weighted** graph, each edge has a numeric cost (or weight) associated with it — a distance, a time, a capacity. In an **unweighted** graph, all edges are equal (or you can think of every edge having weight 1).

Combining these gives four combinations: undirected unweighted, undirected weighted, directed unweighted, directed weighted.

**Cyclic vs acyclic**

A **cycle** is a path that starts and ends at the same vertex. A **Directed Acyclic Graph (DAG)** has no cycles — task dependency graphs and build systems are often DAGs.

**Connected vs disconnected**

A graph is **connected** if there's a path between every pair of vertices. A **disconnected** graph has isolated components — groups of vertices with no edges connecting them to other groups.

**Degree**

The **degree** of a vertex is the number of edges connected to it. In a directed graph, we distinguish **in-degree** (edges pointing in) from **out-degree** (edges pointing out). In a social network, in-degree is like your follower count; out-degree is how many people you follow.

## Representing graphs in code

Two standard representations:

### Adjacency list

An adjacency list stores, for each vertex, a list of its neighbors:

```javascript
// Undirected graph: 0-1, 0-2, 1-2, 2-3
const graph = {
  0: [1, 2],
  1: [0, 2],
  2: [0, 1, 3],
  3: [2],
};

// Weighted version: store {node, weight} pairs
const weightedGraph = {
  0: [
    { node: 1, weight: 4 },
    { node: 2, weight: 1 },
  ],
  1: [
    { node: 0, weight: 4 },
    { node: 2, weight: 2 },
  ],
  2: [
    { node: 0, weight: 1 },
    { node: 1, weight: 2 },
    { node: 3, weight: 5 },
  ],
  3: [{ node: 2, weight: 5 }],
};
```

Space: O(V + E) — one entry per vertex, one entry per edge. Ideal for **sparse** graphs (few edges relative to the number of possible edges).

Getting neighbors of a vertex: O(1) lookup of the list.
Checking if edge (u, v) exists: O(degree(u)) — scan the neighbor list.

### Adjacency matrix

A 2D array where `matrix[u][v] = 1` (or the edge weight) if an edge exists, 0 otherwise:

```javascript
// Same graph as above, 4 vertices
const matrix = [
  [0, 1, 1, 0], // vertex 0's connections
  [1, 0, 1, 0], // vertex 1's connections
  [1, 1, 0, 1], // vertex 2's connections
  [0, 0, 1, 0], // vertex 3's connections
];
```

Space: O(V²) — always a V×V grid. Ideal for **dense** graphs (many edges, close to V² total).
Checking if edge (u, v) exists: O(1) — just `matrix[u][v]`.
Getting all neighbors: O(V) — scan an entire row.

### When to use which

|                    | Adjacency List | Adjacency Matrix           |
| ------------------ | -------------- | -------------------------- |
| Space              | O(V + E)       | O(V²)                      |
| Check edge (u,v)   | O(degree)      | O(1)                       |
| List all neighbors | O(degree)      | O(V)                       |
| Best for           | Sparse graphs  | Dense graphs, edge lookups |

Most real-world graphs (road networks, social networks, web graphs) are sparse — adjacency lists are the default choice.

## Real-world graphs

Graphs model an enormous range of systems:

- **Social networks**: vertices = people, edges = friendships or follows
- **Road maps**: vertices = intersections, edges = roads with distance/time weights
- **Web**: vertices = pages, edges = hyperlinks (directed)
- **Dependency graphs**: vertices = packages/tasks, edges = "requires" (directed, acyclic)
- **Electrical circuits**: vertices = components, edges = wires with resistance
- **Airline routes**: vertices = airports, edges = flights with cost/duration

## What you learned

- A graph is a set of **vertices** (V) and **edges** (E) — trees are a restricted type of graph
- **Directed**: one-way edges; **undirected**: two-way edges; **weighted**: edges have costs
- **Adjacency list**: O(V+E) space, efficient for sparse graphs — the typical default
- **Adjacency matrix**: O(V²) space, O(1) edge lookup — efficient for dense graphs
- Real-world graphs model roads, social networks, web links, dependency chains, and more
