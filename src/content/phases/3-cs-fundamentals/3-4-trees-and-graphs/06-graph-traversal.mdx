## Why this matters

Tree traversal was straightforward: no cycles, clear root, guaranteed finite recursion. Graphs remove all those guarantees. Edges can form loops. There may be no single root. Multiple disconnected components may exist. Graph traversal — properly handling all of these — is the foundation for dozens of practical algorithms: finding the shortest path in a map, discovering connected users in a network, detecting circular dependencies in a build system.

## The problem with graphs

Consider trying to traverse this graph starting at vertex 0:

```
0 -- 1 -- 2
|         |
+---------+
```

A naive recursive "visit each neighbor" approach would loop forever: 0→1→2→0→1→2→...

The solution is a **visited set**: before processing a vertex, mark it as visited. Before visiting a neighbor, check if it's already been visited. This guarantees each vertex is processed exactly once, giving us O(V + E) time complexity.

## Breadth-first search (BFS)

BFS explores layer by layer — first all vertices at distance 1 from the source, then distance 2, and so on. It uses a **queue** (FIFO).

```javascript
function bfs(graph, start) {
  const visited = new Set([start]);
  const queue = [start];
  const order = [];

  while (queue.length > 0) {
    const vertex = queue.shift(); // dequeue from front
    order.push(vertex);

    for (const neighbor of graph[vertex] || []) {
      if (!visited.has(neighbor)) {
        visited.add(neighbor);
        queue.push(neighbor);
      }
    }
  }

  return order;
}
```

Key detail: add to `visited` **when you enqueue**, not when you dequeue. If you wait until dequeue, you may enqueue the same vertex multiple times from different neighbors, wasting work.

**BFS is ideal for**:

- Finding the shortest path (fewest edges) in an unweighted graph
- Finding all vertices within k hops of a source
- Level-by-level exploration

## Depth-first search (DFS)

DFS explores as far as possible down one path before backtracking. It uses a **stack** (LIFO) — either the call stack via recursion, or an explicit stack.

**Recursive DFS**:

```javascript
function dfs(graph, start) {
  const visited = new Set();
  const order = [];

  function explore(vertex) {
    visited.add(vertex);
    order.push(vertex);

    for (const neighbor of graph[vertex] || []) {
      if (!visited.has(neighbor)) {
        explore(neighbor);
      }
    }
  }

  explore(start);
  return order;
}
```

**Iterative DFS** (using an explicit stack):

```javascript
function dfsIterative(graph, start) {
  const visited = new Set();
  const stack = [start];
  const order = [];

  while (stack.length > 0) {
    const vertex = stack.pop(); // pop from top
    if (visited.has(vertex)) continue;
    visited.add(vertex);
    order.push(vertex);

    for (const neighbor of graph[vertex] || []) {
      if (!visited.has(neighbor)) {
        stack.push(neighbor);
      }
    }
  }

  return order;
}
```

Note: iterative DFS may visit neighbors in reverse order compared to recursive DFS, because the stack reverses the order.

**DFS is ideal for**:

- Detecting cycles
- Topological sort (lesson 8)
- Finding connected components
- Maze solving (exploring paths to their end before backtracking)

## Connected components

A **connected component** is a maximal set of vertices such that there's a path between every pair. To count components:

```javascript
function countComponents(graph) {
  const allVertices = Object.keys(graph).map(Number);
  const visited = new Set();
  let components = 0;

  for (const vertex of allVertices) {
    if (!visited.has(vertex)) {
      // Start a new BFS/DFS from this unvisited vertex — it's a new component
      components++;
      const queue = [vertex];
      visited.add(vertex);

      while (queue.length > 0) {
        const v = queue.shift();
        for (const neighbor of graph[v] || []) {
          if (!visited.has(neighbor)) {
            visited.add(neighbor);
            queue.push(neighbor);
          }
        }
      }
    }
  }

  return components;
}
```

The logic: any time you find an unvisited vertex, it starts a new component. Then BFS/DFS from that vertex marks the entire component as visited, so you won't count any of its members again.

## Time and space complexity

Both BFS and DFS:

- **Time**: O(V + E) — each vertex is visited once, each edge is examined once (or twice for undirected graphs)
- **Space**: O(V) for the visited set; additionally O(V) for the queue/stack in the worst case

## What you learned

- Graph traversal requires a **visited set** to handle cycles — trees don't need this
- **BFS** (queue): explores level by level, finds shortest paths in unweighted graphs
- **DFS** (stack or recursion): explores depth-first, useful for cycle detection and topological sort
- Both run in O(V + E) time and O(V) space
- **Connected components**: count how many times you start a fresh BFS/DFS from an unvisited vertex
