## Why this matters

"This algorithm is O(n²)" tells an incomplete story. An algorithm might be O(1) on a lucky input and O(n²) on a pathological one. When you guarantee an SLA to a customer, or when you are building a real-time system that cannot afford spikes, you need to know the _full distribution_ of behavior — not just the average. Understanding case analysis is what separates engineers who can reason about guarantees from those who can only describe happy paths.

## The three cases

Every algorithm has three performance profiles depending on the input it receives:

**Best case** — the most favorable possible input. The algorithm finishes as quickly as it can. This is almost never what you should design or plan around, because you rarely control what input you receive.

**Worst case** — the most adversarial possible input. The algorithm takes as long as it possibly can. This is the number that matters for correctness guarantees, because if the worst case is acceptable, every other input is also acceptable.

**Average case** — the expected performance across the distribution of realistic inputs. This is what matters for benchmarks and throughput calculations. It requires assumptions about the input distribution, which is why it is harder to reason about than worst case.

## Linear search: a clear example

Linear search scans an array until it finds the target or exhausts the array.

```javascript
function linearSearch(arr, target) {
  for (let i = 0; i < arr.length; i++) {
    if (arr[i] === target) return i;
  }
  return -1;
}
```

- **Best case:** Target is at index 0. One comparison. O(1).
- **Worst case:** Target is not in the array. Every element is checked. n comparisons. O(n).
- **Average case:** Target is equally likely to be at any position. On average it is at position n/2. O(n/2) = O(n).

All three cases tell a different story. In practice, when we say "linear search is O(n)," we mean its worst case is O(n), which also happens to equal its average case (modulo constants).

## Quick sort: when worst case is dangerous

Quick sort's average case is O(n log n) — as fast as merge sort. But its worst case is O(n²), and it can be triggered by a specific input pattern.

The pivot selection determines performance. A bad pivot — like always picking the first element of an already-sorted array — produces a degenerate partition:

```
Array: [1, 2, 3, 4, 5, 6, 7, 8]
Pivot: 1 → partition: [] and [2,3,4,5,6,7,8]
Next pivot: 2 → partition: [] and [3,4,5,6,7,8]
... n-1 pivots total, each doing n-k work
```

This reduces to O(n²) — worse than merge sort's guaranteed O(n log n). Real quick sort implementations use random pivot selection or the "median of three" strategy to make the worst case statistically improbable. But the theoretical worst case remains O(n²).

This distinction matters when you need guarantees. If you are sorting user-submitted data in a security-sensitive application, an adversary who knows your pivot strategy can deliberately submit sorted arrays to trigger O(n²) performance and deny service. Merge sort's O(n log n) guarantee is immune to this attack.

## When worst case matters vs. when average case matters

**Prefer worst-case reasoning when:**

- You are building real-time systems (games, trading systems, medical devices) where latency spikes are unacceptable.
- You offer SLA guarantees (response time < 100ms 99th percentile).
- Your input can be adversarial or untrusted.
- Failure is catastrophic (safety systems, payment processing).

**Prefer average-case reasoning when:**

- You are optimizing for throughput and the input is drawn from a known, benign distribution.
- The algorithm is used internally with controlled inputs.
- You are comparing algorithms empirically and want to predict real-world benchmark behavior.

A useful heuristic: design for worst case correctness, optimize for average case performance.

## Amortized analysis preview

Sometimes a single operation can be expensive in the worst case but cheap on average _over a sequence of operations_. This is called **amortized analysis** and it will be covered fully in Lesson 7. The preview: dynamic array `push` is O(n) in the worst case (when it triggers a resize) but O(1) amortized over n pushes. The rare expensive operation is "paid for" by the many cheap ones. This is why you can treat dynamic array push as O(1) in practice.

## Try it

<SandboxExercise language="javascript">
{`// Run linear search on multiple targets and measure actual operation counts.
// This shows real best, worst, and average case behavior.

function linearSearch(arr, target) {
let ops = 0;
for (let i = 0; i < arr.length; i++) {
ops++;
if (arr[i] === target) return { index: i, ops };
}
return { index: -1, ops };
}

function linearSearchStats(arr, targets) {
const results = targets.map(t => linearSearch(arr, t).ops);
const best = Math.min(...results);
const worst = Math.max(...results);
const average = results.reduce((a, b) => a + b, 0) / results.length;
return { best, worst, average: average.toFixed(1), results };
}

// arr = integers 1..100
const arr = Array.from({ length: 100 }, (\_, i) => i + 1);

// Targets: 1 (best), 50 (mid), 100 (near worst), 101 (not found = worst)
const targets = [1, 50, 100, 101];
const stats = linearSearchStats(arr, targets);

console.log("Array size:", arr.length);
console.log("Targets searched:", targets);
console.log("Ops per target:", stats.results);
console.log("Best case ops:", stats.best, "— expected ~1");
console.log("Worst case ops:", stats.worst, "— expected ~100 (n)");
console.log("Average ops:", stats.average, "— expected ~50 (n/2)");`}

## What you learned

- Best case is the fastest input, worst case is the slowest, average case is expected over realistic inputs.
- Worst case is the right anchor for correctness guarantees and adversarial environments.
- Average case drives real-world performance intuition and benchmark comparisons.
- Quick sort is O(n log n) average but O(n²) worst — a real risk when input can be adversarial.
- Amortized analysis (Lesson 7) handles cases where occasional expensive operations are offset by many cheap ones.
