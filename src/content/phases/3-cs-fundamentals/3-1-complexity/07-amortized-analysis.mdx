## Why this matters

If you only look at the worst-case cost of a single operation, you will sometimes reach a conclusion that contradicts observed reality. `Array.push` in JavaScript is O(n) in the worst case — yet every developer treats it as O(1), and benchmarks confirm that pushing n items to an array takes O(n) total. The reason is amortized analysis: reasoning about the _average cost per operation over a sequence_, rather than the worst-case cost of a single operation. This technique appears everywhere in data structures — hash tables, queues built from two stacks, splay trees, and more.

## The dynamic array problem

JavaScript arrays (and Python lists, Java ArrayLists, C++ vectors) are **dynamic arrays**. They store elements contiguously in memory for fast index access, but they need to grow when full. Growing requires:

1. Allocating a new, larger block of memory
2. Copying every existing element into the new block
3. Deallocating the old block

This copy operation costs O(n) — it touches every element. A function that triggers a resize has a worst-case cost of O(n). So is `push` O(n)?

The answer is: worst case yes, amortized no. The key is the **doubling strategy**: when the array needs to grow, it doubles its capacity. This seemingly minor detail changes everything.

## Why doubling makes resizes rare

Suppose you start with capacity 1 and double each time you overflow:

```
Push 1:  capacity 1 → full, no resize.      [1]
Push 2:  capacity 1 → overflow, resize to 2. Copy 1 item. [1, 2]
Push 3:  capacity 2 → overflow, resize to 4. Copy 2 items. [1, 2, 3, _]
Push 4:  capacity 4 → not full.             [1, 2, 3, 4]
Push 5:  capacity 4 → overflow, resize to 8. Copy 4 items. [1, 2, 3, 4, 5, _, _, _]
Push 6-8: capacity 8 → not full.
Push 9:  capacity 8 → overflow, resize to 16. Copy 8 items.
```

Notice the pattern: resizes happen at sizes 1, 2, 4, 8, 16, 32 … The copy costs at each resize are 1, 2, 4, 8, 16, 32 … For n total pushes, the total copy cost is:

```
1 + 2 + 4 + 8 + ... + n/2 + n ≈ 2n
```

This is a geometric series that sums to at most 2n. So across n push operations, the total copy work is O(n). Dividing by n operations: **O(n) / n = O(1) per push, amortized.**

## The banker's analogy

One way to think about amortized analysis is the **credit method** (also called the banker's method). Assign each operation a "credit budget":

- Each push costs 1 unit of actual work but we charge it 3 credits.
- 1 credit pays for the push itself.
- 2 credits are saved in a "bank."
- When a resize happens, it copies n elements — but the bank has been accumulating since the last resize, and there are enough saved credits to pay for all n copies.

Because of the doubling strategy, by the time you resize a capacity-n array, you have pushed roughly n/2 items since the last resize, each contributing 2 credits to the bank — a total of n credits, exactly enough to pay for the n-element copy.

This mental model makes the key point tangible: **cheap operations subsidize the rare expensive operations**. No operation is ever in debt, and the total bill is always O(n).

## Amortized ≠ worst case

This is an important distinction. Amortized O(1) means the _average over a sequence_ is O(1). A single push can still be O(n) in the worst case. This matters when:

- **Latency requirements are strict:** If a specific push must complete in < 1ms, and that push happens to trigger a resize that copies 10 million elements, the amortized guarantee does not help you. Real-time systems sometimes pre-allocate to avoid this.
- **Reporting metrics:** "Operations per second" benefits from amortized analysis. "Worst-case response time" does not.

## Other examples of amortized O(1)

**Hash table insertion** — when load factor exceeds a threshold (e.g., 0.75), the hash table rehashes: it allocates a new, larger table and reinserts all elements. The amortized cost per insert is O(1) by the same geometric series argument as dynamic arrays.

**StringBuilder in Java / string concatenation with a buffer** — appending to a string by building a character array that doubles in capacity amortizes the copy costs. Naive string concatenation (`str += chunk`) in a loop is O(n²) because it copies the whole string each iteration.

**Queue from two stacks** — each element is pushed to the front stack once and transferred to the back stack at most once. Total work across n operations is O(n), giving O(1) amortized per operation — even though individual pops can trigger an O(n) transfer.

## What you learned

- Dynamic arrays double capacity on resize so that resizes become exponentially rarer as the array grows.
- Total copy operations across n pushes form a geometric series summing to at most 2n — giving O(1) amortized per push.
- The credit method explains this intuitively: each cheap push saves credits that pay for the next expensive resize.
- Amortized O(1) is different from worst-case O(1): a single push can still be O(n) when it triggers a resize.
- Hash table rehashing and queue-from-two-stacks use the same amortized reasoning.
