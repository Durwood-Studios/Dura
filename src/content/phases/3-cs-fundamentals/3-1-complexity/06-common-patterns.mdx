## Why this matters

Most interview problems and most real-world optimization opportunities involve a small set of recurring patterns. If you can look at a problem and immediately say "this is a sliding window — it is O(n)," you skip the naive O(n²) solution and write the right code on the first attempt. Pattern recognition is the bridge between understanding Big O abstractly and applying it to new code.

## Two pointers: always O(n)

The two-pointer pattern uses two indices that move through a data structure. The key insight is that **each pointer can only move forward** (or backward) at most n steps. Since the pointers together travel at most 2n steps, the total work is O(n) regardless of how the pointers move relative to each other.

```javascript
// Find a pair that sums to target in a sorted array
function twoSumSorted(arr, target) {
  let left = 0,
    right = arr.length - 1;
  while (left < right) {
    const sum = arr[left] + arr[right];
    if (sum === target) return [left, right];
    if (sum < target)
      left++; // move left pointer forward
    else right--; // move right pointer backward
  }
  return null;
}
// left travels at most n steps, right travels at most n steps
// Total: O(n) — not O(n²)
```

Compare to the naive approach: two nested loops checking every pair is O(n²). The two-pointer pattern achieves O(n) by exploiting the sorted order. The pointers never go backward, so the work per step is bounded.

**Why it works:** The total number of pointer increments/decrements is bounded by n for each pointer. Two pointers means at most 2n operations — which is O(n).

## Sliding window: also O(n)

The sliding window pattern maintains a contiguous subarray (the "window") and expands or contracts it based on a condition. The crucial observation: **each element enters the window at most once and leaves the window at most once**. Total operations = 2n = O(n).

```javascript
// Maximum sum of any k consecutive elements
function maxSumWindow(arr, k) {
  // Build initial window
  let windowSum = 0;
  for (let i = 0; i < k; i++) windowSum += arr[i];

  let maxSum = windowSum;

  // Slide: add right element, remove left element
  for (let i = k; i < arr.length; i++) {
    windowSum += arr[i]; // new element enters window
    windowSum -= arr[i - k]; // old element leaves window
    maxSum = Math.max(maxSum, windowSum);
  }

  return maxSum;
}
// Each element: added once (enters) + removed once (leaves) = 2n ops = O(n)
```

The naive approach for "max sum of any k elements" uses a nested loop: for each starting position (n options), sum k elements. That is O(n × k). The sliding window collapses that to O(n) by reusing the previous window sum instead of recomputing it.

Variable-size sliding windows (where the window size changes based on a condition) still achieve O(n) by the same argument: total number of expand + contract operations = 2n.

## Divide and conquer: O(log n) levels

Divide and conquer algorithms split the input in half (or by some constant fraction), solve each half, and combine the results. The number of levels of splitting is log₂(n), because you can only halve n a logarithmic number of times before reaching a base case of size 1.

The total complexity depends on how much work you do **per level**:

| Algorithm       | Work per level              | Levels   | Total                                |
| --------------- | --------------------------- | -------- | ------------------------------------ |
| Binary search   | O(1) — check one midpoint   | O(log n) | O(log n)                             |
| Merge sort      | O(n) — merge all n elements | O(log n) | O(n log n)                           |
| Naive all-pairs | O(n²) per level             | O(log n) | O(n² log n) — worse than brute force |

This is the intuition behind the **Master Theorem**: the total complexity of a divide-and-conquer algorithm is approximately `levels × work_per_level`. Binary search does O(1) work per level (check midpoint, discard half), giving O(log n). Merge sort does O(n) work per level (merge all elements at that level), giving O(n log n).

```javascript
// Merge sort — O(n log n)
function mergeSort(arr) {
  if (arr.length <= 1) return arr;
  const mid = Math.floor(arr.length / 2);
  const left = mergeSort(arr.slice(0, mid)); // recurse on left half
  const right = mergeSort(arr.slice(mid)); // recurse on right half
  return merge(left, right); // O(n) merge work per level
}
// log n levels × O(n) work per level = O(n log n)
```

## Recognizing patterns from code structure

You can classify most code at a glance by the structure of its loops and recursion:

- **Single linear scan** — one loop, pointer moves one step per iteration: O(n)
- **Two pointers moving toward each other** — total steps bounded by n: O(n)
- **Window that expands/contracts** — each element enters and leaves once: O(n)
- **Loop that searches by halving** — divides remaining range by 2 each time: O(log n)
- **Nested loops, both depend on n** — O(n²) unless proven otherwise
- **Recursion that halves and does O(n) merge** — O(n log n)
- **Recursion that branches twice on n-1** — O(2^n)

When you see a nested loop, your first question should be: "do both loops actually run n iterations, or is one of them bounded by something smaller?" Two pointers look like a while loop with two moving indices — not nested, so not O(n²).

## What you learned

- Two-pointer achieves O(n) because each pointer moves at most n steps total — 2n combined is still O(n).
- Sliding window achieves O(n) because each element enters and exits the window at most once each.
- Divide-and-conquer creates O(log n) levels; total complexity = work per level × log n.
- Pattern recognition — identifying the structure of loops and recursion — lets you classify complexity without counting every operation.
