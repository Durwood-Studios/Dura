## Why this matters

Reading code and immediately knowing its complexity is a fundamental engineering skill. In code review it lets you flag O(n²) bugs before they reach production. In system design it lets you choose between approaches before writing a line. You do not need to run the code — you read it, count the loops, and classify it. This lesson builds that reflex.

## The basic rules for counting

Time complexity is about counting the number of operations relative to the input size n. Three patterns cover the vast majority of code you will encounter.

**Single loop = O(n).** If you iterate through n items once, you do n operations.

```javascript
function sumArray(arr) {
  let total = 0;
  for (const item of arr) {
    // runs n times
    total += item; // O(1) work each iteration
  }
  return total;
}
// Total: O(n)
```

**Nested loops = O(n²).** Each outer iteration triggers a full inner loop. n × n = n² total operations.

```javascript
function printAllPairs(arr) {
  for (let i = 0; i < arr.length; i++) {
    // n times
    for (let j = 0; j < arr.length; j++) {
      // n times each
      console.log(arr[i], arr[j]);
    }
  }
}
// Total: O(n × n) = O(n²)
```

**Loop that halves = O(log n).** When the loop variable is divided (or multiplied) by a constant each iteration, the number of iterations is logarithmic.

```javascript
function countHalvings(n) {
  let count = 0;
  while (n > 1) {
    n = Math.floor(n / 2); // n halves each time
    count++;
  }
  return count; // ≈ log₂(n) iterations
}
// Total: O(log n)
```

## Sequential code: add, then drop lower terms

When code runs one block _after_ another, add their complexities — then drop lower-order terms:

```javascript
function twoPhaseProcess(arr) {
  // Phase 1: O(n)
  for (const item of arr) {
    doConstantWork(item);
  }

  // Phase 2: O(n²)
  for (let i = 0; i < arr.length; i++) {
    for (let j = 0; j < arr.length; j++) {
      doConstantWork(arr[i], arr[j]);
    }
  }
}
// Total: O(n) + O(n²) = O(n²)
// The n² term dominates — O(n) is irrelevant at large n
```

Contrast with **nested** code, where you multiply:

```javascript
// Sequential: two separate loops = O(n) + O(n) = O(n)
for (const a of arr) {
  process(a);
}
for (const b of arr) {
  process(b);
}

// Nested: one loop inside the other = O(n × n) = O(n²)
for (const a of arr) {
  for (const b of arr) {
    process(a, b);
  }
}
```

## Multiple inputs: O(n × m)

When a function takes two separate collections of different sizes, use distinct variables:

```javascript
function findIntersection(arrA, arrB) {
  const result = [];
  for (const a of arrA) {
    // runs |arrA| = n times
    for (const b of arrB) {
      // runs |arrB| = m times each
      if (a === b) result.push(a);
    }
  }
  return result;
}
// Total: O(n × m) — not O(n²), because n ≠ m
```

If the two collections happen to be the same size, O(n × m) simplifies to O(n²). But keep them separate until you know that.

## Recursion: count the call tree

For recursive functions, draw the call tree and count total nodes. Each node is one call doing O(1) work (excluding sub-calls).

```javascript
function factorial(n) {
  if (n <= 1) return 1;
  return n * factorial(n - 1); // one recursive call
}
```

The call tree is a straight line: `factorial(n)` → `factorial(n-1)` → … → `factorial(1)`. Total nodes = n. Complexity: O(n).

```javascript
function fib(n) {
  if (n <= 1) return n;
  return fib(n - 1) + fib(n - 2); // TWO recursive calls
}
```

Here the call tree branches twice at each node. The tree has roughly 2^n nodes. Complexity: O(2^n). This is why naive recursive Fibonacci is catastrophically slow — it recomputes the same values exponentially many times.

**Halving recursion** (like binary search) calls itself once on half the input. Tree depth is log₂(n). Complexity: O(log n).

## The dominant term rule in action

| Code structure              | Complexity | Reason                     |
| --------------------------- | ---------- | -------------------------- |
| Single loop                 | O(n)       | n iterations, O(1) per     |
| Two sequential loops        | O(n)       | O(n) + O(n) = O(2n) = O(n) |
| Loop inside a loop          | O(n²)      | n × n                      |
| Loop that halves            | O(log n)   | log₂(n) iterations         |
| Recursive: one call on n-1  | O(n)       | depth n                    |
| Recursive: two calls on n-1 | O(2^n)     | 2^n nodes                  |
| Recursive: one call on n/2  | O(log n)   | depth log n                |

## What you learned

- Single loop = O(n), nested loops = O(n²), halving loop = O(log n).
- Sequential code blocks add; nested code blocks multiply.
- The dominant term rule: drop lower-order terms and constants once you have the total.
- Recursive complexity equals the number of nodes in the call tree — branching factor and depth determine it.
