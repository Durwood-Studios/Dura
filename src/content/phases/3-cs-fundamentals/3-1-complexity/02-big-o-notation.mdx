## Why this matters

When two engineers debate whether an algorithm is "fast enough," they need shared vocabulary. Big O notation is that vocabulary — it lets you describe algorithm speed in a way that is precise, portable, and independent of hardware. A function that is O(n²) is O(n²) whether it runs on a phone from 2010 or a supercomputer from 2026.

## What Big O actually means

Big O describes the **upper bound on how an algorithm's resource usage grows** as the input size n approaches infinity. The key word is "grows." Big O is not about how long your function takes on your laptop. It is about the _shape_ of the growth curve.

When we say a function is O(n), we mean: double the input, and the work roughly doubles. When we say O(n²), we mean: double the input, and the work roughly quadruples. The growth rate is the story Big O tells.

Two simplification rules make Big O practical:

**Drop constants.** `2n + 100` is O(n), not O(2n + 100). Constants reflect hardware and implementation details, not the algorithm's fundamental behavior. If you move from one server to another that is twice as fast, all your constants halve — but O(n) stays O(n) and O(n²) stays O(n²).

**Drop lower-order terms.** `3n² + 500n + 1000` is O(n²). At large n, the n² term so thoroughly dominates the others that they become noise. At n = 1,000,000, the `3n²` term is 3 trillion. The `500n` term is 500 million — less than 0.02% of the total. Big O focuses on what dominates.

## The complexity classes, from fast to slow

These are the classes you will encounter daily, in order from best to worst:

**O(1) — Constant.** The operation takes the same time regardless of input size. Reading from a hash map by key. Accessing an array by index. Pushing to a stack. These do the same amount of work whether n is 1 or 1,000,000,000.

```javascript
function getFirst(arr) {
  return arr[0]; // one operation, always
}
```

**O(log n) — Logarithmic.** The work grows by one unit each time the input _doubles_. Binary search on a sorted array. Tree traversal in a balanced BST. At n = 1,000,000,000, binary search needs roughly 30 steps.

```javascript
function binarySearch(sortedArr, target) {
  let lo = 0,
    hi = sortedArr.length - 1;
  while (lo <= hi) {
    const mid = Math.floor((lo + hi) / 2);
    if (sortedArr[mid] === target) return mid;
    if (sortedArr[mid] < target) lo = mid + 1;
    else hi = mid - 1;
  }
  return -1; // ~log₂(n) iterations
}
```

**O(n) — Linear.** The work grows proportionally with input. Scanning every element once. Finding a value in an unsorted list. Copying an array. The most common complexity in well-written code.

**O(n log n) — Linearithmic.** The sweet spot for comparison-based sorting. Merge sort, heap sort, and most efficient sorting algorithms live here. It is slightly worse than linear but vastly better than quadratic.

**O(n²) — Quadratic.** Two nested loops, each running n times. Bubble sort, insertion sort, naive duplicate detection. Manageable at small n, dangerous at large n.

```javascript
function hasDuplicate(arr) {
  for (let i = 0; i < arr.length; i++) {
    for (let j = i + 1; j < arr.length; j++) {
      if (arr[i] === arr[j]) return true; // O(n²) comparisons
    }
  }
  return false;
}
```

**O(2^n) — Exponential.** Grows by doubling with each added input element. Naive recursive Fibonacci. Generating all subsets of a set. Becomes unusable quickly — at n = 40, 2^40 is over a trillion.

**O(n!) — Factorial.** Generating all permutations of n items. At n = 20, 20! is over 2 quintillion. Only practical for tiny n.

## How to read Big O expressions

`f(n) = O(g(n))` is read as "f of n is big-O of g of n." It means f grows no faster than g. More precisely: there exist constants C and n₀ such that `f(n) ≤ C × g(n)` for all `n > n₀`. In practice: for large enough n, g captures the dominant growth behavior of f.

You will see Big O used in three contexts:

- **Describing an algorithm:** "merge sort is O(n log n)"
- **Describing data structure operations:** "hash map lookup is O(1) average"
- **Analyzing code you wrote:** "this function is O(n²) — can we do better?"

## What you learned

- Big O describes the growth rate of resource usage as n increases, not the absolute speed.
- Drop constants and lower-order terms — only the dominant term matters for large n.
- The major classes in order: O(1), O(log n), O(n), O(n log n), O(n²), O(2^n), O(n!).
- `f(n) = O(g(n))` means f grows no faster than g — g is an upper bound on f's growth.
