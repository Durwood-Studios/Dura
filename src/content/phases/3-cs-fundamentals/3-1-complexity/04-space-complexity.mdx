## Why this matters

Memory is finite. A sorting algorithm that is blazingly fast but requires a 64 GB array to sort 1 million integers is useless on a mobile device. In embedded systems, game engines, and database internals, space complexity is often the primary constraint — not time. Understanding space complexity lets you make informed tradeoffs and avoid out-of-memory crashes in production.

## What auxiliary space means

Space complexity measures the **extra memory** an algorithm needs beyond the input itself. This is called **auxiliary space** — the additional allocations the algorithm makes as it runs.

The input is not counted because it must exist regardless of which algorithm you choose. Two algorithms that both receive the same 1 million items start at the same baseline. The question is: how much extra space does each allocate?

**O(1) auxiliary space** — the algorithm uses a fixed number of extra variables regardless of input size:

```javascript
function sum(arr) {
  let total = 0; // one variable, regardless of arr.length
  for (const item of arr) total += item;
  return total;
}
// Space: O(1) — total is the only allocation
```

**O(n) auxiliary space** — the algorithm allocates memory that grows linearly with input:

```javascript
function doubled(arr) {
  const result = []; // grows to arr.length elements
  for (const item of arr) result.push(item * 2);
  return result;
}
// Space: O(n) — result array is n elements
```

**O(n²) auxiliary space** — a 2D structure where both dimensions scale with n:

```javascript
function buildGrid(n) {
  const grid = [];
  for (let i = 0; i < n; i++) {
    grid.push(new Array(n).fill(0)); // n × n cells
  }
  return grid;
}
// Space: O(n²) — n rows, each with n columns
```

## Recursion and the call stack

Every recursive function call takes up space on the **call stack** — a region of memory that tracks active function calls. When a function calls itself, the current frame (its local variables and return address) stays on the stack until the recursive call returns.

This means recursion depth directly equals auxiliary space:

```javascript
function recursiveSum(arr, i = 0) {
  if (i === arr.length) return 0;
  return arr[i] + recursiveSum(arr, i + 1); // n frames deep
}
// Time: O(n), Space: O(n) — n stack frames at peak
```

A function that recursively halves its input has O(log n) stack depth:

```javascript
function binarySearch(arr, target, lo = 0, hi = arr.length - 1) {
  if (lo > hi) return -1;
  const mid = Math.floor((lo + hi) / 2);
  if (arr[mid] === target) return mid;
  if (arr[mid] < target) return binarySearch(arr, target, mid + 1, hi);
  return binarySearch(arr, target, lo, mid - 1);
}
// Time: O(log n), Space: O(log n) — log n stack frames
```

A stack overflow occurs when recursion depth exceeds the engine's stack limit — typically a few thousand to tens of thousands of frames in JavaScript. Deep recursion on large arrays is a real production risk.

## In-place algorithms

An **in-place** algorithm modifies the input directly instead of allocating a separate output structure. In-place typically means O(1) auxiliary space because no extra array is created.

```javascript
// In-place reverse — O(1) auxiliary space
function reverseInPlace(arr) {
  let left = 0,
    right = arr.length - 1;
  while (left < right) {
    [arr[left], arr[right]] = [arr[right], arr[left]]; // swap
    left++;
    right--;
  }
}

// Copy reverse — O(n) auxiliary space
function reverseCopy(arr) {
  return arr.slice().reverse(); // allocates a new array
}
```

The tradeoff: in-place operations modify the original data. If you need both the original and the reversed version, you must copy — accepting the O(n) space cost. This is the essence of the time-space tradeoff: sometimes you spend memory to preserve data or gain speed.

## The time-space tradeoff

The most famous example of trading space for time is **memoization**. The naive recursive Fibonacci has O(2^n) time and O(n) space (stack depth). With memoization, you store results you have already computed:

```javascript
function fibMemo(n, memo = {}) {
  if (n in memo) return memo[n]; // O(1) lookup
  if (n <= 1) return n;
  memo[n] = fibMemo(n - 1, memo) + fibMemo(n - 2, memo);
  return memo[n];
}
// Time: O(n) — each value computed once
// Space: O(n) — memo object + O(n) stack = O(n)
```

By spending O(n) space on the memo object, you reduce time from O(2^n) to O(n). That is an astronomical improvement. This pattern — caching results to avoid redundant computation — is called **dynamic programming** and it almost always trades O(n) or O(n²) space to eliminate exponential time.

## What you learned

- Space complexity measures auxiliary space — extra memory beyond the input itself.
- O(1) space uses only fixed variables; O(n) space grows with input; O(n²) space is a 2D structure.
- Recursive functions consume call stack space proportional to their recursion depth.
- In-place algorithms use O(1) auxiliary space by modifying the input; copy algorithms use O(n).
- Memoization is the classic time-space tradeoff: spending O(n) space to eliminate exponential recomputation.
