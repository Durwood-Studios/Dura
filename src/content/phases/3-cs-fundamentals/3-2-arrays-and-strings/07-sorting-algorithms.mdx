## Why this matters

You will rarely implement a sort from scratch in production — `Array.sort()` handles it. But knowing _why_ different sorts have different complexities builds the analytical intuition that carries over to every algorithm you write. Understanding merge sort teaches you divide-and-conquer. Understanding insertion sort teaches you that "best case" matters for nearly-sorted data. Understanding quick sort teaches you that pivot choice is everything.

## Bubble sort — O(n²) — simple but rarely useful

Bubble sort repeatedly steps through the array, comparing adjacent elements and swapping them if out of order. Each full pass "bubbles" the largest unsorted element to its correct position.

```javascript
function bubbleSort(arr) {
  const a = [...arr]; // copy — don't mutate input
  let comparisons = 0;
  for (let i = 0; i < a.length - 1; i++) {
    for (let j = 0; j < a.length - 1 - i; j++) {
      comparisons++;
      if (a[j] > a[j + 1]) {
        [a[j], a[j + 1]] = [a[j + 1], a[j]]; // swap
      }
    }
  }
  return { sorted: a, comparisons };
}
```

For n elements: outer loop runs n-1 times, inner loop runs up to n-1 times. Total: **O(n²)** comparisons. There is no redeeming best-case optimization unless you add an early-exit flag.

Use bubble sort: never in production. Study it to understand what makes a sort slow.

## Insertion sort — O(n²) worst, O(n) best

Insertion sort builds a sorted section from the left. Each new element is inserted into its correct position by shifting sorted elements right.

```javascript
function insertionSort(arr) {
  const a = [...arr];
  let comparisons = 0;
  for (let i = 1; i < a.length; i++) {
    const key = a[i];
    let j = i - 1;
    while (j >= 0 && (++comparisons, a[j] > key)) {
      a[j + 1] = a[j]; // shift right
      j--;
    }
    a[j + 1] = key;
  }
  return { sorted: a, comparisons };
}
```

**Worst case (reverse sorted):** Every element shifts past every sorted element — O(n²).

**Best case (already sorted):** The inner while never executes — each element needs zero shifts. O(n).

**Nearly sorted:** If each element is at most k positions from its final place, insertion sort is O(n × k). For k = 5, that is 5× linear — practically O(n).

Use insertion sort for: arrays with fewer than ~20 elements, nearly-sorted arrays, online sorting (sorting as elements arrive one at a time).

## Merge sort — O(n log n) always, stable

Merge sort is the first **divide-and-conquer** sort. It recursively splits the array in half, sorts each half, then **merges** the two sorted halves:

```javascript
function mergeSort(arr) {
  let comparisons = 0;

  function merge(left, right) {
    const result = [];
    let i = 0,
      j = 0;
    while (i < left.length && j < right.length) {
      comparisons++;
      if (left[i] <= right[j]) result.push(left[i++]);
      else result.push(right[j++]);
    }
    return [...result, ...left.slice(i), ...right.slice(j)];
  }

  function sort(a) {
    if (a.length <= 1) return a;
    const mid = Math.floor(a.length / 2);
    return merge(sort(a.slice(0, mid)), sort(a.slice(mid)));
  }

  const sorted = sort(arr);
  return { sorted, comparisons };
}
```

Why O(n log n)? The split creates log₂(n) levels of recursion. The merge at each level processes every element exactly once — O(n) work per level. Total: **O(n log n)** regardless of input order.

Merge sort is **stable**: elements with equal values appear in the same relative order as in the input. It also uses O(n) auxiliary space for the temporary merge arrays.

Use merge sort for: general-purpose sorting when stability is required, sorting linked lists (O(1) space merge on linked lists), guaranteed O(n log n) behavior.

## Quick sort — O(n log n) average, O(n²) worst

Quick sort also uses divide-and-conquer but **partitions in place** around a pivot element. Elements smaller than the pivot go left; elements larger go right. Then recursively sort each side.

```javascript
function quickSort(arr) {
  const a = [...arr];
  let comparisons = 0;

  function partition(lo, hi) {
    const pivot = a[hi]; // simple last-element pivot
    let i = lo - 1;
    for (let j = lo; j < hi; j++) {
      comparisons++;
      if (a[j] <= pivot) {
        i++;
        [a[i], a[j]] = [a[j], a[i]];
      }
    }
    [a[i + 1], a[hi]] = [a[hi], a[i + 1]];
    return i + 1; // pivot's final position
  }

  function sort(lo, hi) {
    if (lo < hi) {
      const p = partition(lo, hi);
      sort(lo, p - 1);
      sort(p + 1, hi);
    }
  }

  sort(0, a.length - 1);
  return { sorted: a, comparisons };
}
```

**Average case:** A good pivot splits the array roughly in half each time — log₂(n) levels of O(n) partitioning — **O(n log n)**.

**Worst case:** If the pivot is always the min or max (happens with sorted input and a naïve last-element pivot) — O(n) levels, each O(n) — **O(n²)**.

Fix: choose a random pivot or use the "median of three" strategy (pick the median of first, middle, last elements).

Quick sort is **not stable** and uses O(log n) stack space for recursion (O(n) worst case).

Use quick sort for: general-purpose in-place sorting where average performance matters and stability is not required. In practice, a well-implemented quick sort often outperforms merge sort by a constant factor due to better cache locality.

## Timsort — what JavaScript actually uses

V8's `Array.sort()` uses **Timsort**, a hybrid of merge sort and insertion sort developed at Google. It:

1. Scans the input for existing sorted "runs" (subsequences already in order).
2. Uses insertion sort to extend short runs to a minimum size (~32 or 64 elements).
3. Merges runs using a merge sort strategy.

On random data, Timsort is O(n log n). On nearly-sorted data, it approaches O(n) — exploiting the same insight as insertion sort's best case. Timsort is also **stable** as of V8 7.0 (Node.js 11+).

## Complexity comparison

| Algorithm      | Best       | Average    | Worst      | Space    | Stable |
| -------------- | ---------- | ---------- | ---------- | -------- | ------ |
| Bubble sort    | O(n)†      | O(n²)      | O(n²)      | O(1)     | Yes    |
| Insertion sort | O(n)       | O(n²)      | O(n²)      | O(1)     | Yes    |
| Merge sort     | O(n log n) | O(n log n) | O(n log n) | O(n)     | Yes    |
| Quick sort     | O(n log n) | O(n log n) | O(n²)      | O(log n) | No     |
| Timsort        | O(n)       | O(n log n) | O(n log n) | O(n)     | Yes    |

†With an early-exit flag when no swaps occur.

## What you learned

- **Bubble sort** is O(n²) with no practical advantages — its value is pedagogical, not practical.
- **Insertion sort** is O(n²) worst case but **O(n) best case** for already-sorted data — ideal for small or nearly-sorted inputs.
- **Merge sort** is **O(n log n) in all cases**, stable, and uses O(n) auxiliary space — ideal when consistency and stability matter.
- **Quick sort** is O(n log n) average with O(log n) space, but degrades to O(n²) on sorted input with naïve pivots — randomize the pivot to avoid this.
- **Timsort** (used by `Array.sort()`) is a hybrid that exploits existing sorted runs, achieving O(n) on nearly-sorted data and O(n log n) in general.
