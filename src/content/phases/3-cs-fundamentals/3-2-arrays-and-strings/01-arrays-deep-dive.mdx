## Why this matters

You use arrays every day, but do you know why `arr[999]` is just as fast as `arr[0]`? Understanding memory layout turns arrays from a black box into a predictable tool. When you know _why_ `unshift` is slow and `push` is fast, you stop reaching for the wrong operation.

## Memory layout: why indexing is O(1)

When a JavaScript engine allocates an array of numbers, it places the elements side by side in memory — **contiguous bytes**. Think of it like a row of numbered post-office boxes, each the same size.

```javascript
// Conceptually, an array of 4-byte integers at base address 1000:
// Index 0 → address 1000
// Index 1 → address 1004
// Index 2 → address 1008
// Index 3 → address 1012
```

To read element at index `i`, the CPU computes:

```
address = base_address + (i × element_size)
```

One multiplication, one memory read. No scanning. No searching. That is why `arr[i]` is **O(1)** regardless of array length or which index you access.

## Why push is O(1) amortized — dynamic resizing

JavaScript arrays are **dynamic arrays**: they grow automatically. But memory does not teleport — the engine cannot simply extend an existing allocation if the next memory address is taken.

The trick is **amortized resizing**: the engine over-allocates, typically doubling capacity when full.

```javascript
// Rough mental model of what the engine does on push:
// if length < capacity:
//   write element at arr[length], length++   → O(1)
// else:
//   allocate new block of size capacity * 2  → O(n)
//   copy all elements                        → O(n)
//   write new element                        → O(1)
//   free old block
```

Doubling sounds expensive, but it happens so rarely that the **average cost per push across n operations is O(1)**. If you push 1000 elements, resizing happens at capacities 1, 2, 4, 8, 16 ... 512 — a total of ~2000 copy operations for 1000 pushes. That is O(1) per push on average.

## Why unshift and mid-array insertion are O(n)

Contiguous memory cuts both ways. If you insert at index 0, every existing element must slide right to make room.

```javascript
// Inserting at the front of [10, 20, 30]:
// Step 1: slide 30 → index 3
// Step 2: slide 20 → index 2
// Step 3: slide 10 → index 1
// Step 4: write new value at index 0
// 3 slides for 3 elements → O(n) shifts
```

Deletion from the middle is the mirror image: every element to the right slides left to fill the gap.

```javascript
const arr = [10, 20, 30, 40, 50];
arr.splice(2, 1); // deletes 30 — slides 40 and 50 left → 2 shifts
```

For an array of 1,000,000 elements, `arr.unshift(x)` requires 1,000,000 shifts. That is why you should push to the back when possible, and reverse or process from the end when you need front-insertion behavior.

## Sparse vs dense arrays in V8

JavaScript arrays are heterogeneous — they can hold strings, numbers, and objects simultaneously. V8 (Node.js and Chrome's engine) optimizes **dense, homogeneous** arrays into actual contiguous memory similar to C arrays:

```javascript
// Dense integer array — V8 uses SMI (small integer) backing store
const dense = [1, 2, 3, 4, 5]; // optimized

// Sparse array — V8 falls back to a hash map internally
const sparse = [];
sparse[1000] = 42; // only one element, index 1000

// Mixed types — V8 de-optimizes to a generic element array
const mixed = [1, "two", { three: 3 }];
```

Dense arrays get cache-friendly memory access. Sparse arrays pay a hash-map lookup cost. You rarely need to think about this, but it is why creating holes in arrays (`delete arr[i]`) is a performance antipattern.

## Cache locality: the hidden superpower

Modern CPUs load memory in **cache lines** (~64 bytes at once). When you read `arr[0]`, the CPU also loads `arr[1]` through ~`arr[15]` (for 4-byte integers) into cache. Iterating sequentially through an array is fast because each element is probably already in cache.

A linked list stores each node at a random memory address. Traversing it triggers a cache miss on every node. For the same 1000-element iteration, arrays can be **5–20× faster** than linked lists in practice — even though both are theoretically O(n).

```javascript
// Array iteration — sequential addresses, cache-friendly
for (let i = 0; i < arr.length; i++) {
  /* ... */
}

// Linked list traversal — random pointer jumps, cache misses everywhere
let node = head;
while (node !== null) {
  node = node.next;
}
```

Cache locality is why "arrays are simple" is not a limitation — it is a feature.

this.\_data = [];

this.opCount++;

}

// O(n): must shift elements to the right to open a slot
insertAt(i, val) {
for (let j = this.\_data.length; j > i; j--) {
this.\_data[j] = this.\_data[j - 1];
this.opCount++; // count each shift
}
this.\_data[i] = val;
this.opCount++;
}

// O(n): must shift elements left to close the gap
deleteAt(i) {
for (let j = i; j < this.\_data.length - 1; j++) {
this.\_data[j] = this.\_data[j + 1];
this.opCount++; // count each shift
}
this.\_data.pop();
this.opCount++;
}

// O(1) amortized: append to end, no shifting needed
push(val) {
this.\_data.push(val);
this.opCount++;
}

get length() { return this.\_data.length; }
}

// --- Test it ---
const arr = new SimulatedArray();
for (let i = 0; i < 10; i++) arr.push(i \* 10);

arr.get(5);

## What you learned

- Array elements are stored in **contiguous memory**, enabling O(1) index access via address arithmetic.
- `push` is O(1) **amortized** — occasional O(n) resizing copies are rare enough that average cost stays constant.
- `unshift`, `insertAt(0)`, and mid-array deletions are **O(n)** because elements must be shifted to maintain contiguity.
- V8 optimizes **dense, homogeneous** arrays into true contiguous memory; sparse or mixed-type arrays fall back to hash maps.
- **Cache locality** makes arrays 5–20× faster than linked lists in practice for sequential traversal, even though both are O(n).
