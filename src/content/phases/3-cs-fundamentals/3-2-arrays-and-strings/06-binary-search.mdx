## Why this matters

Binary search is one of the most consequential algorithms you will ever learn. It reduces searching a sorted million-element array from 1,000,000 comparisons to just 20. Beyond lookup, the same principle — "eliminate half the candidates" — applies to countless problems: finding where a function changes behavior, locating the first valid configuration, debugging a broken build in a commit history. It is O(log n) thinking applied universally.

## The algorithm

```javascript
function binarySearch(arr, target) {
  let left = 0;
  let right = arr.length - 1;

  while (left <= right) {
    const mid = left + Math.floor((right - left) / 2);

    if (arr[mid] === target) return mid; // found it
    if (arr[mid] < target)
      left = mid + 1; // target is in the right half
    else right = mid - 1; // target is in the left half
  }

  return -1; // not found
}

const sorted = [2, 5, 8, 12, 16, 23, 38, 56, 72, 91];
binarySearch(sorted, 23); // 5
binarySearch(sorted, 100); // -1
```

Each iteration eliminates half the remaining search space. For 1,000,000 elements: 1,000,000 → 500,000 → 250,000 → ... → 1. That is log₂(1,000,000) ≈ 20 steps.

## The integer overflow pitfall

You may have seen binary search written as:

```javascript
const mid = Math.floor((left + right) / 2); // ← potential problem
```

In languages with fixed-size integers (Java, C++), `left + right` can overflow when both are large (e.g., near 2³¹). The safe version avoids the intermediate sum:

```javascript
const mid = left + Math.floor((right - left) / 2); // ← safe
```

JavaScript numbers are 64-bit floats, so overflow is rarely a practical concern here — but writing it the safe way is a good habit that transfers to other languages.

## Off-by-one errors: the classic trap

The two most common mistakes in binary search:

**1. Using `<` instead of `<=` in the loop condition:**

```javascript
while (left < right) { ... }  // WRONG for exact search
// When left === right, you have one candidate remaining.
// Using < skips checking it — you will miss elements at that position.

while (left <= right) { ... } // CORRECT
```

**2. Not excluding the midpoint when moving boundaries:**

```javascript
right = mid; // WRONG — if arr[mid] !== target, mid is still in range → infinite loop
right = mid - 1; // CORRECT — mid already checked and not target, exclude it

left = mid; // WRONG — same infinite loop problem
left = mid + 1; // CORRECT
```

The rule: after eliminating `arr[mid]` from consideration, set `left = mid + 1` or `right = mid - 1`. Never include a position you have already ruled out.

## Lower bound: insertion point

A **lower bound** finds the first index where `arr[i] >= target`. This answers "where would I insert target to keep the array sorted?" — and it handles duplicates gracefully.

```javascript
function lowerBound(arr, target) {
  let left = 0;
  let right = arr.length; // NOTE: right is arr.length, not arr.length - 1

  while (left < right) {
    // NOTE: strict < not <=
    const mid = left + Math.floor((right - left) / 2);
    if (arr[mid] < target) left = mid + 1;
    else right = mid; // NOTE: right = mid, not mid - 1
  }

  return left; // insertion point: all arr[i < left] < target
}

const arr = [1, 3, 3, 5, 7, 9];
lowerBound(arr, 3); // 1 — first index where arr[i] >= 3
lowerBound(arr, 4); // 3 — arr[3]=5 is first element >= 4
lowerBound(arr, 10); // 6 — past the end, safe insertion point
lowerBound(arr, 0); // 0 — insert at the beginning
```

The differences from exact search:

- `right = arr.length` (not `arr.length - 1`) — the result can be one past the last element.
- `while (left < right)` — when `left === right`, we have our answer.
- `right = mid` when `arr[mid] >= target` — mid might still be the answer, so we keep it in range.

## Binary search on the answer space

The most powerful generalization: binary search not on an array, but on a **range of possible answers**. Find the smallest `x` such that some condition `f(x)` is true.

```javascript
// Example: find the minimum speed (pages/hr) to finish a book in `hours`
// (simplified version of a classic interview problem)
function minReadingSpeed(pages, hours) {
  function canFinish(speed) {
    let totalTime = 0;
    for (const p of pages) totalTime += Math.ceil(p / speed);
    return totalTime <= hours;
  }

  let left = 1;
  let right = Math.max(...pages); // worst case: read one chapter at a time at max size

  while (left < right) {
    const mid = left + Math.floor((right - left) / 2);
    if (canFinish(mid))
      right = mid; // mid might be the answer, keep it
    else left = mid + 1; // mid is too slow, exclude it
  }

  return left;
}
```

The pattern: define a monotonic predicate (`canFinish` is false for slow speeds, true for fast), set `left` and `right` to the answer bounds, binary search for the boundary. O(log(range) × cost_of_predicate).

## Edge cases to always test

```javascript
binarySearch([], 5); // empty array → -1
binarySearch([5], 5); // single element, match → 0
binarySearch([5], 3); // single element, no match → -1
binarySearch([1, 2, 3], 0); // target smaller than all → -1
binarySearch([1, 2, 3], 10); // target larger than all → -1
binarySearch([2, 2, 2, 2], 2); // all same elements — returns one of the matches
```

Sorted input is required. Passing an unsorted array to binary search produces nonsense — it is undefined behavior in the algorithm's contract.

## What you learned

- Binary search finds a target in a sorted array in **O(log n)** by halving the search space each step.
- Midpoint computation `left + Math.floor((right - left) / 2)` avoids integer overflow in other languages — a good habit.
- Use `while (left <= right)` and `left = mid + 1` / `right = mid - 1` to avoid infinite loops and off-by-one errors.
- **Lower bound** finds the insertion point — the first index where `arr[i] >= target` — with a slight variation: `right = arr.length` and `right = mid`.
- Binary search generalizes to **answer-space search**: find the minimum value where a monotonic predicate becomes true — no array needed.
